对于在梯形图（LAD）中进行编程的工程师来说，定时器和计数器是最熟悉的工具。我们习惯于直接在 LAD 中拉出 `TON`（接通延时）或 `CTU`（加计数）块，在其上方挂载一个背景数据块（DB），即可实现时间与次数的控制。

然而，在 CPU 底层的系统固件和物理内存中，**定时器与计数器经历了从“硬件ASIC寄存器”向“面向对象数据结构”的深刻演进。**

西门子 PLC 体系中并存着两套完全不同的定时器/计数器架构：
1.  **S5 定时器/计数器（S5T, C）**：基于早期西门子 S5 系统的**硬件专用寄存器表**（如 T0~T255、C0~C255），采用古老的 16 位 BCD 码结构。
2.  **IEC 定时器/计数器（TON, TOF, TP, CTU, CTD）**：基于现代软件工程规范的**系统功能块（SFB4/SFB5/SFB3）**，采用标准的 32 位有符号 `TIME`（毫秒 DINT）或整型结构，支持完美的**多重背景（Multi-Instance）**嵌套。

本教程将带你深入到 CPU 内存分配与时钟中断微码层面，深度解密 S5 定时器的 BCD 码物理布局、IEC 定时器在多重背景下的指针（AR2）调度本质，并手把手教你如何用 STL 在 FB 内部优雅、无索引冲突地封装多路高可靠性时间控制链。

---

### 技术兼容性声明

*   **S7-300 / S7-400**：物理芯片内部集成了 256 个 S5 硬件定时器/计数器寄存器，同时通过固件（SFC）原生支持所有 IEC 定时器。
*   **S7-1500**：为了向下兼容，系统通过软件仿真器保留了对 S5T 的支持。但其原生硬件引擎已经全面转向基于 LREAL/DINT 毫秒级的现代 **IEC 定时器/计数器**。
*   **S7-1200**：**不支持 STL，亦不支持任何原生的 S5 定时器/计数器。** 但本教程第四章将专门提供符合博途最新规范的 **SCL 语言多重背景重构方案**，帮助 S7-1200 用户在项目中实现高度可复用的时间序列控制。

## 1. BCD 码的历史遗存：S5 定时器（`S5T`）与 S5 计数器

在西门子 S5 和早期 S7-300 系统中，CPU 内部有一块专属的物理 SRAM 区域，被称为**定时器/计数器状态表**。这块空间的大小直接限制了整个 PLC 项目中可使用的定时器总数（通常硬件硬性限制为 256 个，即 `T0` 到 `T255`）。

### 1.1 S5T# 时间字 16 位 BCD 码的物理布局与分辨率精度陷阱

S5 定时器（`S5T#`）的时间设定值在内存中占用 **16 位（一个字，WORD）**。这 16 位空间被硬性切割，采用了一种混合了**时间基准（Time Base）**和 **3 位 BCD 码数值** 的特殊编码：

```
+---------------------------------------------------------------------------------+
| Siemens S5T# (S5 Timer Format) - 16-Bit Physical Layout                         |
|                                                                                 |
|  Bits 15...14       Bits 13...12                  Bits 11...0                   |
|  +------------+     +------------+     +----+----+----+----+----+----+----+----+ |
|  |  Not Used  |     | Time Base  |     |      3-Digit BCD Time Value            | |
|  +------------+     +------------+     +----+----+----+----+----+----+----+----+ |
|  Always 00          00, 01, 10, 11     BCD digits: Hundreds, Tens, Ones (0-999) |
+---------------------------------------------------------------------------------+
```

#### 时间基准（Bits 12...13）硬件驱动规则：

*   **`00`**：时间基准为 **10ms**。最大可表示：$10\text{ms} \times 999 = 9.99\text{s}$。
*   **`01`**：时间基准为 **100ms**。最大可表示：$100\text{ms} \times 999 = 99.9\text{s}$。
*   **`10`**：时间基准为 **1s**。最大可表示：$1\text{s} \times 999 = 999\text{s}$ （约16分钟）。
*   **`11`**：时间基准为 **10s**。最大可表示：$10\text{s} \times 999 = 9990\text{s}$ （约2.7小时）。

#### 致命的分辨率精度大坑：
当你声明一个时间 `S5T#2h46m30s`（即 9990 秒）时，CPU 在编译时为了能用 3 位 BCD 码装下这个数，**被迫自动将时间基准调整为最大的 10s**。
这意味着，在定时器运行过程中，**其内部的时间递减和状态刷新只能以 10s 为单位进行！**
在长达近 3 小时的计时中，任何小于 10s 的时间波动都会被硬件直接过滤掉，这在要求精细时间控制的配方转换中是绝对无法接受的。

### 1.2 知识点 1 代码：传统的 S5 接通延时（`SD`）定时器启动与 RLO 示踪

下面的 STL 代码演示如何装载一个符合 BCD 格式的 S5 时间常数，并在 RLO 的驱动下启动经典硬件定时器 `T10`。

#### 完整 STL 源代码

```stl
FUNCTION "S5_Timer_Demo" : VOID
TITLE = 经典 S5 硬件定时器启动与监视
VAR_INPUT
   i_Start_Trigger : BOOL ;   // 输入：启动触发条件 (I0.0)
END_INPUT
VAR_OUTPUT
   q_Timer_Done    : BOOL ;   // 输出：延时到达输出 (Q0.0)
END_VAR
BEGIN
NETWORK
TITLE = 1. 装载时间并以接通延时(SD)模式启动 T10
      A     #i_Start_Trigger; // 1. 读取启动条件。/FC 置 1，写入 RLO
      
      // 2. 装载 S5 时间常数 2s。
      // 在编译时，2s 会自动被转换为 10ms 基准的 200 分度值，
      // 十六进制编码为 16#0200 (二进制 0000_0010_0000_0000)
      L     S5T#2s;           // 3. 将时间控制字读入 ACCU 1
      
      // 4. 以接通延时(SD，Start On-Delay)模式启动硬件定时器 T10
      // 物理动作：SD 指令会读取当前的 RLO 状态。
      // * 若 RLO 从 0 跃迁为 1，CPU 内部时钟控制器激活 T10 状态表并开始递减计数。
      // * 执行完后，/FC 被自动复位为 0。
      SD    T10;              

NETWORK
TITLE = 2. 监视 T10 输出状态
      A     T10;              // 5. 首次检查：直接读取硬件 T10 状态表的输出状态位
      =     #q_Timer_Done;    // 6. 输出。/FC 复位
END_FUNCTION
```

#### 逐行微观寄存器与状态字变化跟踪表

假设：操作员按下启动按钮（`i_Start_Trigger = 1`）。

| 步骤 | 执行指令 | ACCU 1 (32-bit hex) | 当前 `/FC` | 下一步 `/FC` | 当前 `RLO` | 下一步 `RLO` | 物理 CPU 状态字及系统定时器表动作详解 |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| **0** | (开始) | `0000_0000` | `0` | `0` | `0` | `0` | 系统完全干净。 |
| **1** | `A i_Start_T...` | `0000_0000` | `0` | `1` | `0` | `1` | **首检**：启动条件为 `1`。`STA=1` 直接复制到 `RLO=1`。`/FC` 置 `1`。 |
| **2** | `L S5T#2s` | `0000_0200` | `1` | `1` | `1` | `1` | **装载 BCD 字**：将 `16#0200` 装入 ACCU 1 的低 16 位。高 16 位填充 0。非逻辑指令，不改动 RLO。 |
| **3** | `SD T10` | `0000_0200` | `1` | `0` | `1` | `1` | **定时器硬启动**：由于当前 `RLO = 1`，时钟控制器向 T10 硬件寄存器写入 `16#0200`（时间常数），并**开启硬件 10ms 递减脉冲中断**。`/FC` 硬件自动归零。 |
| **4** | `A T10` | `0000_0200` | `0` | `1` | `1` | `0` | **读取状态表**：首检。由于 T10 的 200 个脉冲（2秒）还未递减完毕，定时器未到达。CPU 读取 T10 状态位为 `0` 写入 RLO。`/FC` 置 `1`。 |

### 1.3 工程经验：彻底防范多团队协同开发下的“S5 定时器物理索引重叠冲突”死锁

在维护或调试经典 S7-300 系统时，我们经常遇到如下不可思议的现场故障：**某一个网络的定时器（如 T20）时间还没到，竟然自己莫名其妙地直接输出了。**

*   **痛点成因：物理索引重叠冲突**。
    由于 S5 定时器使用的是绝对硬编码索引（`T0` 到 `T255`）。
    在一个由多个人协作开发的大型项目中：
    *   1 号集成商在编写“输送线 1 控制块”时，使用了 `SD T20`。
    *   2 号集成商在编写“清洗阀门控制块”时，由于缺乏全局沟通，也使用了 `SD T20`。
    当两段代码被同时下装到同一个 CPU 中运行时，由于 **T20 物理上只有一个寄存器空间**，两个完全不相关的逻辑在每一个扫描周期内高频交替改写 T20 的数值和状态，**导致 T20 的时间逻辑发生完全无规则的彻底紊乱**，极易引发机械设备碰撞损坏。
*   **黄金架构规范**：
    在新项目中，**禁止使用任何 S5 绝对定时器（T）和计数器（C）**。一律采用符合现代多重背景规范的 **IEC 定时器/计数器**。

## 2. 现代软件工程的标配：多重背景（Multi-Instance）IEC 定时器（`TON`, `TOF`, `TP`）

为了彻底解决 S5 硬件定时器索引重叠冲突的噩梦，西门子引入了完全符合 **IEC 61131-3 国际软件标准** 的 **IEC 定时器（`TON`、`TOF`、`TP`）**。

### 2.1 硬件设计内幕：为什么 IEC 定时器能够实现无限可复用？

IEC 定时器在底层实际上是西门子系统固件中内置的**系统功能块（SFB，System Function Block）**：
*   **`TON`** 对应 **`SFB4`**（接通延时）。
*   **`TOF`** 对应 **`SFB5`**（关断延时）。
*   **`TP`** 对应 **`SFB3`**（脉冲定时器）。

#### 物理存储原理与多重背景机制：
IEC 定时器不再使用任何特定的硬件寄存器区。
它的时间设定值 `PT`（Preset Time）和当前累积时间 `ET`（Elapsed Time）采用的是标准的 **32 位 DINT（毫秒级双整数，`TIME` 类型，范围高达 $\pm 24$ 天）**。
它的数据结构完全存放在**用户数据块（DB）**中。
当我们将一个 `TON` 定时器声明在 FB 的静态变量（`STAT`）区时：

```
+---------------------------------------------------------------------------------+
| Multi-Instance IEC Timer Storage (SFB4 "TON" in FB Parent DB)                   |
|                                                                                 |
| Parent Instance DB (DI Register -> DB100)                                       |
|                                                                                 |
| 0.0  +---------------------------------------------------------------------+     |
|      | Parent static variables...                                          |     |
| 10.0 +---------------------------------------------------------------------+     |
|      | s_Delay_Timer (SFB4 Multi-instance Base: AR2 = P#10.0)              |     |
|      |                                                                     |     |
|      |  * s_Delay_Timer.IN (BOOL)  Offset: P#10.0                          |     |
|      |  * s_Delay_Timer.PT (TIME)  Offset: P#12.0 (32-bit ms Presets)       |     |
|      |  * s_Delay_Timer.Q  (BOOL)  Offset: P#16.0 (Timer Done Output)       |     |
|      |  * s_Delay_Timer.ET (TIME)  Offset: P#18.0 (32-bit ms Elapsed)       |     |
|      +---------------------------------------------------------------------+     |
+---------------------------------------------------------------------------------+
```

通过这一多重背景机制：
1.  定时器数据完全被封装在 FB 的私有 DB 空间中，**不再占用任何全局硬件 `T` 索引。**
2.  **你可以把这个 FB 实例化调用上千次，在内存中会自动生成上千个彼此完全独立、绝对零索引冲突的 IEC 定时器。**

### 2.2 知识点 2 代码：高精度多阶段电机延时启动序列器（纯多重背景实现）

下面我们使用两个多重背景 `TON` 定时器（`SFB4`），在 STL FB 块中实现两台电机的顺序延时启动序列控制。

#### 完整 STL 源代码（FB40_Motor_Sequencer）

```FUNCTION_BLOCK "Motor_Sequencer"
TITLE = 高精度多重背景顺序起动机
VERSION : 0.1
VAR
   stat_Timer_1 : "TON" ;     // 多重背景 1：1号延时启动器 (SFB4)
   stat_Timer_2 : "TON" ;     // 多重背景 2：2号延时启动器 (SFB4)
   stat_M1_Active : BOOL ;    // 1号电机运行状态
   stat_M2_Active : BOOL ;    // 2号电机运行状态
END_VAR
VAR_TEMP
   t_AR2_Backup : DWORD ;     // 备份 AR2 寄存器，保护多重背景基准
END_VAR
BEGIN
NETWORK
TITLE = 1. 计算 1号电机延时启动 (延时 3s)
      TAR2  #t_AR2_Backup;    // 备份当前多重背景基准指针

      // 当调用多重背景子块时，CPU 必须通过 AR2 动态加上子实例的相对偏移量
      // 我们在 STL 中，利用标准的 CALL 语法调用多重背景实例：
      CALL  #stat_Timer_1 (
         IN := "i_Start_System",       // 启动系统总信号 (I0.2)
         PT := T#3s,                   // 设定延时 3 秒 (32-bit 毫秒: 3000)
         Q  := #stat_M1_Active         // 输出：启动1号电机
      );

NETWORK
TITLE = 2. 计算 2号电机顺控延时启动 (在 1号运行后延时 5s)
      CALL  #stat_Timer_2 (
         IN := #stat_M1_Active,        // 只有 1 号电机运行了，才开启 2 号延时
         PT := T#5s,                   // 设定延时 5 秒 (32-bit 毫秒: 5000)
         Q  := #stat_M2_Active         // 输出：启动2号电机
      );

      // 强力输出到外部通道
      A     #stat_M1_Active;
      =     "q_Motor_1";               // (Q0.1)
      
      A     #stat_M2_Active;
      =     "q_Motor_2";               // (Q0.2)

      LAR2  #t_AR2_Backup;    // 还原 AR2 寄存器
END_FUNCTION_BLOCK
```

#### 逐行剖析与多重背景时钟寻址跟踪表

假设：
父 FB 背景 DB 编号为 `DB100`。
`stat_Timer_1` 被编译器分配在父背景 DB 字节偏移量 `10.0` 处。
`stat_Timer_2` 分配在字节偏移量 `22.0` 处。
当前扫描周期启动总信号 `"i_Start_System" = 1`。

| 步骤 | 执行指令 | ACCU 1 (32-bit hex) | AR2 物理值 | DI 寄存器值 | 状态字 RLO | 寄存器级多重背景定时器跳转详解 |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| **0** | (开始) | `0` | `16#8500_0000` | `100` | `0` | 初始进入父 FB。DI背景 DB 锁死为 100，AR2 物理基准为 `0.0`。 |
| **1** | `TAR2` | `8500_0000` | `16#8500_0000` | `100` | `0` | 备份：将父块基准读入 ACCU 1。 |
| **2** | `CALL #stat_Ti_1` | `0000_0BB8` (3000ms) | `16#8500_0050` | `100` | `1` | **调用定时器 1 瞬间微码动作**：
1. CPU 自动将 `stat_Timer_1` 的相对偏移（`10.0`，十六进制 `50`）加到当前 AR2 上。**新 AR2 被临时重置为 `P#50.0`。**
2. 将输入实参（I0.2）值写入 `DB100.DBX10.0 (stat_Timer_1.IN)`。
3. 将时间常数 `3000` 写入 `DB100.DBD12 (stat_Timer_1.PT)`。
4. 跳转调用系统功能块 `SFB4`。SFB4 内部根据 `DB100` 背景区中的时钟控制字，在系统时钟脉冲中进行高精度递减。
5. 计算完毕，返回。将输出状态写回 `DB100.DBX16.0 (stat_Timer_1.Q)`。 |
| **3** | `CALL #stat_Ti_2` | `0000_1388` (5000ms) | `16#8500_00B0` | `100` | - | **调用定时器 2 瞬间微码动作**：
1. 计算相对偏移量 `22.0`（十六进制 `B0`）。
2. **新 AR2 被临时重置为 `P#B0.0`**（即 `P#22.0`）。
3. 动态打开 `SFB4`，以完全隔离、独立的数据背景，运行 2 号定时器。 |

### 2.3 工程经验：彻底化解“IEC 定时器无法复位（Reset Timer Lock-on）”的隐性 Bug

在现场调试使用 STL 编写的复杂顺控时，很多具有 LAD 基础的工程师经常会遇到一个关于 IEC 定时器的诡异 Bug：**即使停止触发信号 `IN` 已经变为 0，定时器的输出 `Q` 依然固执地保持为 1，导致下一次顺控完全无法重新启动。**

*   **痛点原因：未退块扫描更新**。
    在梯形图（LAD）中，`TON` 定时器只要放在 Network 中，每个扫描周期都会被无条件调用并刷新其状态。
    而在 STL 中，很多工程师喜欢通过跳转指令（如 `JCN`）把整个包含 `CALL #stat_Timer` 的 Network 绕过去。
    这导致：**当触发条件变为 0 时，由于跳转直接跳过了定时器的 `CALL` 指令，定时器在底层根本没有被执行，从而使其内部状态永远锁死在上一扫描周期的“启动/已完成（Q=1）”状态。**
*   **黄金纠错规范**：
    1.  **绝不跳过定时器**：严禁通过跳转指令跳过任何多重背景定时器的调用。
    2.  **显式强制刷新**：如果必须执行重置，必须在逻辑中无条件扫描调用该定时器，且明确向其 `IN` 参数写入 `0`，执行一次**“状态卸载”**：
        ```stl
        // 绝对安全的定时器复位方案：
        A     "i_Reset_System";
        JCN   Normal_Run;
        
        // 强制复位：向 IN 写入 0，PT 写入 0，执行一次 CALL，
        // 强行刷新并洗去背景 DB 内部的时钟残余状态！
        CALL  #stat_Timer_1 (
           IN := FALSE,
           PT := T#0ms
        );
        ```

## 3. BCD 计数器（S5）与 32 位强类型计数器（CTU, CTD, CTUD）

数据累加与工件计数，是工业包装、物流分拣的基础。

### 3.1 计数器内部存储介质与限额差异

*   **S5 计数器（`C0` ~ `C255`）**：
    采用 16 位 BCD 格式。
    **物理极限限额**：**最大只能计数到 999**（3 位 BCD 码最大表示范围）。
    如果在现场生产中，工件计数需要达到上万个，S5 计数器会在溢出后直接锁死在 999，从而导致后续的包装堆垛逻辑完全失效。
*   **IEC 计数器（`CTU` 加计数 / `CTD` 减计数 / `CTUD` 加减计数）**：
    完全声明于 Instance DB 或多重背景中。
    **物理极限限额**：支持 `INT`、`DINT` 强类型。当使用双整数 `DINT` 时，其**最大计数容量高达 +2,147,483,647**。即使是一条每秒生产 100 个工件的超高速流水线，也足以连续运行数年而绝不溢出。

### 3.2 知识点 3 代码：工件包装堆垛顺控计数器（双向加减计数）

下面我们在 STL 内部，使用多重背景加减计数器（`SFB2` / `CTUD`），实现一个高可靠性的工件装箱堆垛控制。当包装箱内装满 50 个工件时触发卸料，卸料完成后自动复位。

#### 完整 STL 源代码

```scl
FUNCTION_BLOCK "Stack_Pack_Control"
TITLE = 基于多重背景加减计数器的高性能包装堆垛控制
VERSION : 0.1
VAR
   stat_Pack_Counter : "CTUD" ; // 多重背景：加减计数器 (SFB2)
   stat_Box_Full     : BOOL ;   // 包装箱满信号
   stat_Cur_Count    : INT ;    // 当前计数
END_VAR
VAR_TEMP
   t_AR2_Backup : DWORD ;
END_VAR
BEGIN
NETWORK
TITLE = 1. 加减计数动态计算
      TAR2  #t_AR2_Backup;    // 备份多重背景基准 AR2

      // 调用 SFB2 多重背景加减计数器
      CALL  #stat_Pack_Counter (
         CU := "i_Workpiece_Sensor",   // 加计数输入：工件到达传感器上升沿 (I1.0)
         CD := "i_Defect_Eject",       // 减计数输入：剔除废品传感器上升沿 (I1.1)
         R  := "i_New_Box_Placed",     // 复位输入：新包装箱就位信号 (I1.2)
         LD := FALSE,                  // 装载输入
         PV := 50,                     // 设定计数值 (满50个工件触发卸料)
         QD := #stat_Box_Full,         // 输出：计数到达下限
         QU := "q_Box_Unload_Trigger", // 输出：计数到达 PV 设定值（满箱，触发卸料 Q1.0）
         CV := #stat_Cur_Count         // 当前计数值暂存到静态变量
      );

      LAR2  #t_AR2_Backup;    // 还原 AR2 寄存器
END_FUNCTION_BLOCK
```

### 3.3 工程经验：处理高速加减计数器时的“双边界防卷回（Rollover Protection）”设计

在工业现场（如多车位立体车库车辆计数、连续长皮带码垛计数）中，如果由于外界高频噪声干扰，导致剔除废品传感器（减计数 `CD`）意外多触发了数次，**使得计数器的当前值（`CV`）在复位前变为了负数（小于0）。**

*   **物理隐患**：
    在某些低端 PLC 的计数器中，**负数在二进制下会发生严重的“符号位反转补码卷回”**（例如：`-1` 在 16位有符号下表现为 `16#FFFF`，如果以无符号形式解析，这个数会瞬间变为 `65535`）。这会导致系统直接误判定为“车库已满”，直接锁死入口道闸。
*   **双边界安全防护规范（绝对防御体系）**：
    在任何需要高可靠运行的加减计数器后，必须紧跟一行对 `CV` 当前值的上下限截断保护：
    ```stl
    // 强制进行下限拦截：
    L     #stat_Cur_Count
    L     0
    <I    ;                   // 若当前计数值由于干扰变为了负数
    JCN   Max_Check;          // 若无，跳转至上限检查
    
    // 强制下限复位
    L     0
    T     #stat_Cur_Count;    // 强行把当前值归零
    T     #stat_Pack_Counter.CV; // 穿透写入计数器内部的当前值寄存器，彻底洗去干扰！
    
    Max_Check: NOP 0;
    ```

## 5. TIA Portal S7-1200 / S7-1500 现代 SCL 定时器与计数器多重背景重构

由于 **S7-1200 原生不支持任何 S5 定时器/计数器，亦不支持 STL 编程**。本章为 TIA Portal 平台用户提供完全等效的、100% 优化块访问的优秀 **SCL 重构方案**。

### S7-1200 / S7-1500 优化的 SCL 多重背景定时器与计数器（SCL Multi-Instance）

在 SCL 中，我们直接在 FB 的 `VAR`（静态变量）区声明标准的 `TON_TIME` 和 `CTUD_INT`。博途编译器会自动将其映射为最高效率的指令。

```scl
FUNCTION_BLOCK "Stack_Pack_Control_SCL"
{ S7_Optimized_Access := 'true' } // 启用优化块访问，物理寻址交由编译器托管
VERSION : 0.1
VAR_INPUT
   i_Workpiece_Sensor   : Bool;       // 加计数传感器 (I1.0)
   i_Defect_Eject       : Bool;       // 减计数传感器 (I1.1)
   i_New_Box_Placed     : Bool;       // 复位信号 (I1.2)
END_INPUT
VAR_OUTPUT
   q_Box_Unload_Trigger : Bool;       // 卸料输出 (Q1.0)
END_VAR
VAR
   stat_Pack_Counter : CTUD_INT;      // 声明多重背景：加减计数器 (IEC Standard)
   stat_Delay_Timer  : TON_TIME;      // 声明多重背景：卸料延时定时器 (IEC Standard)
   stat_Box_Full     : Bool;          // 满箱状态
   stat_Cur_Count    : Int;           // 当前计数值
END_VAR
BEGIN
    // 1. 调用多重背景加减计数器 (SCL 优雅调用)
    #stat_Pack_Counter(CU := #i_Workpiece_Sensor,
                       CD := #i_Defect_Eject,
                       R := #i_New_Box_Placed,
                       LD := FALSE,
                       PV := 50,
                       QU => #stat_Box_Full,
                       CV => #stat_Cur_Count);
                       
    // 2. 双边界安全防御拦截：防止剔除过多导致计数出现负数
    IF #stat_Cur_Count < 0 THEN
        #stat_Cur_Count := 0;
        #stat_Pack_Counter.CV := 0;   // 彻底洗去负数脏数据
    END_IF;
    
    // 3. 满箱后执行延时卸料 (延时 2s 保证工件完全落入箱内)
    #stat_Delay_Timer(IN := #stat_Box_Full,
                      PT := T#2s,
                      Q => #q_Box_Unload_Trigger);
                      
END_FUNCTION_BLOCK
```

## 6. 工业级规范与调试经验总结

在本章也是整个核心指令部分的尾声，我们为具有 LAD 基础的工程师梳理出 2 条在实际调试和优化定时器/计数器时的金牌经验：

1.  **在 TIA 在线监视中使用“时序分析（Trace）”工具定位定时器偶发性不动作故障**：
    在复杂的顺控联锁中，有些定时器的启动信号 `IN` 可能只接通了短短的几十毫秒（由于上游触点高频抖动），随后又断开了。这会导致定时器根本无法正常累计到设定时间 `PT`，输出 `Q` 永远不动作。
    *   **调试大招**：千万不要用普通的监控表去肉眼观察。**立刻右键点击 PLC，选择“添加新趋势（Add New Trace）”**。将定时器的 `IN`、`ET`（累积时间）和 `Q` 信号拉入趋势中。
        Trace 会以 **1ms 级的硬件级分辨率**，将这些信号的波形完整绘制出来。你可以一秒内查明：正是因为启动脉冲宽度太窄，导致了定时器无法正常启动，瞬间锁定现场物理设计缺陷。
2.  **在涉及配方和变时间控制的场合，PT（设定时间）引脚禁止直接写入常量，推荐通过变量动态解耦传入**：
    有些工程师喜欢在定时器引脚上直接写死 `T#10s`。如果现场工艺发生改变（例如因为气温降低，物料需要从混合 10s 延长到 15s），你不得不重新打开程序、修改代码、下载块。
    *   **金牌工程规范**：PT 引脚上**必须统一挂接在 HMI 配方 DB 块中的变量（如 `"gdb_Recipe".Mix_Delay_Time`）上**。让操作员可以直接在触摸屏上根据实际物理工况，无损、无停机、无安全风险地动态修改时间常数，大幅提升整套控制系统的工程智能化水平。
