在 Linux 运维、云服务器管理和 DevOps 的日常工作中，网络带宽异常暴涨是一个高频发生的棘手问题。当服务器带宽跑满、业务响应变慢、甚至遭遇 DDoS 攻击时，如何快速、准确地找出是“哪个 IP”、“哪个端口”在消耗流量，直接决定了故障恢复的时间。

在众多的网络监控工具中，`iftop` 凭借其直观的实时界面、强大的 BPF 过滤功能和低侵入性，成为了排查网络带宽故障的首选利器。

本文将从工作原理、安装部署、基础命令、界面交互、BPF 过滤、高频实战场景、常见问题排查及同类工具对比等多维度，系统且深入地介绍 `iftop` 的使用。

# 1. 什么是 iftop

## 1.1 iftop 的作用
`iftop`（Interface Top）是一款开源的实时网络流量监控工具，运行于 Linux 命令行终端。它类似于系统资源监控工具 `top`，但 `top` 监控的是 CPU、内存和进程，而 `iftop` 监控的是**网卡上的实时网络连接及带宽占用**。

通过 `iftop`，运维人员可以直观地看到：
* 当前网卡正在与哪些外部 IP 进行通信。
* 每一个网络连接的上传（TX）和下载（RX）带宽。
* 过去 2 秒、10 秒和 40 秒内的平均带宽消耗。
* 整体网卡的瞬时和累计流量。

```
              +---------------------------------------------+
              |                 iftop 界面                  |
              +---------------------------------------------+
              |  Host_A  ===>  Host_B    (2s, 10s, 40s 流量) |
              |          <===            (2s, 10s, 40s 流量) |
              +---------------------------------------------+
```

## 1.2 工作原理（基于 libpcap）
`iftop` 的底层核心依赖于 `libpcap` 库（在 Windows 上为 WinPcap/Npcap，Linux 下主要为 `libpcap`）。其流量捕获和统计的工作流如下：

```
+--------------+      +----------------+      +--------------+      +---------------+
|  网络物理介质 | ===> |   网卡驱动层   | ===> | 内核网络协议栈| ===> |   用户态应用  |
+--------------+      +----------------+      +--------------+      +---------------+
                              ||                      ||
                       (网卡设置为混杂模式)            || (系统调用)
                              ||                      \/
                      +----------------------------------+
                      |        libpcap 捕获环形缓冲区     |
                      +----------------------------------+
                                      ||
                                      \/
                      +----------------------------------+
                      |       iftop 流量统计与渲染       |
                      +----------------------------------+
```

1. **网卡混杂模式（Promiscuous Mode）**：默认情况下，网卡只接收目的 MAC 地址是自己的数据包。当 `iftop` 启动时（默认或指定参数下），它会通过 `libpcap` 将指定的网卡置于混杂模式。此时，网卡会接收通过该物理网段的所有数据包，无论其目的地是谁。
2. **内核态包捕获与过滤**：数据包进入网卡后，`libpcap` 利用 Linux 内核的 `packet mmap` 机制或 `AF_PACKET` 套接字，在内核空间直接克隆一份数据包。如果用户指定了过滤规则（BPF 过滤器），内核会直接过滤掉不符合条件的数据包，减少拷贝开销。
3. **用户态数据收集**：`iftop` 从 `libpcap` 的环形缓冲区中提取报文，解析其 IP 头部（获取源 IP 和目的 IP）及传输层头部（获取 TCP/UDP 端口号），并记录报文的长度。
4. **统计与渲染**：`iftop` 在内存中维护着一个以“源IP:源端口 <-> 目的IP:目的端口”为 Key 的哈希表，定时累加流量数据，并通过 `ncurses` 库在终端渲染出动态的条形图和数据表格。

## 1.3 与 top 的区别
* **聚焦对象不同**：`top` 监控的是单机操作系统内部的进程、线程资源消耗（如 CPU 时间片、物理内存、虚拟内存、上下文切换）；而 `iftop` 监控的是系统的网络接口（NIC）与外部世界交互的网络流量数据，不直接关联到操作系统内的进程 ID（PID）。
* **数据来源不同**：`top` 的数据源主要来自 Linux 内核导出的虚拟文件系统 `/proc`（如 `/proc/[pid]/stat`）；而 `iftop` 的数据来源于链路层的网络捕获，属于对网络报文的被动监听。

## 1.4 与其他网络工具的区别

| 工具名称 | 主要数据源 | 监控粒度 | 核心定位与适用场景 |
| :--- | :--- | :--- | :--- |
| **iftop** | libpcap 抓包 | 连接对（IP 对 IP，端口对端口） | 排查“谁在抢占带宽”、“当前与谁通信流量最大”。支持实时交互和 BPF 过滤。 |
| **nload** | `/proc/net/dev` | 网卡全局 | 直观展示单块网卡的总流入、总流出带宽，带有 ASCII 趋势图。无法看具体 IP。 |
| **iptraf-ng** | 原始套接字 | 混合（网卡/协议/IP/端口） | 菜单式交互，提供网卡、协议、TCP/UDP 连接等多种层面的统计。信息全面但稍显繁重。 |
| **nethogs** | `/proc/net/tcp` 关联 | 进程级（PID） | 按进程（PID）或程序名称统计流量。适合定位“哪个本地进程在疯狂下载”。 |
| **vnstat** | `/proc/net/dev` 历史库| 历史统计（天/月/小时） | 后台守护进程运行，不适合实时排查，适合做流量计费核对、长期带宽趋势观察。 |
| **tcpdump** | libpcap 抓包 | 报文级（Packet） | 深度网络协议分析工具。不做流量大小统计，而是把每一个包的细节完整抓取，供后续 Wireshark 分析。 |
| **ss / netstat** | 内核 Socket 状态 | 连接状态（State） | 查看当前活跃的 TCP/UDP 连接状态，非实时流量大小。适合排查连接数爆满或端口监听。 |

# 2. 安装 iftop

在安装之前，建议确保系统已更新包管理器索引，并拥有管理员（root）权限。

## 2.1 RedHat 系列（CentOS / Rocky Linux / AlmaLinux）
在 RHEL 及其衍生版中，`iftop` 并不在官方默认的 BaseOS 仓库中，需要先启用 EPEL（Extra Packages for Enterprise Linux）源。

### 2.1.1 CentOS 7 / 8
```bash
# 安装 EPEL 源
sudo yum install -y epel-release

# 安装 iftop
sudo yum install -y iftop
```

### 2.1.2 Rocky Linux 8/9 & AlmaLinux 8/9
```bash
# 启用 EPEL 仓库
sudo dnf install -y epel-release

# 安装 iftop
sudo dnf install -y iftop
```

## 2.2 Debian / Ubuntu 系列
在 Debian 和 Ubuntu 中，`iftop` 被包含在官方的 main 或 universe 仓库中，可以直接通过 `apt` 安装。

```bash
# 更新本地包索引
sudo apt-get update

# 安装 iftop 及其核心依赖库
sudo apt-get install -y iftop
```

## 2.3 源码编译安装（可选）
当目标系统处于内网无源环境，或者需要使用特定版本的 `iftop` 时，可采用源码编译。

### 2.3.1 安装编译依赖
编译 `iftop` 需要 C 编译器（gcc）、make 工具、libpcap 开发库以及 ncurses 图形库。

* **CentOS/Rocky 系列**：
  ```bash
  sudo yum install -y gcc make libpcap-devel ncurses-devel
  ```
* **Debian/Ubuntu 系列**：
  ```bash
  sudo apt-get update
  sudo apt-get install -y build-essential libpcap-dev libncurses5-dev
  ```

### 2.3.2 下载并编译
```bash
# 下载官方发布的源码包（以最新稳定版 1.0pre4 为例）
wget http://www.ex-parrot.com/~pdw/iftop/download/iftop-1.0pre4.tar.gz

# 解压源码
tar -zxvf iftop-1.0pre4.tar.gz
cd iftop-1.0pre4

# 配置编译参数，生成 Makefile
./configure

# 编译并安装到系统中（默认安装到 /usr/local/sbin/iftop）
make
sudo make install
```

## 2.4 验证安装
安装完成后，可以通过查看版本号或帮助信息来验证是否成功。

```bash
iftop -h
```
如果正常输出类似以下的版本及帮助信息，说明安装成功：
```text
iftop, version 1.0pre4
usage: iftop [-i interface] [-f filter code] [-N] [-P] ...
```

# 3. iftop 基础使用

`iftop` 命令提供了丰富的启动命令行参数，可以灵活地调整其初始运行行为。

```
语法格式：
iftop [选项]
```

下面对每一个常用参数进行深度拆解：

---

### iftop（无参数默认启动）
* **作用**：默认监控系统路由表中默认网关所在的第一块活动网卡（通常是 `eth0`、`ens33` 等），显示所有流量。
* **语法**：`iftop`
* **实际案例**：
  ```bash
  sudo iftop
  ```
* **输出解释**：启动交互式图形界面，展示默认网卡的实时双向连接流量。
* **适用场景**：快速概览当前系统最主要的网络接口流量。
* **注意事项**：若系统有多块网卡（如内网和外网），默认启动可能监控的不是你期望的那块，需通过 `-i` 指定。

---

### iftop -i <interface>
* **作用**：指定要监控的网卡接口。
* **语法**：`iftop -i <interface_name>`
* **参数说明**：`<interface_name>` 为网卡名称（如 `eth1`、`ens18`、`lo`）。
* **实际案例**：监控内网网卡 `eth1` 的流量：
  ```bash
  sudo iftop -i eth1
  ```
* **输出解释**：强制 `iftop` 仅捕获并统计通过 `eth1` 网卡的数据包。
* **适用场景**：多网卡服务器，排查特定网卡（如专线网卡、内网网卡）的流量异常。
* **注意事项**：如果指定的网卡不存在，系统会报错 `pcap_open_live: ... No such device`。

---

### iftop -n
* **作用**：禁用 DNS 反向解析，直接显示 IP 地址。
* **语法**：`iftop -n`
* **参数说明**：`-n` 代表 numeric，即数字显示。
* **实际案例**：
  ```bash
  sudo iftop -n
  ```
* **输出解释**：界面中的主机列不再显示如 `baidu.com` 的域名，而是显示其对应的公网 IP `110.242.140.4`。
* **适用场景**：**线上排查必加参数**。DNS 解析会消耗额外时间并产生 DNS 查询流量，在网络拥堵时，开启 DNS 解析会导致 `iftop` 严重卡顿，甚至误导排查。
* **注意事项**：可以在运行过程中通过快捷键 `n` 随时切换。

---

### iftop -N
* **作用**：禁用服务端口到服务名称的转换，直接显示端口号。
* **语法**：`iftop -N`
* **参数说明**：`-N` 代表不进行端口服务的解析（不读取 `/etc/services`）。
* **实际案例**：
  ```bash
  sudo iftop -P -N
  ```
* **输出解释**：将本应显示为 `ssh`、`http`、`mysql` 的服务名，强制显示为数字形式的端口号 `22`、`80`、`3306`。
* **适用场景**：当需要准确掌握非标准端口的流量时（例如自定义的 8080 端口，不希望它被解析为特定服务名）。
* **注意事项**：必须结合 `-P`（显示端口）参数，`-N` 单独使用时无实际视觉变化。

---

### iftop -P
* **作用**：在界面中显示主机通信的端口号。
* **语法**：`iftop -P`
* **参数说明**：`-P` 代表 Port（端口）。
* **实际案例**：
  ```bash
  sudo iftop -nP
  ```
* **输出解释**：在 IP 地址后方增加一个冒号加端口号，例如 `192.168.1.100:54322`。
* **适用场景**：排查具体是哪个服务（如 MySQL 的 3306，Redis 的 6379）占用了大量带宽。
* **注意事项**：开启端口显示后，界面中的行数会增加（因为一个 IP 对可能存在多个端口连接），信息量会变大。

---

### iftop -B
* **作用**：以字节（Bytes）为单位显示流量，默认是比特（Bits）。
* **语法**：`iftop -B`
* **参数说明**：`-B` 代表 Bytes/sec。
* **实际案例**：
  ```bash
  sudo iftop -B
  ```
* **输出解释**：右上角的刻度尺和右侧的数据列，单位均从 `Mbps/Kbps` 变为 `MB/KB`（例如 8 Mbps 将显示为 1 MB/s）。
* **适用场景**：与业务层文件传输速度（如 wget、nginx 传输速度，通常以 字节/秒 衡量）进行对比时。
* **注意事项**：IDC 带宽和云服务商通常以 **Bit**（比特）为单位计费，而文件大小以 **Byte**（字节）为单位，两者关系为 `1 Byte = 8 Bits`。

---

### iftop -f <filter_code>
* **作用**：使用 BPF（Berkeley Packet Filter）过滤语法，在底层过滤符合特定条件的报文，减少性能开销。
* **语法**：`iftop -f "<filter_expression>"`
* **参数说明**：可以使用 host、port、net 等过滤元。
* **实际案例**：只监控与 IP `1.1.1.1` 之间的流量：
  ```bash
  sudo iftop -f "host 1.1.1.1"
  ```
* **输出解释**：屏幕上只会出现涉及 `1.1.1.1` 这个 IP 的通信对。
* **适用场景**：高并发、大流量服务器。如果不做过滤，全局抓包会导致 `iftop` 自身占用高额 CPU 导致丢包。
* **注意事项**：过滤表达式必须用双引号括起来，否则会被 Shell 解析。

---

### iftop -F <net/mask>
* **作用**：仅监控指定网段的流量（可以是 IPv4 或 IPv6）。
* **语法**：`iftop -F <ip_address/mask_bits>`
* **参数说明**：支持标准的 CIDR 格式。
* **实际案例**：只监控内网 10.0.0.0/8 网段的流量：
  ```bash
  sudo iftop -F 10.0.0.0/8
  ```
* **输出解释**：任何不属于 `10.0.0.0/8` 子网的 IP 通信都会被 `iftop` 过滤掉。
* **适用场景**：排查跨机房、内网集群内部的数据传输。
* **注意事项**：此参数比 `-f` 语法更为简练，专门针对 IP/网段过滤。

---

### iftop -G <net/mask6>
* **作用**：仅监控指定的 IPv6 网段。
* **语法**：`iftop -G <ipv6_address/mask_bits>`
* **实际案例**：
  ```bash
  sudo iftop -G 2001:db8::/32
  ```
* **适用场景**：原生 IPv6 网络环境下的流量监测。

---

### iftop -m <limit>
* **作用**：限制刻度尺（条形图）的最大带宽显示上限。
* **语法**：`iftop -m <limit>[G|M|K]`
* **参数说明**：支持 G（Gbps）、M（Mbps）、K（Kbps）。
* **实际案例**：将刻度尺的最大值固定为 100Mbps：
  ```bash
  sudo iftop -m 100M
  ```
* **输出解释**：最顶部刻度条的最右侧终点被固定为 100M，以便观察当前流量占总可用带宽的百分比。
* **适用场景**：当你知道服务器的带宽上限时（例如公网带宽 10Mbps），强制设置可以让条形图的比例展示更具参考价值。

---

### iftop -t
* **作用**：以非交互的纯文本（text）模式运行，直接输出到标准输出，不渲染图形界面。
* **语法**：`iftop -t [其他参数]`
* **实际案例**：
  ```bash
  sudo iftop -t -n -N -P -L 10
  ```
* **输出解释**：在终端打印一次流量快照，然后自动退出。
* **适用场景**：编写自动化监控脚本，或者通过 cron 定时任务捕获瞬间网络状态，并保存到日志。

---

### iftop -L <number>
* **作用**：在文本模式（`-t`）下，指定输出的流量连接对的最大数量（行数）。
* **语法**：`iftop -t -L <number>`
* **实际案例**：
  ```bash
  sudo iftop -t -n -L 5
  ```
* **输出解释**：只输出当前带宽占用的前 5 个连接。
* **适用场景**：过滤杂音，只在日志或终端中保留最主要的几个流量大户。

---

### iftop -p
* **作用**：在非混杂模式（Non-promiscuous mode）下运行。
* **语法**：`iftop -p`
* **实际案例**：
  ```bash
  sudo iftop -p
  ```
* **输出解释**：网卡不开启混杂模式，只捕获并统计“直接发送给本机”和“由本机发出”的报文。
* **适用场景**：在共享网络介质环境中（如老式集线器 Hub 或特定未做端口隔离的交换机），避免将其他非本机主机的物理流量统计进来；同时也稍微降低了网卡的工作开销。

---

### iftop -b
* **作用**：在界面中默认隐藏顶部刻度条。
* **语法**：`iftop -b`
* **适用场景**：节省宝贵的终端屏幕垂直空间，特别是在移动端 SSH 或小窗口分屏时。

---

### iftop -h
* **作用**：打印帮助信息。
* **语法**：`iftop -h`

---

# 4. iftop 界面详解

启动 `iftop`（不加 `-t` 参数）后，会进入一个全屏的 ncurses 实时交互界面。这个界面包含大量关键指标，掌握其布局和数据逻辑至关重要。

以下是 `iftop` 运行时的经典界面版面：

```text

└───────────────────────────────┴────────────────┴────────────────┴────────────────┴──────────┘
192.168.1.100:ssh         ==> 10.211.55.2:50123         120Kb   85.2Kb   42.1Kb
                          <==                           2.4Kb    1.8Kb    1.2Kb
192.168.1.100:http        ==> 110.242.140.4:443           0b       0b       0b
                          <==                            20Kb     15Kb     10Kb

```

## 4.1 刻度尺（Top Scale Bar）
位于界面最顶部：
```text
TX:             clique          | 1.91Mb         | 3.81Mb         | 5.72Mb         | 7.63Mb   | 9.54Mb
```
* **作用**：它代表当前时刻，条形图（连接右侧的背景白色/灰色横条）所代表的带宽比例尺。
* **自适应机制**：默认情况下，该刻度尺会根据当前最大连接的带宽动态调整。若要锁定，可使用 `-m` 参数。

## 4.2 主机通信行（Connection List）
界面中部的核心区域，每两行代表一个网络连接（双向通道）：

1. **第一行**：
   `192.168.1.100:ssh ==> 10.211.55.2:50123`
   * **`==>`**：表示**发送方向（TX，Transmit）**。即从本地（左侧）发往外部（右侧）。
   * 对应右侧的三个数值（例如 `120Kb  85.2Kb  42.1Kb`），分别表示**发送方向**在过去 **2秒**、**10秒** 和 **40秒** 内的平均发送带宽。
2. **第二行**：
   `<==`
   * **`<==`**：表示**接收方向（RX，Receive）**。即从外部（右侧）发往本地（左侧）。
   * 对应右侧三个数值（例如 `2.4Kb  1.8Kb  1.2Kb`），分别表示**接收方向**在过去 **2秒**、**10秒** 和 **40秒** 内的平均接收带宽。

## 4.3 底部汇总区（Summary Statistics）
位于界面最下方的三行：

* **TX (Transmit - 发送)**：
  * **Cum（Cumulative）**：自 `iftop` 启动以来，**发送**的累计总流量（例如 `45.2MB`）。
  * **peak**：启动以来的**发送**流量峰值（例如 `3.22Mbps`）。
  * **rates**：当前网卡发送方向，在过去 **2秒、10秒、40秒** 的平均发送总带宽（例如 `120Kb  85.2Kb  42.1Kb`）。
* **RX (Receive - 接收)**：
  * **Cum**：自启动以来，**接收**的累计总流量。
  * **peak**：启动以来的**接收**流量峰值。
  * **rates**：当前网卡接收方向，在过去 **2秒、10秒、40秒** 的平均接收总带宽。
* **TOTAL (总计)**：
  * **Cum**：发送和接收流量的累计之和（`TX Cum + RX Cum`）。
  * **peak**：网卡双向总带宽的峰值。
  * **rates**：网卡双向在过去 **2秒、10秒、40秒** 的平均总带宽。

## 4.4 排序规则
默认情况下，`iftop` 按照 **10秒平均总带宽（即中间那一列数值）** 从大到小进行降序排列。
* 可以通过交互按键 `1`、`2`、`3` 分别切换为按 2s、10s、40s 平均流量进行排序。
* 可以通过 `o` 键锁定当前排序状态，防止条目频繁上下跳动。

## 4.5 各种颜色与符号解释
* **白色/反显横条**：在连接行背后的白色横条，代表该连接的瞬时流量占顶部刻度尺上限的比例。横条越长，说明该连接占用的带宽比例越大。
* **`*` 号**：如果主机名或 IP 地址太长，终端宽度不够显示，尾部会出现 `*` 表示被截断。可以通过快捷键 `A` 隐藏或展开。

# 6. BPF 过滤器详解

`iftop` 底层调用 `libpcap`，因此完美支持 **BPF（Berkeley Packet Filter）** 表达式。利用 BPF，你可以像使用 `tcpdump` 一样精准地过滤你想要的报文，这在海量网络连接的服务器上是避免 CPU 跑满和信息淹没的唯一手段。

## 6.1 基本过滤元（Primitives）

* **`host`**：过滤特定主机 IP 或主机名。
  ```bash
  host 192.168.10.12
  ```
* **`src`**：限制源地址。
  ```bash
  src host 10.0.0.5
  ```
* **`dst`**：限制目的地址。
  ```bash
  dst host 8.8.8.8
  ```
* **`net`**：限制网段。
  ```bash
  net 192.168.1.0/24
  # 或者
  net 172.16.0.0 mask 255.255.0.0
  ```
* **`port`**：过滤特定端口（源和目的都算）。
  ```bash
  port 80
  ```
* **`tcp` / `udp` / `icmp`**：限制协议类型。
  ```bash
  tcp
  ```

## 6.2 逻辑运算符
* **`and`（或 `&&`）**：与关系。左右两边条件必须同时满足。
* **`or`（或 `||`）**：或关系。满足其中一个条件即可。
* **`not`（或 `!`）**：非关系。排除符合条件的报文。

## 6.3 复合过滤案例

### 案例 A：监控特定 IP 之外的所有 TCP 流量
```bash
sudo iftop -f "tcp and not host 192.168.1.100"
```
* **原理解释**：捕获所有 TCP 包，但如果这个包的源 IP 或目的 IP 是 `192.168.1.100`，则直接丢弃。

### 案例 B：监控来自于内网某网段，且目标端口是 80 或 443 的流量
```bash
sudo iftop -f "src net 10.0.0.0/8 and (dst port 80 or dst port 443)"
```
* **注意事项**：当 `and` 与 `or` 混用时，必须使用**括号**明确优先级，并且在命令行中括号可能需要转义，但在 `iftop -f` 的双引号内一般可直接使用。

# 7. 20 个高频实战案例

以下实战案例均采用真实业务场景设计，包含完整的排查链路。

### 案例 1：服务器公网上传（TX）带宽突然跑满
* **问题描述**：监控告警提示，某台 Web 服务器的公网流出带宽持续达到 100Mbps 限制线，导致外部用户访问网站极其缓慢。
* **分析思路**：公网流出（TX）大，说明系统正在向外部发送大量数据。可能是：
  1. 正常的静态大文件下载（如 APP 包、视频包被恶意刷流量）。
  2. 遭到黑客入侵，服务器正在向外充当 DDoS 攻击源。
  3. 内部数据备份、同步任务错峰失效。
* **执行命令**：
  ```bash
  # 监控默认公网网卡，不解析域名，显示端口，以字节（Bytes）显示，限制只看前10行
  sudo iftop -i eth0 -n -P -B -L 10
  ```
* **输出分析**：
  ```text
  192.168.1.100:80 ==> 203.208.60.12:45922      0B     11.2MB/s   10.5MB/s
                    <==                          0B      12KB/s     10KB/s
  ```
  通过输出发现，本地的 `80` 端口（HTTP）正源源不断向外部 IP `203.208.60.12` 的随机端口发送高达 `11.2 MB/s`（约 90 Mbps）的数据，基本可以判定是该外部 IP 正在疯狂拉取服务器上的 Web 资源。
* **解决方案**：
  1. 通过 Nginx 日志定位该 IP 访问的具体 URL，若是无意义的恶意下载，在 Nginx 或防火墙中封禁此 IP。
  2. 使用 `iptables` 或 `firewalld` 临时丢弃该主机的连接：
     ```bash
     sudo iptables -I INPUT -s 203.208.60.12 -j DROP
     ```
* **最佳实践**：对 Web 服务器的静态大文件接口设置限速（Nginx `limit_rate`），并引入 CDN 缓解源站压力。

### 案例 2：服务器下载（RX）带宽异常暴涨
* **问题描述**：运维平台显示主机入网流量（RX）突增，可能影响同网段其他业务响应。
* **分析思路**：系统在疯狂下载东西。可能是内部定时任务拉取大镜像、大备份文件，也可能是服务器被注入木马正在下载恶意负载。
* **执行命令**：
  ```bash
  sudo iftop -n -P -t -L 5
  ```
* **输出分析**：
  ```text
  192.168.1.100:41238 ==> 185.190.140.10:443       12b      15b      10b
                      <==                         12MB/s   11.5MB/s 10.2MB/s
  ```
  外部 IP `185.190.140.10` 的 `443` 端口（HTTPS）正向本机的临时端口 `41238` 发送超过 `11 MB/s` 的数据。
* **解决方案**：
  结合 `lsof` 或 `ss` 定位本地临时端口对应的进程：
  ```bash
  sudo lsof -i :41238
  ```
  输出：
  ```text
  COMMAND   PID USER   FD   TYPE DEVICE SIZE/OFF NODE NAME
  wget    31201 root    3u  IPv4 981223      0t0  TCP 192.168.1.100:41238->185.190.140.10:https (ESTABLISHED)
  ```
  确认是 PID 31201 的 `wget` 进程在后台下载。若非管理员操作，应立即 `kill -9 31201`，并排查计划任务（cron）。

### 案例 3：遭遇 SYN Flood / DDoS 攻击排查
* **问题描述**：服务器响应极慢，网站报 502/504 错误。
* **分析思路**：疑似遭遇拒绝服务攻击，需确认是否有大量外部 IP 涌入，且流量特征呈现“只有入包，没有出包”（单向大流）。
* **执行命令**：
  ```bash
  sudo iftop -n -P
  ```
* **输出分析**：
  在界面中，看到大量不同的外部 IP 极速涌向本机的 `80` 或 `443` 端口，但 `<==` 接收方向有流量（尽管每个 IP 只有几 Kbps），而 `==>` 发送方向流量几乎为零（0b），且整体连接数呈爆炸式增长。
* **解决方案**：
  1. 开启系统的 TCP SynCookies：
     ```bash
     sudo sysctl -w net.ipv4.tcp_syncookies=1
     ```
  2. 若攻击流量过大（超过公网带宽物理上限），需立即联系云厂商启用黑洞策略或接入高防 IP。

### 案例 4：排查疑似挖矿程序通信
* **问题描述**：安全态势感知平台告警，某服务器 CPU 跑满，疑似失陷并运行了挖矿程序。
* **分析思路**：挖矿程序需要连接矿池（通常通过非标准端口或 Stratum 协议）。利用 `iftop` 配合端口过滤，定位异常的外部连接。
* **执行命令**：
  ```bash
  # 排除常用的 SSH (22), HTTP (80), HTTPS (443), DNS (53) 端口
  sudo iftop -n -P -f "not port 22 and not port 80 and not port 443 and not port 53"
  ```
* **输出分析**：
  ```text
  192.168.1.100:59124 ==> 51.15.63.12:3333      1.2Kb    1.1Kb    1.0Kb
                      <==                       0.5Kb    0.4Kb    0.4Kb
  ```
  本地发起连接，目标 IP 为外网 `51.15.63.12`，目标端口为 `3333`（高危矿池常用端口，如 Monero 矿池）。
* **解决方案**：
  1. 通过 `ss -nap | grep 3333` 定位进程 PID。
  2. 杀死异常进程，清理 `/var/spool/cron/*` 下的恶意定时脚本，删除挖矿二进制文件。

### 案例 5：排查反弹 Shell / 木马后门通信
* **问题描述**：服务器被黑客植入后门，可能正在回传机密数据。
* **分析思路**：反弹 Shell 的特征是：本地主动发起连接，向外网某个高位随机端口发送控制流，或者高频发送微小报文维持心跳。
* **执行命令**：
  ```bash
  # 监控所有主动向外发起的非标准服务端口连接
  sudo iftop -n -P -f "tcp and src host 192.168.1.100 and not dst port 80 and not dst port 443"
  ```
* **输出分析**：
  发现一条长连接：
  ```text
  192.168.1.100:44321 ==> 45.76.x.x:4444       230b     230b     210b
                      <==                      150b     150b     140b
  ```
  `4444` 端口是经典的 Metasploit/Netcat 反弹监听端口。
* **解决方案**：
  阻断该 IP 通信，保留现场，分析 `/proc/` 下对应进程的符号链接，重置系统。

### 案例 7：定位大量短连接（TCP Time-Wait）导致的带宽损耗
* **问题描述**：服务器带宽并不大，但是网卡每秒包数（PPS）极高，CPU 软中断（softirq）占比居高不下。
* **分析思路**：大量短连接在快速建立与释放，频繁的 TCP 三次握手和四次挥手包占用了大量链路带宽。
* **执行命令**：
  ```bash
  sudo iftop -n -P
  ```
* **输出分析**：
  在主界面，看到满屏的连接对，右侧 2 秒平均值很小（几百字节），但行数极多，不断有新的端口号闪现又消失。
* **解决方案**：
  1. 优化内核 TCP 参数，回收 Time-Wait 连接：
     ```bash
     sudo sysctl -w net.ipv4.tcp_tw_reuse=1
     ```
  2. 业务侧改用 HTTP 长连接（Keep-Alive）。

### 案例 8：监控后端数据库（MySQL）流量爆发
* **问题描述**：应用服务器提示“数据库响应变慢”，怀疑有大 SQL 导出任务抢占了 MySQL 网卡带宽。
* **分析思路**：MySQL 默认端口为 `3306`。只关注 `3306` 端口的流入流出情况。
* **执行命令**：
  ```bash
  sudo iftop -n -P -f "port 3306"
  ```
* **输出分析**：
  ```text
  192.168.1.50:3306 ==> 192.168.1.101:45322      5.2MB/s   4.8MB/s   3.1MB/s
                    <==                           12KB/s    10KB/s     8KB/s
  ```
  数据库 IP `192.168.1.50` 正在向应用服务器 `192.168.1.101` 的临时端口疯狂推送数据（5.2MB/s）。
* **解决方案**：
  在 MySQL 中执行 `show full processlist;`，找出正在执行 `SELECT *` 且未加限制（Limit）的慢查询线程，予以杀死（kill query）。

### 案例 11：排查 Docker 桥接网络中的容器异常流量
* **问题描述**：Docker 宿主机网络 IO 异常，需要确定是哪一个容器在传输数据。
* **分析思路**：Docker 容器默认通过 `docker0` 网桥（或自定义网桥如 `br-xxxxx`）进行通信。在宿主机上，我们可以直接指定该虚拟网卡进行监控。
* **执行命令**：
  ```bash
  # 查看当前宿主机的所有网卡，找到 docker0
  ip addr
  # 监控 docker0 网卡的流量
  sudo iftop -i docker0 -n -P
  ```
* **输出分析**：
  ```text
  172.17.0.2:8080 ==> 172.17.0.3:51224      3.4MB/s   2.8MB/s   1.5MB/s
  ```
  通过 IP 地址，可直接锁定是容器 `172.17.0.2` 的 `8080` 端口在与 `172.17.0.3` 频繁通信。
* **解决方案**：
  使用 `docker ps -q | xargs docker inspect --format '{{.NetworkSettings.IPAddress}} {{.Name}}'` 批量匹配容器名，定位后进入容器进行整治。

### 案例 12：定位 Kubernetes Pod 级流量故障
* **问题描述**：Kubernetes 集群中某个节点上的网络组件频繁丢包，带宽报警。

# 8. iftop 常见问题（30 个 FAQ）

### Q1：为什么执行 iftop 提示 "command not found"？
**A**：说明系统未安装该软件。请参考本文第 2 节，根据系统版本（CentOS 使用 yum/dnf，Ubuntu 使用 apt-get）安装 `iftop`，或源码编译安装。

### Q2：为什么运行 iftop 提示 "pcap_open_live: eth0: You don't have permission to capture on that device"？
**A**：因为数据链路层抓包（libpcap）需要调用操作系统的特权系统调用，普通用户无此权限。必须使用 `sudo iftop` 或切换到 `root` 账户下运行。

### Q3：为什么我的 iftop 界面没有任何流量显示？
**A**：
1. 请确认指定的网卡（`-i` 参数）是否正确。可以通过 `ip link` 或 `ifconfig` 查看活跃网卡名。
2. 确认当前网卡确实有数据传输。可以尝试在另一个终端执行 `ping baidu.com`。
3. 检查系统防火墙或安全策略（如 SELinux）是否阻断了 pcap 抓包。

### Q4：为什么开启 iftop 后，界面反应非常迟钝、卡顿？
**A**：默认情况下，`iftop` 会试图把所有的 IP 地址反向解析（Reverse DNS）为主机名/域名。在网络异常拥堵或 DNS 服务器响应慢时，这会导致严重的线程阻塞。**请务必加上 `-n` 参数启动**（或者在交互界面按 `n` 键关闭解析）。

### Q5：为什么 iftop 统计的流量和 sar、nload 统计的有一些偏差？
**A**：
1. **统计层级不同**：`nload`/`sar` 从内核 `/proc/net/dev` 中直接读取网卡字节累加器，包含所有物理层/链路层的帧头尾开销；而 `iftop` 是在链路层/网络层抓包（libpcap），可能会因为系统 CPU 繁忙导致**部分丢包（packet dropped by filter）**，从而使得 `iftop` 统计的数据偏小。
2. **时间滑窗算法**：`iftop` 使用指数加权移动平均（EWMA）计算 2s/10s/40s 平均值，而 `sar` 通常是绝对平均，两者计算机制有轻微时差。

### Q6：如何查看具体是哪个进程（PID）产生了这些流量？
**A**：`iftop` 属于网络层/传输层流量观测工具，**原生不支持显示进程 PID**。若需要知道进程名，请：
1. 通过 `iftop -nP` 记录高流量的端口号。
2. 运行 `sudo lsof -i :<端口号>` 或 `sudo ss -nap | grep :<端口号>` 找出关联进程。
3. 或者直接改用 `nethogs` 工具。

### Q7：iftop 会导致服务器网络性能下降（网络延迟增加）吗？
**A**：在低流量（千兆以下）时，性能影响微乎其微。但在万兆（10Gbps）及以上的高并发链路中，开启混杂模式抓包会导致网卡向 CPU 发送大量的硬中断，且 `libpcap` 需要在内核与用户态之间拷贝大量数据，**会对系统性能产生明显的负面影响（引发丢包或 CPU 软中断跑满）**。

### Q8：为什么 iftop 显示的主机名后面有个星号（*）？
**A**：这代表主机名太长，当前终端窗口宽度有限，`iftop` 自动对文本进行了截断。可以通过调大终端窗口，或者在交互界面按 `A` 键关闭主机名解析来显示完整 IP。

### Q9：在 iftop 中，Tx 和 Rx 分别代表什么？
**A**：
* **TX（Transmit）**：发送流量，即从本机网卡流出去的流量。
* **RX（Receive）**：接收流量，即从外部网络流入本网卡的流量。

### Q10：为什么 iftop 显示有很多没有端口号的连接？
**A**：因为这些连接不是 TCP 或 UDP 协议。例如，ICMP 协议（如 ping 测速）、ESP/AH 协议（IPsec 虚拟专用网络）或者 ARP 报文没有端口概念，`iftop` 会显示 IP 级的对等流，但不显示端口。

### Q11：普通用户可以通过配置 sudoers 免密执行 iftop 吗？
**A**：可以。在 `/etc/sudoers` 中添加如下配置：
```text
username ALL=(root) NOPASSWD: /usr/sbin/iftop
```

### Q12：为什么使用 BPF 过滤器（-f）时，系统提示语法错误？
**A**：通常是因为过滤表达式中含有特殊字符，或者没有用双引号包裹。例如：`iftop -f host 1.1.1.1` 是错的，应当写成 `iftop -f "host 1.1.1.1"`。

### Q13：iftop 能够保存历史监控数据吗？
**A**：**不能**。`iftop` 定位为实时交互观测工具。如果需要历史流量审计，推荐使用 `vnstat`（基于内核统计并入库）或结合 Prometheus+Grafana。

### Q14：可以同时打开多个 iftop 进程监控同一个网卡吗？
**A**：可以。`libpcap` 支持多个捕获实例并存。但要注意，开启多个实例会成倍增加网卡报文克隆的 CPU 消耗。

### Q15：为什么即使我按了 N 键，端口还是显示为 SSH，而不是 22？
**A**：请确认是否已经开启了端口显示。在交互界面中，需要先按大写 `P` 开启端口展示，然后按大写 `N` 才能在“数字端口”与“服务名称”之间进行切换。

### Q16：如何在 iftop 启动时默认就是 Bytes（字节）单位，而不用手动按 B？
**A**：在启动时直接附加 `-B` 参数，例如：`sudo iftop -B`。或者在用户家目录下配置 `.iftoprc` 配置文件。

### Q17：.iftoprc 配置文件怎么写？
**A**：你可以在 `~/.iftoprc` 中定义默认参数，例如：
```text
dns-resolution: no
port-resolution: no
show-ports: yes
port-display: numeric
```
这样每次直接输入 `sudo iftop` 就会默认采用以上专业排查设置。

### Q18：为什么有些连接的 IP 后面显示的端口是星号（*）？
**A**：这是因为 `iftop` 已经合并了该 IP 对之间的多个连接，或者该协议不提供特定的端口号。通过快捷键 `S` 或 `D` 可以调整其聚合策略。

### Q19：iftop 能否抓取 Loopback（lo，本地回环）的流量？
**A**：可以。执行 `sudo iftop -i lo` 即可。这在排查本地进程（例如 Nginx 反向代理本地的 Tomcat/Node.js 服务）之间的内部流量时非常有用。

### Q20：在云服务器（如阿里云、腾讯云）上，iftop 的流量和云控制台显示的为什么不一样？
**A**：
1. **统计点不同**：云控制台监控的是云平台物理宿主机交换机上的端口流量（包括了云平台内网通信、挂载云盘等底层的管理流量）；而虚拟机的 `iftop` 只能看到操作系统网卡可见的虚拟网络报文。
2. **抽样步长**：云监控控制台通常是 1 分钟或 5 分钟抽样一次，呈现的是平均值，而 `iftop` 是秒级更新，展现的是波峰。

### Q21：非混杂模式（-p）有什么用，为什么要用它？
**A**：非混杂模式（Non-Promiscuous）下，网卡只接收目标 MAC 是自己，或者是广播/组播的帧。这在大流量的共享交换网络（如内网未做物理隔离）中，可以避免捕获并计算其他服务器的流量，降低 CPU 的中断压力。

### Q22：为什么 iftop 界面下方显示的 Cum 累计流量会清零？
**A**：当 `iftop` 退出或者检测到网卡状态断开重连时，内存中的数据结构会被销毁，累计值将清零。

### Q23：如何让 iftop 界面中某些不重要的 IP 通信行直接消失？
**A**：在交互模式下按 `l` 键（小写 L），会弹出 `Local filter` 输入框，输入不需要看见的 IP 的正则，回车即可在当前界面过滤。

### Q24：iftop 在高流量下（如 10G 光口）丢包严重怎么办？
**A**：
1. 使用 `-f` 参数添加极为精准的 BPF 过滤器，降低传递到用户态的报文数。
2. 增大 `libpcap` 的内核缓冲区大小（需要通过底层调优）。
3. 这种情况建议不要使用 `iftop` 交互界面，改用 `tcpdump -w` 保存到文件并在离线端用 Wireshark 渲染。

### Q25：为什么 iftop 界面上的条形图（比例尺）突然缩得很短？
**A**：因为网络中出现了一个瞬间的超大流量连接，拉高了整个刻度尺的上限（Top Scale）。可以按 `b` 键关闭比例尺，或者用 `-m` 参数在启动时锁定比例尺的最大值。

### Q26：如何查看 iftop 目前支持哪些快捷键？
**A**：在交互界面中，按 `h` 或 `?`，会弹出一个遮罩帮助菜单，列出当前版本所有可用的按键映射。按任意键退出帮助界面。

### Q27：iftop 支持监控 IPv6 吗？
**A**：支持。`iftop` 自 1.0 版本起完全支持 IPv6 数据报文的解析。

### Q28：为什么有时候 iftop 会显示一些奇怪的网卡名称，比如 vethxxxx？
**A**：这些是 Linux 虚拟网卡对（Virtual Ethernet Pair）。它们通常是由容器技术（Docker、Kubernetes、LXC）创建的，用于连接容器内部和宿主机的网络命名空间。

### Q29：如何把 iftop 的输出直接导出到 HTML 或图片？
**A**：`iftop` 原生不支持导出。但可以通过 `-t` 文本模式输出数据，然后利用第三方脚本（如 `ansi2html`）转换为 HTML 网页。

### Q30：为什么 iftop 显示的 TOTAL 流量不等于 TX + RX？
**A**：`TOTAL` 确实等于 `TX + RX`。如果你发现不相等，请检查数值单位（例如一个是 Kbps，一个是 Mbps，或者因为四舍五入保留小数导致的视觉微小误差）。

| :--- | :--- | :--- | :--- | :--- |
| **iftop** | 实时排查网络带宽被谁占满 | 直观的 IP/端口 对等流量展示，支持 BPF 过滤，交互性极佳 | 在万兆大流量下 CPU 消耗高，不记录进程 PID | **定位“哪个外部 IP/端口抢占了带宽”** |

# 10. 最佳实践与避坑指南

为了保证在生产环境（特别是核心业务服务器）中安全、高效地使用 `iftop`，建议遵守以下最佳实践：

## 10.1 线上生产环境启动建议参数
在生产服务器上，尤其是网络繁忙时，**千万不要直接运行 `sudo iftop`**。强烈推荐以下“排查专用”组合参数：

```bash
sudo iftop -n -P -N -m 100M
```
* **参数组合原理**：
  * `-n`：关闭 DNS 反向解析，防止因 DNS 查询导致响应卡死或产生额外解析流量。
  * `-P` 与 `-N`：显示数字端口，不解析服务名（如保持 `3306` 而不显示为 `mysql`），避免读取系统的 `/etc/services` 文件。
  * `-m 100M`：锁定刻度尺上限，让波峰图比例符合真实的物理带宽（如 100Mbps 限制）。

## 10.2 性能开销防范（避坑常识）
1. **大流量限用**：当服务器网卡流量超过 **3Gbps** 且包量（PPS）极高时，不建议运行 `iftop`。因为它基于 `libpcap` 软件抓包，此时开启混杂模式会导致大量的网络报文克隆及上下文切换，可能直接把服务器 CPU（软中断）挤爆，导致业务出现真实的网络丢包。
2. **多核时代注意**：`iftop` 的数据包捕获和界面渲染大部分在单核上运行。即使你的服务器有 64 个 CPU 核心，若主网卡流量过载，`iftop` 仍会把其中一个核心跑满（100%），进而出现丢包。

## 10.3 权限最小化与安全
1. **避免长期后台挂起**：不要在终端中通过 `nohup iftop &` 长期挂起运行。`iftop` 的内存哈希表在长时间累积大并发连接后，会导致内存开销变大。
2. **审计日志备份**：如需要长期监控，应使用系统级 `sar`，或者将 `iftop -t` 命令配合 crontab 每分钟导出一次快照，但要设置定时清理任务（Logrotate），避免撑爆磁盘。

## 10.4 配合自动化排查脚本
为了更高效地应急响应，可将 `iftop` 与 `ss`、`iptables` 结合，编写简易的“突发流量自动防御脚本”。例如，当检测到某个 IP 带宽超过 10MB/s 且连接数异常时，自动封禁 1 小时。

# 11. 总结

`iftop` 是 Linux 运维武器库中不可或缺的尖兵。

### 它的核心优势
* **实时直观**：对等网络流（Host-to-Host）的展现形式，让我们能在一秒内揪出抢占带宽的罪魁祸首。
* **BPF 赋能**：强大的过滤器语法，允许我们在海量报文中实施极其精确的切片式监控。
* **低侵入**：不需要在操作系统或应用层修改任何配置，随装随用。

### 它的局限性
* **缺乏进程绑定**：无法直接指出流量是由哪个 PID 产生的，需要配合 `lsof`/`ss` 辅助定位。
* **高负载开销**：不适用于超大规模（萬兆级）流量的线上长期常态监控。

**推荐使用场景**：
