在上一章中，我们攻克了扫码枪 ASCII 文本的清洗与标定转换。今天，我们要去降伏工业自动化中最无形、也最具决定性的物理维度——**时间（Time）**。

在任何自动化工厂里，时间都是隐形的协调员。
你很难想象一个没有时间控制的 PLC 项目：
*   “按钮按下去，警铃先响 3 秒（预报警），电机才允许合闸起动。”
*   “传感器检测到液位超标，不能立刻报警，得持续 2 秒以上（防抖过滤），防止液面波动引起误报。”
*   “主泵停机后，冷却风扇必须继续维持运行 30 秒（延时关断），把余热排掉才能停。”

这些动作，在 SCL 中主要通过时间数据类型 **`TIME`**、**`LTIME`** 以及西门子经典的**时间三剑客（`TON`、`TOF`、`TP`）**来实现。

很多刚从 LAD 转过来的徒弟，由于不理解定时器内部的**异步数据刷新机制**，写出的 SCL 定时器程序经常发生“时间死锁”、“无法二次触发”或者“计时器莫名卡住”。

今天，师父将带你从二进制底层剖析西门子时间类型的本质，揭开 SCL 定时器在编译期的多重背景机理，并手写一个生产级的**“液压泵站智能防抖、预警铃脉冲与风扇延时停机系统”**。

## 1. 西门子时间数据类型的二进制物理底层

在 SCL 中，我们最常打交道的时间类型有两大家族。

### 1.1 TIME（32位标准时间）
*   **物理本质**：在西门子 CPU 内部，`TIME` **本质上就是一个带符号的 32 位双整型（DINT）**！
*   **物理单位**：**1 毫秒（ms）**。
*   **SCL 常量书写格式**：`T#5s`（代表 5 秒）、`T#2d10h30m20s500ms`（天/小时/分/秒/毫秒）。

#### 💡 高级底层机理：
当你写下 `#tDelay := T#5s;` 时，CPU 在编译期会将它翻译成无符号整数 **`5000`**。
因为其底层是 DINT，所以 TIME 的最大极限范围是：
`T#-24d_20h_31m_23s_648ms` 到 `T#24d_20h_31m_23s_647ms`（正好对应 DINT 最小和最大整型边界）。

---

### 1.2 LTIME（64位高精度时间，仅限 S7-1500）
随着锂电池、半导体、超高速包装线对精度的要求达到微秒级，32位毫秒级的 `TIME` 已经捉襟见肘。
*   **物理本质**：带符号的 64 位整型（LInt）。
*   **物理单位**：**1 纳秒（ns）**！
*   **SCL 格式**：`LT#2ms_500us_100ns`（代表 2 毫秒 500 微秒 100 纳秒）。

---

### 1.3 师父的高效心法：时间直接数学运算
由于 `TIME` 底层就是整型，你在 SCL 里**可以直接对时间进行加减乘除数学运算**，这比在 梯形图（LAD）里拖放 `T_ADD`、`T_SUB` 块要优雅太多了！

```scl
// SCL 时间直接运算示范
#tTotalDelay := #tDelay1 + #tDelay2; // 两个时间直接相加
#tCycleHalf := DINT_TO_TIME(TIME_TO_DINT(#tDelay) / 2); // 时间除以 2 
```

## 2. 时间控制三剑客：TON、TOF、TP

工业现场的绝大多数时序逻辑，都是由这三种标准的 IEC 定时器构建而成的。我们先用最直观的**波形通断图**来看清它们的物理动作差别：

```
                    西门子定时器三剑客波形对比图
                    
 1. TON (接通延时):
    输入 (IN):  ───┌────────────────┐───
    输出 (Q):   ───────[ 延时t ]┌───┐─── (IN 变为 TRUE 后，延时 t 纳秒，Q 才变为 TRUE)
    
 2. TOF (关断延时):
    输入 (IN):  ───┌────────────────┐───
    输出 (Q):   ───┌────────────────────┐─── [ 延时t ] (IN 变为 FALSE 后，Q 继续坚守 t 纳秒才变为 FALSE)
    
 3. TP (脉冲触发):
    输入 (IN):  ───┌──┐────────────────────
    输出 (Q):   ───┌────────────┐────────── (不管 IN 闪烁多久，只要触发一下，Q 强行输出一个宽度完美等于 t 的脉冲)
```

### 2.1 定时器在博途底层的“数据骨架”
在博途优化内存中，当你实例化一个定时器（如 `tonDelay : TON_TIME;`）时，系统在后台实际上创建了一个含有 **4 个核心成员的结构体**：

*   `tonDelay.IN` (Bool)：启动计时输入引脚。
*   `tonDelay.PT` (Time)：设定时间输入引脚。
*   `tonDelay.Q` (Bool)：定时器状态输出引脚。
*   `tonDelay.ET` (Time)：当前已经流逝的时间（当前值）。

## 3. SCL 中定时器调用的黄金准则与致命假死漏洞

在 SCL 中调用定时器，语法形式非常像高级语言中的“函数对象调用”：

```scl
#tonDelay(IN := #bStart, PT := T#3s); // 驱动定时器运行
#bOut := #tonDelay.Q;                 // 获取输出状态
```

### 3.1 ⚠️ 致命红线：IF 条件嵌套内的“定时器假死失忆”

这是刚接触 SCL 的徒弟 **100% 会犯的低级错误**。
如果在现场，你为了图省事，把定时器写在了 `IF` 分支内部：

```scl
// ❌ 导致定时器彻底瘫痪的错误写法！
IF #bAutoMode THEN
    #tonDelay(IN := #bSensor, PT := T#5s); // 如果切出了自动模式，定时器将直接被 CPU 挂起“冷冻”！
END_IF;
```

*物理机理故障解析*：
定时器（TON）在 PLC 底层的工作原理是：**每个周期不断去比对当前 CPU 硬件系统时钟值，并寻找 IN 接口从 FALSE 变为 TRUE 的“上升沿沿变化”来触发计时**。

如果你把整个定时器塞进了 `IF #bAutoMode` 内部。一旦外部切出了自动模式（`bAutoMode` 变为 `FALSE`）：
1.  CPU 扫描时，会直接跳过整个 `IF` 分支。
2.  `#tonDelay` 物理实体在内存中**彻底停止了运行（被冻结在最后一刻）**。
3.  当外部 `#bSensor` 恢复，且重新切回自动模式时，定时器由于在中间缺失了刷新周期，无法捕捉到 `IN` 引脚的上升沿沿状态，从而导致**定时器彻底假死，计时永远不累计，输出永远不响应。**

**黄金铁律：在 SCL 编程中，所有的定时器实体（`TON`、`TOF`、`TP`），必须无条件摆放在最外层（无任何 IF 嵌套的裸露区域），确保每一个 CPU 扫描周期都能无条件刷新到定时器内部的时钟数据寄存器！**

## 4. 工业级综合案例：液压泵站智能起停延时、防抖与声光报警系统

现在，我们把本章讲的 `TIME` 算术计算、多重背景定时器（`TON`、`TOF`、`TP`）以及上两章学过的设备建模 UDT 彻底熔炼，解决一个重工业现场的液压主泵调度程序。

### 4.1 工业现场工艺要求
某轧钢厂配有一套大功率高压液压泵站。主泵控制系统需要设计一个控制功能块 **`FB_HydraulicPump`**：
1.  **开机警铃预报警（TP 脉冲应用）**：
    由于液压泵压力高达 31.5MPa，起动前必须强制拉响现场的电铃 **5 秒钟（预报警）**。
    *   **控制逻辑**：当操作员按下起动按钮后，电铃立刻拉响，且**不管操作员后续是否释放按钮**，电铃必须在物理上响完 5 秒，一毫秒都不能多，也不能少。响完后，主泵接触器自动合闸起动。
2.  **油压过低防抖延时（TON 接通延时应用）**：
    管道中由于液压阀切换会产生瞬间“压力下瞬冲击（气蚀波动）”。压力传感器反馈“低油压（`bFbk_PressureLow`）”时：
    *   **控制逻辑**：不能立刻报警停机，必须**持续处于低压达 2 秒以上**，才判定为“真实缺油故障”，自锁停机并报警。排除由于管道瞬间空气泡引起的瞬时误报。
3.  **风扇余热排出延时（TOF 关断延时应用）**：
    主泵电机（110KW）发热巨大，配有强制风冷冷却风扇。
    *   **控制逻辑**：当主泵运行反馈消失（停机）后，冷却风扇绝对不能立刻关停。风扇必须**继续维持强制运转 30 秒（余热排除延时）**，把电机内部和阀台附近的余热完全排空后，才允许自动关停。

### 4.2 步骤一：块接口声明（FB_HydraulicPump）

我们在静态变量区（Static）直接声明三剑客的多重背景，零背景 DB 碎片。

```
VAR_INPUT
    bAutoMode : Bool := FALSE;      // 自动模式 (TRUE=自动, FALSE=手动)
    bStart_Trigger : Bool;          // 综合启动请求脉冲 (来自自动逻辑或手动按钮)
    bStop_Trigger : Bool;           // 综合停止请求脉冲
    bFbk_PumpRunning : Bool;        // 主泵接触器物理吸合反馈信号 (DI)
    bFbk_PressureLow : Bool;        // 管道压力低行程开关信号 (DI, 正常时为FALSE, 低压时为TRUE)
    bFbk_Overload : Bool;           // 热继电器过载保护信号 (DI)
    bReset : Bool;                  // 故障一键复位
    tPreAlarmBell : Time := T#5s;   // 起动前电铃响铃时间设定
    tPressureDebounce : Time := T#2s; // 低压力防抖时间设定
    tFanDelayStop : Time := T#30s;  // 冷却风扇延时停机时间设定
END_VAR

VAR_OUTPUT
    bOut_PumpContactor : Bool;      // PLC 驱动主泵合闸接触器输出 (DO)
    bOut_AlarmBell : Bool;          // PLC 驱动现场报警电铃 (DO)
    bOut_CoolingFan : Bool;         // PLC 驱动冷却风扇接触器 (DO)
    bAlarm_Fault : Bool;            // 主泵汇总报警锁死线圈
    iErrorCode : Int;               // 错误代码 (0:正常, 1:过载, 2:真实缺油故障)
END_VAR

VAR
    // ==========================================
    // 静态变量区 (Private STAT) - 多重背景定时器
    // ==========================================
    bRunLatch : Bool;               // 起停控制自锁标志
    bFaultActive : Bool;            // 故障汇总锁存标志
    
    // 核心：时间三剑客的多重背景实例化 (直接嵌入主 FB背景中)
    tpPreAlarm : TP_TIME;           // 预警铃脉冲定时器
    tonPressureCheck : TON_TIME;    // 低压力接通延时防抖定时器
    tofFanDelay : TOF_TIME;         // 风扇余热排出延时关断定时器
END_VAR
```

### 4.3 步骤二：SCL 核心代码实现

徒弟，请特别注意：**我们在这段代码里，将所有的定时器实体调用全部无条件放到了 SCL 的最外侧，确保每一周期都能完美扫过它们的数据寄存器，杜绝假死。**

```scl
FUNCTION_BLOCK "FB_HydraulicPump"
{ S7_Optimized_Access := 'TRUE' }
VERSION : 0.1
BEGIN
	// ==========================================
	// 1. 无条件过载安全保护（最高优先级）
	// ==========================================
	IF #bFbk_Overload THEN
	    #bRunLatch := FALSE;
	    #bFaultActive := TRUE;
	    #iErrorCode := 1; // 错误码 1: 电机热过载
	END_IF;
	
	// ==========================================
	// 2. 主泵起停自锁控制（行为决定）
	// ==========================================
	IF NOT #bFaultActive THEN
	    // 正常的起保停锁定
	    IF #bStart_Trigger THEN
	        #bRunLatch := TRUE;
	    ELSIF #bStop_Trigger THEN
	        #bRunLatch := FALSE;
	    END_IF;
	ELSE
	    #bRunLatch := FALSE;
	END_IF;
	
	// ==========================================
	// 3. 【三剑客之一：TP 脉冲应用】启动警铃 5s 逻辑
	// ==========================================
	// 无论操作员手多抖，只要 #bRunLatch 变为 TRUE 的那一瞬间，
	// tpPreAlarm 的输入端 IN 捕捉到上升沿，Q 端立刻强力输出 5s 的高电平
	#tpPreAlarm(IN := #bRunLatch AND (NOT #bFbk_PumpRunning),
	            PT := #tPreAlarmBell);
	
	// 电铃输出驱动：当脉冲定时器正在计时（Q 为 TRUE），且电机尚未运行时，电铃响
	#bOut_AlarmBell := #tpPreAlarm.Q AND (NOT #bFbk_PumpRunning);
	
	// 主泵真实物理合闸接触器：必须在警铃 5 秒响完后（tpPreAlarm.Q 变为 FALSE 的瞬间），才允许合闸起动
	#bOut_PumpContactor := #bRunLatch AND (NOT #tpPreAlarm.Q) AND (NOT #bFaultActive);
	
	// ==========================================
	// 4. 【三剑客之二：TON 接通延时应用】油压低 2s 防抖保护逻辑
	// ==========================================
	// 只有在泵已经在运行状态下，低油压检测才有物理意义
	// 启动接通延时：当低压信号持续存在，且泵运行反馈在线时，开始累积 2s 倒计时
	#tonPressureCheck(IN := #bFbk_PressureLow AND #bFbk_PumpRunning,
	                  PT := #tPressureDebounce);
	                  
	IF #tonPressureCheck.Q THEN
	    // 2s 延时到，说明低压不是管道瞬时波动，而是真实缺油！触发保护关泵！
	    #bRunLatch := FALSE;
	    #bFaultActive := TRUE;
	    #iErrorCode := 2; // 错误码 2: 液压站真实缺油故障
	END_IF;
	
	// ==========================================
	// 5. 【三剑客之三：TOF 关断延时应用】冷却风扇 30s 延时关机
	// ==========================================
	// 当主泵在运行反馈在线时，风扇必须跟着运行（IN 为 TRUE，此时 Q 立刻变为 TRUE）。
	// 当主泵停止（IN 变为 FALSE），风扇的 Q 端继续坚守 30 秒。
	#tofFanDelay(IN := #bFbk_PumpRunning,
	             PT := #tFanDelayStop);
	             
	// 冷却风扇物理驱动线圈直接与 TOF 定时器的输出端 Q 挂钩
	#bOut_CoolingFan := #tofFanDelay.Q;
	
	// ==========================================
	// 6. 故障复位一键复位
	// ==========================================
	IF #bReset THEN
	    IF NOT #bFbk_Overload THEN
	        #bFaultActive := FALSE;
	        #iErrorCode := 0;
	        // 复位所有内部定时器
	        #tpPreAlarm(IN := FALSE, PT := #tPreAlarmBell);
	        #tonPressureCheck(IN := FALSE, PT := #tPressureDebounce);
	    END_IF;
	END_IF;
	
	#bAlarm_Fault := #bFaultActive;
	
END_FUNCTION_BLOCK
```

## 6. 常见错误与避坑指南 (Senior Tips)

### 6.1 错误一：对定时器输入端（IN）的“电平保持”误区导致无法二次起动

西门子 `TON` 定时器要开始下一次计时，它的 `IN` 引脚必须**经历一个完整的、从 FALSE 变为 TRUE 的边沿变化**。

如果你写了这段代码：
```scl
// ❌ 导致定时器卡死的 Bug 代码
#tonDelay(IN := TRUE, PT := T#5s); // 你以为只要一直保持 TRUE，时间到了它能重新计时？
```
当 5s 时间到，`tonDelay.Q` 变为 `TRUE`。由于 `IN` 一直长亮为 `TRUE`，它绝对不会自发复位。下一次你想重新计时，你必须**先给 `IN := FALSE` 刷新一个周期**，然后再给 `IN := TRUE`，计时才会重新触发。

---

### 6.2 错误二：直接在 HMI（触摸屏）上显示 TIME 变量产生的“秒分转换”混乱
如果你直接把 `tonPressureCheck.ET`（当前计时值，格式为 TIME）拉到触摸屏上。因为 `TIME` 的底层物理单位是**毫秒**，屏幕上会显示一个巨大的数字 `"3000"`（代表 3 秒）。操作员会以为是 3000 秒，造成大范围误解。
**正确的 HMI 标定处理**：
```scl
//  高雅的 HMI 标定转换
// 将毫秒 DINT 值除以 1000.0，优雅折算为以“秒”为单位的 REAL 变量输出给 HMI
#rHmiSecondsDisplay := DINT_TO_REAL(TIME_TO_DINT(#tonPressureCheck.ET)) / 1000.0;
```

### 练习 1：双轴伺服防扭自适应延时同步监测 FB (TON + TOF 练习)
在上一章的练习中，我们写过防龙门架扭偏的逻辑。现在，我们进行时间升级：
当两个伺服轴的位置差值绝对值 $> 2.0\text{mm}$ 时：
*   **输入参数**：
    *   `tFilterTime` : Time := T#500ms (不一致防抖过滤时间，防止瞬间噪点干扰)。
    *   `tRecoveryDelay` : Time := T#2s (恢复正常延时判断时间)。
*   **控制逻辑**：
    1. 偏差 $> 2.0\text{mm}$ 必须**持续存在 500 毫秒**以上，才判定为“真实扭偏”，发出报警。
    2. 当故障排除、偏差恢复到 $< 2.0\text{mm}$ 时，不能立刻清除故障，必须**持续保持正常达 2 秒**以上，才允许解除安全刹车信号。
