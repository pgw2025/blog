## 1. 在线监控的视觉艺术与潜规则（Online Monitor）

在博途工具栏中，点击那个经典的 **“眼镜”图标（启用/禁用监视 - Monitor on/off）**，你就开启了上帝视角。

### 1.1 SCL 在线监视的视觉呈现
在 SCL 编辑器中，在线监视以极度直观的形式呈现：
*   **布尔量（Bool）**：如果是 `TRUE`，整行代码左侧的实线会亮起绿色；如果是 `FALSE`，则显示为灰色虚线。
*   **数值变量（Int / Real）**：在代码的最右侧，会自动出现一列实时的虚线框，显示当前扫描周期内该变量的物理值。

```scl
// 博途在线监控下的视觉效果
IF #stMotor.Ctrl.bAutoMode THEN       | TRUE   <--- 绿色粗实线亮起，代表满足
    #stMotor.Sts.bOutStart := TRUE;   |        <--- 这一行也变为绿色
    #stCurrent := #stMotor.Sts.rCurr;  | 12.5 A <--- 右侧实时显示电流值 12.5
END_IF;                               |
```

---

### 1.2 ⚠️ 致命的视觉盲区：通信采样率与 CPU 刷新率的物理差

徒弟，你必须时刻保持冷静：**你在博途眼镜里看到的，绝对不是 PLC 内部最真实的瞬间快照！**

*物理底层真相*：
西门子 CPU 执行一次主循环（OB1）只需要 **1 ~ 3 毫秒（ms）**。
但是，由于以太网物理带宽限制和博途软件的刷新负荷，博途在线监视的通信刷新频率通常只有 **100 ~ 200 毫秒（ms）**。

这意味着：
如果一个报警位 `#bAlarm` 在 SCL 循环内由于噪点干扰，**亮起了 2ms 随后又熄灭**。
你的眼睛透过博途眼镜**绝对捕捉不到它**（它闪烁的那一刻被博途通信自动过滤掉了）。
所以，绝对不要因为在线监控里“变量没亮过”，就断定它在物理上没有触发过！

## 2. 局部断点调试：时间停止的黑客领域（Breakpoints）

如果你玩过 C# 或 C++，你一定对“断点（Breakpoint）”视若珍宝。西门子 S7-1500 硬件底层原生支持 SCL 的断点单步调试。

### 2.1 ⚠️ 铁律：严禁在生产运转的物理真机上设置断点！

听好了，这是自动化行业的安全红线：
当你在博途里对某一行 SCL 代码激活断点，且 PLC 扫描运行到该行时：
**整个 CPU 的主循环（OB1）用户程序会被瞬间“强行冻结”（挂起挂起）！**
此时：
1.  **PLC 停止了对所有输入/输出（I/O）的扫描刷新！**
2.  正在飞速旋转的变频器电机、正在喷火加热的喷枪，会保持断点前半个微秒的最后一刻输出状态（或者是由于失联引发通信故障）。
3.  这极易导致设备冲出轨道、加热盘过热起火等重大的现场设备损毁和人员伤亡事故！

**断点调试的唯一合法作案现场：办公室 PLCSIM / PLCSIM Advanced 仿真器，或者在现场空载调试、且安全拔掉所有外设物理插头的前提下进行。**

---

### 2.2 动作套路：Step-by-Step 调试

在 PLCSIM 运行状态下，在线连接 SCL 块，左键单击行号左侧的灰色边缘，会出现一个小红点（断点）。

```
 SCL 断点调试动作控制面板
 ┌──────────────┬──────────────────────────────────────────────┐
 │    按钮名    │                 物理调试动作                 │
 ├──────────────┼──────────────────────────────────────────────┘
 │ 单步跳过     │ 只运行当前这一行代码，随后立即冻结在下一行。
 │ (Step Over)  │
 ├──────────────├──────────────────────────────────────────────┐
 │ 单步进入     │ 如果当前行调用了一个子 FC，点击后，CPU 会    │
 │ (Step Into)  │ 强行拨开迷雾，带你进入该子 FC 的第一行代码。  │
 ├──────────────├──────────────────────────────────────────────┤
 │ 单步跳出     │ 从子 FC 中功成身退，返回主调 FBs 块。         │
 │ (Step Out)   │                                              │
 └──────────────┴──────────────────────────────────────────────┘
```

利用断点单步，你可以像放大镜一样，看清你的 `FOR` 循环里的计数器 `#i` 每次递增时，数组内部数据发生的每一步微妙变化。

## 3. 高速 Trace 跟踪示波器：降维打击的故障捕捉神器

既然博途在线监控有 100ms 的延迟，无法抓取高频高瞬时的瞬时信号，我们该如何抓取现场那些闪烁 2ms 的断线故障？

西门子为我们配备了降维打击级的工具——**Trace 跟踪示波器**。

```
                       博途硬件 Trace 示波器运作原理
                       
 [ 物理传感器 2ms 闪烁 ] ───> [ PLC 高速 SRAM 缓冲区 ] <─── CPU 每个扫描周期无条件记录
                                       │ 记录完毕
                                       ▼ 一键打包上传
                               ┌──────────────┐
                               │  博途 HMI    │ ──> 绘制出精细到 1ms 的高精度物理波形图
                               └──────────────┘
```

*工作机理*：
`Trace` 不是博途软件在监视，而是 **PLC 操作系统在硬件层级执行的嵌入式示波器**。
当你配置好 Trace 变量（如电压、编码器物理坐标）并启动后：
1.  CPU 的操作系统会在每个周期（如 1ms 周期）结束时，**无条件把这些变量的实时值写入到 CPU 芯片内置的高速 RAM 缓冲区中**。
2.  当达到你设定的**触发条件**（如：报警位 `#bAlarm` 发生上升沿的那一刻）时，WDT 硬件会自动记录前后 200 个周期的所有波形。
3.  最后，将缓冲区的数据整体一键上传给博途，在屏幕上绘制出精细到 **1 毫秒**、绝对无失真的高精度物理波形图。

在现场排查“伺服位置超差抖动”、“传感器高频断线”时，**Trace 是你唯一的、也是最强大的武器。**

## 4. 变量监控表（Watch Table）与 强制表（Force Table）的魔力

在现场，我们常常需要人为干预一些信号的走向。博途提供了两张表格。

### 4.1 监控表（Watch Table）
*   **作用**：在线输入想要观察的变量、数组元素的符号，实时观察。
*   **干预手段**：可以通过“修改值（Modify Value）”功能，在当前扫描周期结束时，强行把某个变量改写为你想要的数值（例如，强行把水位改为 `50.0m` 以测试后续逻辑）。

---

### 4.2 强制表（Force Table）：物理层级的越权强置（Force）

许多徒弟分不清“修改值”和“强制值”的区别。

*   **修改值**：只在当前这一刻写一次。随后，PLC 程序如果继续执行，程序逻辑（如 `#Out := FALSE;`）会**立刻覆盖改写**你刚刚修改过的值。
*   **强制值**：**无视程序逻辑的最高优先级越权。**
    一旦你对某个物理输入点（例如 `%I0.0`）强制了 `TRUE`：
    **无论该输入点实际物理接线是通还是断，也无论 PLC 内部程序如何改写它，在整个 CPU 运行期，该点的状态被硬件芯片死死锁存为 `TRUE`！**

*高级警告*：强制功能会直接屏蔽所有的软件安全互锁判定，极易由于误操作导致设备直接失控暴走。
**在工业现场，点击“启动强制”之前，你的右手必须放在物理紧急停止（急停）按钮上！**

## 5. 实战排故：诊断一台卡死在第 30 步的重载桁架机械手状态机

这是师父在某汽车总装车间（ noisy, 充斥着火花与喧嚣）遇到过的一个极其经典的真实故障案例。

### 5.1 现场灾难现场描述
现场一台三轴伺服直角坐标重载桁架机械手（XYZ轴，使用第二十章学过的状态机 FB 控制），在运行了 3 天后，突然毫无征兆地卡死在了半空中。
*   **触摸屏（HMI）状态显示**：当前步骤码为 **`30`（ST_MAN_Z_UP1：Z轴携带工件全速上行中）**。
*   **现场物理表现**：Z 轴伺服电机确实没有旋转，机械臂悬在半空，气动爪死死咬合着重达 50kg 的电芯。
*   **报警提示**：无报警，系统没有抛出任何 Fault，也没有看门狗停机。主 PLC 的 OB1 依然平稳运行，但机械手就是卡在第 30 步不肯动。

生产经理在一旁疯狂催促：“小张，这条线一分钟产值 5 万块，已经停线 10 分钟了，到底是什么问题？什么时候能搞好？”

---

### 5.2 侦探式排故第一步：调阅诊断缓冲区（Diagnostic Buffer）

作为高级工程师，现场排故的第一准则永远是：**先看诊断缓冲区！**

1.  在线连接 CPU，右键点击“在线和诊断”。
2.  调阅“诊断缓冲区”，从上往下依次看最近的几条 CPU 系统硬件记录。
3.  *结果*：缓冲区内一片安静，没有任何“编程出错（OB121）”、“I/O 访问错误（OB122）”或硬件掉线记录。
4.  *结论*：**CPU 硬件完好，这 100% 是一次由于软件逻辑条件未满足导致的“死锁卡步”故障。**

---

### 5.3 侦探式排故第二步：深入 SCL 块执行在线监控

1.  在博途中打开核心控制块 `FB_GantryManipulator`。
2.  点击“眼镜”图标启动在线监视。
3.  利用鼠标中轮向下滚动，直接定位到状态机第 `30` 步的代码区：

```scl
    // 博途眼镜下的真实故障监控画面
    #ST_MAN_Z_UP1:
        #bZ_Move_Up := TRUE;                             | TRUE
        
        // 跃迁退出条件 (故障核心区)
        IF #bZ_Limit_Up THEN                             | FALSE <── 闪烁着灰色虚线，代表不满足！
            #bZ_Move_Up := FALSE;                        |
            #iState := #ST_MAN_X_WORK;                   |
        END_IF;
```

4.  *诊断发现*：
    Z 轴上升驱动线圈 `#bZ_Move_Up` 确实是 `TRUE`。但是，跳转到下一步的退出条件 `#bZ_Limit_Up`（Z 轴物理上限位限位开关）为 **`FALSE`**！
    因为这个开关没有闭合，所以步骤计数器 `#iState` 被死死卡在 `30`，无法跳转到步骤 `40`。

---

### 5.4 侦探式排故第三步：利用监控表进行“闭环物理核对”

为什么 Z 轴已经升到了最高处，而物理上限位 `#bZ_Limit_Up` 却依然为 `FALSE`？
1.  我们新建一张监控表，把以下变量拉入表中进行实时监控：
    *   `%I2.4`（Z轴物理上限位常开接近开关输入点）
    *   `"Axis_Z".ActualPosition`（Z轴工艺对象实时物理编码器坐标值）
    *   `"Axis_Z".StatusWord.X5`（Z轴坐标系是否建立/已回零标志）

```
                     在线变量监控表实时快照
  变量名称               修改值       实时监视值
  ┌────────────────────┬──────────┬───────────┐
  │ %I2.4 (物理接近开关)│          │  TRUE     │ <── 震惊！接近开关在物理上已经闭合亮灯了！
  ├────────────────────┼──────────┼───────────┤
  │ Axis_Z.ActualPos   │          │  198.5 mm │ <── 编码器显示高度为 198.5mm
  ├────────────────────┼──────────┼───────────┤
  │ Axis_Z.StatusWordX5│          │  FALSE    │ <── 惊天雷！X5 回零标志居然是 FALSE ！！！
  └────────────────────┴──────────┴───────────┘
```

2.  *逻辑疑点*：
    既然物理接近开关 `%I2.4` 已经为 `TRUE`，为什么在 SCL 的 `#bZ_Limit_Up` 判定中依然是 `FALSE`？
    我们急忙往上追溯代码，看看 `#bZ_Limit_Up` 在 SCL 内部是怎么赋值的：
    ```scl
    #bZ_Limit_Up := %I2.4 AND "Axis_Z".StatusWord.X5; // 只有在接近开关触发，且 Z 轴坐标系健全（已回零）的前提下，才认为上限位有效
    ```
    
    *真相大白*：
    **由于 Z 轴工艺对象的“回零坐标系建立标志（`X5`）”莫名其妙丢失了（变成了 `FALSE`），导致程序判定上限位信号无效！**

---

### 5.5 侦探式排故第四步：用 Trace 高速示波器还原“丢失的 2 毫秒”

为什么运行得好好的机械手，会在半路上突然把 `X5`（回零标志）丢掉？
为了抓住这个转瞬即逝的偶发干扰：
1.  我们在博途中，在 Z 轴工艺对象下新建一个 **Trace（跟踪）**。
2.  配置需要高频抓拍记录的变量：
    *   `"Axis_Z".StatusWord.X5`（回零标志）
    *   `"Axis_Z".ActualVelocity`（实际运行速度）
    *   `"Axis_Z".Error`（轴报错）
    *   `"Axis_Z".ErrorDetail`（详细故障代码）
3.  **设置触发条件（Trigger）**：当 `"Axis_Z".StatusWord.X5` 发生下行沿（即从 `TRUE` 变为 `FALSE`）的那一瞬间，触发 Trace 抓拍。
4.  启动 Trace，让设备再次空载运行。
5.  在第 5 次循环时，设备再次卡死，Trace 波形一键上传。我们看到了以下这幅震憾的物理波形：

```
                    高频 Trace 示波器抓拍波形图
                    
 1. ActualVelocity: ───────┌─────────┐────────────────── (速度在高速上升中)
                           │         │ 
 2. StatusWord.X5:  ───────┴─────────┐                  (在速度达到最高点时，X5 瞬间掉为 FALSE)
                                     │
 3. ErrorDetail:    ─────────────────┴─── [ 16#0008 ]   (同时，ErrorDetail 瞬间暴出 16#0008 通信丢失)
                                      
 4. 2#00001000 干扰区：                 [ 干扰发生2ms ]
```

6.  *真凶锁定*：
    Trace 波形表明，在 Z 轴电机全力加速到最高转速时（此时电机驱动电流最大，周围电磁辐射极强），由于动力电缆与编码器电缆靠得太近，且**屏蔽层没有双端接地**。
    大电流高频高次谐波干扰侵入了编码器物理通道，导致 **反馈脉冲丢失持续了 2 毫秒**。
    西门子工艺对象（TO）极其敏感，一旦检测到 2ms 的通信硬件闪烁，为了安全起见，**自动将轴的坐标系拉失效（X5 变为 FALSE），但并不触发 CPU 报警停机。** 这直接导致了状态机被卡死在第 30 步。

---

### 5.6 救灾解决方案与 SCL 防御性补丁

1.  **硬件层面**：
    要求现场施工人员立刻将 Z 轴伺服电机的编码器屏蔽层，在控制柜侧和电机侧进行标准的 **360度金属夹环接地（EMC 屏蔽优化）**。实现物理隔离。
2.  **软件（SCL）防御性补丁**：
    为了防止未来因类似干扰产生“无声的卡死”，我们在状态机第 30 步增加了**防丢零防御判定**：

```scl
    //  安全的 SCL 状态机防御补丁
    #ST_MAN_Z_UP1:
        #bZ_Move_Up := TRUE;
        
        // 防御检测：如果轴在运动中，坐标系意外丢失
        IF NOT #Axis_Z.StatusWord.X5 THEN
            #bZ_Move_Up := FALSE;
            #bFaultActive := TRUE;
            #iErrorCode := 109; // 抛出明确 HMI 报警：伺服上升中丢失坐标原点
            #iState := #ST_FAULT; // 强行切入故障，安全刹车
            RETURN;
        END_IF;
        
        IF #bZ_Limit_Up THEN
            #bZ_Move_Up := FALSE;
            #iState := #ST_MAN_X_WORK;
        END_IF;
```

通过这套软硬件双重闭环，不仅彻底根治了偶发性的电磁干扰，还保证了即使在极端工况下发生原点丢失，设备也能在一毫秒内安全刹车并抛出精准报警，再也没有发生过“无声的卡死”事故。整条生产线的运行效率得到了最严密的保障。

## 6. 高级工程师的现场调试守则（Senior Rules）

作为一名手握大权、在现场主持调试的高级工程师，你必须时刻牢记以下调试规范：

1.  **备份第一**：
    在修改 PLC 代码前，**必须无条件对当前的 PLC 变量和块进行一次完整的“离线另存备份（Backup）”**。永远给自己留一条退路。
2.  **不瞎猜，不盲动**：
    遇到故障，绝对不要在没找到真因的前提下就去瞎改代码。必须首先通过诊断缓冲区、监控表、Trace 提取出确定无疑的物理数据支持，定位到真因后再动刀。
3.  **高空与移动防护**：
    调试运动控制和阀门时，**必须配备一名助手，手死死按在现场的物理急停按钮上**。一旦发生轨迹偏差，两秒内切断物理电源，这是自动化工程师保障生命安全的底线。
