# Git 终极通关指南：从底层原理到高阶骚操作

如果你只会 `git add` 和 `git commit`，那你就相当于只学会了开车踩油门，一旦遇到复杂的路况（冲突、回滚、历史污染），你就会束手无策。

这篇博客将带你从**底层模型**出发，深度拆解每一个命令的高阶参数，让你彻底掌握 Git 这个强大的工具。

---

## 0. 灵魂地基：Git 的“三层楼”模型

在学命令之前，你脑子里必须有这三层楼：
1.  **一楼：工作区 (Working Directory)** —— 你正在改的代码。
2.  **二楼：暂存区 (Staging Area/Index)** —— 你准备提交的代码（快递打包区）。
3.  **三楼：本地仓库 (Local Repository)** —— 已经存入数据库的版本历史。
4.  **云端：远程仓库 (Remote Repository)** —— 托管在 GitHub/GitLab 的代码。

---

## 1. 基础配置：工欲善其事

### `git config` —— 你的身份证
*   `git config --global user.name "Your Name"`
*   `git config --global user.email "mail@example.com"`
*   **高阶：**
    *   `--list`: 查看所有配置。
    *   `--local`: 只给当前项目设置（比如公司项目用公司邮箱）。
    *   `alias.<name> "<cmd>"`: 设置别名。例：`git config --global alias.st status`，以后输 `git st` 即可。

---

## 2. 精细化操作：add 与 commit

### `git add` —— 精准打包
*   `-u`: 只加被修改和被删除的文件（不加新文件）。
*   `-A`: 所有改动（包括新文件、修改、删除）。
*   **`-p` (Patch mode)**: **神级参数**。Git 会把改动切成一个个小块询问你。你可以只把某一行代码加入暂存区，而留下另一行。

### `git commit` —— 更有质量的记录
*   `-m "..."`: 提交消息。
*   `--amend`: **追加修改**。如果你刚 commit 完发现少改了一个标点，改完后执行这个，它会把改动合并到上一个 commit，而不会产生难看的“fix bug”提交记录。
*   `--allow-empty`: 允许提交一个没有任何改动的 commit（常用于触发 CI/CD 流程）。

---

## 3. 穿越平行宇宙：分支与合并的高级艺术

### `git merge` —— 殊途同归
*   `--ff` (Fast-forward): 默认。如果可能，直接把指针移过去，不产生新提交。
*   `--no-ff`: **推荐**。强行产生一个合并提交，清晰记录“这里曾经合并过”。
*   `--squash`: 把 A 分支的所有 commit 压缩成一个，再合并过来。适合把杂乱的开发分支合并到干净的主分支。

### `git rebase` —— 时光平滑术
`merge` 是合并，`rebase` 是“变基”。它把你的改动拆下来，接在目标分支的最末尾。
*   **`-i` (Interactive)**: **最强高阶命令**。
    *   你可以合并 (squash) 过去的提交。
    *   你可以修改过去的提交消息。
    *   你可以删除中间的某个提交。
*   *警告：永远不要对已经推送到远程的分支执行 rebase！*

### `git cherry-pick` —— 精准摘取
你想把 B 分支里的**某一个** commit 拿过来，而不是整个合并？
*   `git cherry-pick <commit-id>`: 把那个特定的“零件”装到当前分支。

---

## 4. 远征与安全：Remote 操作

### `git push` —— 安全发货
*   `-f`: 强制推送（危险！）。
*   **`--force-with-lease`**: **安全强制推送**。如果远程有别人的新代码，它会拒绝执行，避免你把同事的代码覆盖掉。
*   `--tags`: 把本地的所有标签（如 v1.0）一次性推送到服务器。

---

## 5. 后悔药的深度拆解：Reset 与 Revert

### `git reset` —— 毁尸灭迹 vs 退居二线
*   **`--soft <commit>`**: 仅移动 HEAD 指针。代码还在，且在暂存区。
*   **`--mixed`**: 默认。代码还在工作区，但暂存区空了。
*   **`--hard`**: **最狠一招**。代码、暂存区全部回到过去，之后的努力全部消失。

### `git checkout -- <file>`
*   把工作区某个文件恢复到最后一次提交的状态。

---

## 6. 救命稻草：当你以为一切都毁了

### `git reflog` —— Git 的“黑匣子”
即使你执行了 `git reset --hard` 误删了代码，即使你删除了分支，只要你在本地操作过，`reflog` 就有记录。
*   输入 `git reflog` 找到那个消失的哈希值。
*   `git reset --hard <hash>`: 奇迹发生，代码回来了。

### `git stash` —— 临时避难所
*   `git stash save "message"`: 给临时存放的东西起个名。
*   `git stash list`: 看看存了多少东西。
*   `git stash apply stash@{1}`: 恢复特定的那次。
*   `git stash clear`: 清空避难所。

---

## 7. 历史审查：谁干的？

### `git blame <file>` —— 找人对线
*   每一行代码显示作者、提交时间和哈希。

### `git log` —— 定制化报表
*   `-L 10,20:file.txt`: 只看某个文件第 10 到 20 行的变动历史。
*   `--author="Will"`: 只看某人的提交。
*   `--since="2 weeks ago"`: 只看最近两周的。

---

## 总结：Git 进阶心法

1.  **提交要“小而美”**：一次 commit 只干一件事。
2.  **消息要清晰**：别写 "update"，写 "fix: 修复登录页面的背景偏移问题"。
3.  **多用分支**：主分支永远是稳定的，开发必须在 Feature 分支。
4.  **学会 rebase -i**：在推送到远程之前，把自己的历史整理得漂漂亮亮。

**掌握了这些，你就不再只是代码的搬运工，而是代码历史的导演！**
