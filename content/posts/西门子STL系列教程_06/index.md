在梯形图（LAD）编程中，我们习惯于直接拉出触点、关联变量名。在这种直观的“物理符号化”操作背后，CPU 内部有一套极为严密、甚至带有鲜明“物理存储器分区”痕迹的**地址体系（Address System）**。

西门子 S7 处理器在物理架构上，是一台典型的**哈佛存储器结构（Harvard Architecture）**变体或高度分区的微处理器。它将内存划分为不同的物理区域，以满足高速逻辑执行、周期性 I/O 同步、局部变量临时缓存以及长周期数据配方保存等不同的工业控制需求。

在 STL 中，我们不仅会与全局的 `I`（输入）、`Q`（输出）、`M`（位存储）打交道，还会频繁操纵底层的 `DB`（全局数据块）、`L`（局部数据栈），甚至是极其神秘的 `V`（前序局部数据栈）以及 FB 背景下的 `DI` 寄存器。

本教程将带你拆解西门子 S7 CPU（兼容 S7-300、S7-400、S7-1500）的物理寻址体系，深入到存储器芯片的硬接线与编译器映射层面，解析这些存储器分区为什么会这样设计，以及它们在 STL 中是如何被读取的。

---

### 技术兼容性及声明

*   **S7-300/400**：采用基于物理微码地址总线的硬分区设计。本文所述的 `I`、`Q`、`M`、`DB`、`DI`、`L`、`V` 等物理区域在经典 CPU 内部有着对应的物理高速 SRAM 缓存区。
*   **S7-1500**：虽然内部芯片架构转为现代通用处理器，但其虚拟机在微秒级完整模拟了本文所述的所有地址体系行为。
*   **S7-1200**：**不支持 STL，亦不支持 V 存储区寻址**。但理解本教程中关于过程映像区（I/Q）与物理外设（PI/PQ）的读取差异、FB 中 `STAT`（静态变量）与 `TEMP`（临时变量）的物理位置差异，对于在 S7-1200 中编写零 Bug 的 SCL 代码、避免变量覆盖冲突具有极高的工程指导价值。

## 1. 全局存储区：I、Q、M

这是最基础的、与外部物理世界和全局状态直接关联的三个存储区域。

### 1.1 物理过程映像区（Process Image）与外设直接访问（Direct Peripheral Access）

很多初学者认为，当我们在 STL 中写下 `A I0.0` 时，CPU 会瞬间通过背板总线去读取那个物理输入模块上的电平。**这是一个极大的误解。**

#### 过程映像输入（PII）与外设直接输入（PI）的物理原理

为了保证在一个扫描周期内，所有的逻辑计算都基于同一组稳定的输入状态（防止在扫描中途由于物理触点抖动导致前后逻辑冲突），西门子设计了**过程映像区（Process Image）**：

```
+-----------------------------------------------------------------------------------+
| S7 CPU Cyclic I/O Synchronization Model                                           |
|                                                                                   |
|                                ┌──────────────────┐                               |
|                                │ 1. Physical Input│                               |
|                                └────────┬─────────┘                               |
|                                         │ (Scan Start: Fast DMA Transfer)         |
|                                         ▼                                         |
|                                ┌──────────────────┐                               |
|                                │ PII (I-Area RAM) │ <=== "A I0.0" reads here      |
|                                └────────┬─────────┘                               |
|                                         │                                         |
|                                         ▼                                         |
| ┌───────────────────────────────────────────────────────────────────────────────┐ |
| │ User Program execution (OB1)                                                  │ |
| │                                                                               │ |
| │  * "A  I0.0"   -> Reads from stable PII RAM (Fast, jitter-free)               │ |
| │  * "L  PIW128" -> Forces direct backplane bus read from module (Slow, raw)     │ |
| └───────────────────────────────────────────────────────────────────────────────┘ |
+-----------------------------------------------------------------------------------+
```

*   **I (Process Image Input - PII)**：
    在 OB1 扫描开始前，CPU 硬件通过直接内存存取（DMA）通道，极速将外部物理输入模块的当前状态一次性拷贝到 CPU 内部的 **I 存储区高速 RAM** 中。程序中所有的 `A I`、`L IB` 指令，全都是在读取这块内部 RAM，速度极快且无抖动风险。
*   **PI / PQ (Peripheral Input / Output - 外设直接访问)**：
    如果我们在高频中断（如 OB35 10ms 中断）中需要读取绝对实时的压力值或模拟量，不能等待 OB1 周期结束才更新的过程映像区。此时必须使用外设直接访问，指令在德语/经典环境中写作 `L PEW 128`（或在博途中写作 `L "Analog_In":P`）。
    **执行该指令时，CPU 的程序计数器（PC）会暂停，强行通过背板总线向物理模块发出读取指令，等待硬件电平稳定、A/D 转换完成、返回数据后，CPU 才会继续向下执行。** 这一过程的硬件开销（耗时）通常是直接读 I 区的高达数十倍。

*   **M (Bit Memory - 位存储区)**：
    M 区是 CPU 内部的一块全局辅助高速 RAM。在 S7-300 早期，M 区是非常珍贵的资源（如 CPU 313 仅有 256 字节的 M 空间）。在现代博途项目中，M 区主要用于临时全局标志，**西门子官方强烈建议在大型项目中用全局 DB 块代替 M 区，以获得更好的符号交叉引用保护和块封装性。**

### 1.2 知识点 1 代码：过程映像区（I）与直接外设（PIW）高速捕获对比

本例展示如何在 STL 中，分别使用过程映像区和外设直接访问，来捕获一个高速运转的压机在下降中途产生的微秒级压力波形峰值。

#### 完整 STL 代码

```stl
FUNCTION "Capture_Pressure" : VOID
TITLE = 高速压力数据实时捕获
VAR_INPUT
   PI_Addr : INT ;            // 物理外设模拟量通道基地址 (如 256)
END_INPUT
VAR_IN_OUT
   Max_Pressure : INT ;       // 记录到的历史最大压力值
END_VAR
VAR_TEMP
   t_Instant_Pressure : INT ; // 实时读取到的模拟量值
   t_AR1_Backup : DWORD ;     // 备份 AR1
END_VAR
BEGIN
NETWORK
TITLE = 1. 利用外设直接访问（PEW）强行读取硬件模块，绕过过程映像区延时
      TAR1  #t_AR1_Backup;    // 备份 AR1

      // 计算物理外设地址对应的指针
      L     #PI_Addr;         // 装载基地址 (如 256)
      SLD   3;                // 左移 3 位，转为位寻址指针格式
      LAR1  ;                 // 将指针载入 AR1

      // 强行通过外设寻址读入 (等效于 L PEW [AR1, P#0.0])
      // 注意：使用 P (Peripheral) 标识符强行进行背板总线即时读取
      L     PIW [AR1, P#0.0]; // 在博途或标准 STL 中，外设输入通过 PIW 访问
      T     #t_Inter_Press;   // 暂存至临时变量

NETWORK
TITLE = 2. 比较并更新峰值
      L     #t_Inter_Press;   // 装载刚刚强行读入的实时压力
      L     #Max_Pressure;    // 装载历史最大值
      >I    ;                 // 比较：如果当前实时值大于历史最大值
      JCN   Skip;             // 若不满足，跳转至 Skip 标号
      
      L     #t_Inter_Press;   // 满足条件，更新历史最大值
      T     #Max_Pressure;

Skip: LAR1  #t_AR1_Backup;    // 恢复 AR1 现场
END_FUNCTION
```

#### 逐行剖析与寄存器状态跟踪表

假设输入：`PI_Addr = 256`（代表外设模拟量通道 `PIW256`），当前物理通道上由于压机撞击瞬间，传感器实际电平已达十进制 `21500`（但此时 OB1 开始前拷贝的过程映像区 `IW256` 中还残存着上一周期的旧值 `5000`）。

| 步骤 | 指令 | ACCU 1 | ACCU 2 | AR1 指针值 | RLO | 状态字 /FC | 硬件与寄存器级微观细节解析 |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| **0** | (开始) | `0` | `0` | `0.0` | `0` | `0` | 系统开始，等待指令。 |
| **1** | `L #PI_Addr` | `256` | `0` | `0.0` | `0` | `0` | 将基地址 `256` 读入 ACCU 1。 |
| **2** | `SLD 3` | `2048` | `0` | `0.0` | `0` | `0` | **左移 3 位**。`256 * 8 = 2048`。转换为二进制指针格式（低 3 位为 bit 偏移，高位为 byte 偏移），即 `P#256.0`。 |
| **3** | `LAR1` | `2048` | `0` | `P#256.0` | `0` | `0` | 将计算出的指针写入硬件地址寄存器 AR1。 |
| **4** | `L PIW [AR1, P#0.0]`| `21500`| `2048` | `P#256.0` | `0` | `0` | **外设直接读取指令**。CPU 暂停程序，通过背板总线实时读取 `PIW256` 通道。获取最新硬件电平 `21500` 存入 ACCU 1。原指针 `2048` 移入 ACCU 2。 |
| **5** | `T #t_Inter_Press` | `21500`| `2048` | `P#256.0` | `0` | `0` | 数据传送，累加器无变化。 |
| **6** | `L #Max_Pressure` | `8000` | `21500`| `P#256.0` | `0` | `0` | 装载历史最大值 `8000`。实时值 `21500` 滑入 ACCU 2。 |
| **7** | `>I` | `8000` | `21500`| `P#256.0` | `1` | `1` | 比较指令：比较 ACCU 2（21500）是否大于 ACCU 1（8000）。条件成立，**RLO 被硬件置 1**。 |

### 1.3 工程经验：严禁在常规控制回路中滥用外设直接访问（PI/PQ）

虽然直接外设访问（`PIW`/`PQW`）能够提供没有过程映像延时的绝对实时数据，但在常规的逻辑控制（如电机联锁、普通阀门切换）中，**严禁图省事滥用它们**。

*   **技术隐患**：
    由于外设访问是物理级直接操作背板总线，其执行时间（几十微秒）远远大于过程映像区（小于一微秒）。如果你在程序中对几十个传感器通道都使用 `PIW`，会**导致 PLC 扫描周期（Scan Time）瞬间飙升**。
    更危险的是，如果在同一扫描周期中，多处使用 `PIW` 访问同一通道，由于外界物理信号是连续跳变的，**会产生“程序前十行读取是 1，后十行读取是 0”的极度混乱逻辑冲突**，给设备安全带去不可预知的隐患。
*   **黄金准则**：
    95% 的常规控制回路，必须使用标准的 `I` 和 `Q` 高速映像区。只有在 1% 的高速运动控制、精密工艺峰值捕获、或是在循环中断中，才允许针对特定的单个通道使用 `PIW` / `PQW` 访问。

## 2. 共享与背景数据块：DB、DI

在西门子存储架构中，数据块（Data Block）是保存长周期、结构化数据的“总仓库”。

为了在 CPU 芯片内部实现最高效率的读写，西门子硬件设计了**两个独立的数据块寄存器**：**DB 寄存器** 和 **DI 寄存器**。

### S7 CPU 双数据块寄存器硬件寻址架构

```
+-----------------------------------------------------------------------------------+
| S7 CPU Dual Data Block Register Architecture                                      |
|                                                                                   |
|  +---------------------------------------+  +----------------------------------+  |
|  | DB Register (Shared DB Pointer)       |  | DI Register (Instance DB Pointer)|  |
|  | Open via: "OPN DB"                    |  | Open via: "OPN DI" or "CALL FB"  |  |
|  | Active Area: [DBX / DBB / DBW / DBD]   |  | Active Area: [DIX / DIB / DIW /  |  |
|  |                                       |  |               DID]               |  |
|  +---------------------------------------+  +----------------------------------+  |
|                       ▲                                      ▲                    |
|                       │                                      │                    |
|  +────────────────────┴──────────────────────────────────────┴─────────────────+  |
|  | CPU Address Decoder (物理地址解码器)                                       |  |
|  |                                                                             |  |
|  |  * "L DBW 10" -> Decodes through DB Register (Loads from Global DB)          |  |
|  |  * "L DIW 20" -> Decodes through DI Register (Loads from Instance DB)        |  |
|  +────────────────────────────────────────────────----------------─────────────+  |
+-----------------------------------------------------------------------------------+
```

### 2.1 为什么 CPU 设计两个独立的 DB 寄存器？

在调用带背景数据块（Instance DB）的功能块（FB）时，CPU 在执行 FB 内部逻辑的同时，经常需要去读取全局数据块（Shared DB）里的工艺配方。

*   **如果只有一个 DB 寄存器**：
    CPU 在读取全局 DB 变量前，必须执行一次 `OPN DB 10`。读取完毕，要写回 FB 的背景数据时，又必须执行一次 `OPN DB 20`（打开背景 DB）。**这种频繁在两个数据块之间切换的操作，在底层总线上会产生巨大的读写时钟损耗。**
*   **双寄存器完美设计**：
    西门子设计了 **DB 寄存器（代表全局共享）** 与 **DI 寄存器（代表背景 DB，注意德语中 I 代表 Instanz）**。
    *   在 FB 运行时，其专属背景 DB 的号被自动锁死在 **DI 寄存器** 中，程序内部所有针对背景变量（`STAT`、`IN`、`OUT`）的访问，全部通过 `DIX`、`DIW` 寻址。
    *   此时，如果需要读写全局共享 DB，只需将共享 DB 号锁在 **DB 寄存器** 中，通过 `DBX`、`DBW` 寻址。
    *   **两个寄存器并行工作，互不干扰，实现了零等待的双通道数据交互。**

### 2.2 知识点 2 代码：在 STL 中实现双寄存器高效并行动作

以下 FC 函数展示如何在 STL 中，利用双寄存器并行优势，将一个背景数据块（DI）中的一批过程值，极速、无等待地拷贝到一个全局共享数据块（DB）中。

#### 完整 STL 代码

```stl
FUNCTION "Fast_Block_Copy" : VOID
TITLE = 双寄存器通道高效对拷
VAR_INPUT
   Shared_DB_No   : INT ;     // 全局目标共享 DB 编号 (锁入 DB 寄存器)
   Instance_DB_No : INT ;     // 背景源数据块编号 (锁入 DI 寄存器)
   Copy_Len_Bytes : INT ;     // 拷贝字节长度 (偶数，按字拷贝)
END_INPUT
VAR_TEMP
   t_AR1_Backup   : DWORD ;   // 现场保护
   t_AR2_Backup   : DWORD ;
   t_Loop_Count   : INT ;
END_VAR
BEGIN
NETWORK
TITLE = 1. 打开双寄存器并定位指针
      TAR1  #t_AR1_Backup;    // 备份 AR1 和 AR2
      TAR2  #t_AR2_Backup;

      // 双通道打开数据块
      OPN   DB [#Shared_DB_No];   // 将目标共享 DB 锁入 DB 寄存器
      OPN   DI [#Instance_DB_No]; // 将源背景 DB 锁入 DI 寄存器

      // 初始化指针
      LAR1  P#0.0;            // AR1 指向源 DB (DI) 的首字节 0.0
      LAR2  P#0.0;            // AR2 指向目标 DB (DB) 的首字节 0.0

      // 计算循环字（WORD）个数
      L     #Copy_Len_Bytes;
      SRW   1;                // 除以 2，转为字个数
      T     #t_Loop_Count;

NETWORK
TITLE = 2. 双寄存器免切换极速对拷循环
Loop: T     #t_Loop_Count;

      // 重点动作：利用双寄存器的并行指令，一行载入 DI，一行写入 DB
      L     DIW [AR1, P#0.0]; // 从 DI 寄存器锁定的背景 DB 中装载一个字 (高效率)
      T     DBW [AR2, P#0.0]; // 写入由 DB 寄存器锁定的全局 DB 中 (免切换)

      // 指针后移一个字 (2 字节)
      +AR1  P#2.0;
      +AR2  P#2.0;

      L     #t_Loop_Count;
      LOOP  Loop;             // 循环控制

      // 还原现场
      LAR1  #t_AR1_Backup;
      LAR2  #t_AR2_Backup;
END_FUNCTION
```

#### 逐行剖析与硬件解码细节

假设运行参数：`Shared_DB_No = 10`，`Instance_DB_No = 20`，`Copy_Len_Bytes = 4`。

*   **`OPN DB [#Shared_DB_No]`**：
    CPU 将数据块号 `10` 载入内部物理 **DB 寄存器**。
*   **`OPN DI [#Instance_DB_No]`**：
    CPU 将数据块号 `20` 载入内部物理 **DI 寄存器**。此时，两个高压物理通道在 RAM 中同时打开。
*   **`L DIW [AR1, P#0.0]`**：
    **指令解码器微码动作**：
    1. 识别操作数前缀为 `DIW`。
    2. 地址解码器直接调取 **DI 寄存器** 里的值（`20`）。
    3. 结合 AR1 指针，读取 `DB20.DBW0`。
*   **`T DBW [AR2, P#0.0]`**：
    **指令解码器微码动作**：
    1. 识别操作数前缀为 `DBW`。
    2. 地址解码器直接调取 **DB 寄存器** 里的值（`10`）。
    3. 结合 AR2 指针，将 ACCU 1 里的数据写入 `DB10.DBW0`。
    整个载入和传送过程**没有任何重新打开数据块的总线握手开销**，完全是底层的、无缝的寄存器级对调。

### 2.3 工程经验：防范因“DI 寄存器污染”导致的子 FB 局部变量崩溃故障

在编写使用地址寄存器（AR1/AR2）的高级 STL 代码时，有一个极其隐蔽且破坏性极强的技术大坑：**DI 寄存器污染**。

*   **故障场景**：
    在一个复杂的 FB 内部（该 FB 会使用背景变量，其背景 DB 被锁在 DI 寄存器中）。你编写了一段 STL 逻辑，其中为了读取外部某个配置 DB 块，你使用了一条 `OPN DI [#My_Config_DB]` 指令（由于粗心，误将 `DB` 写成了 `DI`）。
    
    一旦这条指令执行，**DI 寄存器中原本锁定的当前 FB 的背景 DB 号，瞬间被强行篡改为了外部的 `My_Config_DB`**。
*   **毁灭性的后果**：
    在这行代码之后，只要该 FB 试图访问任何其自身的局部变量（例如 `#Start_Button`，输入输出参数，静态变量），**CPU 在底层寻址时，仍会无脑依据 DI 寄存器去寻址，从而在错误的外部 `My_Config_DB` 中读写数据**。
    这不仅会破坏外部 DB 块的干净状态，还会导致当前 FB 内部的逻辑完全坍塌，并且因为访问越界，随时引发 `OB122` 导致 PLC 崩溃骤停。
*   **工程规避铁律**：
    在任何 FB（特别是多重背景 FB）中，**绝对禁止手动执行 `OPN DI` 指令**。所有的共享/配置 DB 块打开操作，必须 100% 局限在 `OPN DB` 寄存器通道上。确保 DI 寄存器的控制权牢牢掌握在 CPU 系统的块调度引擎手中。

## 3. 本地与前序局部变量区：L、V

`L`（Lokaldaten，局部数据）和 `V`（Vorgänger-Lokaldaten，前序局部数据）代表了 CPU 内部对子程序堆栈的高效调度。

### 3.1 本地局部数据（L）与前序局部数据（V）的物理关系

当 CPU 在 OB1 中调用 FC1，再在 FC1 中调用 FC2 时，局部变量堆栈（L-Stack）在内存中会发生以下变化：

```
+-----------------------------------------------------------------------------------+
| L-Stack Transition during Nested Block Calls                                      |
|                                                                                   |
|  +-----------------------------+                                                  |
|  | OB1 L-Stack Frame           |                                                  |
|  +-----------------------------+                                                  |
|  | FC1 L-Stack Frame           | <=== This is L-Area inside FC1                    |
|  +-----------------------------+                                                  |
|  | FC2 L-Stack Frame           | <=== This is L-Area inside FC2                    |
|  |                             |                                                  |
|  |  * Within FC2:              |                                                  |
|  |    - Accessing "L0.0"       -> Points to FC2's L-Stack Frame                   |
|  |    - Accessing "V0.0"       -> Points directly to FC1's L-Stack Frame!         |
|  +-----------------------------+                                                  |
+-----------------------------------------------------------------------------------+
```

*   **L (Local Data Area - 本地局部数据区)**：
    当前正在执行的块（FC 或 FB）的局部变量区（`VAR_TEMP`）。
    当程序在 FC2 内部访问 `L10.0` 时，读取的是 FC2 自身的临时变量。
*   **V (Previous Local Data Area - 前序局部数据区)**：
    **这是西门子经典寻址机制中最具极客（Geek）色彩的设计**。
    当处于 FC2 中时，通过 `V` 前缀（如 `V10.0`），可以直接跨越块边界，**直接读取或改写调用它的上级块（即 FC1）的临时变量 L-Stack 空间**！
    *   在德语中，`V` 代表 **Vorgänger（前序、先驱）**。
    *   在英文中，它常被直观地解释为 **Virtual / Previous Local**。

### 3.2 知识点 3 代码：利用 V 存储区实现跨子程序参数“隐式反向追踪”

本例设计一个深度嵌套的诊断小工具 FC20，该工具专门被下级子程序调用。它能直接突破子程序的边界，通过 `V` 区反向追踪并提取出上级父程序（如 OB1 或 父FC）内部定义的临时报警状态。

#### 完整 STL 代码

```stl
FUNCTION "Back_Trace_Diag" : VOID
TITLE = 诊断诊断工具：跨子程序反向追踪 L-Stack
VAR_OUTPUT
   Caller_Temp_Word : WORD ;  // 追踪到的上级块在某个位置的临时字状态
END_VAR
BEGIN
NETWORK
TITLE = 跨越边界读取前序 L-Stack
      // 假定我们的父程序（调用者）在其局部变量 TEMP 区的开头定义了一个 16位的临时字 (LW0)
      // 在本子程序内部，我们可以直接使用 V 区域前缀，无损、无延迟地读取该状态。
      
      L     VW 0;             // 直接装载调用者 (Previous Block) 的局部字 LW0
      T     #Caller_Temp_Word;// 将其输出，用于故障诊断或深度调用链分析
END_FUNCTION
```

#### 逐行剖析与堆栈物理定位机制

假设 OB1 作为调用者。OB1 内部在临时变量声明表的起始位置定义了一个 `Temp_Err_Word : WORD`（其在 OB1 堆栈中物理占用 `LW0` 地址）。

当 OB1 执行到某一网络，调用了 `FC "Back_Trace_Diag"`：

*   **`L VW 0` 执行瞬间**：
    1. CPU 内部的指令译码器识别出 `V` 寻址前缀。
    2. 堆栈控制器（Stack Controller）提取当前的堆栈指针（SP）。
    3. 通过减去当前子程序 `FC "Back_Trace_Diag"` 的 L-Stack 帧长度，计算出**上级父块（OB1）的 L-Stack 物理基地址**。
    4. 从该基地址的 `0` 偏移处，读取两个字节（`VW0`）。
    5. 将读取到的 OB1 临时报警状态装入 ACCU 1，并成功输出。

### 3.3 工程经验：在常规业务逻辑中严禁使用 V 区寻址

虽然 `V` 区寻址提供了极具艺术感的底线逆向穿透能力，但在日常的工业控制逻辑（如工艺联锁、普通的自动步骤控制）中，**严禁使用任何 `V` 寻址**。

*   **致命的设计隐患**：
    `V` 寻址会导致块与块之间产生**深度寄生、高耦合、极度不安全的依赖关系**。
    如果哪一天，另一个不知道底细的工程师修改了父程序（OB1），在局部变量表的最前方增加了一个布尔临时变量，导致原本的 `Temp_Err_Word` 物理位置从 `LW0` 偏移到了 `LW2`。
    由于 `V` 寻址是硬编码的物理绝对偏移（`VW0`），**子程序将直接读取到错误的、错位后的垃圾数据**。这会导致诊断逻辑完全失效，并且极难查出原因。
*   **黄金准则**：
    `V` 区寻址的生存空间仅限于：**高度封装的系统级诊断库函数、通用的 CPU 宕机现场数据黑匣子捕获工具**。常规业务逻辑块之间的数据传递，必须严格通过标准的 **Input / Output 接口传参**完成。

## 4. 块接口变量的物理本质：TEMP、STAT、IN、OUT、IN_OUT

在编写 FC 和 FB 的程序时，我们每天都在接口表中定义各种变量。但在 CPU 的物理内存上，它们的实现机制截然不同。

### 4.1 接口变量在物理存储介质上的本质区别

下表详细解密了不同接口变量在 PLC 底层硬件上的真实宿主位置：

| 接口变量类别 | 在 FC（功能）中的物理宿主 | 在 FB（功能块）中的物理宿主 | 物理保持性（Non-Volatile） | 读写物理机制差异 |
| :--- | :--- | :--- | :--- | :--- |
| **TEMP** | **本地局部数据高速 RAM (L-Stack)** | **本地局部数据高速 RAM (L-Stack)** | 否（断电/退块即失） | 由本地 L 寻址通道直接高速读写。 |
| **STAT** | *(FC 不允许存在 STAT 变量)* | **背景数据块 (Instance DB)** | **是**（只要背景 DB 保持） | 物理上存放在 DI 背景区域，通过 AR2 寄存器的绝对偏移进行读写。 |
| **IN** | 基本类型值传递或复杂类型指针传递 | **背景数据块 (Instance DB)** | 是（在 FB 背景 DB 中） | *   FC 中通过局部堆栈或寄存器。
*   FB 中由系统先拷贝至 DI 区，再在内部读写。 |
| **OUT** | 调用结束时执行物理值传送 | **背景数据块 (Instance DB)** | 是（在 FB 背景 DB 中） | *   FC 内部写，退块时写入实参。
*   FB 内部直接通过 DI 区物理写入。 |
| **IN_OUT** | **6 字节或 10 字节物理指针传参** | **背景数据块 (Instance DB)** | 是（在 FB 背景 DB 中） | 极其关键的区别。在 FC 中，**对 IN_OUT 的读写，底层全部是利用物理指针执行的实参地址间接寻址。** |

---

### 4.2 知识点 4 代码：深入底层剖析 FC 的 IN_OUT 变量（物理指针间接读写）

本例设计一个在 FC 内部累加 64 位有符号大整数的函数。我们将看到，在底层的 STL 指令中，对 `IN_OUT` 变量的每一次访问，是如何转化为复杂的指针间接操作的。

#### 完整 STL 代码

```stl
FUNCTION "Add_Big_Int" : VOID
TITLE = 模拟大整数累加，剖析 IN_OUT 变量的指针本质
VAR_INPUT
   Increment_Val : DINT ;     // 待加的增量值
END_INPUT
VAR_IN_OUT
   Base_Big_Int  : DINT ;     // 大整数基数 (IN_OUT，传址引用)
END_VAR
VAR_TEMP
   t_AR1_Backup  : DWORD ;
END_VAR
BEGIN
NETWORK
TITLE = 指针级读写 IN_OUT 变量
      TAR1  #t_AR1_Backup;    // 备份 AR1

      // 重点：在 FC 内部，直接装载 IN_OUT 变量的指针
      L     P##Base_Big_Int;  // 装载 Base_Big_Int 实参在当前 L-Stack 中的指针 (6字节格式)
      LAR1  ;                 // 将其装入 AR1。此时 AR1 正式指向实参的物理存放地址！

      // 从 AR1 指向的实参物理地址中，装载当前的基数
      L     D [AR1, P#0.0];   // 这是物理间接读取！(等效于读取 Base_Big_Int 实参值)
      
      // 执行加法计算
      L     #Increment_Val;
      +D    ;                 // 累加
      
      // 重点：将计算结果，直接写回到实参指针指向的物理地址中
      T     D [AR1, P#0.0];   // 物理间接写入！实时更新了外部实参值

      LAR1  #t_AR1_Backup;    // 还原 AR1
END_FUNCTION
```

#### 逐行剖析与指针传址追踪

假设我们在 OB1 中调用该函数：
`CALL "Add_Big_Int" ( Increment_Val := 10, Base_Big_Int := "DB30".DBD20 );`

*   **`L P##Base_Big_Int` 执行瞬间**：
    由于 `Base_Big_Int` 被声明为 `IN_OUT` 复杂类型，OB1 调用时传入的是 `DB30.DBD20` 的地址。
    FC10 在被调用时，其临时局部变量栈（L-Stack）中，会自动生成一个指向该实参的**指针参数单元**（存放了指向 `DB30.DBD20` 的指针信息）。
    本指令直接装载这个指针数据。`ACCU 1` 获得：`16#8400_00A0`（即 `P#DBX20.0`）。同时系统将 DB30 锁定在 DB 寄存器。
*   **`LAR1`**：
    `AR1` 获得物理目标实参地址：`AR1 = P#DBX20.0`。
*   **`L D [AR1, P#0.0]`**：
    执行物理跨区域读取。由于 DB30 锁在 DB 寄存器，`AR1` 指向 `20.0`。
    实际装载的是外部 `DB30.DBD20` 里的真实物理数值。
*   **`T D [AR1, P#0.0]`**：
    将累加后的结果，直接通过 `AR1` 指针物理写入外部的 `DB30.DBD20` 内存空间。

### 4.3 工程经验：在 FC 内部，IN_OUT 变量与 OUTPUT 变量在意外停机时的表现差异

在工业安全和事故追溯中，深刻理解 `IN_OUT` 与 `OUTPUT` 在 FC 底层的执行差异，具有决定性的工程设计价值。

*   **OUTPUT 变量的底层机制：值传递（退块传送）**：
    在 FC 内部，当你对一个 `OUTPUT` 类型的变量（例如 `#Motor_Out`）执行赋值（`=`）时，**在块退出（`BEU` 或 `BEC`）前，该数据只是存放在本 FC 临时分配的本地 L-Stack 中，并没有物理写回到 OB1 调用时的外部实参中。**
    只有当 FC 正常运行到最后一行，成功执行退栈并返回 OB1 时，CPU 才会将 L-Stack 中的最终计算值，批量拷贝覆盖到外部物理引脚（如 `Q0.0`）上。
    
    *   **意外停机隐患**：如果在 FC 执行中途，因为数组越界等原因，CPU 突发故障崩溃停机，或者内部执行了直接退块（不完整返回），**你的 `OUTPUT` 实参状态将处于悬空未更新状态，可能无法输出断电安全状态。**
*   **IN_OUT 变量的底层机制：地址引用（实时写入）**：
    如 4.2 节代码所示，由于 `IN_OUT` 是指针地址直通的。你在 FC 内部对 `IN_OUT` 执行的每一次 `T` 指令，都是**穿透块边界、实时写入到外部实参（如全局 DB 或 Q 点）中**。
*   **工程选型建议**：
    对于涉及设备人身安全的连锁输出、关键的应急切断逻辑，建议**优先选用 `IN_OUT` 接口或直接使用全局符号寻址**，确保关键输出的实时穿透写能力，防范 FC 异常挂起时的输出延时故障。

## 5. 绝对地址寻址（Absolute） vs. 符号地址寻址（Symbolic）

最后，我们来探讨西门子寻址体系的表象与本质。

### 5.1 编译器视角下的绝对地址与符号地址

*   **绝对地址寻址（Absolute Addressing）**：
    例如：`A I0.0`、`L DB5.DBD20`。
    *   **特点**：直接指向物理内存单元。对于 CPU 的指令解码单元（IDU）来说最直观。
    *   **弊端**：代码脆弱。如果在博途开发中，中间删除了一个数据，导致后续所有数据在物理 DB 中的偏移位置发生了移动，你必须人工重新修改所有关联的 STL 绝对地址指令，极易漏改发生生产事故。
*   **符号地址寻址（Symbolic Addressing）**：
    例如：`A "Tag_Emergency_Stop"`、`L "Motor_Data".Actual_Speed`。
    *   **特点**：高可读性、高可维护性。
    *   **博途的编译器优化**：在 TIA Portal 中，如果你开启了**“优化块访问”**，符号化寻址的性能达到了最高。博途编译器在编译时，会根据编译符号字典表（Symbol Dictionary），将每一个符号直接分配给特定的、对齐的高速 RAM 寄存器或内存偏移量。这使得符号化代码在执行效率上不仅不输绝对地址，还具有更强的底层越界保护屏障。

### 5.2 知识点 5 代码：全符号化、零绝对地址污染的高性能 STL FC

本例展示如何在博途中设计一个符合现代“符号化、强类型、模块化”规范的 STL 程序。该函数完全不涉及任何具体的物理绝对寄存器（如 `M10.0` 等魔术常数），全部通过强符号变量和接口引脚进行逻辑拼装。

#### 完整 STL 代码

```stl
FUNCTION "Symbolic_Interlock" : VOID
TITLE = 100% 符号化无污染控制逻辑
VAR_INPUT
   Enable_System : BOOL ;     // 符号化输入：系统使能
   Safety_OK     : BOOL ;     // 符号化输入：安全门OK
   Reset_Cmd     : BOOL ;     // 符号化输入：复位指令
END_INPUT
VAR_OUTPUT
   Interlock_Active : BOOL ;  // 符号化输出：联锁动作
END_VAR
VAR_TEMP
   t_Logic_State    : BOOL ;  // 符号化临时变量：中间状态
END_VAR
BEGIN
NETWORK
TITLE = 逻辑判断
      // 全符号化寻址。博途编译器会自动解析这些变量在局部 L-Stack 中的精确对齐偏移，
      // 并生成经过极致优化的机器码。
      
      A     #Enable_System;   
      A     #Safety_OK;       
      =     #t_Logic_State;   // 1. 写入临时变量

NETWORK
TITLE = 锁存并输出
      A     #t_Logic_State;   
      O     #Reset_Cmd;       
      =     #Interlock_Active;// 2. 写入输出引脚
END_FUNCTION
```

### 5.3 工程经验：大型项目的“解耦式”零绝对地址开发规范

在现代化的大型博途项目（特别是由多名工程师协同开发的联合生产线项目）中，建议制定严格的**“解耦式”零绝对地址开发规范**：

```
                    【全局控制总线 (全符号化)】
                                │
         ┌──────────────────────┼──────────────────────┐
         ▼                      ▼                      ▼
    【FC 1: 控制块】        【FC 2: 诊断块】        【FC 3: 通信块】
   (100% 接口引脚传参)     (100% 接口引脚传参)     (100% 接口引脚传参)
         │                      │                      │
         └──────────────────────┼──────────────────────┘
                                ▼
                       【物理 I/O / 绝对 DB】
                     (仅在最外层的 OB1 中进行符号关联)
```

1.  **逻辑 FC/FB 内部彻底去物理地址化**：
    在编写任何 FC、FB 内部的业务逻辑时，**禁止使用任何类似 `I0.0`、`Q0.0`、`M100.0`、`DB5.DBX2.0` 的绝对物理变量**。所有的外部关联信号，必须无一例外地通过 IN、OUT、IN_OUT 接口引脚，以符号形式定义并读取。
2.  **物理引脚仅在 OB1/调用层关联**：
    所有的物理绝对地址（如真实的 `I/O` 硬件引脚）关联动作，必须**仅被局限在最顶层的 OB1 或者是专用的接口封装网络中进行引脚传递**。
3.  **巨大的工程红利**：
    这样做能够让你的底层 STL / SCL 算法逻辑与现场的物理硬件配置彻底解耦。
    如果在调试后期，现场因为端子烧毁，需要将启动按钮从 `I0.0` 更改到 `I1.5`。你**完全不需要修改任何核心功能 FC/FB 的内部代码**，只需在 OB1 的引脚关联处，将符号 `"Tag_Start_Btn"` 绑定的物理引脚在变量表中修改一次即可。整个项目结构将具有无与伦比的健壮性、可移植性与可复用性。
