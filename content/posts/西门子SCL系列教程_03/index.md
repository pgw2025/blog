在上一章中，我们熟悉了博途中的 SCL 开发环境，并且拆解了临时变量、静态变量以及输入/输出变量在 CPU 内存中的运行机制。

今天，我们要攻克工业编程中最基础、最重要，但也最容易让初学者栽跟头的阵地——**数据类型（Data Types）**。

很多从梯形图（LAD）转过来的工程师，在刚开始写 SCL 时，最痛苦的就是遇到编译器报出一堆“数据类型不匹配”的红叉。在 LAD 中，你可能习惯了用一条 `MOVE` 指令把一个无符号的 `WORD` 强行塞进一个有符号的 `INT` 里，因为底层都是 16 位二进制，CPU 不会阻拦你。

但在 SCL 这种符合 **IEC 61131-3** 标准的强类型文本语言中，**类型安全（Type Safety）是底线**。编译器会极其严格地审查每一个变量的血统。如果你对数据类型的物理本质（内存占用、二进制编码方式）没有清晰的认识，你的 SCL 代码就会充满隐蔽的溢出 Bug、精度丢失故障，甚至导致 CPU 直接停机（Stop）。

这一章，师父会带你逐一剖析 SCL 中最常用的 10 种核心数据类型，从它们的底层物理内存结构，一直讲到它们在真实工业项目中的应用和避坑指南。


## 1. 核心数据类型物理本质对照表

在展开讲解之前，我们先给这 10 种数据类型拉一张“底细清单”。建议你将这张表打印出来贴在电脑旁，这是你编写任何 SCL 算法时的基础物理依据。

| 数据类型 | 占用物理内存 (S7-1200/1500) | 二进制编码形式 | 最小值 | 最大值 | 典型工业应用场景 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **BOOL** | 1 Bit (物理打包) | 布尔二进制 (0 或 1) | `FALSE` (0) | `TRUE` (1) | 开关量、按钮、线圈、电磁阀反馈 |
| **BYTE** | 1 Byte (8 Bits) | 无符号二进制 / 字符 | `B#16#00` (0) | `B#16#FF` (255) | Modbus 字节拆分、ASCII 字符、状态码 |
| **WORD** | 2 Bytes (16 Bits) | 无符号二进制 / 位串 | `W#16#0000` (0) | `W#16#FFFF` (65535) | 16位通信寄存器、控制字/状态字、掩码 |
| **DWORD** | 4 Bytes (32 Bits) | 无符号二进制 / 位串 | `DW#16#0` (0) | `DW#16#FFFFFFFF` | 32位指针偏移量、故障汇总掩码 |
| **INT** | 2 Bytes (16 Bits) | 二进制补码 (带符号) | -32768 | 32767 | 循环计数、数组索引、模拟量通道值 |
| **DINT** | 4 Bytes (32 Bits) | 二进制补码 (带符号) | -2,147,483,648 | 2,147,483,647 | 高速计数器脉冲、产量累计、大数组索引 |
| **REAL** | 4 Bytes (32 Bits) | IEEE 754 标准单精度浮点 | $\approx \pm 1.18 \times 10^{-38}$ | $\approx \pm 3.40 \times 10^{38}$ | 温度、压力、流量、流量计瞬时值 |
| **TIME** | 4 Bytes (32 Bits) | 32位双精度带符号整数 | `T#-24d_20h_31m_23s_648ms` | `T#24d_20h_31m_23s_647ms` | 定时器设定值（PT）与当前值（ET） |
| **DATE** | 2 Bytes (16 Bits) | 自1990年1月1日起的天数 | `D#1990-01-01` | `D#2168-12-31` | 设备保养周期计算、排班时间判断 |
| **STRING** | 2 ~ 256 Bytes (动态) | ASCII/字符数组 (带头) | 空字符串 | 默认 254 个字符 | 扫码枪条码解析、RFID 数据读取 |

## 2. 核心数据类型逐个击破

现在，我们把这 10 种数据类型分成四大阵营，从**内存结构**、**转换细节**、**实战应用**和**常见错误**四个维度进行深度剖析。

---

### 第一阵营：位与位串类型（BOOL, BYTE, WORD, DWORD）

这一阵营的变量，本质上是**没有数值符号的二进制位集合**。它们在工业控制中主要用于表达状态、通信报文以及底层的位操作。

#### 1. BOOL（布尔量）
*   **内存占用**：在博途优化块（Optimized DB）中，CPU 会在物理上打包存储 BOOL 变量（多个 BOOL 压缩存储在同一个字节中，仅占 1 Bit 的寻址空间）。但在非优化块中，为了地址对齐，单个独立的 BOOL 变量会奢侈地占用 **1 个字节（8 Bits）** 的空间。
*   **工业场景**：数字量输入输出（DI/DO）、程序控制流的条件判定、设备运行/停止状态。
*   **转换**：在 SCL 中，BOOL 不能直接通过 `BOOL_TO_INT` 隐式转换。如果你需要把一个 Bool 的 `TRUE` 变成数字 `1`，必须进行显式转换：
    ```scl
    #iStatus := BOOL_TO_INT(#bMotorRunning); // TRUE 转为 1，FALSE 转为 0
    ```

#### 2. BYTE（字节）/ WORD（字）/ DWORD（双字）
徒弟，请特别注意：**BYTE、WORD、DWORD 在 IEC 标准里被称为“位串（Bit String）”，它们是没有正负号概念的。** 很多工程师把 WORD 当作 `UINT`（无符号整型）来用，这在西门子体系里虽然常见，但它们在逻辑上有着本质区别（WORD 支持位移、按位与、按位或，而数学运算应当使用 INT/DINT）。

*   **内存占用**：BYTE（8位/1字节）、WORD（16位/2字节）、DWORD（32位/4字节）。
*   **工业场景**：
    *   **WORD / DWORD** 极其常用于 **通信控制字（Control Word）** 和 **状态字（Status Word）**。例如，变频器通信报文的第一字通常是控制字，其中的第 0 位代表启动，第 1 位代表正反转，第 3 位代表故障复位。
*   **底层位操作示范**：
    在 SCL 中，你可以直接对 WORD 进行按位逻辑运算：
    ```scl
    #wControlWord := #wControlWord AND 16#FFFE; // 按位与：将最后一位（Bit 0）强制清零
    #wControlWord := #wControlWord OR 16#0002;  // 按位或：将第二位（Bit 1）强制置一
    ```
*   **常见错误（Sign Extension 符号延伸陷阱）**：
    当你尝试把一个代表传感器状态的 `WORD` 转换成 `INT` 时：
    ```scl
    #wData := 16#8000; // 十六进制的 8000，二进制最高位为 1
    #iValue := WORD_TO_INT(#wData); // 显式转换
    ```
    你期望得到 `32768`，但由于 `INT` 是带符号的，最高位 `1` 被当成了负号补码，最终 `#iValue` 的值会变成 **`-32768`**！这会导致你的温度或压力标定程序瞬间算出一个极其荒谬的负数。

### 第二阵营：数值与浮点类型（INT, DINT, REAL）

这一阵营专门用于处理**数学运算**、**模拟量标定**以及**精密控制算法**。

#### 1. INT（整型）/ DINT（双整型）
*   **内存占用**：INT 为 16 位（2 字节），DINT 为 32 位（4 字节）。
*   **编码方式**：采用 **二进制补码（Two's Complement）**。最高位（MSB）为符号位（0 代表正，1 代表负）。这就是为什么 INT 的范围是 `-32768` 到 `32767` 的物理原因。
*   **工业场景**：
    *   **INT**：西门子 AI/AO 模块的物理通道值范围永远是 `-27648` 到 `27648`，必须用 INT 存储。此外，`FOR` 循环的计数器也首选 INT。
    *   **DINT**：用于高速计数器（如编码器读数，脉冲值会迅速超过 32767），以及生产产量的累计值。
*   **类型转换与溢出（Overflow）**：
    在 SCL 中，将 `INT` 赋给 `DINT` 是安全的，这被称为 **隐式向上转换（Widening Conversion）**：
    ```scl
    #diLargeValue := #iSmallValue; // 编译器隐式允许，无安全隐患
    ```
    但是，反过来将 `DINT` 强行赋给 `INT`，如果数据超过了 32767，就会发生**截断溢出**：
    ```scl
    #diLargeValue := 40000;
    #iSmallValue := DINT_TO_INT(#diLargeValue); // 显式转换，但结果会溢出变成 -25536！
    ```

#### 2. REAL（实数 / 浮点数）
*   **内存占用**：32 位（4 字节），完全遵循 **IEEE 754 单精度浮点数标准**（1 位符号位 + 8 位指数位 + 23 位尾数位）。
*   **工业场景**：温度、压力、PID 控制器的输出（0.0 ~ 100.0%）、流量计的瞬时流量。

##### ⚠️ 致命漏洞：浮点数的“累积墙”（Precision Loss Limit）

这是很多半吊子工程师写累积流量算法时最容易制造的惊天 Bug。

假设你正在编写一个流量计累积程序。瞬时流量非常小，每个周期（比如 100ms）累积进来的增量 `#rDelta` 只有 `0.001` 升。当前的总累积量 `#rTotal` 已经运行了半年，达到了 `100000.0` 升。

在 SCL 中，你写了这一行代码：
```scl
#rTotal := #rTotal + #rDelta; // 即 100000.0 + 0.001
```
**结果是：`#rTotal` 永远停留在 `100000.0`，不再增加！**

*底层的数学残酷真相*：
IEEE 754 单精度浮点数（REAL）只有 **7 位有效十进制精度**。
当 `100000.0`（7位有效数字）与 `0.001` 相加时，由于两者的指数级差太大，在进行二进制“对阶”操作时，小数字 `0.001` 的尾数会被完全移出有效存储区（被直接舍弃）。
**这就是著名的“浮点数相加丢失”。**

*解决方案*：
1.  在 S7-1500 中，将 `#rTotal` 升级为 **`LREAL`（双精度浮点数，64位，15位有效精度）**。
2.  如果不支持 LREAL，应采用“双整型累积法”：将整数部分和小数部分拆开，用 DINT 存储。

### 第三阵营：时间与日期类型（TIME, DATE）

在工业控制中，一切都与时间息息相关。西门子 PLC 内部对时间的处理极其精准。

#### 1. TIME（时间）
*   **内存占用**：32 位（4 字节）。
*   **底层物理本质**：**TIME 类型在 CPU 内部，本质上就是一个带符号的 `DINT`！它的单位是毫秒（ms）。**
    例如：当你写下 `#tDelay := T#1s;` 时，CPU 内部存储的实际二进制数据就是整型数值 `1000`。
*   **工业场景**：各种定时器（TON、TOF）的设定值（PT）和当前值（ET）。
*   **SCL 的时间运算便利性**：
    既然 TIME 本质是 DINT，在 SCL 中你可以非常优雅地对时间进行数学运算：
    ```scl
    #tTotalDelay := #tDelay1 + #tDelay2; // 两个时间直接相加
    #tHalfTime := DINT_TO_TIME(TIME_TO_DINT(#tDelay) / 2); // 时间除以 2
    ```

#### 2. DATE（日期）
*   **内存占用**：16 位（2 字节）。
*   **底层物理本质**：内部是一个无符号的 `INT`，**记录的是自 1990 年 1 月 1 日以来所度过的天数**。
    例如：`D#1990-01-01` 对应数值 `0`；`D#1990-01-02` 对应数值 `1`。

### 第四阵营：字符与字符串（STRING）

在跟扫码枪、RFID 或者上位机 MES 系统通信时，字符串处理是躲不开的硬骨头。

#### STRING（字符串）
*   **内存占用**：默认长度为 254 个字符，但在内存中它实际占用 **256 字节**。
*   **底层物理结构（Siemens String Header）**：
    西门子的 STRING 类型拥有一个非常特殊的 **“头部结构（Header）”**，占用最前面的 2 个字节：
    *   **Byte 0**：**最大容量（Max Length）**。声明该字符串最大能装多少个字符。
    *   **Byte 1**：**当前实际长度（Current Length）**。当前字符串里实际装了多少个有效字符。
    *   **Byte 2 ~ 255**：真正的 ASCII 字符数据。

```
 字节地址:     Byte 0      Byte 1      Byte 2      Byte 3      Byte 4    ...
            ┌───────────┬───────────┬───────────┬───────────┬───────────┐
 内容:      │ Max_Len   │ Cur_Len   │  'H'      │  'E'      │  'L'      │ ...
            │ (例如 254)│ (例如 3)  │  (16#48)  │  (16#45)  │  (16#4C)  │
            └───────────┴───────────┴───────────┴───────────┴───────────┘
```

*   **工业应用**：解析扫码枪发送的 TCP 条码报文、格式化日志输出。
*   **在 SCL 中声明自定义长度的字符串**：
    如果你不需要默认 254 那么长，千万不要浪费宝贵的 DB 块空间。你可以指定长度：
    ```scl
    VAR
        sBarcode : String[30]; // 仅占用 32 字节（30字符 + 2字节头部）
    END_VAR
    ```
*   **常见错误（手动覆盖内存导致的空字符 Bug）**：
    如果你用 `BLKMOV` 或者 Modbus 通信直接把一串 ASCII 数据强行写入了 `sBarcode` 的物理存储区（比如从 Byte 2 开始写入了 `"12345"`），但你**没有修改 Byte 1（当前长度）的值**。此时，你在博途里监视 `sBarcode`，它依然会显示为 `""`（空字符串）！
    *原因*：PLC 系统函数在读取字符串时，只认 Byte 1 里的数值。如果 Byte 1 是 0，它就认为这个字符串长度为 0。
    *解决方案*：在用指针强行写入字符后，必须手动用 SCL 修正 Byte 1 的值，或者使用博途自带的 `Chars_TO_Strg` 指令。

## 3. 显式与隐式类型转换（Type Conversion）

在 SCL 中，处理不同类型变量之间的赋值有两条通道：

### 3.1 隐式转换（Implicit Conversion）
当赋值是从**低精度/低范围**向**高精度/高范围**流动时，博途编译器会自动帮你在底层做位填充，无需你写任何转换指令。
*   `BOOL` -> 无隐式转换（必须显式）
*   `INT` -> `DINT` -> `REAL`
*   `BYTE` -> `WORD` -> `DWORD`

```scl
#diValue := #iValue; // 安全，隐式转换
#rValue := #diValue; // 安全，隐式转换
```

### 3.2 显式转换（Explicit Conversion）
只要存在**数据精度丢失、截断、或者跨越数据阵营（如位串转数值）**的风险，编译器就会罢工，强制你使用显式转换指令：`<源类型>_TO_<目标类型>`。

```scl
#iValue := DINT_TO_INT(#diValue);   // 风险：高位截断
#rValue := INT_TO_REAL(#iValue);     // 推荐：消除隐式警告
#wData  := INT_TO_WORD(#iValue);     // 跨阵营：数值转位串
```

#### 💡 高级技巧：浮点数转整型的四种截整方式

当我们需要把一个 `REAL` 转换成 `INT` 时，绝对不能拍脑袋就转。SCL 提供了四种不同的内置截整函数，它们的底层逻辑大不相同：

```
                    输入 REAL 值:  2.6                    输入 REAL 值: -2.6
                          │                                     │
         ┌────────────────┼────────────────┐   ┌────────────────┼────────────────┐
         ▼ (ROUND)        ▼ (TRUNC)        ▼ (CEIL)            ▼ (TRUNC)        ▼ (FLOOR)
      [四舍五入]        [截断小数]       [向上取整]          [截断小数]       [向下取整]
         │                │                │                   │                │
         ▼                ▼                ▼                   ▼                ▼
     结果: 3          结果: 2          结果: 3             结果: -2         结果: -3
```

1.  **`ROUND`（四舍五入）**：
    遵循 IEEE 标准的“最近偶数舍入（Round half to even）”。例如，`ROUND(2.5)` 会变成 `2`，而 `ROUND(3.5)` 会变成 `4`。这在统计学上能减少累积误差，但常常让初学者感到困惑。
2.  **`TRUNC`（截断 / 去尾法）**：
    直接把小数点后面的部分无脑扔掉。`TRUNC(2.9)` 的结果是 `2`，`TRUNC(-2.9)` 的结果是 `-2`。
3.  **`CEIL`（向上取整）**：
    返回大于或等于该浮点数的最小整数。`CEIL(2.1)` 的结果是 `3`，`CEIL(-2.9)` 的结果是 `-2`。
4.  **`FLOOR`（向下取整）**：
    返回小于或等于该浮点数的最大整数。`FLOOR(2.9)` 的结果是 `2`，`FLOOR(-2.1)` 的结果是 `-3`。

## 4. 工业实战案例：非标 Modbus 电力仪表数据解析器

现在，我们把本章所学的位串（WORD/DWORD）、数值（DINT/REAL）和显式类型转换融会贯通，写一个在工业现场非常经典的程序。

### 4.1 工业背景
在现场，你通过 Modbus RTU 协议读取了一台非标的智能电量仪表。该仪表通过 Modbus 寄存器（16位无符号 WORD）发回了以下数据：
1.  **电网频率**：存储在 `Modbus_Reg[0]` 中，单位是 $0.01\text{Hz}$。我们需要将其转换为 `REAL` 并除以 100.0，得到真实的频率值（如 $50.02\text{Hz}$）。
2.  **累计电能 (KWh)**：由于累计值非常大，仪表使用了两个寄存器 `Modbus_Reg[1]`（高 16 位）和 `Modbus_Reg[2]`（低 16 位）拼成了一个 **32 位无符号的双字（DWORD）**。我们需要将这两个 WORD 拼接起来，并转换成 `REAL` 输出。
3.  **仪表报警字**：存储在 `Modbus_Reg[3]` 中。其中的 Bit 0 代表过压报警，Bit 1 代表过流报警，Bit 2 代表逆相序故障。我们需要把它们解包成 3 个独立的 `BOOL` 变量输出。

### 4.2 块接口声明（FC_ModbusParser）

这是一个纯粹的算法块，我们选择 **FC (函数)**，语言选择 **SCL**。

| 接口类型 | 变量名 | 数据类型 | 注释 |
| :--- | :--- | :--- | :--- |
| **Input** | `arrModbusRegs` | `Array[0..3] of Word` | 从仪表读回的 4 个原始寄存器 |
| **Output** | `rGridFrequency`| `Real` | 解析后的真实电网频率 (Hz) |
| **Output** | `rTotalEnergy` | `Real` | 解析后的累计电量 (KWh) |
| **Output** | `bOverVoltage` | `Bool` | 过压报警输出 |
| **Output** | `bOverCurrent` | `Bool` | 过流报警输出 |
| **Output** | `bPhaseError`  | `Bool` | 逆相序故障输出 |
| **Temp** | `dwTempEnergy` | `DWord` | 临时双字用于拼接 |

---

### 4.3 SCL 代码实现

```scl
FUNCTION "FC_ModbusParser" : Void
{ S7_Optimized_Access := 'TRUE' }
VERSION : 0.1
   VAR_INPUT 
      arrModbusRegs : Array[0..3] of Word;   // 传入的 Modbus 原始数据
   END_VAR

   VAR_OUTPUT 
      rGridFrequency : Real;   // 真实电网频率 (Hz)
      rTotalEnergy : Real;     // 真实累计电量 (KWh)
      bOverVoltage : Bool;     // 过压警报
      bOverCurrent : Bool;     // 过流警报
      bPhaseError : Bool;      // 逆相序故障
   END_VAR

   VAR_TEMP 
      dwTempEnergy : DWord;    // 用于 32 位拼接的临时双字
   END_VAR


BEGIN
	// ==========================================
	// 1. 解析电网频率 (16位 WORD 转为 32位 REAL 并除以 100.0)
	// ==========================================
	
	// 在西门子体系中，WORD 不能直接隐式转换为 REAL
	// 我们需要路线：WORD -> INT -> REAL (或者 WORD -> UINT -> REAL)
	// 假设频率寄存器永远是正数
	#rGridFrequency := INT_TO_REAL(WORD_TO_INT(#arrModbusRegs[0])) / 100.0;
	
	
	// ==========================================
	// 2. 拼接解析累计电能 (两个 16位 WORD 拼接为 32位 DWORD)
	// ==========================================
	
	// 我们使用逻辑位移操作：将高位寄存器左移 16 位，然后与低位寄存器进行“按位或”
	// 1) 先将第一个寄存器转换成 32位双字并左移 16 位
	#dwTempEnergy := SHL(IN := WORD_TO_DWORD(#arrModbusRegs[1]), N := 16);
	
	// 2) 与第二个寄存器转换成的双字进行“或”运算，填补低 16 位
	#dwTempEnergy := #dwTempEnergy OR WORD_TO_DWORD(#arrModbusRegs[2]);
	
	// 3) 将得到的 32位无符号双字转换为浮点数 REAL
	// 注意：在 S7-1500 中，推荐使用 UDINT_TO_REAL 进行无符号转换，避免最高位 1 导致负数
	#rTotalEnergy := UDINT_TO_REAL(DWORD_TO_UDINT(#dwTempEnergy));
	
	
	// ==========================================
	// 3. 解析仪表警报字 (WORD 位拆分)
	// ==========================================
	
	// 在博途中，我们可以使用内置的 Slice（片段访问）语法：变量名.Xn
	// 这种方法性能极高，不需要使用复杂的指针和指针移位
	#bOverVoltage := #arrModbusRegs[3].X0; // 解析第 0 位
	#bOverCurrent := #arrModbusRegs[3].X1; // 解析第 1 位
	#bPhaseError  := #arrModbusRegs[3].X2;  // 解析第 2 位
	
END_FUNCTION
```

### 4.4 代码深度剖析与高级指令释义

徒弟，这段代码展现了西门子 SCL 处理工业数据的两项“看家本领”。

#### 1. 位移函数 `SHL`
在第二部分，我们将高 16 位寄存器强转为 32 位的 `DWORD` 之后，执行了 `SHL(..., N:=16)`：
```
 转换前:  0000 0000 0000 0000 [ 16位 ModbusReg[1] 原始数据 ]
 SHL 16: [ 16位 ModbusReg[1] 原始数据 ] 0000 0000 0000 0000
```
然后，通过 `OR` 运算：
```
   [ 16位 ModbusReg[1] ] 0000 0000 0000 0000
OR 0000 0000 0000 0000 [ 16位 ModbusReg[2] ]
─────────────────────────────────────────────
=  [ 16位 ModbusReg[1] ] [ 16位 ModbusReg[2] ]  --> 完美拼接！
```
这种纯位操作不需要任何中间寄存器和乘除法运算，直接在 CPU 的算术逻辑单元（ALU）里完成，**执行时间是微秒级的**。

#### 2. 杀手级语法：Slice 寻址（片段访问）
在第三部分，我们使用了：
`#bOverVoltage := #arrModbusRegs[3].X0;`
这是博途的一项伟大发明。在经典的 S7-300 时代，如果你想提取一个字（Word）里的某一个位（Bit），你需要用 STL 或者是写一个复杂的掩码过滤器。
而在博途中，只要你的变量是 `BYTE / WORD / DWORD` 类型，你就可以使用 `.X` 语法直接提取其中的任意一位：
*   `.X0` 到 `.X15` 代表字（Word）的 16 个位。
*   `.B0` / `.B1` 代表提取其中的字节。

*注意*：**Slice 寻址只能用于优化块中的位串类型（BYTE, WORD, DWORD），不能直接用于带符号的 INT！** 这也就是为什么我们在前面的设计中，坚持把 Modbus 寄存器定义为 `WORD` 而不是 `INT` 的原因之一。

## 5. 常见数据类型错误与避坑指南 (Senior Tips)

### 5.1 错误一：用 `120.0` 代替 `120`，反之亦然
在 SCL 中，数据常量的书写格式直接决定了它的数据类型：
*   `120` 默认是整型 `INT`。
*   `120.0` 默认是浮点型 `REAL`。

如果你写了这一行：
```scl
#rResult := #rSensorValue * 2; // 假设 rSensorValue 为 REAL
```
博途编译器可能会报出警告，甚至在某些严格配置下报错。因为你试图让一个 `REAL` 去乘以一个 `INT` 常量 `2`。
**正确习惯**：**只要是给 REAL 类型的计算，所有的常数都要加上小数点和后缀。**
`#rResult := #rSensorValue * 2.0;`

### 5.2 错误二：除法结果不合常理地归零
看下面这段代码：
```scl
#rRatio := 3 / 4; // 你期望得到 0.75
```
运行后，你会震惊地发现 `#rRatio` 的值是 **`0.0`**！
*原因*：在 SCL 编译器眼中，`3` 和 `4` 都是 `INT` 类型。`INT` 与 `INT` 做除法，执行的是**整数除法**，保留商的整数部分。`3/4` 的商是 `0`，余数是 `3`。商 `0` 随后被转换成 `0.0` 赋给了 `#rRatio`。
**避坑指南**：做除法时，至少要保证分子或分母其中一个是 `REAL` 格式：
`#rRatio := 3.0 / 4.0;` 或者 `#rRatio := INT_TO_REAL(#iVal1) / INT_TO_REAL(#iVal2);`
