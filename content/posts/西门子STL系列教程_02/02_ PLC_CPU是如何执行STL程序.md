在第一篇教程中，我们探讨了西门子 STL 语言的历史背景、存在价值以及它与硬件寄存器的紧密联系。在本篇教程中，我们将更进一步，深入到西门子 S7 CPU 的物理/虚拟运行机制中，研究 CPU 是如何在硬件层面调度、解析并执行 STL 程序的。

作为具有 LAD（梯形图）基础的工程师，我们习惯于“电信号从左流向右”的直观逻辑。但在 CPU 内部，这一过程被拆解为微观的指令获取、寄存器状态更新、累加器数据移位以及堆栈调度。

本教程将基于 **STEP 7 Classic** 与 **TIA Portal V16** 平台，重点剖析 S7 CPU（兼容 S7-300、S7-400 及 S7-1500 兼容模式）的内部执行机理。

---

### 技术兼容性及声明

*   **S7-300/400**：采用基于 ASIC 芯片的硬件微循环架构，其原生指令集（MC7）与 STL 指令一一对应。
*   **S7-1500**：虽然采用现代通用微处理器架构，但其虚拟机监控器（Hypervisor）或编译解析器在后台完整模拟了本文所述的寄存器机制（ACCU、AR、Status Word）。因此，本文的原理分析完全适用于 S7-1500。
*   **S7-1200**：由于硬件设计未引入经典 S7 的累加器和地址寄存器模型，**不支持 STL 语言**。但理解本文中关于扫描周期、程序调用和局部变量栈（L-Stack）的物理本质，对于在 S7-1200 中编写高性能的 SCL 或 LAD 同样具有指导意义。

## 1. 扫描周期与 OB1 的宿主环境

PLC 与通用计算机在执行程序上的最大区别在于其**周期性循环扫描**的机制。STL 程序并非孤立运行，它宿主在由操作系统（OS）管理的周期性循环进程中。

### 1.1 经典 S7 CPU 的物理扫描周期循环

CPU 的运行周期由操作系统控制，主要包含以下五个阶段：

```
       +--------------------------------------------+
       |   1. 写入物理输出到模块 (PIQ -> Q-Area)    |
       +--------------------------------------------+
                             │
                             ▼
       +--------------------------------------------+
       |   2. 读取物理输入到映像区 (I-Area -> PII)   |
       +--------------------------------------------+
                             │
                             ▼
       +--------------------------------------------+
       |   3. 操作系统服务 (通信、定时器、自检)     |
       +--------------------------------------------+
                             │
                             ▼
       +--------------------------------------------+
       |   4. 执行用户程序 (调用 OB1 以及其子程序)   |
       +--------------------------------------------+
                             │
                             ▼
       +--------------------------------------------+
       |   5. 循环监控与看门狗复位 (Reset Watchdog) |
       +--------------------------------------------+
                             │
                             └──────────────────────┘
```

在这个循环中，所有的用户 STL 代码都在**阶段 4（执行用户程序）**中被执行。OB1（组织块 1）是用户程序的起点。

### 1.2 异步中断对寄存器环境的物理冲击

在阶段 4 的执行过程中，CPU 随时可能受到异步中断源的干扰。例如：
*   **OB35（100ms 循环中断）**：时间到达，强行中断 OB1。
*   **OB40（硬件中断）**：某个物理输入点产生上升沿，强行中断 OB1。
*   **OB121/OB122（编程/I-O 访问错误中断）**：程序运行出错时触发。

当这些中断发生时，CPU 必须立即跳转到对应 OB 块执行紧急代码。这时产生了一个核心的硬件设计问题：**中断 OB 同样会使用 ACCU1/ACCU2、AR1/AR2 以及状态字。如果它们改写了这些寄存器，当中断结束返回 OB1 时，OB1 的计算逻辑岂不是会崩溃？**

#### 西门子 CPU 的保护机制：上下文切换（Context Switch）

为了解决这一冲突，西门子 CPU 在硬件/操作系统级设计了**现场保护与恢复机制**。
1.  **中断触发**：操作系统暂停当前 OB1 的程序计数器（PC）。
2.  **压栈保存**：将当前 OB1 运行时的 ACCU1、ACCU2、AR1、AR2、状态字（Status Word）、当前打开的 DB 块编号（DB/DI 寄存器）以及 L 堆栈指针，自动压入系统的**上下文物理堆栈（Context Stack）**。
3.  **执行中断**：将上述寄存器全部清空或初始化，跳转执行中断 OB（如 OB35）。
4.  **弹栈恢复**：中断 OB 执行完毕，系统执行返回指令，将刚才保存在上下文堆栈中的寄存器值全部**弹出（Pop）并恢复**到物理寄存器中。
5.  **恢复执行**：OB1 从被中断的一行指令继续无缝执行。

### 1.3 知识点 1 代码：在循环中断（OB35）中安全进行高精度计时

下面的例子展示了如何在 OB35 中编写一个累加高精度运行时间的 STL 块。我们将看到，在中断环境中，代码是如何与 OB1 的主循环逻辑做到“互不干扰”的。

#### 完整 STL 代码（OB35 内部逻辑）

```stl
// 假定 OB35 的调用周期为固定的 100ms（由CPU属性设置决定）
// 每次执行，将累积时间增加 0.1 秒
ORGANIZATION_BLOCK "Cyclic_Interrupt_100ms"
TITLE = 100ms 高精度工艺时间累加
VERSION : 0.1
SYSTEM_FLAGS : Active
VAR_TEMP
   OB35_EV_CLASS : BYTE ;   // 事件类别
   OB35_STRT_INF : BYTE ;   // 启动信息
   OB35_PRIORITY : BYTE ;   // 优先级
   OB35_OB_NUMBR : BYTE ;   // 组织块编号
   OB35_RESERVED_1 : BYTE ;
   OB35_RESERVED_2 : BYTE ;
   OB35_PHASE_OFFSET : INT ;
   OB35_RESERVED_3 : INT ;
   OB35_EXC_FREQ : INT ;    // 执行频率 (ms)
END_VAR
BEGIN
NETWORK
TITLE = 累加器安全数学运算（0.1s 累加）
      OPN   "Process_Data"            // 打开全局共享数据块 DB10
      L     "Process_Data".Accum_Time // 装载当前的累积时间 (REAL, 单位：秒)
      L     1.000000e-001             // 装载实数常数 0.1
      +R                              // 执行实数加法运算 (ACCU2 + ACCU1 -> ACCU1)
      T     "Process_Data".Accum_Time // 将累加后的结果写回，存入 DB10.DBD0
      
      // 注意：此处不需要手动保存 RLO 或 ACCU，
      // 操作系统在退出 OB35 时会自动将它们还原为 OB1 被中断瞬间的值。
END_ORGANIZATION_BLOCK
```

#### 逐行深度剖析与寄存器状态跟踪表

假设进入 OB35 前，`Process_Data.Accum_Time` 的当前值为 `1250.4`。

| 步骤 | 指令 | ACCU 1 (32-bit hex/float) | ACCU 2 (32-bit hex/float) | RLO | /FC | 寄存器机制及物理意义解析 |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| **0** | (中断切入) | `0.000000e+000` | `0.000000e+000` | `0` | `0` | 进入 OB35 时，CPU 已自动完成了现场保护。此时寄存器处于干净的初始状态。 |
| **1** | `OPN "Process_Data"` | `0.000000e+000` | `0.000000e+000` | `0` | `0` | 打开 DB10。该操作将 `10` 写入内部 DB 寄存器。累加器和状态字保持不变。 |
| **2** | `L "Process_Data".Accum_Time` | `1.250400e+003` | `0.000000e+000` | `0` | `0` | 将原 ACCU1（0.0）推入 ACCU2。将实数 `1250.4` 装入 ACCU1。 |
| **3** | `L 1.000000e-001` | `1.000000e-001` | `1.250400e+003` | `0` | `0` | 将原 ACCU1（1250.4）推入 ACCU2。将实数 `0.1` 装入 ACCU1。 |
| **4** | `+R` | `1.250500e+003` | `1.250400e+003` | `0` | `0` | 硬件浮点运算单元将 ACCU2 与 ACCU1 相加，结果 `1250.5` 写入 ACCU1。ACCU2 保持原值不变。 |
| **5** | `T "Process_Data".Accum_Time` | `1.250500e+003` | `1.250400e+003` | `0` | `0` | 将 ACCU1 中的结果传输回 DB10.DBD0。累加器中的数据仍旧保留。 |

### 1.4 工程经验：如何应对多任务并发时的寄存器资源冲突

在多任务、多中断并发的工业场景中，有些寄存器是**不被操作系统自动保护**的。最典型也是最危险的，就是**地址寄存器 AR1 和 AR2**。

*   **危险场景**：
    假设你在 OB1 中编写了一段使用 `LAR1` 指针寻址的大型循环逻辑。当循环执行到中途时，OB35（或 OB40 硬件中断）突然切入。如果在 OB35 的程序中同样使用了 `LAR1` 且**没有进行手动备份和恢复**，那么在退出 OB35 返回 OB1 时，OB1 面临的将是一个**被篡改的 AR1 指针**。这会导致 OB1 接下来的指针寻址完全错乱，直接引发 `OB122`（I/O 访问错误）或导致 PLC 骤停（STOP）。
*   **黄金法则**：
    在编写任何**中断 OB 调用的 FC/FB**，或者任何可能被中断高频切入的底层通用函数时，如果使用了 `AR1` 或 `AR2`，**必须在函数入口处备份，并在出口处还原**。

## 2. 状态字（Status Word）与 RLO 的物理决断过程

状态字是西门子 S7 架构中极其精妙的控制核心，它实际上是 PLC CPU 内部运算控制器的“神经末梢”。其中，**RLO（逻辑运算结果）** 和 **/FC（首次检查）** 位的设计尤为特殊。

### 2.1 状态字寄存器的 16 位物理布局

虽然状态字总长为 16 位，但在 STL 中实际使用的只有低 9 位。其硬件结构图如下：

```
+----+----+----+----+----+----+----+----+----+----+----+----+----+----+----+----+
| 15 | 14 | 13 | 12 | 11 | 10 |  8 |  7 |  6 |  5 |  4 |  3 |  2 |  1 |  0 | -  | -> Bit
+----+----+----+----+----+----+----+----+----+----+----+----+----+----+----+----+
|  0 |  0 |  0 |  0 |  0 |  0 |  0 | BR |CC1 |CC0 | OS | OV | OR |STA |RLO |/FC | -> Name
+----+----+----+----+----+----+----+----+----+----+----+----+----+----+----+----+
```

#### 关键控制位的硬件驱动逻辑：

*   **/FC (首次检查位，First Check)**：
    为什么西门子不需要像其他语言那样输入“IF”或“START”来开始一个逻辑链？就是因为有 `/FC` 的存在。
    *   当 `/FC = 0` 时，表明当前执行的布尔指令（如 `A`、`O`）是**新一轮逻辑链的第一条指令**。CPU 不会将此指令的值与旧的 RLO 进行逻辑运算，而是直接将该点物理状态（STA）复制到 RLO 中，并自动将 `/FC` 置 `1`。
    *   当 `/FC = 1` 时，后续的布尔指令（如 `A`、`ON`）将按照对应的逻辑关系，与当前的 RLO 进行合并计算，并更新 RLO 的值。
    *   当遇到赋值指令（如 `=`、`S`、`R`）或某些跳转指令时，逻辑链结束，CPU 自动将 `/FC` 清零（`0`）。

*   **OR (或缓存位)**：
    专门用于处理“先与后或”的多组复合分支逻辑。当 CPU 遇到特殊的布尔组合时，`OR` 位扮演临时寄存器的角色，用来缓存上一段“与”链的结果。

*   **BR (二进制结果位，Binary Result)**：
    它是 STL 代码与 LAD/FBD 混合编程的桥梁。在 LAD 中，每个块都有使能输入（EN）和使能输出（ENO）。
    在底层，**ENO 的输出状态完全等同于子程序执行完毕时，状态字中 BR 位的值**。

### 2.2 知识点 2 代码：复合逻辑块及状态字微观变化示踪

为了深入分析状态字的变化，我们设计了一段包含与、或、非以及条件跳转的复合逻辑程序。该程序用于监控一台大功率电机的启动联锁条件。

#### 完整 STL 代码

```stl
A     "Low_Oil_Level"       // 检查油位是否过低 (I0.4)
NOT                         // 取反 (油位不过低表示正常)
A(                          // 开启第一级括号 (暂存 RLO)
A     "Start_Button"        // 启动按钮按下 (I0.5)
O     "Remote_Start"        // 或者是远程启动信号 (I0.6)
)                           // 结束括号，与前面的正常状态执行与操作
AN    "Emergency_Stop"      // 且不能有紧急停止信号 (I0.7)
=     "Motor_Ready"         // 驱动电机就绪中间状态 (M4.0)
A     "Motor_Ready"         // 再次读取就绪状态
JC    M001                  // 若 RLO = 1 (就绪)，跳转到标号 M001 执行启动逻辑
SET                         // 若 RLO = 0 (未就绪)，强制将 RLO 置 1
R     "Motor_Contactor"     // 复位运行接触器输出 (Q0.1)
BEU                         // 块无条件结束 (不执行后续代码)

M001: S     "Motor_Contactor"     // 标号位置：置位运行接触器输出 (Q0.1)
```

#### 逐行深度剖析与寄存器状态跟踪表

假设当前物理触点状态为：
`I0.4 (油位过低) = 0`（取反后代表正常）、`I0.5 (启动按钮) = 1`、`I0.6 (远程启动) = 0`、`I0.7 (紧急停止) = 0`。

| 步骤 | 指令 | 物理状态 | /FC | RLO | STA | OR | BR | 状态字逻辑控制器微观动作详解 |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| **0** | (开始) | - | `0` | `0` | `0` | `0` | `0` | 系统准备就绪，迎接首检。 |
| **1** | `A I0.4` | `0` | `1` | `0` | `0` | `0` | `0` | `/FC=0`。CPU 识别出这是首检指令。STA 被读入为 `0`，并被复制到 RLO。然后 CPU 自动将 `/FC` 置 `1`。 |
| **2** | `NOT` | - | `1` | `1` | `1` | `0` | `0` | RLO 状态取反。旧 RLO 为 `0`，新 RLO 变为 `1`。STA 变为当前 RLO 的值。 |
| **3** | `A(` | - | `0` | `1` | `1` | `0` | `0` | 开启括号。当前 RLO（`1`）及 `/FC` 被推入嵌套堆栈。**内部 `/FC` 重新被置为 `0`**，标志着括号内部将开启一个全新的逻辑链。 |
| **4** | `A I0.5` | `1` | `1` | `1` | `1` | `0` | `0` | 括号内的首检。STA（`1`）写入 RLO。`/FC` 置 `1`。 |
| **5** | `O I0.6` | `0` | `1` | `1` | `0` | `0` | `0` | 或操作。将 RLO（`1`）与 STA（`0`）进行“或”运算，RLO 保持 `1`。 |
| **6** | `)` | - | `1` | `1` | `1` | `0` | `0` | 括号结束。从嵌套堆栈弹出之前的 RLO（`1`），与括号内部的计算结果（`1`）执行“与”操作。新 RLO 为 `1`。此时外部 `/FC` 恢复为 `1`。 |
| **7** | `AN I0.7`| `0` | `1` | `1` | `0` | `0` | `0` | 与非操作。STA（`0`）取反后变为 `1`。再与当前 RLO（`1`）相与，最终 RLO 保持 `1`。 |
| **8** | `= M4.0` | - | `0` | `1` | `1` | `0` | `0` | 赋值指令。将 RLO（`1`）写入 `M4.0`。该逻辑链执行完毕，**`/FC` 自动被复位为 `0`**。 |
| **9** | `A M4.0` | - | `1` | `1` | `1` | `0` | `0` | 新一轮逻辑链首检。STA 值为 `1`，复制到 RLO，`/FC` 置 `1`。 |
| **10** | `JC M001`| - | `0` | `1` | `1` | `0` | `0` | 条件跳转指令。因为此时 `RLO = 1`，条件成立，CPU 执行跳转到 `M001`。**注意：跳转指令会强制将 `/FC` 复位为 `0`**。 |
| **11** | `S Q0.1` | - | `0` | `1` | `1` | `0` | `0` | 标号处指令。虽然上一步跳转清空了 `/FC`，但置位（`S`）和复位（`R`）指令属于控制输出指令，不依赖于 `/FC` 的首检判定，它们直接依据当前的 RLO（`1`）对 `Q0.1` 进行置位。 |

### 2.4 工程经验：如何规避跳转（Jump）指令后的“状态字盲区”

在 STL 编程中，跳转指令（如 `JC`、`JCN`、`JU`）非常频繁。但跳转指令会导致一个初学者极易踩坑的现象：**状态字盲区**。

*   **技术机制**：
    当 CPU 执行了跳转，无论跳转是否成功，目标标号（Label）之后的第一条逻辑指令，其 `/FC` 位都会被 CPU 的硬件自动清零（`0`）。这意味着，**标号后的第一条指令，无论写的是 `A` 还是 `O`，都将被视为首次检查**。
*   **潜在隐患**：
    ```stl
    JU    NEXT
    
    NEXT: O     "Switch_1"   // 虽然写的是 O 指令，但由于是跳转后的首行，其物理等价于 A 指令！
          A     "Switch_2"
          =     "Output"
    ```
    在上述代码中，由于 `NEXT` 是跳转目标，CPU 执行到 `O "Switch_1"` 时将 `/FC` 视为 `0`。即使 `"Switch_1"` 为 `1`，它也只是完成首检赋值，而不会和任何前序逻辑执行“或”运算。
*   **工程规避方法**：
    在任何跳转目标标号（Label）的下一行，如果要开启新的逻辑判定，建议**一律使用 `A`（与）指令作为逻辑链的起点**。避免使用 `O` 或否定指令作为标号后的首行指令，从而保持代码逻辑与人类直觉的一致性。

## 3. 累加器（ACCU）工作方式与数学运算管道

在西门子 S7 CPU 内部，算术逻辑单元（ALU）不直接与外部的物理输入输出、数据块直接打交道。所有的加、减、乘、除、移位、逻辑字运算，其操作数都必须先被载入到**累加器（ACCU）**中。

### 3.1 累加器在内存装载（Load）时的滑移（Shift）原理

S7 CPU 拥有两个 32 位的物理寄存器：`ACCU 1` 和 `ACCU 2`。它们之间具有单向滑移的硬件电路连接。

#### `L`（Load，装载）指令的物理步骤：

```
1. 执行装载前:
   +-----------------------+       +-----------------------+
   | ACCU 2:  [ 旧数据 B ] |       | ACCU 1:  [ 旧数据 A ] |
   +-----------------------+       +-----------------------+

2. 执行 L 指令过程中（滑移）:
   +-----------------------+       +-----------------------+
   | ACCU 2:  [ 旧数据 A ] | <==== | ACCU 1:  [ 旧数据 A ] | (数据被复制到 ACCU 2，旧数据 B 丢失)
   +-----------------------+       +-----------------------+

3. 完成装载后:
   +-----------------------+       +-----------------------+
   | ACCU 2:  [ 旧数据 A ] |       | ACCU 1:  [ 新装载数据]| (新数据写入 ACCU 1)
   +-----------------------+       +-----------------------+
```

---

#### 16位数据（INT）装载时的“符号位扩展”设计

在工业控制中，我们经常从模拟量模块（如 AI 模块）读取 16 位的有符号整数（INT），其范围是 `-32768 ~ +32767`。
当执行 `L "Analog_Input"`（将 16 位有符号数装入 32 位的 ACCU 1）时，CPU 会采用怎样的硬件设计？

*   **CPU 的处理机制：符号位扩展（Sign Extension）**
    CPU 不会简单地将 ACCU 1 的高 16 位填 `0`。因为如果是负数，高位填 `0` 将导致数值的符号彻底发生改变（例如，16位有符号数 `-1` 的二进制补码是 `1111_1111_1111_1111`。如果高 16 位补 `0`，数值将变成 `65535`，从而发生严重的数学运算错误）。
    因此，CPU 硬件在装载 16 位有符号数时，会**复制 16 位数据中的最高位（Bit 15，即符号位），并将其填充到 32 位累加器 ACCU 1 的高 16 位（Bit 16 到 Bit 31）中**。
    *   正数 `+100`（十六进制 `0064`）装载后变为：`0000_0064`
    *   负数 `-100`（十六进制 `FF9C`）装载后变为：`FFFF_FF9C`

### 3.2 知识点 3 代码：模拟量工程量转换（FC105 经典算法复现）

下面这段 STL 代码，是西门子系统自带的 FC105 模拟量标定算法的底层复现。该程序读取原始 AI 信号（0~27648），并将其线性转换为工程量实际物理值（例如 0.0 ~ 100.0 ℃）。

#### 完整 STL 代码

```stl
FUNCTION "Scale_Analog" : VOID
TITLE = 模拟量线性标定算法（纯寄存器级实现）
VERSION : 0.1
VAR_INPUT
   Raw_Value : INT ;        // 原始 AI 输入通道值 (0 ~ 27648)
   Hi_Limit : REAL ;        // 工程量上限 (例如 100.0)
   Low_Limit : REAL ;       // 工程量下限 (例如 0.0)
END_INPUT
VAR_OUTPUT
   Scaled_Value : REAL ;    // 标定后的实数工程量
END_OUTPUT
BEGIN
NETWORK
TITLE = 线性转换数学公式: Out = [(Raw / 27648.0) * (Hi - Low)] + Low
      // 1. 将 16位有符号有源输入 Raw_Value 转换为双整数，并转换为实数
      L     #Raw_Value;     // 装载 16位 INT，触发符号位扩展，推入 ACCU 1
      ITD   ;               // 累加器1中的 16位有符号整数 转换为 32位双整数 (DINT)
      DTR   ;               // 累加器1中的 32位双整数 转换为 32位浮点数 (REAL)
      
      // 2. 限制原始值范围，除以最大分度值 27648.0
      L     2.764800e+004;  // 装载浮点数常数 27648.0，原 ACCU1 值被压入 ACCU 2
      TAK   ;               // 交换 ACCU 1 与 ACCU 2 的值
      /R    ;               // 执行实数除法: ACCU2 / ACCU1 -> ACCU1 (即 Raw_Float / 27648.0)
      
      // 3. 计算极差 (Hi_Limit - Low_Limit)
      L     #Hi_Limit;      // 装载上限，上一步的中间结果被压入 ACCU 2
      L     #Low_Limit;     // 装载下限，上限值被压入 ACCU 2，中间结果被压入系统外部堆栈（或覆盖）
      -R    ;               // 计算 (Hi - Low)，结果存入 ACCU 1
      
      // 4. 乘以极差，再加上下限
      TAK   ;               // 交换累加器，使得 ACCU 1 为刚才算出的 Raw_Ratio，ACCU 2 为极差值
      *R    ;               // 乘法计算: Raw_Ratio * (Hi - Low)，结果存入 ACCU 1
      L     #Low_Limit;     // 装载下限
      +R    ;               // 最终相加得到实数工程量值
      T     #Scaled_Value;  // 写入输出通道
END_FUNCTION
```

#### 逐行深度剖析与寄存器状态跟踪表

假设：`Raw_Value = 13824`（正好是一半分度值），`Hi_Limit = 100.0`，`Low_Limit = 0.0`。

| 步骤 | 指令 | ACCU 1 (32-bit hex/float) | ACCU 2 (32-bit hex/float) | 状态字 CC1/CC0 | 寄存器滑移及数学管道微观剖析 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **0** | (开始) | `0000_0000` | `0000_0000` | `0/0` | 寄存器处于干净的初始状态。 |
| **1** | `L #Raw_Value` | `0000_3600` (13824) | `0000_0000` | `0/0` | `13824` 装入 ACCU1 低 16 位。由于最高位为 `0`（正数），高 16 位填充 `0`。 |
| **2** | `ITD` | `0000_3600` (13824) | `0000_0000` | `0/0` | 指令将 ACCU 1 中的 16 位整数转换为 32 位 DINT。对值本身无改动。 |
| **3** | `DTR` | `4658_0000` (13824.0) | `0000_0000` | `0/0` | 将双整数转换为实数表示。ACCU 1 中的二进制码转为 IEEE 754 标准浮点格式。 |

| **6** | `/R` | `3F00_0000` (0.5) | `46D7_E000` (27648.0) | `1/0` (大于0)| 执行实数除法：`13824.0 / 27648.0`。结果 `0.5` 存入 ACCU 1。 |

### 3.3 工程经验：处理除以零（Division by Zero）等数学异常保护

在连续的、无监督工业现场，模拟量断线、传感器故障会导致数据采集出现非正常值。如果在执行 `/R` 指令时，**除数（ACCU 1 里的值）不幸为 0**，会发生什么？

*   **硬件物理后果**：
    CPU 的硬件浮点计算器会产生严重错误（算术异常）。状态字中的 **OV（溢出位）** 和 **OS（存储溢出位）** 会瞬间被置 `1`，而计算出的结果值将变为 **NaN（非数字，Not a Number）**。
    如果该 `NaN` 传递给随后的 PID 运算，或者执行了某些数据强制转换，在经典 S7-300 系统中，会**直接触发 OB121 中断**。如果用户未下载 OB121 错误处理块，**PLC 会瞬间进入 STOP（停止）状态**，造成生产线意外停机。
*   **工程层面的预防手段**：
    在执行任何除法指令（`/I`、`/D`、`/R`）前，必须使用比较指令排除除数为 0 的状况：
    ```stl
    L     #Divisor_Value        // 装载除数
    L     0.0                   // 装载零常数
    ==R                         // 比较是否为 0
    JC    ERR                   // 若为0，跳转至错误处理标号 ERR，绕过除法计算
    
    // 正常计算路径
    L     #Dividend_Value
    L     #Divisor_Value
    /R
    T     #Result_Value
    JU    END
    
    ERR:  L     0.0
    T     #Result_Value         // 出错时输出安全默认值
    SET   
    =     "Calc_Error_Flag"     // 输出错误联锁信号
    
    END:  NOP   0
    ```

## 4. 程序调用机制：FB、FC 与局部数据堆栈（L-Stack）

程序块之间的调用机制是探寻 PLC CPU 运行原理的“核心拼图”。当我们写下 `CALL FB1, DB1` 时，CPU 在内存中正在进行着复杂的参数投递和指针漂移。

### 4.1 局部数据堆栈（L-Stack）的生命周期

**L-Stack（Local Data Stack，局域数据堆栈）** 是 CPU 内部高速 RAM 中的一块特殊物理存储区。每个程序块声明的临时变量（`VAR_TEMP`）在块被调用（`CALL`）时，就会在 L-Stack 中被动态分配一段内存空间。

```
+-------------------------------------------------------------+
| System L-Stack (局部数据栈总空间)                           |
|                                                             |
|  +---------------------------+                              |
|  | OB1 Frame                 |                              |
|  | - Temp Vars for OB1       |                              |
|  +---------------------------+                              |
|  | FC1 Frame (当 OB1 调用 FC1 时动态开辟)                    |
|  | - Temp Vars for FC1       | <=== 块结束瞬间，该空间物理释放 |
|  +---------------------------+                              |
|  | FB1 Frame (当 FC1 调用 FB1 时继续开辟)                    |
|  | - Temp Vars for FB1       |                              |
|  +---------------------------+                              |
+-------------------------------------------------------------+
```

*   **特点 1：动态生存期**。当该块退出时，为该块开辟的 L-Stack 空间立刻被系统回收。
*   **特点 2：初始随机性**。西门子 CPU 为了追求执行速度，**在开辟 L-Stack 空间时，默认不会对其进行清零（0）初始化**。该物理地址中存放的，完全是上一个刚刚退出的函数块残留下来的“垃圾数据”。这就是为什么**临时变量（TEMP）在未经过显式初始化赋值前，绝对不能直接读取**的原因。

---

### 4.2 TIA Portal 的“优化块”与“非优化块”的调用性能本质

*   **非优化块（标准访问）**：
    在调用非优化 FB 时，所有的参数传递都是通过**绝对物理地址偏移**实现的。
    例如，当你传递一个复杂的结构体（STRUCT）给非优化的 FB 时，CPU 会在后台生成一个 **ANY 指针**。在每次读取该参数时，CPU 必须调用 AR1/AR2，执行两级寻址：先找到 ANY 指针所指的物理 DB 地址，再根据偏移量读取实际变量。**多次指针间接寻址会导致较大的时钟周期开销。**
*   **优化块（符号访问）**：
    在 S7-1500 的优化访问块中，CPU 在编译阶段就已经通过编译表，将所有变量的寄存器引用进行了静态解析。
    在调用优化 FB 并传递结构体参数时，操作系统底层不再使用复杂的 ANY 物理指针，而是直接传递该结构体在优化内存区中的**直接指针（Ref）**。CPU 访问优化块参数时，就像访问本地局部变量一样快，从而实现了执行效率的飞跃。

### 4.3 知识点 4 代码：深入底层研究多重背景（Multi-Instance）FB 的指针漂移

在多重背景（Multi-Instance）设计中，子 FB（作为成员）被内嵌入父 FB 的背景 DB 中。此时，CPU 在调用子 FB 时，是如何知道当前的局部变量在背景 DB 中的具体物理偏移地址的？

这背后离不开 **AR2 寄存器** 的关键协助。

#### 完整 STL 代码

```stl
FUNCTION_BLOCK "Parent_Block"
TITLE = 父功能块：管理多重背景调用
VERSION : 0.1
VAR
   Valve_1 : "Sub_Valve_FB" ; // 子背景1：控制1号阀门
   Valve_2 : "Sub_Valve_FB" ; // 子背景2：控制2号阀门
END_VAR
BEGIN
NETWORK
TITLE = 启动1号阀门与2号阀门
      // 当我们在 LAD 中拉出一个多重背景块，
      // 底层编译器实际生成了如下所示的 STL 调用机制：
      
      TAR2  ;                 // 读取当前父块执行时的基准 AR2（多重背景偏移基准）
      L     P##Valve_1;       // 装载 1号阀门 实例在背景DB中的相对偏移地址 (例如 P#0.0)
      +D    ;                 // 将两地址相加，计算出 1号阀门 在物理背景DB中的绝对偏移量
      LAR2  ;                 // 将绝对偏移量载入 AR2。此时，AR2 成为 Valve_1 执行时的基准指针！
      
      CALL  #Valve_1 ;        // 物理调用子块逻辑。子块内部所有关于 #Start 的寻址，
                              // 在底层都会被编译为 [AR2, P#offset]
                              
      // 调用结束后，必须恢复父块的寻址环境
      TAR2  ;
      L     P##Valve_2;       // 加上2号阀门实例的物理相对偏移 (例如 P#10.0)
      +D    ;
      LAR2  ;                 // 重新定位 AR2 为 2号阀门 的绝对偏移地址
      CALL  #Valve_2 ;        // 物理调用 2号阀门
END_FUNCTION_BLOCK
```

#### 逐行深度剖析与寄存器状态跟踪表

假设：
父背景 DB 的基准物理地址为 `DB100`。
`Valve_1` 定义在父 FB 的起始字节偏移 0 处（即 `P##Valve_1 = P#0.0`），长度为 10 字节。
`Valve_2` 定义在父 FB 偏移 10 字节处（即 `P##Valve_2 = P#10.0`）。

| 步骤 | 指令 | AR 2 物理值 | ACCU 1 | DB / DI 寄存器 | 指针漂移与多重背景物理机制深度解读 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **0** | (父块被调用) | `P#DBX0.0` | `0` | `DI: 100` | 进入父 FB。系统将父块的 DI 背景寄存器设置为 100，AR2 物理偏移指向本实例基准 `P#DBX0.0`。 |
| **1** | `TAR2` | `P#DBX0.0` | `8400_0000` (`P#0.0`) | `DI: 100` | **传送 AR2 到累加器**。将当前 AR2 的物理基准地址复制到 ACCU 1 中。 |
| **2** | `L P##Valve_1`| `P#DBX0.0` | `0000_0000` (`P#0.0`) | `DI: 100` | 装载 Valve_1 在父块内部的相对偏移。由于排在最前面，其偏移值为 `P#0.0`。原 ACCU 1 的数据移入 ACCU 2。 |
| **3** | `+D` | `P#DBX0.0` | `8400_0000` (`P#0.0`) | `DI: 100` | 将相对偏移与父块基准相加，计算得到子块 `Valve_1` 在当前 DI100 中对应的绝对地址基准（`P#DBX0.0`）。 |
| **4** | `LAR2` | `P#DBX0.0` | `8400_0000` | `DI: 100` | 将计算出的物理基准重新载入 AR2。 |
| **5** | `CALL #Valve_1`| `P#DBX0.0` | - | `DI: 100` | 进入子功能块。子功能块内部如果有一行 `= #Output_Bit`（假定子块定义该变量偏移为 2.0），CPU 在底层实际执行：`= DIX [AR2, P#2.0]`。由于此时 `AR2 = P#0.0`，实际写入的是 `DB100.DBX2.0`。 |
| **6** | (返回父块) | `P#DBX0.0` | - | `DI: 100` | 调用结束返回。 |
| **7** | `TAR2` | `P#DBX0.0` | `8400_0000` (`P#0.0`) | `DI: 100` | 重新准备为 2 号阀门寻址。 |
| **8** | `L P##Valve_2`| `P#DBX0.0` | `0000_0050` (`P#10.0`) | `DI: 100` | 装载相对偏移：10 字节偏移量，即 80 个位偏移（十六进制 `50`，表示 `P#10.0`）。 |
| **9** | `+D` | `P#DBX0.0` | `8400_0050` (`P#10.0`) | `DI: 100` | 计算绝对基准：基准 `P#0.0` + 偏移 `P#10.0` = `P#10.0`。 |
| **10**| `LAR2` | `P#DBX10.0`| `8400_0050` | `DI: 100` | 重新定位 AR2 为 `P#DBX10.0`。 |
| **11**| `CALL #Valve_2`| `P#DBX10.0`| - | `DI: 100` | 进入 Valve_2。子块执行相同逻辑 `= #Output_Bit`，底层同样执行 `= DIX [AR2, P#2.0]`。但由于此时 `AR2` 已变更为 `P#10.0`，最终实际写入的是 `DB100.DBX12.0`。 |

### 4.5 工程经验：在多重背景 FB 内部使用 AR2 寻址的致命禁忌

在研究完多重背景的底层原理后，我们可以得出一条极为重要的、关乎生死安全的 PLC 编程禁忌：

*   **危险的操作**：
    在具有**多重背景（Multi-Instance）**属性的 FB 内，**绝对禁止直接使用 `LAR2` 指令来改写 AR2 寄存器**，除非你在执行改写前，完整保存了 AR2，并在退出前对其进行了恢复。
*   **物理后果**：
    如果你在子 FB 内部为了自己的某些指针逻辑改写了 AR2，且没有恢复现场。那么在执行完这段代码后，只要这个子 FB 试图读写其自身的局部变量（如输入参数、输出参数、静态变量 Static），**由于 AR2 基准已经被你改写，CPU 将完全定位到错误的物理内存空间上**。这不仅会导致后续的变量评估全部乱套，极有可能造成物理设备发生不受控制的非预期动作，引发安全事故。

## 5. CPU 执行顺序与指令管道优化

在最后一章中，我们将研究 CPU 顺序执行的底层逻辑，以及如何利用 STL 语言特有的**跳转表（Jump Table）**机制来设计高效率的状态机（Sequence Controller）。

### 5.1 顺序执行与程序计数器（Program Counter）

PLC 内部的控制运算器包含一个**程序计数器（PC，Program Counter）**。在没有遇到跳转指令前，PC 会随着每一行指令的物理执行而依次递增，使得程序按顺序向下扫描。

而当我们执行各种跳转指令时：
*   **`JU`（无条件跳转）**：强行修改 PC 寄存器的值为目标标号的绝对指令地址，实现直接分叉。
*   **`JC`（条件跳转）**：CPU 首先判断当前 `RLO` 位的值。
    *   若 `RLO = 1`，则更新 PC，跳转到目标。
    *   若 `RLO = 0`，则忽略本跳转，PC 递增，正常执行下一行紧跟的代码。

---

### 5.2 知识点 5 代码：基于 STL 跳转表（JL）的高效状态机（Sequence Controller）

在工业生产中，顺序控制逻辑（如：装料 -> 混料 -> 卸料 状态转换）非常普遍。在 LAD 中，我们不得不使用大量的比较指令并联来搭建状态机。而在 STL 中，我们可以使用极为底层的、物理级的**跳转表指令（JL）**来实现。

`JL` 指令可以直接根据当前的累加器 1（ACCU 1）中存储的整型状态代号，**在一条扫描周期内一步定位并直接跳转到对应的处理代码段**。其执行开销极低，没有多余的空闲逻辑扫描。

#### 完整 STL 代码

```stl
FUNCTION_BLOCK "Sequence_Controller"
VERSION : 0.1
VAR
   Current_State : INT := 0;  // 当前状态寄存器: 0-初始化, 1-装料, 2-混料, 3-卸料
   Timer_DB_No   : INT := 20; // 混料计时器 DB块编号
END_VAR
BEGIN
NETWORK
TITLE = 核心跳转路由表 (State Router)
      L     #Current_State;   // 装载当前状态值 (0, 1, 2, 3...) 到 ACCU 1
      JL    STATE_ERR;        // 跳转保护：如果 ACCU 1 越界（大于标号清单的总数），跳转至错误处理标号
      JU    STATE_INIT;       // ACCU 1 = 0 时的跳转目的地
      JU    STATE_FILL;       // ACCU 1 = 1 时的跳转目的地
      JU    STATE_MIX;        // ACCU 1 = 2 时的跳转目的地
      JU    STATE_DISCHARGE;  // ACCU 1 = 3 时的跳转目的地
STATE_ERR: JU  RESET_SEQ;     // 越界错误分支，强制将状态机重置

NETWORK
TITLE = 状态 0 逻辑：系统初始化 (STATE_INIT)
STATE_INIT: NOP 0;
      A     "System_Start";   // 系统启动信号 (I1.0)
      JC    GO_FILL;          // 若满足条件，迁移至状态 1
      BEU   ;                 // 否则，直接退出本块执行，不再执行后续无关的网络！

GO_FILL: L   1;
      T     #Current_State;   // 状态变更为 1
      BEU   ;

NETWORK
TITLE = 状态 1 逻辑：产品装料阶段 (STATE_FILL)
STATE_FILL: NOP 0;
      A     "Material_Full";  // 装料完成检测开关 (I1.1)
      JC    GO_MIX;
      SET   ;
      S     "Fill_Valve";     // 开启装料阀物理输出 (Q1.0)
      BEU   ;

GO_MIX: R   "Fill_Valve";     // 关阀
      L     2;
      T     #Current_State;   // 状态变更为 2
      BEU   ;

NETWORK
TITLE = 状态 2 逻辑：电机高速混料阶段 (STATE_MIX)
STATE_MIX: NOP 0;
      // 假定混料时间到了，跳转至卸料
      A     "Mix_Time_Done";  // 混料延时到 (I1.2)
      JC    GO_DISCH;
      SET   ;
      S     "Mixer_Motor";    // 启动混料电机 (Q1.1)
      BEU   ;

GO_DISCH: R "Mixer_Motor";    // 停电机
      L     3;
      T     #Current_State;   // 状态变更为 3
      BEU   ;

NETWORK
TITLE = 状态 3 逻辑：成品卸料阶段 (STATE_DISCHARGE)
STATE_DISCHARGE: NOP 0;
      A     "Material_Empty"; // 卸料空检测开关 (I1.3)
      JC    RESET_SEQ;
      SET   ;
      S     "Discharge_Valv"; // 开启卸料阀物理输出 (Q1.2)
      BEU   ;

RESET_SEQ: R "Discharge_Valv"; // 关阀
      L     0;
      T     #Current_State;   // 状态回归 0，开始下一轮循环
      BEU   ;
END_FUNCTION_BLOCK
```

#### 逐行深度剖析与寄存器状态跟踪表

假设当前状态：`Current_State = 2`（代表正在执行混料控制阶段）。

| 步骤 | 指令 | ACCU 1 (十进制值) | RLO | /FC | 程序计数器 (PC) 指向的执行流向 | 状态机底层微观物理动作解析 |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| **1** | `L #Current_State` | `2` | `0` | `0` | 路由段起点 | 将状态值 `2` 装入 ACCU 1 中。准备进行多路跳转匹配。 |
| **2** | `JL STATE_ERR` | `2` | `0` | `0` | 多路分支判定 | `JL` 指令检查跳转表容量。表内共定义了 4 个目标（第 4、5、6、7 行）。由于 `ACCU1 = 2` 未超出范围（`0`到`3`），CPU 不会跳转到 `STATE_ERR`。 |
| **3** | `JU STATE_INIT` | `2` | `0` | `0` | 越过（ACCU1=0 对应行） | CPU 自动计算：`2 *（跳转指令宽度）`。定位跳转到列表中的第三个物理目标。 |
| **4** | `JU STATE_FILL` | `2` | `0` | `0` | 越过（ACCU1=1 对应行） | 忽略该条无条件跳转。 |
| **5** | `JU STATE_MIX` | `2` | `0` | `0` | 跳转指令被触发 | **命中匹配**。CPU 强行修改 PC 指针，使其指向 `STATE_MIX` 标号对应的内存物理起始点。接下来的指令全部被跨越，不参与本周期扫描。 |
| **6** | `STATE_MIX: NOP 0` | `2` | `0` | `0` | 跨越跳转，直接降落于此 | 标号占位符指令。由于直接跳跃至此，之前无关的网络（初始化、装料等网络）的物理代码完全没有被 CPU 扫描读取，节省了宝贵的扫描时钟周期。 |
| **7** | `A "Mix_Time_Done"`| `2` | `0` / `1` | `1` | 判定混料是否完成 | 进行新一轮首检。如果未完成（信号为 `0`），则 RLO 变为 `0`。 |
| **8** | `JC GO_DISCH` | `2` | `0` | `0` | 分支判定 | 因为此时 `RLO = 0`，条件不成立。CPU 跳过跳转，正常向下执行。 |
| **9** | `SET` | `2` | `1` | `1` | 强制置 1 | 强行置 RLO 为 `1`，为下一步的置位（`S`）准备就绪。 |
| **10**| `S "Mixer_Motor"` | `2` | `1` | `0` | 电机保持输出 | 驱动 Q1.1 物理接触器置位。 |
| **11**| `BEU` | `2` | `1` | `0` | **块无条件结束退出** | **极其关键的硬件截断机制**。无条件提前结束当前块的运行。不再执行后续任何关于状态 3 的网络代码。本周期程序扫描安全完结。 |

### 5.4 工程经验：如何设计高鲁棒性、零死锁的状态机

跳转表（`JL`）在提供极致扫描速度的同时，也隐藏着一个巨大的工程设计隐患：**如果由于偶发性干扰、指针计算错误或人为修改，导致 `Current_State` 出现了一个非法、非预期的数据（例如 4、10 或 -12），状态机应当如何自处？**

如果没有安全熔断机制，程序将发生**死锁**：CPU 将无法跳转到对应的状态段。如果跳转表的参数校验不严格，甚至会引发严重的寻址溢出崩溃。

#### 鲁棒性设计三板斧：
1.  **首选无条件防越界保护（Guard Check）**：
    在 `JL` 路由表的顶部，必须提供越界捕获标号。在我们的代码中，由 `JL STATE_ERR` 承担这一防线。一旦 `Current_State` 的值超出了后面定义的跳转标签总数，`JL` 会强制让 CPU 跳转至 `STATE_ERR` 错误处理，而不去访问非法路由。
2.  **避免非法负数穿透**：
    在装载 `Current_State` 后，如果其可能存在负数值（INT 类型允许），建议加一行逻辑判断：
    ```stl
    L     #Current_State
    L     0
    <I                          // 判断是否小于 0
    JC    STATE_ERR             // 一旦小于 0，立刻强制跳转到错误复位保护
    ```
3.  **在每段状态的末尾强制使用 `BEU`（无条件结束）**：
    由于跳转表的机制是物理分流，**每一段处理状态的尾部必须加上 `BEU`。**
    如果不加 `BEU`，当 CPU 跳转到 `STATE_INIT` 执行完初始化代码后，会继续向下漏（顺序扫描）执行 `STATE_FILL`、`STATE_MIX` 的代码，从而导致整个顺序控制流程发生灾难性的并发逻辑混乱。
