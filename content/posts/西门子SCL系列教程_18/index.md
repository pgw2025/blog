# 


在上一章中，我们一起攻克了 TIA 博途时代的强类型武器——`VARIANT` 指针引用。那是在高级数据架构和变长数组管理中的降维打击。

今天，我们要跨入一个全新，但在现代数字化工厂（尤其是汽车、锂电、食品、医药包装）中出镜率极高的领域——**字符串（String）数据处理**。

在当今工业 4.0 的浪潮下，PLC 不再仅仅和阀门、电机打交道。我们需要通过以太网（TCP/IP）、串口（Modbus）或者总线，去读取扫码枪（条码/二维码）、RFID 读写头、电子秤、以及上位机 MES 系统发送过来的 ASCII 文本报文。

如果你尝试用梯形图（LAD）去解析一串诸如 `"SN_20231024_A15_9999"` 的文本，你得画出密密麻麻、无休无止的数组转移和字符比较网络，这简直是折磨。

而在 SCL 中，西门子为我们配备了完全比肩高级开发语言（如 C#、Python）的一整套**高阶字符串处理函数**。

今天，师父带你彻底扒开西门子字符串在物理内存中的“头部结构”，剖析字符串操作对 CPU 扫描周期的算力损耗，并带你手写一个生产级的**“汽车总装线扫码枪报文深度解析、清洗与自适应标定系统”**。

## 1. 物理本质：西门子 String 字符串的物理骨架

在西门子 S7-1200/1500 内部，`STRING` 类型的存储方式并不是一串无脑排列的字符。它拥有一个非常特殊的 **“双字节头部结构（Header）”**。

我们用下面这张物理内存结构图，彻底拆解一个声明为 `String[10]` 的变量，在装载了 `"HELLO"` 文本时的真实物理面貌：

### 1.1 ⚠️ 致命红线：网口直接通讯写入导致的“空字符串 Bug”

徒弟，请务必记住这个在现场调试网络通信时，**高达 90% 发生率的经典大坑**：

如果你通过 TCP（`TRCV`）或者串口自由口通信，直接把外部扫码枪发回来的 5 个 ASCII 字节 `"HELLO"`，覆盖写入到了 `String` 变量的 `Byte 2` 到 `Byte 6` 中。

当你随后在 SCL 中调用 `LEN("你的String")` 或者把它显示在触摸屏上时，**系统会固执地认为这个字符串是个“空字符串（""）”！**

*原因*：
因为网络接收函数是无脑写入物理内存的，它**没有去修改 Byte 1（当前长度，Current_Length）的值**。此时 Byte 1 依然是默认的 `16#00`。PLC 内部的所有字符串指令只认 Byte 1。既然 Byte 1 是 0，它就认为这个字符串长度为 0！

*高级防范手段*：
在接收到物理报文后：
1.  要么在 SCL 内部，手动用绝对偏移寻址强行改写该字符串的 `Byte 1` 为真实的接收长度。
2.  **或者（推荐）**：使用西门子标准的转换系统指令 **`Chars_TO_Strg`**（将字符数组安全标定组装为字符串）。

## 2. 五大核心 SCL 字符串处理兵器

在 SCL 中，我们主要利用以下五大金刚对文本进行切割、合并、长度探针操作：

### 2.1 `LEN(IN := <STRING>)`
*   **物理动作**：直接读取字符串头部的 **`Byte 1`**（当前实际长度），并以整型 `INT` 输出。

---

### 2.2 `LEFT(IN := <STRING>, L := <INT>)`
*   **物理动作**：从字符串的**最左侧开始**，截取指定长度 `#L` 的子字符串并输出。
*   **SCL 示例**：
    ```scl
    #sSub := LEFT(IN := 'SN_123', L := 3); // 截取最左边 3 个字符，输出 '#sSub' = 'SN_'
    ```

---

### 2.3 `RIGHT(IN := <STRING>, L := <INT>)`
*   **物理动作**：从字符串的**最右侧开始**，截取指定长度 `#L` 的子字符串并输出。

---

### 2.4 `MID(IN := <STRING>, L := <INT>, P := <INT>)`
*   **物理动作**：从指定的物理起始位置 `#P` 开始，截取长度为 `#L` 的子字符串并输出。
*   **注意（1位偏移对齐）**：西门子字符串中的字符位置 `#P` 也是 **从 1 开始** 的（1 代表 `Byte 2` 即第一个字符，2 代表 `Byte 3`，以此类推）。

---

### 2.5 `CONCAT(IN1 := <STRING>, IN2 := <STRING>, ...)`
*   **物理动作**：将多个独立的字符串，首尾无缝拼接成一根大长字符串并输出。

---

## 3. 性能深度解剖：字符串操作在 PLC 内部的算力暴增隐患

在 PC 端写 C# 或者是 Python，我们对字符串的使用肆无忌惮。**但在 PLC 内部，字符串操作是极度消耗 CPU 扫描时间的“重体力劳动”。**

*硬件微观真相*：
当你在 SCL 内部执行一次 `CONCAT` 拼接时，CPU 必须：
1.  在 L-Stack 中开辟临时缓冲区。
2.  通过底层的微循环（类似于 FOR 循环），将第一个字符串的每个字节依次拷贝过去。
3.  接着将第二个字符串的每个字节依次拷贝过去。
4.  计算总长度，重新改写新字符串的 Byte 1 头信息。

如果你在 OB1 中同时对 50 个 `String[254]`（默认长度）执行 `CONCAT` 和 `MID` 截取。由于需要频繁拷贝大量的 RAM 内存块，**PLC 的扫描周期可能会从正常的 2ms 骤然暴涨到 15ms 以上！**

**师父的优化守则**：
*   **切忌无脑使用默认的 `String`（默认长度254字节）**。
*   如果你的扫码枪条码最大只有 20 位，请务必将其精准声明为 **`String[20]`**。
*   这不仅能节省 10 倍以上的 DB 工作内存，还能成倍减少 CPU 在底层进行内存拷贝时的算力开销。

## 4. 工业级综合案例：汽车总装线扫码枪报文深度清洗与数据格式化系统

现在，我们将展现大厂高级项目经理的顶层设计能力。

### 4.1 工业背景与工艺要求
在某知名新能源汽车总装线上，当白车身（BIW）流转到底盘合装工位时，工业扫码枪会抓拍扫描车身上的钢印二维码。
*   **扫码枪通过 TCP 传输发回的原始未加工字符串格式**：
    `"$02SN_20231024_A15_9999$03"`
    *   其中 `"$02"` 代表 ASCII 控制字符 STX（Start of Text，帧头，十六进制 `16#02`）。
    *   其中 `"$03"` 代表 ASCII 控制字符 ETX（End of Text，帧尾，十六进制 `16#03`）。
    *   有效核心条码为：`"SN_20231024_A15_9999"`。
*   **条码包含以下核心工艺数据（必须清洗提炼）**：
    *   **头部校验**：必须是以 `"SN_"` 开头，否则判定为“无效条码”。
    *   **生产日期（Date Code）**：条码中第 4 位开始的 8 位字符（如 `"20231024"`），我们需要将其截取出来，并**转换为 DINT 整数**输出，以便与 PLC 内部的排班时间做代数比较。
    *   **车型代号（Model Code）**：日期后、下划线之间的 3 位字符（如 `"A15"`）。
    *   **生产序列流水号（Serial Number）**：条码最后 4 位（如 `"9999"`），我们需要将其**转换为 INT 整数**，以便统计当班产线负荷。
3.  **结果反馈要求**：
    *   提取出车型代号后，通过 `CONCAT` 拼接成一根清爽的反馈控制字输出给上位机 MES：`"MODEL:A15_OK"`。

---

### 4.2 步骤一：块接口声明（FB_BarcodeCleaner）

因为我们需要使用系统转换指令以及可能的时间监控，所以我们采用 **FB**。我们对输入的字符串指定长度为 `35`，以此优化内存开销。

```
VAR_INPUT
    sRawBarcode : String[35];   // 扫码枪发回的原始带帧头尾报文
    bTriggerClean : Bool;       // 触发清洗解析脉冲
END_VAR

VAR_OUTPUT
    bCleanSuccess : Bool;       // 解析清洗成功
    bBarcodeError : Bool;       // 扫码枪数据格式异常警报
    diProductionDate : DInt;    // 提炼并转换后的生产日期整型 (例如 20231024)
    sModelCode : String[5];     // 提炼出的车型代号 (例如 'A15')
    iSerialNumber : Int;        // 提炼出的流水号 (例如 9999)
    sFeedbackMes : String[25];  // 拼接好发给 MES 的反馈字符串
END_VAR

VAR
    bCleanTrigger_FP : Bool;    // 边沿锁存
END_VAR

VAR_TEMP
    iTotalLen : Int;            // 原始报文总长度
    sCleanedCode : String[30];  // 剥离了帧头帧尾后的核心有效条码
    sHeader : String[3];        // 截取出的头部
    sDateTemp : String[10];     // 临时日期文本
    sSerialTemp : String[6];    // 临时流水号文本
    
    // 系统转换函数所需的状态变量
    iConvRetVal : Int;          // STRG_VAL 转换返回值
END_VAR
```

### 4.3 步骤三：SCL 代码实现

```scl
FUNCTION_BLOCK "FB_BarcodeCleaner"
{ S7_Optimized_Access := 'TRUE' }
VERSION : 0.1
BEGIN
	// ==========================================
	// 1. 触发边沿捕捉与安全防御
	// ==========================================
	IF #bTriggerClean AND NOT #bCleanTrigger_FP THEN
	    
	    // 初始化输出，防尘底牌
	    #bCleanSuccess := FALSE;
	    #bBarcodeError := FALSE;
	    #diProductionDate := 0;
	    #sModelCode := '';
	    #iSerialNumber := 0;
	    #sFeedbackMes := '';
	    
	    // 获取原始报文实际长度
	    #iTotalLen := LEN(#sRawBarcode);
	    
	    // 安全诊断：标准报文长度 "$02SN_20231024_A15_9999$03" 应该正好是 23 个字符
	    // 如果长度低于 20，直接判定为“断线或残缺报文”，安全退出
	    IF #iTotalLen < 20 THEN
	        #bBarcodeError := TRUE;
	        RETURN;
	    END_IF;
	    
	    // ==========================================
	    // 2. 剥离 ASCII 帧头 "$02" 与 帧尾 "$03" (数据清洗)
	    // $02 在第 1 位， $03 在最后一位。
	    // 核心有效长度为：总长度 - 2 字节。起算位置在第 2 位。
	    // ==========================================
	    #sCleanedCode := MID(IN := #sRawBarcode, L := #iTotalLen - 2, P := 2);
	    
	    // 此时 sCleanedCode 应为 "SN_20231024_A15_9999"
	    
	    // ==========================================
	    // 3. 头部合法性校验 (首码验证)
	    // 截取最左边 3 个字符，验证是否是 "SN_"
	    // ==========================================
	    #sHeader := LEFT(IN := #sCleanedCode, L := 3);
	    
	    IF #sHeader <> 'SN_' THEN
	        #bBarcodeError := TRUE; // 头部不对，判定为非法其他条码
	        RETURN;
	    END_IF;
	    
	    // ==========================================
	    // 4. 工艺字段高速截取与提炼
	    // ==========================================
	    // A. 截取生产日期：从第 4 位开始，截取 8 个字符（对应 "20231024"）
	    #sDateTemp := MID(IN := #sCleanedCode, L := 8, P := 4);
	    
	    // B. 截取车型代号：从第 13 位开始，截取 3 个字符（对应 "A15"）
	    #sModelCode := MID(IN := #sCleanedCode, L := 3, P := 13);
	    
	    // C. 截取生产流水号：从第 17 位开始，截取 4 个字符（对应 "9999"）
	    #sSerialTemp := MID(IN := #sCleanedCode, L := 4, P := 17);
	    
	    // ==========================================
	    // 5. 跨数据阵营转换：文本安全转整型
	    // 我们利用系统标定指令 STRG_VAL（字符串转换为数值）
	    // ==========================================
	    // A. 将日期文本 "20231024" 转换为双整型 20231024
	    STRG_VAL(IN := #sDateTemp,
	             FORMAT := 16#0000, // 16#0000 代表无符号整数/普通数字格式
	             OUT => #diProductionDate);
	             
	    // B. 将流水号文本 "9999" 转换为整型 9999
	    STRG_VAL(IN := #sSerialTemp,
	             FORMAT := 16#0000,
	             OUT => #iSerialNumber);
	             
	    // ==========================================
	    // 6. 反馈消息拼接 (CONCAT 高效合成)
	    // 目标格式: "MODEL:A15_OK"
	    // ==========================================
	    #sFeedbackMes := CONCAT(IN1 := 'MODEL:',
	                            IN2 := #sModelCode,
	                            IN3 := '_OK');
	                            
	    // 宣布全面解析清洗成功！
	    #bCleanSuccess := TRUE;
	    
	END_IF;
	#bCleanTrigger_FP := #bTriggerClean; // 锁存
	
END_FUNCTION_BLOCK
```

## 5. 深度解剖实战代码的“工业标准化设计思维”

徒弟，这套扫码枪清洗逻辑在重工业流水线上极其通用。我们来看看它是如何体现出“大厂标准级”的防御性编程的。

### 5.1 剥离控制字符的二分法偏置（第 28 行）
在第二部分中，我们写了这一行：
`#sCleanedCode := MID(IN := #sRawBarcode, L := #iTotalLen - 2, P := 2);`
为什么长 L 是 `#iTotalLen - 2`，起算点 P 是 `2`？

*物理对齐解析*：
因为原始数据自带帧头 `"$02"`（占 1 字节）和 帧尾 `"$03"`（占 1 字节）。
我们要把最头上的 1 字节扔掉，所以起点 P 必须设为 `2`（跳过第 1 个控制字符）。
因为头尾各占 1 字节，所以核心有效长度必须是总长度减去 2（`#iTotalLen - 2`）。
**只有这样，算出的物理切片，才能够严丝合缝、不偏不倚地精准提取出 `"SN_20231024_A15_9999"`，绝不会产生前后错位 1 字节导致的“多出下划线”或“字符残缺”。**

---

### 5.2 跨越数据阵营的桥梁：`STRG_VAL` 指令
在第五部分，我们截取出的 `#sSerialTemp` 是一个 **`String`** 类型的 `'9999'`。
如果后面我们需要判定：“当流水号累计到 9999 时，说明这一班的产量已满，触发封箱动作。”
你绝对不能在 SCL 里写：`IF #sSerialTemp = '9999' THEN ...`（那是字符串比较，速度极慢，且无法做代数比较）。
我们必须通过 **`STRG_VAL`** 系统转换指令，将它转换为真正的 **`INT`（9999）** 存储在物理寄存器中。只有转换为整型，PLC 才能以微秒级的速度执行高速数学代数运算。

## 6. 常见错误与避坑指南 (Senior Tips)

### 6.1 错误一：用 `String` 作为引脚时的“动态超频”报警
在博途中编写 FC/FB 接口时，如果你声明的输入输出引脚是 `String`，外部实参传递的是一个 `String[254]`。
如果你没有指定局部变量长度，直接在内部进行多次 `CONCAT` 拼接。
*致命后果*：
由于拼接后的大字符串可能超出了你接收变量的最大限制（比如你只声明了 `sFeedbackMes : String[25]`，但你拼接出的字符实际上有 30 个）。
**西门子 CPU 不会截断它，而是会瞬间抛出“字符串长度超出定义上限”运行时故障，并将 CPU 直接拉入 STOP 状态！**
**防范机制**：**在任何时候，只要进行 CONCAT 拼接，必须首先在纸上计算出可能拼接出的最大极限物理长度。确保接收变量的 STRING[N] 中的 N，绝对大于或等于这个最大物理极限！**

---

### 6.2 错误二：频繁在循环（FOR/WHILE）内部去读取 `LEN()`
```scl
// ❌ 导致 CPU 扫描周期暴涨的愚蠢写法
FOR #i := 1 TO LEN(#sMyString) DO
    // 在循环体内多次调用 LEN
END_FOR;
```
*原因*：每次循环，CPU 都要去读写一次 String 变量的 Byte 1，这相当于每次都要执行一次指针访问。
*高阶优化*：
```scl
//  高效率缓存寻址
#iStringLen := LEN(#sMyString); // 只在循环前读取一次，存入临时变量
FOR #i := 1 TO #iStringLen DO
    ...
END_FOR;
```

### 练习 2：RFID 物流托盘配方一键追溯路径合成器 (CONCAT 练习)
在立体仓库的辊筒线上，托盘上贴有 RFID 电子标签。标签内读出的条码格式为：`"BATCH_05"`（代表批次号），以及工件类型 `"TYPE_A"`。
为了形成完整的 MES 系统追溯路径，我们需要合成一个标准的服务器 API 检索地址格式：
`"SERVER_IP/QUERY?BATCH=05&TYPE=TYPE_A"`。
请编写一个 SCL FC，使用 **`CONCAT`** 拼接大法：
*   **输入**：

*   **要求**：
    1. 在 SCL 内部，使用 `MID` 或 `RIGHT` 先从 `"BATCH_05"` 中提取出 `"05"`。
    2. 一键拼接出完整的 URL：`"SERVER_IP/QUERY?BATCH="` + `"05"` + `"&TYPE="` + `"TYPE_A"`。
