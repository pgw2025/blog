# Linux iostat 使用教程：系统级磁盘 I/O 性能分析与调优指南

在 Linux 系统的性能调优与故障排查中，磁盘 I/O 性能通常是制约系统整体吞吐量和响应延迟的常见瓶颈之一。无论是运行高并发事务的数据库，还是承载高频文件读写的 Web 应用，一旦磁盘 I/O 出现堆积，系统往往表现为响应缓慢、CPU `iowait` 飙升甚至服务直接卡死。

作为 `sysstat` 工具包中最核心的成员之一，`iostat`（Input/Output Statistics）是每一位 Linux 运维工程师、SRE、DBA 必须熟练掌握的系统级排查利器。它能够以极低的系统开销，实时展现 CPU 的工作状态以及各物理磁盘、逻辑分区的读写吞吐量、I/O 请求延迟和队列堆积情况。

本文将从 Linux 内核 I/O 协议栈的设计原理出发，层层剖析 `iostat` 的数据来源、各项关键指标的数学定义，并结合 25 个真实的生产环境排查案例，帮助你建立一套结构化的磁盘性能分析与调优思维。

# 第一章 什么是 iostat

## 1.1 为什么需要 iostat 与解决什么问题
在现代计算机架构中，CPU 的处理速度（纳秒级）与传统机械硬盘（毫秒级）或固态硬盘（微秒级）的读写速度存在着几个数量级的鸿沟。为了缓解这种速度不对称，Linux 内核设计了极其复杂的页面缓存（Page Cache）、I/O 调度器（I/O Schedulers）和块设备驱动。

然而，这种复杂的设计也带来了一个副作用：**应用层的读写操作并非直接对应底层的物理 I/O**。当生产环境发生以下性能问题时，我们很难直接通过应用日志定位：
* **系统响应突然变慢**，但 CPU 占用率却处于低位，内存也未被占满。
* **磁盘 %util 达到 100%**，是否意味着硬件性能已达极限？
* **数据库执行事务耗时剧增**，究竟是由于写入量超载，还是由于磁盘硬件开始出现老化？
* **RAID 阵列或云盘**在高并发下是否达到了其标称的 IOPS 和吞吐量上限？

`iostat` 正是为了回答这些问题而设计的。它不仅能提供宏观的磁盘吞吐量（MB/s）和每秒传输次数（tps），更能深入展现每一次 I/O 操作的**平均等待时间（await）**、**请求队列长度（avgqu-sz）**，帮助我们精准定位系统的 I/O 边界。

## 1.2 iostat 的定位
`iostat` 在 Linux 性能调优工具矩阵中扮演着“**第一阶段全局探针**”的角色。
* **它不直接追踪具体的进程**，这是它与 `iotop` 或 `pidstat` 的区别。
* **它专注于揭示“硬件与操作系统”层面的整体健康状态**。当系统性能出现滑坡时，典型的分析链路通常是先通过 `top` 观察全局 CPU `iowait` 状态，接着调用 `iostat` 锁定具体故障的设备与 I/O 特征，最后通过 `iotop`、`pidstat`、`strace`、`bcc-tools` 等追踪到具体的进程与文件描述符。

## 1.3 工作原理与内核数据来源
`iostat` 本质上是一个用户态的数据聚合与渲染工具，它对系统的 CPU 和磁盘并无主动探针消耗，而是通过周期性读取 Linux 内核在虚拟文件系统中公开的**性能计数器**来完成统计。

```
+------------------+
|      iostat      |  (读取并计算时间差值)
+--------+---------+
         |
         v
+--------+---------+
|  /proc/diskstats |  <--- 记录每个块设备的 I/O 状态计数器
|  /proc/stat      |  <--- 记录 CPU 状态计数器
+--------+---------+
         ^
         |  (内核定时/事件累加)
+--------+---------+
|   Linux 内核     |  (I/O 调度层、块设备驱动层、CPU 调度器)
+------------------+
```

### 1.3.1 CPU 数据来源：`/proc/stat`
`/proc/stat` 记录了系统自启动以来 CPU 在各种模式下运行的累积 tick 数（时间片单位）。`iostat` 通过在两次采样区间内获取该文件数值的差值，计算出各种 CPU 指标的比率。

### 1.3.2 磁盘数据来源：`/proc/diskstats`
对于每个块设备，内核在 `/proc/diskstats` 中维护了 10 多个计数器。以下是内核源码中记录的关键字段：

| 字段编号 | 字段名称 | 描述 |
| :--- | :--- | :--- |
| 1 | `num_reads` | 成功完成的读请求总数 |
| 2 | `num_reads_merged` | 被合并（Merge）的读请求数 |
| 3 | `sectors_read` | 读取的扇区总数 |
| 4 | `time_reading` | 读请求所消耗的总时间（毫秒） |
| 5 | `num_writes` | 成功完成的写请求总数 |
| 6 | `num_writes_merged` | 被合并的写请求数 |
| 7 | `sectors_written` | 写入的扇区总数 |
| 8 | `time_writing` | 写请求所消耗的总时间（毫秒） |
| 9 | `io_in_progress` | 当前正在处理中的 I/O 请求数（队列深度） |
| 10 | `time_in_queue` | I/O 活动及等待的总时间（毫秒） |

`iostat` 记录两次采样的计数值差值 $\Delta t$，将其除以采样间隔时间，便可计算出吞吐量、tps、单个 I/O 的响应时间等衍生指标。

## 1.4 sysstat 套件介绍
`iostat` 是 Linux 性能监控工具包 `sysstat` 的一部分。该套件包含了许多经典监控命令：
* **`sar`**：系统活动报告，可记录和回溯系统多维度性能指标的历史数据。
* **`mpstat`**：多核 CPU 性能统计。
* **`pidstat`**：进程级性能统计。
* **`sa1` / `sa2`**：后台定时数据收集脚本。

## 1.5 与常见 I/O 性能工具的区别

```
                 [系统全局监控]
                       |
        +--------------+--------------+
        |                             |
  [硬件与系统层]                 [进程与事务层]
   - iostat (I/O全局趋势)         - iotop (谁在读写)
   - vmstat (内存与I/O页交换)     - pidstat (进程资源占用)
   - sar (历史趋势回溯)          - perf / bcc (内核内核栈分析)
```

* **`vmstat`**：专注于内存页交换（swap in/out）、系统上下文切换和 CPU 整体负载，磁盘方面仅显示宏观的 `bi`（块输入）和 `bo`（块输出），不提供单个磁盘设备的详细分析。
* **`sar`**：虽然可以使用 `sar -d` 采集磁盘指标，但它更适用于非交互式的历史趋势回溯，而 `iostat` 则更适合高频度、交互式的实时排查。
* **`top`**：只能提供 CPU %wa（iowait）这一间接的、可能存在误导性的指标，完全无法感知是哪块硬盘遇到了何种读写瓶颈。
* **`iotop`**：直接追踪用户态进程。它能够展现“哪个 PID 正在以多少 MB/s 读写磁盘”，但由于其采用进程行为轮询或 Netlink 连接，在高吞吐的生产环境下开启 `iotop` 本身会产生明显的性能损耗；而 `iostat` 仅读取内核计数器，其开销可以忽略不计。

# 第二章 安装

在绝大多数 Linux 发行版中，`iostat` 并不会默认作为独立命令存在，而是封装在 `sysstat` 软件包内。

## 2.1 各发行版安装步骤

### CentOS 7/8, Rocky Linux, AlmaLinux
在 RHEL 及其衍生版本中，可以直接使用 `yum` 或 `dnf` 安装：
```bash
# CentOS 7 使用 yum
sudo yum install -y sysstat

# Rocky Linux / AlmaLinux 使用 dnf
sudo dnf install -y sysstat
```

### Ubuntu, Debian
在 Debian 系发行版中，使用 `apt` 软件包管理器：
```bash
sudo apt update
sudo apt install -y sysstat
```

* **特别提示**：在 Debian/Ubuntu 下，安装完 `sysstat` 后，其默认可能并未激活后台收集。你可以通过修改 `/etc/default/sysstat`，将 `ENABLED="false"` 修改为 `ENABLED="true"`，然后重启服务：
```bash
sudo systemctl enable --now sysstat
```

## 2.2 验证安装与查看版本
安装完成后，通过查看版本号来验证工具是否可以正常使用：
```bash
iostat -V
```
* **输出示例**：
  ```text
  sysstat version 12.5.4
  (C) Sebastien Godard (sysstat <at> orange.fr)
  ```
* **版本兼容避坑**：不同版本的 `sysstat` 对部分磁盘指标的命名或计算方式可能有所微调（例如 `svctm` 指标在较新版本中已被明确标注为废弃，不再推荐使用；且某些版本增加了 `%dscl` 丢弃请求字段）。在生产排查时，建议优先通过 `iostat -V` 确认当前运行的版本。

# 第三章 基础使用

`iostat` 命令的参数非常丰富，支持对输出粒度、数据单位、特定设备以及输出格式进行高自由度的定制。

---

### iostat（默认无参数）
* **作用**：显示系统自启动以来的 CPU 平均利用率和所有磁盘设备的平均 I/O 状态汇总。
* **语法**：`iostat`
* **输出解释**：默认输出由两部分组成。第一部分是 `avg-cpu` 统计，第二部分是 `Device` 统计，单位默认为 blocks/s（块/秒）。
* **注意事项**：不带任何采样间隔参数的 `iostat` 输出，其数据是**系统自开机以来的历史累积平均值**。如果系统已经运行了数月，开机以来的均值会彻底掩盖当前的瞬时突发瓶颈。因此，在日常性能分析中，**几乎从不使用无参数命令**。

---

### iostat 1
* **作用**：每隔 1 秒实时刷新一次 CPU 和磁盘的 I/O 状态。
* **语法**：`iostat <interval>`
* **适用场景**：实时观测当前业务高峰对系统磁盘造成的冲击，观察流量的波动。
* **注意事项**：输出会无休止地在屏幕上滚动，直至手动按下 `Ctrl + C`。

---

### iostat 1 5
* **作用**：每隔 1 秒进行一次数据采集并打印，在连续采集 5 次后自动退出。
* **语法**：`iostat <interval> <count>`
* **适用场景**：脚本化收集、性能测试期间的定时采样。
* **最佳实践**：第一次输出的数据依然是开机以来的平均值，而**从第二次输出开始，展示的才是真正的 1 秒采样区间内的瞬时状态**。因此，做性能分析时，应当忽略第一次的输出，重点看后续采样。

---

### iostat -x
* **作用**：输出扩展属性（Extended statistics），展示最全面的磁盘性能细节指标。
* **语法**：`iostat -x`
* **适用场景**：**线上排查的核心参数**。只有带上 `-x`，才会输出 `await`（延迟）、`avgqu-sz`（队列）、`%util`（繁忙度）等至关重要的排障指标。
* **最佳实践**：推荐使用 `iostat -xz 1` 组合。

---

### iostat -xz
* **作用**：排除那些在采样周期内**完全没有任何 I/O 活动**的空闲设备。
* **语法**：`iostat -xz [interval] [count]`
* **适用场景**：当物理服务器挂载了数十个甚至上百个块存储设备（如多路径外挂存储或 Kubernetes 节点挂载的大量 PV）时，如果不加 `-z`，屏幕会被零值设备霸屏，很难一眼找到真正繁忙的那个磁盘。

---

### iostat -d
* **作用**：仅输出磁盘设备（Device）的 I/O 指标，屏蔽顶部的 CPU 数据。
* **语法**：`iostat -d [interval] [count]`
* **适用场景**：当我们只专注于磁盘排查，不希望屏幕被 CPU 信息占用空间时。

---

### iostat -c
* **作用**：仅输出 CPU 数据，屏蔽所有的磁盘设备信息。
* **语法**：`iostat -c [interval] [count]`
* **适用场景**：需要轻量级查看 CPU 运行和 iowait 比例，且不想受到磁盘信息干扰时。

---

### iostat -m
* **作用**：强制以兆字节/秒（MB/s）为单位显示数据。
* **语法**：`iostat -m [interval]`
* **适用场景**：现代存储设备读取吞吐极高。默认的 KB/s 会产生极长的数据字符（如 500000 kB/s），不利于人眼快速读取。强制 `MB` 显示（如 488.28 MB/s）能让输出更加直观。

---

### iostat -k
* **作用**：强制以千字节/秒（kB/s）为单位显示数据。
* **语法**：`iostat -k [interval]`
* **注意事项**：在较新的 `iostat` 版本中，kB 是默认单位，但某些老旧发行版中默认是 blocks/s，此时推荐显式指定 `-k`。

---

### iostat -t
* **作用**：在每次输出的数据头部打印当前系统的精确时间戳。
* **语法**：`iostat -t [interval]`
* **适用场景**：需要将性能指标与应用的运行日志（如 MySQL 慢查询日志、Nginx Access 慢日志）在时间维度上进行精确对齐时。

---

### iostat -y
* **作用**：在输出中**忽略并跳过**系统自开机以来的历史累积平均值。
* **语法**：`iostat -y [interval] [count]`
* **参数说明**：只能在使用 `interval` 时生效。
* **适用场景**：如果我们只需要看实时的瞬时变化，且不希望被第一行无意义的系统开机平均数据干扰。
* **最佳实践**：`iostat -xyz 1 10`

---

### iostat -o JSON
* **作用**：将性能指标数据输出为标准的 JSON 格式。
* **语法**：`iostat -o JSON [interval] [count]`
* **适用场景**：用于编写自动化监控、报警过滤脚本，或将数据直接推送至 ELK、Prometheus Pushgateway 等平台。

---

### iostat -p [device|ALL]
* **作用**：指定特定磁盘设备及其对应所有分区（如 `sda1`, `sda2`）的 I/O 状态细节。
* **语法**：`iostat -p sda 1`
* **适用场景**：确认某个具体的应用写入是集中在操作系统的根分区，还是独立的日志/数据盘分区上。

---

### iostat -N
* **作用**：在输出中将 LVM 逻辑卷显示为其实际的注册名称，而非内核默认映射的 `dm-X`（Device Mapper）节点名称。
* **语法**：`iostat -N 1`
* **适用场景**：在使用 LVM 的物理服务器或虚拟机上，默认的 `dm-0`、`dm-1` 非常抽象。使用 `-N` 会直接显示为 `/dev/mapper/vg0-lv_root` 等清晰的命名，极大提升排障效率。

---

### iostat -h
* **作用**：以易读格式（Human-readable）显示，根据数据量自动在 `K`、`M`、`G` 之间切换。
* **语法**：`iostat -h`

# 第四章 CPU 指标详解

当我们在 `iostat` 的默认输出或包含 CPU 的输出中查看时，会得到如下的 `avg-cpu` 块：

```text
avg-cpu:  %user   %nice %system %iowait  %steal   %idle
           2.15    0.00    1.50   12.40    0.00   83.95
```

这一行指标表面上是 CPU 的运行指标，但它们与底层磁盘 I/O 存在着极其紧密的联动关系。

---

## 4.1 %user
* **定义**：CPU 处于用户态（User space）运行应用程序代码的时间百分比，不包含通过 nice 改变过优先级的进程。
* **内核计算机制**：CPU 时间片被分配给非特权级（Ring 3）用户进程的比例。
* **高值场景**：计算密集型应用（如高频排序、加解密、复杂业务逻辑处理）。
* **对 I/O 的提示**：当 `%user` 很高，且 `%iowait` 较低时，说明当前系统瓶颈主要在计算端，而非 I/O 端。

---

## 4.2 %nice
* **定义**：CPU 处于带 nice 值的用户态进程的运行时间百分比。
* **内核计算机制**：即通过 `nice` 命令调整了进程优先级（nice > 0）的进程在用户态下消耗的时间。
* **高值场景**：后台低优先级的定时批处理、压缩备份任务。

---

## 4.3 %system
* **定义**：CPU 处于内核态（Kernel space）运行系统级代码的时间百分比。
* **内核计算机制**：CPU 执行系统调用（Syscalls）、软中断（softirqs）、硬中断（irqs）的时间开销。
* **高值场景**：高频的网络包收发、大量的线程上下文切换、**高频的文件系统元数据读写（如不断新建/删除微小文件，触发 VFS 层内存锁竞争）**。
* **异常判定**：如果 `%system` 伴随 I/O 指标长期超过 30%，表明系统调用开销过大，或者是内核在处理大量碎片 I/O 请求时出现锁等待。

---

## 4.4 %iowait
* **定义**：CPU 在空闲状态下，**等待未完成的磁盘 I/O 请求**的时间百分比。

```
                  +-----------------------------------+
                  | CPU 处于空闲状态 (Idle == True)   |
                  +-----------------+-----------------+
                                    |
                    是否存在未完成的磁盘 I/O 请求？
                                    |
                  +-----------------+-----------------+
                  |                                   |
               [ 是 ]                              [ 否 ]
                  |                                   |
                  v                                   v
           记录为 %iowait                       记录为 %idle
```

* **深度剖析（避坑重点）**：**`%iowait` 的本质是 CPU 的一种“特殊 Idle（空闲）”状态**！
  * 当 CPU 处于空闲（即没有就绪进程可执行），且当前**系统中有尚未完成的磁盘 I/O 请求**时，这段空闲时间被标记为 `%iowait`。
  * 如果系统中没有任何未完成的 I/O 请求，同样的空闲时间会被标记为普通的 `%idle`。
  * **结论**：`%iowait` **并不代表 CPU 忙碌**。高 `%iowait` 只能说明**磁盘慢限制了应用，导致 CPU 无事可做**。如果你的系统是多核 CPU，即使某一个单线程任务被磁盘 I/O 阻塞导致其占用的核心出现高 `%iowait`，其他核心依然可以正常承载计算任务，此时系统整体仍有计算余量。

---

## 4.5 %steal
* **定义**：虚拟机（VM）运行期间，被物理宿主机（Hypervisor）的调度器抢占，分配给其他租户/虚拟机运行的 CPU 时间百分比。
* **异常判定**：此指标若长期大于 5%，在云服务器上被称为“超卖严重”或遭遇了“吵闹邻居”（Noisy Neighbor）。即便你的 VM 内部磁盘压力正常，由于底层 CPU 资源被剥夺，依然可能造成应用程序间接的 I/O 调度延迟。

---

## 4.6 %idle
* **定义**：CPU 处于完全空闲状态，且系统中没有任何尚未完成的磁盘 I/O 请求的时间百分比。

---

# 第五章 磁盘指标详解

通过 `iostat -x` 获得的磁盘设备性能指标是性能诊断的黄金数据源。以下将逐一介绍这些指标。

```text
Device            r/s     w/s     rkB/s     wkB/s   rrqm/s   wrqm/s  %rrqm  %wrqm r_await w_await aqu-sz rareq-sz wareq-sz  svctm  %util
sda             12.00   85.00    240.00   1024.00     2.00    15.00  14.29  15.00    4.20    8.50   1.20    20.00    12.05   1.10  10.67
```

---

### Device
* **是什么**：被监控的块设备名称，对应 `/dev/` 下的文件名。如 `sda`（物理机械盘/SATA SSD）、`nvme0n1`（NVMe SSD）、`vda`（KVM 虚拟盘）或 `dm-0`（LVM 逻辑卷）。

---

### tps
* **是什么**：Transactions Per Second，每秒传输次数。代表了每秒向该设备发送的 I/O 请求总次数。
* **如何计算**：$tps = r/s + w/s + d/s$（读、写、丢弃请求每秒总和）。
* **正常范围**：取决于介质物理极限：
  * 7200 转 SATA 机械硬盘：100 - 150 左右。
  * 企业级 SATA SSD：50,000 - 90,000 左右。
  * 企业级 NVMe SSD：可达 500,000 - 1,000,000。
* **异常判定**：当 $tps$ 长期逼近介质物理极限且 `await` 急剧上升时，说明磁盘已达硬件瓶颈。

---

### kB_read/s & kB_wrtn/s
* **是什么**：每秒从设备读取和写入的数据量，单位为千字节。
* **如何计算**：直接从内核 `/proc/diskstats` 的读写扇区数计算而来（1 扇区通常等于 512 字节）。
* **换算**：$MB/s = kB/s \div 1024$。

---

### kB_dscd/s
* **是什么**：每秒由于 Trim/Discard 操作被释放（丢弃）的数据量，多用于 SSD 设备的垃圾回收与空间释放监控。

---

### await
* **是什么**：Average Wait，平均 I/O 响应时间。代表**每一个 I/O 请求从进入内核队列到最终被物理磁盘完成的总时间**，单位为毫秒（ms）。

```
           [ I/O 请求产生并下发 ]
                     |
                     v
   +-----------------+-----------------+
   |   内核 I/O 调度队列 (Wait Time)     |  ======>  等待时间 (Wait Time)
   +-----------------+-----------------+
                     |
                     v  (I/O 调度器派发)
   +-----------------+-----------------+
   |   物理磁盘硬件执行 (Service Time)   |  ======>  服务时间 (Service Time)
   +-----------------------------------+
                     |
                     v
             [ I/O 完成返回 ]

   -----------------------------------------------------------------
   总响应时间 (await) = 等待时间 (Wait Time) + 服务时间 (Service Time)
   -----------------------------------------------------------------
```

* **计算公式**：$await = (time\_in\_queue \div num\_io\_requests)$。
* **正常范围**：
  * 高速 NVMe SSD：通常 < 1.0 ms。
  * SATA SSD：通常 < 3.0 ms。
  * 机械硬盘：通常在 5.0 - 15.0 ms 之间。
* **异常判定**：无论何种物理介质，**一旦 `await` 持续超过 20ms**，通常预示着队列堆积严重，应用层已能感受到明显的卡顿。

---

### r_await & w_await
* **是什么**：分别代表**读请求**和**写请求**的平均响应时间（ms）。
* **异常判定**：
  * 在数据库等读敏感场景中，`r_await` 应保持在极低水平（最好 < 5ms）。如果 `r_await` 很高，代表读请求被写请求堵在内核队列中。
  * 如果 `w_await` 远大于 `r_await`，往往表明写缓冲区已满、文件系统刷盘策略太激进、或磁盘内部垃圾回收（GC）占用了物理带宽。

---

### svctm（Service Time）
* **是什么**：平均每次 I/O 请求被物理硬件直接服务（执行）的时间，不含队列等待。
* **废弃警告**：**自 sysstat v11.x 及更高版本起，`svctm` 已被明确废弃，其计算逻辑已不可靠**。
* **为什么废弃**：因为现代 SSD 和多路径 RAID 阵列支持高度的**并发并发处理**（一个请求还未结束，另一个请求已进入硬件管道）。`iostat` 无法单凭 `/proc/diskstats` 的简单计数器准确剥离出纯粹的硬件执行时间。**在线上分析时，直接忽略该指标，转为分析 `await` 和 `aqu-sz`**。

---

### %util
* **是什么**：采样周期内，**至少有一个** I/O 请求在进行中的时间比例。也称为磁盘设备的繁忙度百分比。
* **计算公式**：$\%util = (time\_active \div total\_time\_of\_interval) \times 100\%$。
* **深度剖析（核心避坑）**：
  * **对于单通道的机械硬盘**，一个时刻只能处理一个磁头寻道。当 `%util` 达到 100% 时，磁盘确实已经饱和。
  * **对于现代 SSD、RAID 阵列或分布式网络存储（Ceph、AWS EBS）**，它们内部拥有数十个通道支持并发写入。例如，一个设备可以同时支持 32 个并发 I/O。如果采样周期内，始终至少有一个 I/O 在运行，则 `%util` 会显示为 100%；但实际上，**该设备还有处理剩余 31 个并发请求的富余能力**。
  * **结论**：**`%util` 达到 100% 绝不代表磁盘性能跑满**，必须要配合 `avgqu-sz` 和 `await` 来综合研判。

---

### aqu-sz & avgqu-sz
* **是什么**：Average Queue Size，平均 I/O 队列长度。代表采样周期内，在内核队列中等待以及正在被设备执行的 I/O 请求的平均个数。
* **正常范围**：
  * 对于单块普通机械硬盘，合理的平均队列长度应 $\le 2$。
  * 对于支持并发的 SSD 块设备，合理的队列长度计算公式为：$合理队列长度 \approx 磁盘通道数 \times 队列深度$。
* **异常判定**：如果 `%util` 是 100%，且 `avgqu-sz` 已经远超其并发处理通道数（如数值达到数十甚至上百），说明队列发生了严重积压，磁盘已经彻底过载。

---

### rareq-sz & wareq-sz
* **是什么**：分别表示平均每次**读**和**写** I/O 请求的数据包大小（单位：kB）。
* **计算公式**：$rareq\_sz = rkB/s \div r/s$。
* **应用意义**：
  * 如果该值很小（如 4kB、8kB），说明当前磁盘上运行着大量**随机 I/O**（常见于关系型数据库索引查询、小文件读写）。
  * 如果该值很大（如 256kB、1MB），说明是**顺序 I/O**（常见于视频流传输、日志顺序刷盘、数据库全表扫描）。
  * 帮助优化：根据 `req-sz` 的大小，可以针对性地调整文件系统的块大小（Block size）、I/O 调度器参数（如 `read_ahead_kb` 预读大小）。

---

### r/s & w/s
* **是什么**：设备每秒接收到的读、写请求数。

---

### rrqm/s & wrqm/s
* **是什么**：每秒被操作系统 I/O 调度器合并（Merge）的读、写请求数。
* **什么是 I/O 合并**：当应用下发了多个物理上连续的相邻扇区的读写请求时，Linux 内核为减少磁盘磁头位移，会将多个小 I/O 合并为一个大 I/O 统一发送给块设备驱动。
* **%rrqm & %wrqm**：合并请求占总下发请求的百分比。如果合并比例很高，说明程序产生的主要是顺序 I/O，这对机械硬盘是极大的性能解救。

---

# 第六章 await、svctm、%util 深入解析

这三个指标最容易让初学者产生混淆。为了彻底理清它们的内幕，我们使用一个直观的**“超市结账”**模型来做类比：

```
                    【超市结账收银排队模型】
  
  [顾客到达] ===> [排队等待区 (Queue)] ===> [收银窗口 (SSD/HDD)] ===> [结账完成离开]
                 (Wait Time)              (Service Time)
                 
  * 顾客 (I/O 请求)
  * 收银窗口 (磁盘物理通道)
  * 整个结账过程所花总时间 (await) = 排队时间 + 扫码付款时间
```

* **顾客（I/O 请求）**：不断来到收银台前。
* **收银窗口（磁盘物理通道）**：负责扫码收款。
* **整个结账过程所花总时间（await）**：顾客排队的时间 + 扫码付款完成的时间。
* **收银员干活时间（svctm）**：仅仅是收银员拿起商品扫码的物理时间。
* **收银台繁忙度（%util）**：只要收银台前有至少一个顾客，收银台就是“忙碌”的。

---

## 6.1 深度剖析：为什么 await 会很高？
根据上述模型，$await = 等待时间 + 硬件服务时间$。
1. **原因一：硬件性能太慢（服务时间长）**。收银员动作太慢，导致扫码付款时间极长。对于机械盘，代表磁头物理寻道和转动延迟太大。
2. **原因二：并发请求太高（排队时间极长）**。即使收银员动作很快（每次只需 1ms），但如果瞬间涌入 100 个顾客，排在最后的顾客依然需要等待很长时间才能轮到。**在线上，由于并发请求堆积造成的队列等待，是 `await` 偏高最主要的原因**。

---

## 6.2 深度剖析：为什么 %util 100% 不代表磁盘跑满？
我们来看“单收银台”与“多收银台”的对比：

```text
【单通道设备（传统机械硬盘 HDD）】
 [窗口1] ── 只能同时为 1 个人服务。只要有人，%util = 100%。此时确实达到了处理能力上限。

【多通道并发设备（现代固态硬盘 SSD / RAID 阵列）】
 [窗口1] [窗口2] [窗口3] [窗口4] [窗口5] ... [窗口32]
 ── 虽然任何时刻都有人在其中一个窗口结账，导致 %util = 100%。
 ── 但实际上，可能只有 1-2 个窗口有顾客，剩下的 30 个窗口全都处于闲置状态。
 ── 此时，系统依然有非常庞大的并发吞吐余量！
```

* **研判法则**：
  要判断现代并发存储设备的真正饱和度，应使用以下公式：
  $$\text{真实饱和度} = \frac{\text{当前平均队列长度 (avgqu-sz)}}{\text{设备的最大并发处理能力 (Queue Depth / Channels)}}$$
  只要 $avgqu-sz$ 显著小于设备支持的最大深度，哪怕 `%util` 保持 100%，磁盘也未达到瓶颈。

---

# 第七章 实战案例（25 个高频案例）

在生产环境中，纯粹的数字是枯燥的。这一章我们将通过 25 个完全仿真的生产环境性能故障场景，演示如何通过 `iostat` 抽丝剥茧，锁定底层的性能元凶。

---

### 案例 1：服务器变慢，整体响应延迟变长
* **问题现象**：Web 接口平均耗时从 50ms 突增至 3s。
* **分析思路**：运行 `top` 观察到系统的 `%wa`（iowait）高达 25%，表明 CPU 正在大量空闲等待磁盘。接着使用 `iostat` 定位高延迟磁盘。
* **执行命令**：`iostat -xz 1 3`
* **输出**：
  ```text
  Device            r/s     w/s     rkB/s     wkB/s   rrqm/s   wrqm/s  r_await  w_await aqu-sz  %util
  sdb              0.00  450.00      0.00   1800.00     0.00   120.00     0.00   85.20   12.50  98.00
  ```
* **原因分析**：设备 `sdb` 的写响应时间 `w_await` 达 `85.20ms`，队列深度高达 `12.50`。系统存在写入瓶颈。
* **解决方案**：定位写入大户。
* **最佳实践**：在业务盘上对数据和日志进行物理隔离，使用 SSD 替代 `sdb`。

---

### 案例 2：数据库卡顿，查询延迟剧增（随机读瓶颈）
* **问题现象**：MySQL 执行简单 `SELECT` 主键查询耗时突增。
* **分析思路**：查询卡顿通常对应随机读高。
* **执行命令**：`iostat -xz 1 3`
* **输出**：
  ```text
  Device            r/s     w/s     rkB/s     wkB/s  rareq-sz  r_await  aqu-sz  %util
  sdc            850.00    5.00   3400.00     20.00      4.00    38.00    6.20  99.00
  ```
* **原因分析**：`r/s` 高达 850，但是读取吞吐 `rkB/s` 仅为 3400 kB（约 3.3MB/s）。平均单次读取包大小 `rareq-sz` 恰好是 `4.00 kB`，`r_await` 高达 `38.00 ms`。这是非常典型的**小数据块高并发随机读瓶颈**，瞬间查询已经榨干了传统硬盘磁头的极限。
* **解决方案**：增加内存，调大 MySQL `innodb_buffer_pool_size` 缓存数据，减少物理磁盘读取。

---

### 案例 3：MySQL IOPS 跑满，写入积压
* **问题现象**：Innodb 脏页刷新跟不上，事务提交被阻塞。
* **分析思路**：观察写请求和合并情况。
* **执行命令**：`iostat -x -p sdb 1 3`
* **输出**：
  ```text
  Device            r/s     w/s     rkB/s     wkB/s   wrqm/s  w_await  %util
  sdb              2.00 3500.00   14000.00  56000.00   200.00    18.50  100.00
  ```
* **原因分析**：`w/s` 达 3500，且 `wkB/s` 达 56MB/s，磁盘处于持续高载。
* **解决方案**：将单盘改组 RAID-10 或升级为企业级 NVMe SSD。

---

### 案例 4：Redis 在后台执行 BGSAVE 期间业务严重抖动
* **问题现象**：每隔数小时 Redis 响应就会出现一次大范围毫秒级超时。
* **分析思路**：Redis BGSAVE 会 fork 子进程进行内存快照，将数据写入磁盘。
* **执行命令**：`iostat -xt 1 10` (重点盯防 BGSAVE 执行时刻)
* **输出**：
  ```text
  12:00:01: sda              0.00  3200.00      0.00  128000.00   w_await 150.00  aqu-sz  45.00  %util 100.00
  ```
* **原因分析**：BGSAVE 期间，由于数据量庞大且写操作极度密集，单盘 `sda` 的写等待 `w_await` 直接飙升至 `150.00ms`，造成系统严重卡顿。
* **解决方案**：调大 `/proc/sys/vm/dirty_background_ratio`，或者将 AOF/RDB 文件写入专用 SSD。

---

### 案例 5：Docker 容器内部跑批导致宿主机全局 I/O 阻塞
* **问题现象**：多租户环境下，一个容器内跑分析脚本，导致其他容器服务假死。
* **分析思路**：确定流量是否来源于 Docker 虚拟块设备。
* **执行命令**：`iostat -xz 1 3`
* **输出**：
  ```text
  Device            r/s     w/s     rkB/s     wkB/s   r_await  w_await  %util
  dm-4             0.00 8500.00      0.00  340000.00     0.00   112.00  100.00
  sda              0.00 8500.00      0.00  340000.00     0.00   112.00  100.00
  ```
* **原因分析**：代表容器 LVM 虚拟分区的 `dm-4` 与物理盘 `sda` 承载了相同的巨大写入，且写入延迟极高。
* **解决方案**：在 Docker 启动或 Compose 文件中，限制单容器的 I/O 带宽（`--device-write-bps` / `--device-write-iops`）。

---

### 案例 6：Kubernetes 节点因 PV（持久卷）过载导致 Node 变为 NotReady
* **问题现象**：K8s Node 节点间歇性假死，提示磁盘承载过高。
* **执行命令**：`iostat -xN 1 3` (使用 -N 识别 LVM 盘)
* **输出**：
  ```text
  Device                         r/s     w/s     rkB/s     wkB/s   w_await  %util
  kubernetes--vg-kubernetes--lv  0.00  6500.00      0.00  260000.00   85.00   100.00
  ```
* **解决方案**：使用 `StorageClass` 配置 I/O 限速，或将高写入应用（如 ElasticSearch）隔离到专用节点。

---

### 案例 7：日志刷盘策略太激进导致整体性能下滑
* **问题现象**：高并发 Web 服务，随着流量上升，响应延迟成倍放大。
* **分析思路**：观察写请求的大小（wareq-sz）和频率（w/s）。
* **执行命令**：`iostat -xz 1`
* **输出**：
  ```text
  Device            r/s     w/s     rkB/s     wkB/s  wareq-sz  w_await  %util
  sda              0.00 12000.00   48000.00  48000.00      4.00    25.00  100.00
  ```
* **原因分析**：写频率高达 `12000.00 w/s`，但每次平均写包大小 `wareq-sz` 仅仅是 `4.00 kB`。这说明应用频繁执行 `fsync()` 强制将每一条小日志实时落盘。
* **解决方案**：修改应用日志刷盘机制，启用异步批量刷盘（如 Nginx 开启 `buffer=32k flush=1s`）。

---

### 案例 8：SSD 性能老化导致写入吞吐骤降（写放大与 GC 问题）
* **问题现象**：生产固态硬盘在满盘运行一段时间后，写入速度只剩标称的 1/5。
* **分析思路**：观察写延迟 `w_await` 是否异常偏高。
* **执行命令**：`iostat -xz 1 3`
* **输出**：
  ```text
  Device            r/s     w/s     rkB/s     wkB/s   w_await  %util
  nvme0n1          0.00 1200.00      0.00  12000.00    82.00   99.00
  ```
* **原因分析**：NVMe SSD 的 `w_await` 竟达到恐怖的 `82.00ms`。代表块设备内部没有足够的空白块可写，导致主控高频触发垃圾回收（GC），造成严重的“写放大”。
* **解决方案**：对固态盘执行 `fstrim`，或留出 10%-20% 的预留空间（Over-provisioning）。

---

### 案例 9：RAID-5 阵列因“写惩罚”导致大文件写入卡顿
* **问题现象**：新组建的 RAID-5 存储系统，在大文件持续拷入时，速度波动极大。
* **执行命令**：`iostat -xz 1 3`
* **输出**：
  ```text
  Device            r/s     w/s     rkB/s     wkB/s   r_await  w_await  %util
  md0            1800.00 1800.00   7200.00   7200.00    12.00    45.00  100.00
  ```
* **原因分析**：在没有读取任务时，底层成员磁盘却出现了大量的读请求 `r/s`，这是由于 RAID-5 阵列在写入非整条带数据时，必须先读取旧数据和校验区算校验，触发了“写惩罚（Write Penalty）”。
* **解决方案**：更换为带强力电池电容保护的硬件 RAID 卡并开启 Write-Back 缓存，或者改组 RAID-10。

---

### 案例 10：NFS 网络挂载文件系统高延迟排查
* **问题现象**：读取局域网内的 NFS 挂载目录极慢。
* **分析思路**：对比网络磁盘与本地物理盘表现。
* **执行命令**：`iostat -xz 1`
* **输出**：
  ```text
  Device            r/s     w/s     rkB/s     wkB/s   r_await  w_await  %util
  nfs1             5.00    0.00     20.00      0.00   320.00     0.00   0.00
  ```
* **原因分析**：NFS 网络盘的 `r_await` 居然是 `320.00ms`，但 `%util` 却是 `0.00%`。这直接证明了**延迟的根本原因在于网络抖动、丢包或 NFS 服务器过载**，而不在本地。
* **解决方案**：排查 NFS 服务器端的 CPU 和存储瓶颈，调整 NFS 挂载参数（如 `rsize`, `wsize` 为 1MB）。

---

### 案例 11：Ceph 分布式存储 OSD 磁盘过载
* **问题现象**：Ceph 集群报警，提示部分 OSD 延迟高，导致整体 IOPS 下滑。
* **执行命令**：`iostat -xz 1`
* **输出**：
  ```text
  Device            r/s     w/s     rkB/s     wkB/s   w_await  aqu-sz  %util
  sdh              5.00 2500.00  10000.00  85000.00    58.00   35.00  100.00
  ```
* **原因分析**：该 OSD 对应的物理盘 `sdh` 写入延迟高达 `58.00ms`，且请求积压。
* **解决方案**：将 Ceph DB/WAL 写入 SSD，数据写入 HDD，以此来加速元数据写响应。

---

### 案例 12：KVM 虚拟机“吵闹邻居”抢占宿主机 I/O
* **问题现象**：一台云宿主机上的部分客户虚拟机投诉磁盘变慢。
* **分析思路**：查看多路虚拟盘之间的并发开销。
* **执行命令**：`iostat -xz 1`
* **输出**：
  ```text
  Device            r/s     w/s     rkB/s     wkB/s   r_await  w_await  %util
  vda              2.00    5.00      8.00     20.00    48.00    92.00  100.00
  ```
* **原因分析**：该虚拟机自身仅有微小的吞吐，但是 `w_await` 高达 92ms。这表明受到了宿主机上其他虚拟机的强烈 I/O 干扰。
* **解决方案**：使用 `virsh blkiotune` 限制各个虚拟机的最高磁盘权重和 IOPS，隔离业务。

---

### 案例 13：云服务器 EBS 云盘触发限速瓶颈
* **问题现象**：云主机上的应用写入达到某一固定速度后再也无法上升。
* **分析思路**：云盘有严格的吞吐和 IOPS 上限，超出会被云平台实施“限速门限制（Throttling）”。
* **执行命令**：`iostat -xz 1`
* **输出**：
  ```text
  Device            r/s     w/s     rkB/s     wkB/s   w_await  aqu-sz  %util
  xvda             0.00 3000.00      0.00  120000.00    35.00   28.00  100.00
  ```
* **原因分析**：连续数次采样中，写入量精准卡死在 `120000.00 kB/s`（约 120MB/s）或 `3000.00 tps`。同时队列和延迟飙升。这正是非常典型的**触碰了云厂商对该云盘设定的吞吐/IOPS 阈值上限**。
* **解决方案**：在云服务商后台升级云盘规格。

---

### 案例 14：CPU 跑空，iowait 飙升至异常水位
* **问题现象**：系统看起来非常卡顿，`top` 中 `%wa` 高达 40%，但物理内存还剩很多。
* **执行命令**：`iostat -c 1` 以及 `iostat -xz 1`
* **输出**：
  ```text
  avg-cpu:  %user   %nice %system %iowait  %steal   %idle
             1.00    0.00    2.00   45.00    0.00   52.00
  Device            r/s     w/s     rkB/s     wkB/s   r_await  aqu-sz  %util
  sda            150.00    0.00    600.00      0.00    42.00    5.50   99.00
  ```
* **原因分析**：CPU 消耗极低，但 `iowait` 极高。磁盘 `sda` 上的读响应延迟已经上升至 `42.00ms`。CPU 核心由于长期等待这 150 个慢读取请求而彻底挂起。
* **解决方案**：优化应用，引入 Redis/Memcached 缓存阻断高频物理读取。

### 案例 15：磁盘 %util 100% 但吞吐极低，磁盘未过载
* **问题现象**：开发人员反馈监控显示磁盘 `util 100%`，急需扩容。
* **分析思路**：评估磁盘真实负载。
* **执行命令**：`iostat -xz 1 3`
* **输出**：
  ```text
  Device            r/s     w/s     rkB/s     wkB/s   r_await  w_await aqu-sz  %util
  nvme0n1          5.00    1.00     20.00      4.00     0.15     0.10   0.01  100.00
  ```
* **原因分析**：虽然 `%util` 显示 `100%`。但仔细看核心数据：`r/s` 为 5，`w/s` 为 1，读写平均延迟都在 `0.1ms` 级别。平均队列 `aqu-sz` 仅仅为 `0.01`。这说明磁盘**绝无可能过载**。
* **解决方案**：无需任何硬件扩容。

### 案例 16：await 指标异常偏高但 TPS 极低
* **问题现象**：一个几乎没有读写业务的备机，其磁盘 `await` 高达 180ms。
* **分析思路**：低吞吐下的高延迟通常来自网络存储或特殊的块设备。
* **执行命令**：`iostat -xz 1`
* **输出**：
  ```text
  Device            r/s     w/s     rkB/s     wkB/s   w_await  aqu-sz  %util
  sdd              0.00    0.20      0.00      0.80   180.00    0.04   1.20
  ```
* **原因分析**：5 秒内才有一个写请求（0.20 w/s）。因为写请求极其稀疏，虽然单个请求由于云盘唤醒或网络层寻道花掉了 180ms（导致平均值偏高），但整体没有积压（队列仅为 `0.04`），对系统并无实质负面影响。
* **解决方案**：此为统计学上的均值漂移现象，属于正常状态。

### 案例 17：高并发批量写入下，tps 异常偏低
* **问题现象**：在性能压测期间，数据库写入速度极低。
* **执行命令**：`iostat -xz 1`
* **输出**：
  ```text
  Device            r/s     w/s     rkB/s     wkB/s   w_await  aqu-sz  %util
  sda              0.00   80.00  81920.00  81920.00    35.00    8.50  100.00
  ```
* **原因分析**：写入次数 `w/s` 仅为 80，但写入量 `wkB/s` 达到了 80MB/s。单次写平均大小为 1MB。由于请求包过大，磁盘通道瞬间饱和，导致无法接收更多的并发。
* **解决方案**：降低批量写入的单次数据块大小，将大块拆分为中等块写入。

---

### 案例 18：高频随机读把缓存击穿（Cache Miss）
* **问题现象**：服务器在某次运营活动开始后，整体响应开始雪崩。
* **执行命令**：`iostat -xz 1`
* **输出**：
  ```text
  Device            r/s     w/s     rkB/s     wkB/s   r_await  aqu-sz  %util
  sda            4500.00   0.00  18000.00      0.00     18.50    8.00  100.00
  ```
* **原因分析**：高频的 `4500.00 r/s`，但读取数据块都很小。说明缓存彻底被穿透，全部流量直达物理介质。
* **解决方案**：使用布隆过滤器防穿透，提高 Redis 预热命中率。

---

### 案例 19：单线程大文件写引发队列“假堵塞”
* **问题现象**：一个单线程程序在做日志转储（Logrotate）时系统突然抖动。
* **执行命令**：`iostat -xz 1`
* **输出**：
  ```text
  Device            r/s     w/s     rkB/s     wkB/s   w_await  aqu-sz  %util
  sda              0.00  200.00  204800.00 204800.00     5.00    1.00   99.00
  ```
* **原因分析**：写延迟仅为 `5.00ms`，队列大小 `aqu-sz` 精准等于 `1.00`。说明当前**只有一个线程在进行独占性的顺序写入**。此时虽然 `%util` 接近 100%，但对于其他并发线程并不会造成灾难性的队列等待。
* **解决方案**：对大日志写启用 `ionice -c3`（限制其使用空闲 I/O 级别）。

### 案例 20：队列长度（aqu-sz）严重积压引发内核崩溃（Hang）
* **问题现象**：应用抛出 "task blocked for more than 120 seconds" 内核恐慌。
* **执行命令**：`iostat -xz 1`
* **输出**：
  ```text
  Device            r/s     w/s     rkB/s     wkB/s   w_await  aqu-sz  %util
  sda              0.00   50.00    200.00    200.00   4500.00  220.00  100.00
  ```
* **原因分析**：在极低的写入量（50 w/s）下，写延迟 `w_await` 高达 `4500ms`（4.5秒），排队队列高达 `220`。说明底层硬件已经发生故障、掉盘或者发生了严重的链路锁死。
* **解决方案**：硬件排查，检查 SAS 控制器和固件。

### 案例 21：inode 用满导致元数据高频读写，I/O 暴涨
* **问题现象**：写入极小文件时，系统报错 "No space left on device"，但 `df -h` 还有空间。
* **分析思路**：由于 inode 耗尽，系统高频地在磁盘扇区中检索空余 inode。
* **执行命令**：`iostat -xz 1`
* **输出**：
  ```text
  Device            r/s     w/s     rkB/s     wkB/s   r_await  %util
  sda            2800.00 1200.00  11200.00   4800.00     15.00  100.00
  ```
* **原因分析**：伴随了大量的系统开销（iostat 顶部 %system > 35%），磁盘进行大量的乱序读写。
* **解决方案**：清理零字节小临时文件，释放 inode 空间。

### 案例 22：Ext4 日志提交机制引发频繁同步卡顿
* **问题现象**：在进行大吞吐文件操作时，每隔数秒就有一次周期性应用卡顿。
* **分析思路**：排查文件系统内部机制。
* **执行命令**：`iostat -xt 1`
* **输出**：
  ```text
  15:00:05: sda      0.00  150.00     0.00  150000.00    w_await  2.10
  15:00:10: sda      0.00 4500.00     0.00   18000.00    w_await 45.00
  ```
* **原因分析**：由于 Ext4 默认在后台每隔 5 秒提交一次 journal 日志，造成 I/O 在第 10 秒发生爆发。
* **解决方案**：在 `/etc/fstab` 中，将该分区挂载参数调整为 `noatime,nodiratime,commit=60`（将提交间隔放宽至 60 秒）。

### 案例 23：系统频繁发生 OOM（内存溢出）引发大量页交换（Swap I/O）
* **问题现象**：当系统内存快用完时，磁盘读写突然出现火山喷发。
* **分析思路**：对比 swap 空间读写。
* **执行命令**：`iostat -xz 1` 并结合 `vmstat 1`
* **输出**：
  ```text
  # vmstat 看到 si (swap in), so (swap out) 极高
  si 15000   so 22000
  # iostat 看到系统盘：
  Device            r/s     w/s     rkB/s     wkB/s   w_await  %util
  sda            1200.00 1800.00  48000.00  72000.00    55.00  100.00
  ```
* **原因分析**：因为物理内存耗尽，Linux 开始强行把不活跃的进程内存置换到磁盘 swap 分区中，由此产生的虚拟内存页交换彻底拖死了系统。
* **解决方案**：关闭 swap 分区（`swapoff -a`），调低系统的内核 `swappiness` 参数为 10，并排查进程内存泄露。

### 案例 24：rsync 大规模文件同步期间本地 I/O 被挤爆
* **问题现象**：每天半夜两点执行 rsync 异地备份时，前台用户访问报错。
* **执行命令**：`iostat -xz 1`
* **输出**：
  ```text
  Device            r/s     w/s     rkB/s     wkB/s   r_await  w_await  %util
  sda            1800.00 1500.00  72000.00  60000.00    15.00    25.00  100.00
  ```
* **解决方案**：限制 rsync 同步带宽上限：`rsync --bwlimit=10000`（限制为 10MB/s 吞吐）。

### 案例 25：病毒或恶意挖矿木马在后台大量扫盘/写日志
* **问题现象**：服务器在完全无业务的时间段，磁盘灯狂闪。
* **执行命令**：`iostat -xz 1`
* **输出**：
  ```text
  Device            r/s     w/s     rkB/s     wkB/s   r_await  %util
  sda            2500.00    0.00  10000.00      0.00     12.00   98.00
  ```
* **原因分析**：半夜持续存在 `2500.00 r/s` 且吞吐恒定。
* **解决方案**：配合 `iotop` 锁定异常的可执行程序，使用杀毒工具清除木马，并关闭异常外联端口。

# 第八章 常见误区与 FAQ

---

### Q1：为什么 %util 达到 100% 时，系统的响应（await）还是很快？
**A**：参考第 6 节。这是现代 SSD、NVMe 固态硬盘或多盘 RAID 阵列具有高并发多通道能力的明证。只要这些通道未全部被塞满，队列（$avgqu-sz$）很短，平均延迟就会依然维持在极低的物理安全水平（如 < 1ms）。

### Q2：为什么 await 只有 1ms，但应用却报错“磁盘写入超时”？
**A**：有些应用在发起写入时，没有开启异步线程，而是直接使用同步阻塞方式在用户态下执行。如果应用在用户态处理逻辑慢（例如高频发生互斥锁竞争），即便底层磁盘处理物理 I/O 只要 1ms，应用整体感知的逻辑耗时依旧会非常长。需要通过 `strace` 检测系统调用耗时。

### Q3：为什么磁盘的 TPS 很高，却完全不用担心？
**A**：高 TPS 如果伴随着大比例的 I/O 合并（$rrqm/s$ 或 $wrqm/s$ 很强），说明大部分是连续的、对硬件非常友好的顺序读写。

### Q4：为什么云服务器上的磁盘延迟（await）比物理机明显要高？
**A**：因为云主机挂载的大多是网络分布式存储（如 AWS EBS、阿里云云盘）。每一次磁盘读写都需要通过网络协议栈封装，跨越物理交换机到达存储集群，因此其天然自带了网络传输延迟（通常在 1.0ms - 3.0ms 之间）。

### Q5：为什么 iostat 中的 kB_read/s 加上 kB_wrtn/s 算出来的总吞吐和 vmstat 的 bi + bo 指标对不上？
**A**：
1. **单位转换差异**：`vmstat` 的 `bi`/`bo` 的单位通常是系统块大小（Block），在大多数 x86 系统中，1 Block = 1024 字节。
2. **统计范畴**：`iostat` 直接在物理块设备驱动层抓取统计，而 `vmstat` 在虚拟内存页写回层统计，包含了对 Swap 分区的操作，导致由于中间层的合并或缓存影响产生轻微的数额差异。

### Q6：iostat 中 Device 显示的 dm-0、dm-1 是什么？
**A**：这些是 Linux 逻辑卷管理（LVM）或多路径软件（Multipath）映射的 Device Mapper 虚拟节点。使用 `iostat -N` 参数可以让它们直接转译显示为真实的逻辑卷名称。

### Q7：开机第一次执行 iostat 输出的信息有参考价值吗？
**A**：**完全没有**。无参数或第一次采样输出的数据是自系统启动以来的累计均值。要获得准确数据，必须加采样频率（如 `iostat 1`），并从第二秒的输出看起。

### Q8：iostat 的 kB_dscd/s 是什么意思？
**A**：这是新版 `sysstat` 的新增字段，代表系统由于执行 `Trim` 操作，每隔一秒通知 SSD 物理释放的废弃数据大小。

### Q9：SSD 上的 w_await 为什么通常高于 r_await？
**A**：因为固态硬盘的物理特性是“写前必须擦除”。当写入数据时，主控如果无法找到直接可写的干净块，必须先擦除一整个物理页，从而导致写延迟大幅上升。

### Q10：为什么 iowait 很高，但是磁盘 util 却很低？
**A**：因为 `iowait` 仅仅表示 CPU 在空闲等待 I/O，但也许由于系统设计原因（例如单线程同步读取），此时在盘上只有极个别的慢 I/O 请求，因此磁盘的整体繁忙度（%util）不高。这也印证了 `iowait` 属于间接指标，必须用 `iostat` 核实硬件。

### Q11：iostat 怎么看随机 I/O 还是顺序 I/O？
**A**：通过平均请求包大小 `rareq-sz` 和 `wareq-sz` 判断。
* 若平均单次请求包 $\le 8.00 kB$，基本是**随机 I/O**。
* 若平均单次请求包 $\ge 128.00 kB$，说明是**顺序 I/O**。

### Q12：svctm 在较新版本的 sysstat 里被标为废弃，我们该拿什么代替它？
**A**：无需代替，直接看 `await`（总时长）和 `aqu-sz`（积压度）。这两个指标组合足以判断磁盘饱和度。

### Q13：iostat 的 kB_read 和 kB_read/s 有何区别？
**A**：`kB_read` 是采样区间内读取的**累积总量**（kB），而 `kB_read/s` 是**单位时间速率**（kB/s）。

### Q14：如何在不安装 sysstat 的情况下，裸看 Linux 的磁盘性能数据？
**A**：可以直接通过命令 `cat /proc/diskstats` 读取内核计数器，并手动计算时间周期差值。

### Q15：%steal 很高和磁盘 I/O 有什么间接关系？
**A**：在云服务器环境下，`%steal` 高说明宿主机 CPU 被严重争抢。这不仅会导致 VM 的算力下降，也会拖累 VM 虚拟网络协议栈的封装速度，使得挂载的网络云盘的 `await` 大幅拉长。

### Q16：RAID-0、RAID-1、RAID-10 下的 await 特征有何不同？
**A**：
* RAID-0：拥有最佳的并发和最低的 `await`（无镜像或校验损耗）。
* RAID-1：写 `await` 与单盘一致，但读 `await` 会减半（双盘轮询读取）。
* RAID-10：完美平衡，读写延迟在多路并发下表现都极其稳定。

### Q17：在 SSD 上开启 noatime 挂载参数对 iostat 有何影响？
**A**：开启 `noatime`（不更新文件的访问时间）后，系统能减少大量的元数据小写入，这在 `iostat` 上的直接表现为系统的 `w/s` 将会有显著下降。

### Q18：iostat -xz 输出里的 rrqm/s 很小说明什么？
**A**：说明应用程序产生的 I/O 地址在物理上极其分散、完全乱序，内核调度器无法将其进行就近合并。通常对应大范围的随机读写（如数据库索引未命中）。

### Q19：如何通过 iostat 判断磁盘硬件介质开始发生物理坏道？
**A**：如果在低吞吐下，`await` 突然出现间歇性飙升（达到数秒级别），而队列长度却不大，常常是由于磁头在物理上遇到了无法读取的坏道，正在进行反复重试和错误恢复动作。

### Q20：有些虚拟盘在 iostat 里显示为 loop0，这是什么？
**A**：这是回环设备（Loop Device）。多见于通过一个大文件挂载虚拟化出来的文件系统（如 Docker 的部分存储驱动、或通过 snap 安装的服务）。

### Q21：iostat 输出如何格式化为更友好的 CSV 格式导入 Excel 绘图？
**A**：可以使用 `iostat -o JSON` 参数输出，并用 Python 的 `json` 和 `pandas` 库直接导出为 Excel。

### Q22：iostat 对 SSD 执行的 Trim/Discard 操作能否准确统计？
**A**：可以。较新版本的 `iostat` 中会单独输出 `kB_dscd/s` 和 `dscd/s` 字段。

### Q23：当内存 Page Cache 频繁写回（Writeback）时，iostat 指标有什么特征？
**A**：`w/s` 和 `wkB/s` 会出现大范围、阶段性的周期性暴涨，通常伴随着 CPU 顶部 `%system` 利用率的短时攀升。

### Q24：为什么 iostat -p sda 输出的信息没有包括 sda1、sda2 分区？
**A**：因为默认情况下，如果分区完全没有读写，`iostat` 不会主动输出它们。可以使用 `-p sda` 组合 `-y 1` 定向抓取。

### Q25：系统的内存不足（OOM）会导致 %iowait 飙升吗？
**A**：是的，正如实战案例 23。因为内存耗尽会高频触发 swap（交换分区）读写，磁盘读写响应会直接把系统拖向高延迟状态。

### Q26：如何查看 iostat 自身的系统性能开销？
**A**：`iostat` 仅仅是读取 `/proc`，其开销相当于执行一次轻量级的文本读取，CPU 占用率一般低于 0.01%，对生产环境无任何负荷影响。

### Q27：当 %user 占满而 I/O await 极低时，如何调优？
**A**：代表系统当前瓶颈纯在计算，磁盘配置已经足够富余，无需对存储进行任何物理改造，重点优化代码或进行 CPU 扩容。

### Q28：云厂商的 IOPS 限制（如 3000 IOPS）指的是 iostat 的什么指标？
**A**：精准对应 `iostat` 输出中的 `tps`（或者是 $r/s + w/s$）。

### Q29：iowait 异常高但没有进程在读写，可能是什么原因？
**A**：可能是由于系统内核中发生了僵尸线程锁（如 D 状态进程，处于不可中断睡眠）。

### Q30：为什么 SSD 的平均队列大小（aqu-sz）长期保持在 32 左右是合理的？
**A**：因为现代企业级 NVMe/SATA SSD 的内部主控一般配置了深度为 32 或更高（如 NVMe 支持最大 64000 个队列，每个队列 64000 深度）的硬件并行队列。

# 第九章 与其他工具对比

性能调优并不是一叶障目。下面将 `iostat` 与 Linux 常见的监控排查工具进行横向多维对比：

| 工具名称 | 适用场景 | 优势 | 缺点 | 什么时候使用 | 什么时候不要使用 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **`iostat`** | 磁盘 I/O 整体性能诊断 | 全面、极轻量、完美展示延迟与队列 | 无法看具体进程，无历史保存能力 | **怀疑磁盘过载，需评估硬件健康度与吞吐极限时** | 想看具体是哪个 PID 进程导致写入时 |
| **`top`** | CPU 与内存全局监控 | 系统内置标配，直观展示全局核心负载与 `iowait` | 磁盘维度的信息为零，iowait 指标不准确 | **进入系统的第一眼初步排查** | 深入诊断磁盘、网络或 I/O 细节时 |
| **`vmstat`** | 内存、页交换、上下文切换统计 | 极速，完美呈现 swap 交换和 CPU 整体上下文开销 | 磁盘设备仅有两个宏观数值（bi/bo），无设备级延迟 | **排查内存爆满、OOM 导致系统颠簸时** | 确定是哪块磁盘或分区变慢时 |
| **`sar`** | 历史性能趋势回溯与事后审计 | 自动记录历史指标，支持全方位参数 | 无法进行超高频的实时交互式分析 | **排查半夜 3 点系统异常卡顿等事后追溯场景** | 故障正在发生、需高频交互实时观察时 |
| **`iotop`** | 精确定位写入进程 PID | **能直接展示具体哪个进程（CMD）在读写** | 开启时会向内核注册轮询，大流量下有明显的性能损耗 | **已通过 iostat 确认磁盘慢，需抓出罪魁祸首进程时** | 生产环境 I/O 已经极度饱和、系统快卡死时 |
| **`dstat` (PCP)**| 混合多维指标实时展示 | 界面极度友好，将 CPU、内存、磁盘、网络合并在同一行展示 | 系统往往不默认安装，需额外依赖 | **在测试环境进行压力测试，想一眼看清全链路水位时** | 编写标准化无依赖的线上排障脚本时 |
| **`atop`** | 全天候系统级深度性能记录 | 自动高亮标红系统瓶颈（CPU、I/O等），信息量大且支持回放 | 界面排版紧凑繁复，学习曲线陡峭 | **服务器发生疑难杂症，常规监控漏报时** | 仅需要快速查看某块磁盘速率时 |
| **`pidstat`** | 细粒度进程/线程性能分解 | 可以把 I/O 拆分到具体进程、线程和 CPU 关联 | 需要 sysstat 套件，输出多为瞬时 | **需要单独追踪某个 Java/Go 进程内部多线程写入细节时** | 系统卡死、只想看整体物理磁盘时 |
| **`perf`** | 内核级函数调用及性能剖析 | 极度底层，能直接抓出内核中具体哪段 C 函数拖慢了 I/O | 使用门槛高，需要 debug 符号表，解析复杂 | **深度调优存储引擎，开发文件系统或内核模块时** | 日常的常规应用运维排障时 |
| **`bcc-tools`** | 基于 eBPF 的磁盘 I/O 延迟追踪 | **能统计每一次 I/O 调用的直方图分布（biolatency）** | 依赖高版本 Linux 内核（>= 4.9），安装开销大 | **需要精确掌握每一个 bio 级别延迟分布和调用路径时** | 处于旧版 CentOS 7 环境，或只需看吞吐时 |

# 第十一章 最佳实践

在长达数年的高负载运维实践中，以下几条经验是确保系统稳定运行的黄金准则：

## 11.1 生产环境推荐监控参数
* **日常线上巡检与故障排查推荐命令**：
  ```bash
  sudo iostat -xtmyz 1
  ```
  * **参数拆解**：
    * `-x`：查看包含 `await`、`avgqu-sz` 的完整扩展指标。
    * `-t`：带上系统时间戳，方便与日志对齐。
    * `-m`：以 MB 为单位显示，让吞吐量一目了然。
    * `-y`：丢弃开机累计均值，第一行即显示当前瞬时数据。
    * `-z`：剔除空闲无流量磁盘，精简屏幕。

## 11.2 长效监控体系架构设计
单靠人工登录查看是远远不够的。在现代化可观测性平台中，我们推荐：

```
+--------------------+
| 被监控主机 (Node)  |
|  - node_exporter   |  =====>  使用 /proc/diskstats 采集核心指标
+---------+----------+
          |
          |  (HTTP Pull / 15s 周期)
          v
+---------+----------+
| Prometheus 存储库  |  =====>  计算并存储磁盘趋势
+---------+----------+
          |
          v
+---------+----------+
| Grafana 可视化大屏 |  =====>  绘制实时监控曲线（await, IOPS, %util）
+--------------------+
```

### 推荐的核心告警 PromQL 表达式：
1. **磁盘平均响应时间（await）告警**：
   ```promql
   rate(node_disk_read_time_seconds_total{device!~"^(loop|ram|dm-).*"} [1m]) 
   / 
   rate(node_disk_reads_completed_total{device!~"^(loop|ram|dm-).*"} [1m]) 
   > 0.02
   ```
   * **含义**：如果过滤掉虚拟盘后的真实物理磁盘，其读平均延迟持续 1 分钟超过 20ms（0.02s），触发告警。
2. **磁盘队列积压严重告警**：
   ```promql
   node_disk_io_time_weighted_seconds_total > 5
   ```

# 第十二章 总结

通过对 `iostat` 详尽、全方位的学习，我们不仅认识了它在系统排障中的核心地位，更洞悉了它背后深邃的 Linux 内核设计哲学。

## iostat 的优缺点总结

### 优点
1. **极简低损**：仅通过读取内核已经累加好的 `/proc` 数据，其性能损耗在生产环境中几乎可以忽略不计。
2. **直观科学**：将吞吐、频率、队列和延迟剥离得极为清晰，提供了最科学的瓶颈判定链路。
3. **高自由度**：借助 BPF 过滤和多参数组合，能够完美胜任各种复杂存储介质的评估。

### 缺点
1. **进程隔离缺失**：无法直接定位到具体的进程名和 PID。
2. **历史记录匮乏**：不能保留历史数据以供事后审计（需要配合 `sar` 或 Prometheus）。

**学习建议**：
在日常工作中，多在业务正常运行、压测高峰期及系统空闲期分别运行 `iostat`，记录下属于你自身服务器的 **“性能基线（Baseline）”**。只有熟悉了健康的运行数值，当故障发生、数值产生异常漂移时，你才能在第一时间凭直觉一针见血地锁定病灶，保障业务高效稳定地运行。
