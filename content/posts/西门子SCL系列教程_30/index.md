## 1. 第一维度：底蕴之源——PLC 底层运行物理机理

一个普通程序员和高级架构师最底层的技术分野在于：**你是否能看透代码在硅芯片、寄存器以及工作内存里的微观运行状态**。只有懂底层，你写出的代码才具有极限的效率和稳定性。

```

 ├────────────────────────────────────────────────────────────────────────┤
 │ 1.2 物理存储器三分区规划 (SRAM / NVRAM / Flash)                         │
```

### 1.1 核心修炼心法：
*   **不要只在软件表面看数值**：每当你定义一个 `Real` 或是 `DInt` 变量时，你要在脑子里勾勒出它占用 4 个字节，在 Big-Endian（大端）格式下它是高位字节在前的二进制排列形式。
*   **警惕物理内存浪费**：在大项目中，尽量使用优化数据块（Optimized DB），避免在标准 DB 块中因为变量乱序声明产生的“内存对齐空隙（Padding）”。

### 2.1 师父推荐的一段“标准防御性标定 SCL 范式”：

```scl
// 一个高级工程师写出的标定函数，必须自带零除、限幅、越界三重保护
FUNCTION "FC_SafeScale" : Real
VAR_INPUT
    in_rRaw : Real;             // 原始输入值
    in_rRawMax : Real;          // 通道上限 (如 27648.0)
    in_rRawMin : Real;          // 通道下限 (如 0.0)
    in_rOutMax : Real;          // 标定上限 (如 100.0)
    in_rOutMin : Real;          // 标定下限 (如 0.0)
END_VAR
VAR_TEMP
    temp_rDen : Real;           // 分母缓存
    temp_rRawClamped : Real;    // 夹逼限幅后的原始值
END_VAR
BEGIN
    // 1. 物理越界防御：强制将输入值卡在 [Min..Max] 范围内，防输入污染
    #temp_rRawClamped := LIMIT(MN := #in_rRawMin, IN := #in_rRaw, MX := #in_rRawMax);
    
    // 2. 零分母防零除防御（防 CPU STOP）
    #temp_rDen := #in_rRawMax - #in_rRawMin;
    IF ABS(#temp_rDen) < 1.0E-6 THEN
        #FC_SafeScale := #in_rOutMin; // 分母为零时，输出安全默认下限值
        RETURN; // 极速退出，防止除零故障
    END_IF;
    
    // 3. 高效率乘法标定
    #FC_SafeScale := #in_rOutMin + ((#temp_rRawClamped - #in_rRawMin) / #temp_rDen) * (#in_rOutMax - #in_rOutMin);
END_FUNCTION
```

### 3.1 架构设计黄金法则：
*   **不相信任何 M 点**：在大型项目中，彻底遗忘 M 区（位存储区）。所有的全局状态交换、数据交互，一律使用带有 UDT、结构化封装的**“全局优化 DB 数据块（符号寻址）”**。
*   **统一公司级的变量前缀**：坚决推行公司级的变量命名美学（如 `#in_`、`#out_`、`#io_`、`#stat_`、`#temp_` 与驼峰数据类型标记）。代码写得规整，团队成员在协同开发时，出错概率能降低 80% 以上。

### 4.1 师父的现场干货：屏蔽层接地的物理哲学（EMC）
*   **高频干扰（如变频器、伺服动力电缆辐射）**：编码器或通信电缆屏蔽层必须 **“双端接地”**（且必须在金属夹上进行 360 度大面积环形接地）。双端接地能阻断高频磁场干扰。
*   **低频干扰（如传感器大地电势差信号干扰）**：电缆屏蔽层采用 **“单端接地”**。防止两端大地由于电位差产生微弱电流形成“地回路环流噪声”。
*   在写模拟量处理时，**一律无条件在 SCL 中加入 PT1 一阶低通滤波器**，这是用软件算法对抗恶劣电磁环境的最佳防火墙。
