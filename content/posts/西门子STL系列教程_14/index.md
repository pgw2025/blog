对于在梯形图（LAD）中进行编程的工程师来说，整数的加减乘除不过是把变量填入 `ADD` 或 `SUB` 框中。但在 CPU 底层的有源算术逻辑单元（ALU）内部，**整数运算是一场关于二进制补码（Two's Complement）物理逻辑门的电平博弈**。

PLC 的硬件核心是为高频、确定性控制而设计的。在执行整数运算时，ALU 内部的加法电路会实时驱动状态字寄存器中的 **CC0（条件码 0）**、**CC1（条件码 1）**、**OV（溢出位）** 和 **OS（存储溢出位）**。

在 STL 中，如果忽视了二进制补码的物理极限，编写出的数学计算程序极易发生**隐式正负溢出（Wrap-around）**、**除零崩溃（Division by Zero）**、或者**最窄负数取反溢出（Negation Overflow）**等严重故障。

本教程将带你深入到 CPU 的微码与硬件寄存器层面，深度剖析 16 位有符号整数运算指令 `+I`、`-I`、`*I`、`/I`（以及如何提取余数 `MOD`）、`NEGI`（取反）的运行本质，教你如何用最少的指令周期构建高可靠性的整数绝对值（`ABS`）算法。

### 技术兼容性声明

*   **S7-300 / S7-400**：原生支持本文所有 16 位整数 STL 算术指令，由内部有源 ALU 硬件直接执行。
*   **S7-1500**：完整支持本文所有 STL 指令，通过高速寄存器仿真执行。

## 1. 补码加减器与状态字防线：加法（`+I`）与减法（`-I`）

在计算机体系结构中，16 位有符号整数（`INT`）的表示范围是：$-32768 \sim +32767$。其中，最高位（Bit 15）是符号位（`0` 代表正数，`1` 代表负数）。

### 1.1 硬件设计内幕：为什么 CPU 采用补码（Two's Complement）进行加减运算？

西门子 S7 CPU 与现代计算机一样，在物理上只设计了**加法器电路**，并没有专门设计“减法器电路”。

为了用加法器实现减法，所有的有符号整数全部采用**二进制补码**存储。
*   **减法变加法**：计算 $A - B$ 在 CPU 底层等效于计算 $A + (-B)$（即 $A$ 的补码加上 $B$ 的相反数补码）。
*   **硬件优势**：这种设计极大地节省了 CPU 芯片的硅片面积，降低了功耗，并且使得加减法能在 **1 个时钟周期**内极速完成。

#### 算术结果对状态字 CC0 和 CC1 的硬件驱动规则：

每次执行完 `+I` 或 `-I`，ALU 都会根据当前 ACCU 1 中的计算结果，瞬间更新状态字中的 **CC1** 和 **CC0** 位：

```
+---------------------------------------------------------------------------------+
| Condition Code (CC1, CC0) Generation after Integer Math                         |
|                                                                                 |
|                      ALU Calculation Result in ACCU 1                           |
|                                     │                                           |
|         ┌───────────────────────────┼───────────────────────────┐               |
|         ▼ (Result = 0)              ▼ (Result < 0)              ▼ (Result > 0)  |
|      CC1 = 0, CC0 = 0            CC1 = 0, CC0 = 1            CC1 = 1, CC0 = 0   |
+---------------------------------------------------------------------------------+
```

如果计算结果超出了 $-32768 \sim +32767$ 的物理极限，状态字中的 **OV（溢出）** 和 **OS（存储溢出）** 会被立即置 `1`。

### 1.2 知识点 1 代码：带硬件溢出保护的 PID 手动输出限制器

我们编写一段高可靠性的代码：在手动模式下，操作员可以通过“增加”和“减少”按钮调节比例阀的开度（DINT或INT范围），但在运算中一旦发生溢出或超出 $0 \sim 10000$（代表 0.00%~100.00%）范围，系统必须立刻拦截并实施硬限幅保护。

#### 完整 STL 源代码

```stl
FUNCTION "Safe_Manual_Adjust" : VOID
TITLE = 高可靠性手动输出限制器
VAR_INPUT
   i_Incr_Cmd : BOOL ;        // 输入：增加开度指令 (I0.0)
   i_Decr_Cmd : BOOL ;        // 输入：减少开度指令 (I0.1)
   i_Step_Val : INT ;         // 输入：调节步长 (例如 50)
END_INPUT
VAR_IN_OUT
   iq_Current_Out : INT ;     // 输入输出：当前开度值 (0~10000)
END_VAR
VAR_OUTPUT
   q_Over_Limit   : BOOL ;    // 输出：超出限幅报警 (Q0.0)
END_VAR
BEGIN
NETWORK
TITLE = 1. 执行加法调节并立即捕获溢出
      A     #i_Incr_Cmd;      
      JCN   Decr;             // 若无增加指令，跳转至 Decr

      L     #iq_Current_Out;  // 装载当前值
      L     #i_Step_Val;      // 装载调节步长，原当前值移入 ACCU 2
      +I    ;                 // 执行 16位有符号整数加法 (ACCU 2 + ACCU 1 -> ACCU 1)
      
      // 物理自检：如果加法导致了溢出（例如越界到了 32768 变成负数）
      A     OV;               // 检查状态字中的 OV 标志位
      JC    ERR_LIMIT;        // 若发生溢出，强行跳转至错误限幅区

      T     #iq_Current_Out;  // 正常状态，更新当前开度
      JU    Range_Check;      // 跳转至物理范围限幅检查

Decr: A     #i_Decr_Cmd;      
      JCN   Range_Check;      // 若无减少指令，跳转至范围检查

      L     #iq_Current_Out;  
      L     #i_Step_Val;      
      -I    ;                 // 执行减法 (ACCU 2 - ACCU 1 -> ACCU 1)
      A     OV;               
      JC    ERR_LIMIT;        
      T     #iq_Current_Out;  

NETWORK
TITLE = 2. 物理极限范围限幅检查 (0 ~ 10000)
Range_Check: NOP 0;
      L     #iq_Current_Out;  
      L     10000;            
      >I    ;                 // 若当前值 > 10000
      JC    ERR_MAX;          

      L     #iq_Current_Out;  
      L     0;                
      <I    ;                 // 若当前值 < 0
      JC    ERR_MIN;          
      
      CLR   ;                 // 正常无越界
      =     #q_Over_Limit;    
      BEU   ;                 // 安全退块

ERR_LIMIT: NOP 0;             // 发生算术溢出时的惩罚性保护
      // 根据前序计算趋势，判定是正向溢出还是负向溢出
      // 检查 CC1 和 CC0。若 CC1=1, CC0=0 说明结果原本为正（正溢出）
      A     CC_1;             
      JC    ERR_MAX;          
      JU    ERR_MIN;          

ERR_MAX: L  10000;            // 强行限幅至 10000
      T     #iq_Current_Out;  
      SET   ;                 
      =     #q_Over_Limit;    // 报出限幅警告
      BEU   ;

ERR_MIN: L  0;                // 强行限幅至 0
      T     #iq_Current_Out;  
      SET   ;                 
      =     #q_Over_Limit;    
END_FUNCTION
```

#### 逐行微观寄存器状态变化跟踪表

假设当前状态：`iq_Current_Out = 32750`（接近 16 位正整数物理极限 `32767`），`i_Step_Val = 50`。执行加法调节：$32750 + 50 = 32800$。这超出了 `+32767` 的上限，在二进制层面会发生**符号位回绕**，结果会变成负数。

| 步骤 | 执行指令 | ACCU 1 (16-bit dec) | ACCU 2 (16-bit dec) | 当前 RLO | `/FC` | `OV` | `CC1 / CC0` | 算术溢出微观物理动作解析 |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| **0** | (开始) | `0` | `0` | `0` | `0` | `0` | `0/0` | 初始化。 |
| **1** | `L #iq_Current_Out` | `32750` | `0` | `1` | `0` | `0` | `0/0` | 将 `32750`（十六进制 `16#7FEE`）装入 ACCU 1。 |
| **2** | `L #i_Step_Val` | `50` | `32750` | `1` | `0` | `0` | `0/0` | 将 `50`（十六进制 `16#0032`）装入 ACCU 1，`32750` 滑入 ACCU 2。 |
| **3** | `+I` | `-32736` | `32750` | `1` | `0` | **`1`** | **`0/1`** | **溢出发生**：有源加法器执行 `32750 + 50`。二进制补码相加结果为 `16#8020`。由于最高位（Bit 15）变为了 `1`，被解读为负数 `-32736`。**ALU 判定结果越界，强行将 OV 硬件置 1，由于结果为负，CC1/CC0 变为 0/1**。 |
| **4** | `A OV` | `-32736` | `32750` | **`1`** | `1` | `1` | `0/1` | 首检：读取 `OV = 1`，RLO 被置为 `1`。`/FC` 置 `1`。 |
| **5** | `JC ERR_LIMIT` | `-32736` | `32750` | `1` | `0` | `1` | `0/1` | 跳转成立。程序计数器（PC）强行跳转至 `ERR_LIMIT` 标号分支。/FC 自动清零。 |
| **6** | `A CC_1` | `0` | `-32736` | `0` | `1` | `1` | `0/1` | 检查状态字。由于当时是正向溢出，在进入跳转前，由于溢出回绕前的趋势是正数，通过 CC1 判定是否发生正向越界。最终强行跳转到 `ERR_MAX` 将值强行复位为 `10000`。 |

### 1.3 工程经验：在中断或高速处理中，如何快速清除“存储溢出（OS）”标志

在西门子状态字中，**OS（Stored Overflow，存储溢出）** 位具有锁存特性。一旦在你的块内部某处计算触发过一次 `OV`，`OS` 就会永久保持为 `1`。

*   **痛点**：
    如果你试图在块的最后用 `A OS` 来判定全局计算是否有过错，一旦 OS 被锁死，后续即使所有的计算都正常，`A OS` 依然会高频报错。
*   **消除 OS 的微码指令大招**：
    西门子没有直接提供“复位 OS”的专用指令。
    在 STL 中，**执行任何一次正常的、不产生溢出的双精度浮点数除法或特定的 32 位乘法，都会自动将 OS 复位清零。**
    最常用的、不产生任何副作用的复位 OS 的硬核写法是：
    ```stl
    // 在块的入口处，强行执行一次 0.0 * 0.0 实数乘法，
    // 该计算绝不产生溢出，系统固件在执行时会自动将 OS 位强制清零（0）！
    L     0.0
    L     0.0
    *R    ;                   // 这一行执行完，状态字中的 OS 被完美复位清零
    ```

## 2. 乘法与除法双输出通道：乘法（`*I`）与除法（`/I`、`MOD`）

在 16 位有符号整数除法中，西门子 CPU 采用了一个极其精妙、在微处理器设计史上非常经典的**“双返回通道”**设计。

### 2.1 硬件设计内幕：除法 `/I` 如何同时计算出“商”和“余数（MOD）”？

对于普通的 32 位计算机，执行除法和求模（取余数）是两个完全独立的汇编指令。但在 PLC 控制中，执行除法（如将“总秒数”换算为“分钟+剩余秒数”）时，我们几乎总是同时需要“商”和“余数”。

因为除法是 CPU 所有整数指令中**耗时最长、执行周期最多**的指令。如果执行完除法后，再执行一次单独的模运算，会造成一倍以上的 ALU 时钟周期浪费。

为了解决这一痛点，西门子 S7 CPU 的除法器微码设计为：**执行 `/I` 指令时，ALU 会在 1 个指令周期内同时计算出“商”和“余数”，并将它们打包存放在一个 32 位的累加器 1（ACCU 1）中：**

```
+---------------------------------------------------------------------------------+
| ACCU 1 Memory Layout after Integer Division (/I)                                |
|                                                                                 |
|  ACCU 1 High Word (Bits 31...16)             ACCU 1 Low Word (Bits 15...0)       |
|  +--------------------------------+          +--------------------------------+ |
|  |     Remainder (余数 / MOD)     |          |       Quotient (商)            | |
|  +--------------------------------+          +--------------------------------+ |
|  16-bit Signed Integer                       16-bit Signed Integer              |
+---------------------------------------------------------------------------------+
```

*   **ACCU 1 的低 16 位（Low Word）**：存放除法计算的**商（Quotient）**。
*   **ACCU 1 的高 16 位（High Word）**：存放除法计算的**余数（Remainder / MOD）**。

### 2.2 知识点 2 代码：高精度时钟解析器（总秒数换算为“分+秒”）

我们编写一段代码，接收一个不断累加的总秒数（0~32767），利用 `/I` 指令的双返回通道特征，在最少的指令周期内，极速拆解出当前的“分钟值”和“剩余秒值”。

#### 完整 STL 源代码

```stl
FUNCTION "Time_Parser_16Bit" : VOID
TITLE = 高速时间解析器 (总秒数 -> 分 + 秒)
VAR_INPUT
   i_Total_Seconds : INT ;    // 输入：累计的总秒数 (0 ~ 32767)
END_INPUT
VAR_OUTPUT
   q_Minutes       : INT ;    // 输出：分钟值
   q_Rem_Seconds   : INT ;    // 输出：剩余秒值 (0 ~ 59)
END_VAR
BEGIN
NETWORK
TITLE = 1. 执行有源 16 位整数除法
      L     #i_Total_Seconds; // 将总秒数 (如 125) 装载入 ACCU 1 (低16位填充，高16位补0)
      L     60;               // 将除数 60 装载入 ACCU 1，总秒数被滑动挤入 ACCU 2
      
      // 执行 16位除法：ACCU 2 / ACCU 1 -> ACCU 1
      /I    ;                 // 计算: 125 / 60。 
                              // 此时 ACCU 1 的状态变为:
                              // * 低16位 (Low Word)  = 2  (商，即分钟数)
                              // * 高16位 (High Word) = 5  (余数，即剩余秒数)

NETWORK
TITLE = 2. 提取并分发 商（分钟）与 余数（秒）
      // 提取低 16 位的商直接传送。
      // 因为传送指令 T 在传送 16位有符号数（INT）时，默认只读取 ACCU 1 的低 16 位！
      T     #q_Minutes;       // 此时 q_Minutes 获得数据 2

      // 提取高 16 位的余数。
      // 我们需要将高 16 位的数据移动到低 16 位。
      // 采用双字向右逻辑移动 16 位的指令 SRD 16 
      SRD   16;               // 将整个 ACCU 1 双字向右移动 16 位，高位补 0。
                              // 移动后：ACCU 1 的低 16 位变成了原先高 16 位的余数（5）！
      
      T     #q_Rem_Seconds;   // 此时 q_Rem_Seconds 获得数据 5
END_FUNCTION
```

#### 逐行微观寄存器状态变化跟踪表

假设输入：`i_Total_Seconds = 125`。

| 步骤 | 执行指令 | ACCU 1 (32-bit hex) | ACCU 2 (32-bit hex) | RLO | `/FC` | 状态字 CC1/CC0 | 寄存器级双通道返回变化详情 |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| **0** | (开始) | `0000_0000` | `0000_0000` | `0` | `0` | `0/0` | 寄存器清空。 |
| **1** | `L #i_Total_S...` | `0000_007D` (125) | `0000_0000` | `0` | `0` | `0/0` | `125`（十六进制 `7D`）装入 ACCU 1 的低 16 位。高 16 位填充 0。 |
| **2** | `L 60` | `0000_003C` (60) | `0000_007D` (125) | `0` | `0` | `0/0` | `60`（十六进制 `3C`）装入 ACCU 1，`125` 滑动到 ACCU 2。 |
| **3** | `/I` | `0005_0002` | `0000_007D` | `0` | `0` | `1/0` (大于0) | **除法器极速双返回**：ALU 除法器执行 `125 / 60`。商为 `2`（十六进制 `0002`），余数为 `5`（十六进制 `0005`）。它们被打包塞入 ACCU 1：高 16 位为 `0005`，低 16 位为 `0002`，即 `16#0005_0002`。 |
| **4** | `T #q_Minutes` | `0005_0002` | `0000_007D` | `0` | `0` | `1/0` | **提取商**：T 指令写入 16 位有符号变量时，默认只拷贝 ACCU 1 的低 16 位（`0002`）。商 `2` 成功写出。 |
| **5** | `SRD 16` | `0000_0005` | `0000_007D` | `0` | `0` | `1/0` | **提取余数（MOD）**：执行双字右移 16 位。高 16 位的 `0005` 被移动到了低 16 位。ACCU 1 变为了 `16#0000_0005`。 |
| **6** | `T #q_Rem_Sec...` | `0000_0005` | `0000_007D` | `0` | `0` | `1/0` | 传送：余数 `5` 成功写出。整个过程只执行了 1 次除法！ |

### 2.3 工程经验：防范“除数为 0”引发的 CPU 闪红灯直接崩溃停机故障

在工业控制中，由于模拟量断线、操作员误设参数，除数 `#Divisor` 极易在运行中变为 `0`。

在 S7-300/400 物理 CPU 中，一旦执行 `/I` 且除数为 0，系统会直接判定为 **Hardware Division by Zero (除零硬件故障)**，直接强行跳转调用 **`OB121`（编程错误中断）**。
如果你的 PLC 内部没有下装 `OB121` 块，**CPU 会在微秒级瞬间强行进入 STOP（停止）状态！** 导致整条生产线意外停机。

#### 绝对安全防范规范代码（每一位合格工程师的标配防线）：

在任何 `/I` 或 `/D` 指令前，**必须首先对除数进行零值拦截**：

```stl
// 工业标准安全除法防御盾牌：
L     #i_Divisor
L     0
==I   ;                       // 比较除数是否为 0
JC    ERR_DIV_ZERO;           // 若为 0，强行拦截，跳转至错误安全处理

L     #i_Dividend
L     #i_Divisor
/I    ;                       // 此时执行除法，100% 免疫除零停机风险
T     #q_Quotient
BEU   ;                       // 安全退出

ERR_DIV_ZERO: NOP 0;          // 故障恢复分支
L     0
T     #q_Quotient;            // 强行输出安全默认值 0
SET   ;                       
=     #q_Div_Zero_Alarm;      // 报出除零严重故障报警
```

## 3. 符号位翻转与绝对值：取反（`NEG`）与绝对值（`ABS`）

在 16 位整数运算中，取反（`NEGI`，求相反数）和绝对值（`ABS`）是常用的信号调理指令。

### 3.1 补码取反的物理细节与 $1/32768$ 的溢出深层大坑

在二进制补码体系下，对一个有符号整数进行取反（`NEGI`）的物理机制是：**“按位取反（Invert all bits），然后加 1”**。

这引入了一个在计算机科学中非常经典的、能让 90% 程序员踩雷的硬件大坑：**$-32768$ 的取反溢出。**

*   **大坑成因**：
    16 位有符号整数的表示范围是不对称的（因为 0 占用了正数的一位空间，导致负数比正数多一个）。其范围是 $-32768 \sim +32767$。
    如果你对最小负数 **`-32768`（十六进制 `16#8000`）** 执行 `NEGI` 指令，试图求其相反数：

```
1. 载入原始值 -32768:
   ACCU 1 = 1000_0000_0000_0000 (16#8000)

2. 执行 NEGI (第一步: 按位取反):
   ACCU 1 变为了 0111_1111_1111_1111 (16#7FFF, 即 +32767)

3. 执行 NEGI (第二步: 硬件加 1):
   ACCU 1 变为了 1000_0000_0000_0000 (16#8000, 符号位重新回绕变成了 1 !)

4. 最终结果:
   ACCU 1 = -32768 ! 并且 状态字中 OV = 1, OS = 1 (溢出触发)
```

*   **物理结果**：**在 16 位整数下，$-32768$ 的相反数依然是 $-32768$！并且伴随严重的算术溢出！**

### 3.2 知识点 3 代码：超低指令开销的高性能 16 位整数绝对值（`ABS`）函数

西门子 STL 并没有提供专门针对 16 位整数的 `ABSI` 指令（只提供了针对浮点数的 `ABS`）。
有些工程师喜欢用“判断正负，如果是负数就乘 -1”的方式来实现绝对值。但乘法指令（`*I`）的物理执行周期极其漫长，会严重拖慢扫描速度。

下面我们利用状态字中的 **CC1 和 CC0 符号标志位**加 `NEGI` 组合，编写一个**仅消耗 4 个时钟周期、零乘法开销的极限速度 16 位整数绝对值算法。**

#### 完整 STL 源代码

```stl
FUNCTION "ABS_Int_Fast" : VOID
TITLE = 高速 16 位整数绝对值函数 (ABS)
VAR_INPUT
   i_Input_Val : INT ;        // 输入的有符号整数 (INT)
END_INPUT
VAR_OUTPUT
   q_ABS_Out   : INT ;        // 输出：绝对值
END_VAR
BEGIN
NETWORK
TITLE = 1. 利用状态字符号位，判定并执行取反
      L     #i_Input_Val;     // 1. 装载输入值。此时状态字 CC1/CC0 自动更新
      
      // 2. 特殊保护：若输入刚好为最小负数 -32768，取反会溢出。
      // 我们必须强行将其限幅至 -32767，以防止溢出回绕。
      L     -32768;
      ==I   ;                 // 比较是否等于 -32768
      JC    ERR_MIN_INT;      // 若等于，跳转至特殊处理

      L     #i_Input_Val;     // 重新载入
      // 3. 检查 CC1 和 CC0 的状态。
      // 若 CC1=0, CC0=1，表明当前装载的数据为负数（Result < 0）
      A     CC_0;             // 检查 CC0 是否为 1
      AN    CC_1;             // 且 CC1 是否为 0
      JCN   Save;             // 若不为负（大于等于0），直接跳转至 Save 传出

      NEGI  ;                 // 4. 重点：若为负数，执行取反（按位取反加1，仅需1个时钟周期！）

Save: T     #q_ABS_Out;       // 保存绝对值结果
      BEU   ;                 // 安全退块

ERR_MIN_INT: L 32767;         // 极限越界安全保护：将绝对值强行限制在最大上限 32767
      T     #q_ABS_Out;
END_FUNCTION
```

## 5. TIA Portal S7-1200 / S7-1500 的现代 SCL 替代与优化

由于 **S7-1200 不支持直接操作状态字条件码（CC0/CC1）**。为了在现代 TIA 博途平台下实现同等高效且具备越界防御机制的有源整型算法，本章提供 100% 优化的 **SCL 语言重构方案**。

### 5.1 知识点 2 的 SCL 规范重构（时间解析器，Modulo 求余）

在 SCL 中，我们不需要手动右移 16 位，可以直接使用内置的 `/`（整除）与 `MOD`（取模）关键字。博途编译器会自动将其优化为最简机器码。

```scl
// S7-1200 / 1500 优化的 SCL 时间解析器
#q_Minutes := #i_Total_Seconds / 60;        // SCL 整除运算
#q_Rem_Seconds := #i_Total_Seconds MOD 60;  // SCL 取模运算
```

// 100% 优化访问、带 -32768 越界防御的 SCL ABS 块
IF #i_Input_Val = -32768 THEN

## 6. 工业级规范与调试经验总结

在本章整数运算的尾声，我们为具有 LAD 基础的工程师梳理出 2 条在实际工程现场处理算术故障时的金牌经验：

1.  **善用“在线状态监视（Online Monitor）”追踪 OV 与 OS 的跃迁**：
    当你在现场遇到诸如“配方计算偶尔不对”、“阀门开度偶尔乱跳”等偶发性算术故障时，**立刻打开计算块的在线 STL 视图，紧紧盯着右侧状态字列中的 OS 位**。
    *   **诊断绝招**：只要 OS 变红（`1`），就说明在长达数小时的运行中，由于某种极限工艺工况触发，你的代码中**必然发生过至少一次加减越界或除零故障**。根据 OS 出现的逻辑网络，顺藤摸瓜，即可在半小时内彻底定位故障。
2.  **在涉及除法、移位等高开销计算时，坚持“能用移位不用除法”的优化准则**：
    在编写高频扫描的底层算法时，如果你需要除以 2、除以 4 或除以 8。**绝对禁止写 `/I`（除法指令）**。
    *   **优化方案**：
        除以 2，等效于**有符号右移 1 位（`SSW 1`）**。
        除以 4，等效于**有符号右移 2 位（`SSW 2`）**。
        `SSW`（算术右移）属于 CPU 的原生单指令位操作，其物理执行耗时**仅为除法指令 `/I` 的 1/15 左右**。这一习惯能让你的系统扫描周期始终保持在极佳的黄金线以内。
