*   “现场有 100 个轴承温度传感器，需要找出哪个温度最高，并把超出 80°C 的传感器全部筛选出来报警。”
*   “配方系统里有 50 种原料的配比参数，需要一键将它们全部清零或进行系数微调。”

如果用梯形图（LAD）写这类逻辑，你得复制粘贴几百行，核对一个变量就能让你眼花缭乱；更别提一旦数组长度从 100 变成 120，你就得重画整个梯形图。

## 1. FOR 循环的语法与底层运行机制

`FOR` 循环是一种**计数型循环**。它用于在已知循环次数的情况下，重复执行某段代码。

### 1.1 基础语法结构

```scl
FOR <循环变量> := <初始值> TO <目标值> [BY <步长>] DO
    <循环体代码>;
END_FOR;
```

*   **循环变量**：用来记录当前循环到第几次的变量（如 `#i`）。
*   **初始值** 和 **目标值**：循环的起点和终点。
*   **BY <步长>**：可选参数。每次循环结束后，循环变量增加的数值。如果省略，默认步长为 `1`。

---

### 1.2 💡 极其重要：循环在 PLC 里的“时间本质”

徒弟，这是很多刚从 PC 端编程转过来的程序员最容易犯的原则性错误，你必须牢牢记住：

**在 PLC 中，整个 `FOR` 循环（哪怕循环 1000 次），也是在同一个 CPU 扫描周期内全部执行完毕的！它不会跨越多个周期执行！**

```
 正常 PLC 扫描周期:
 ┌─────────────────┬──────────────────────────────────────────┬─────────────────┐
 │   1. 输入采样   │               2. 执行程序                │   3. 输出刷新   │
 └─────────────────┴────────────────────┬─────────────────────┴─────────────────┘
                                        │
                                        ▼
                         SCL 中的 FOR 循环在这里瞬间爆发：
                         FOR #i := 0 TO 99 DO
                            // 100 次重复计算在微秒内一气呵成！
                         END_FOR;
```

*底层物理真相*：
当 CPU 执行到 `FOR` 语句时，它会不断进行“比较、累加、往回跳转”的硬件指令操作。在整个循环彻底结束（循环变量大于目标值）之前，**CPU 会被死死锁在这段代码里，不会去刷新物理输出（DO 映像区），也不会去响应看门狗复位**。

因此：
1.  **绝不能在 `FOR` 循环内部调用延时等待函数（如 `WAIT` 等，PLC 中也不存在这类函数）**。
2.  **绝不能写循环次数无法预测或极度巨大的循环。** 如果一个循环要跑 100 万次，CPU 会因为在单次扫描周期内耗时太长，直接触发“看门狗超时（Watchdog Timeout）”故障并强行停机（STOP）！

## 2. 循环变量的底层奥秘与安全规范

很多新手写循环，变量声明极其随意。在高级工业编程中，循环变量有着极其严苛的规范。

### 2.1 规范一：循环变量必须声明在 `VAR_TEMP`（临时变量区）

```scl
VAR_TEMP
    i : Int; // 完美的循环变量声明
END_TEMP
```

*为什么不能声明在 `VAR`（静态变量区）或全局 DB 中？*
1.  **读写速度**：`VAR_TEMP` 分配在 CPU 的高频局部数据堆栈（L-Stack）中。循环变量在每次循环中都会被频繁读写，声明在 `TEMP` 区可以获得最高的硬件访问速度。
2.  **重入安全**：如果把循环变量声明为全局变量，当你的程序被多重调用或者在中断块（如 OB30）中被强行打断时，全局循环变量的值可能会被篡改，从而导致主循环陷入**死循环**或产生越界 Bug。

### 2.2 规范二：绝不要在循环体内部去修改循环变量的值

```scl
// ❌ 极其危险的自杀式写法！
FOR #i := 0 TO 9 DO
    #i := #i + 1; // 严禁在内部手动改写循环变量！这会导致循环逻辑彻底失控
END_FOR;
```
*原因*：`FOR` 语句在每次执行到 `END_FOR` 时，会自动按照设定的 `BY` 步长去累加循环变量。你在内部强行修改，不仅会破坏编译器的寄存器优化，还极易制造无法跳出的无限循环。

### 2.3 提前终止循环：`EXIT` 指令
如果在循环中途，你已经找到了想要的数据，不需要再往后找了，可以使用 `EXIT` 立即跳出当前循环，节省 CPU 算力。

```scl
FOR #i := 0 TO 99 DO
    IF #arrData[#i] = 16#FF THEN
        #iTargetIndex := #i;
        EXIT; // 找到了，立刻跳出循环，不再执行后续的第 11..99 次比较
    END_IF;
END_FOR;
```

## 3. 数组遍历：SCL 与数据结构的完美合体

`FOR` 循环最强大的搭档就是 **数组（Array）**。将循环变量作为数组的**下标索引**，就可以用极其简练的代码横扫成千上万个数据。

### 3.1 💡 工业级规范：使用 `LOWER_BOUND` 与 `UPPER_BOUND` 规避硬编码

很多新手写数组遍历，喜欢把上限写死：
```scl
// ❌ 埋下隐患的硬编码写法
FOR #i := 0 TO 99 DO // 万一以后数组大小改成 120，这里就漏掉了 20 个数据
    #arrSensors[#i] := 0.0;
END_FOR;
```

在西门子 S7-1200/1500 中，系统提供了动态获取数组边界的内置指令：
*   **`LOWER_BOUND(ARR := #Array, DIM := 1)`**：获取数组的第一维下限。
*   **`UPPER_BOUND(ARR := #Array, DIM := 1)`**：获取数组的第一维上限。

```scl
//  高弹性的自适应遍历写法
#diLowBound := LOWER_BOUND(ARR := #arrSensors, DIM := 1);
#diHighBound := UPPER_BOUND(ARR := #arrSensors, DIM := 1);

FOR #i := #diLowBound TO #diHighBound DO
    #arrSensors[#i] := 0.0; // 无论数组大小如何调整，程序自适应运行，永不越界
END_FOR;
```

---

### 3.2 ⚠️ 致命红线：数组下标越界（Index Out of Bounds）

如果你的数组定义为 `Array[0..99] of Real`，而你的 `FOR` 循环由于逻辑错误访问了 `arrSensors[100]`。
在 S7-1500 中，**默认情况下 CPU 会瞬间报错并进入 STOP 状态，现场设备会由于强制切断输出而瞬间瘫痪！**

**防范机制**：
1.  **必须保证** 循环计数器的范围被严格限制在数组物理边界之内。
2.  如果在现场调试阶段，可以在 CPU 的组织块（OB）中，添加 **“外设访问出错组织块 (OB122)”** 和 **“编程出错组织块 (OB121)”**。这样当发生越界时，CPU 只会报错，而不会立刻停机，给工程师留出挽救的机会（但在正式交付时，必须从代码层杜绝越界）。

## 4. 工业级综合案例：100个传感器报警检测与汇总分选系统

现在，我们把 `FOR` 循环、结构体数组（Array of UDT）、边界自适应检索以及最值筛选算法拧成一股绳，写一个真正符合现代化大厂交付标准的生产程序。

### 4.1 工业场景描述
某化工厂的加热炉管道上均匀分布着 100 个温度传感器。我们需要编写一个 SCL 控制块：
1.  **输入**：包含 100 个传感器的物理测量数据包。每个传感器包含：当前温度值（Real）、温度上限（Real）、温度下限（Real）、屏蔽启用位（Bool）。
2.  **批量检测与报警**：
    *   遍历所有 100 个传感器。如果某个传感器未被屏蔽，且其温度超出上限或低于下限，立即置位该通道的报警标志位。
    *   统计当前处于报警状态的传感器总数。
3.  **最值筛选（核心算法）**：
    *   找出所有非屏蔽的传感器中，**当前温度最高（Hottest）** 和 **当前温度最低（Coldest）** 的传感器，并记录它们的数值以及对应的传感器编号（ID）。

---

### 4.2 步骤一：创建自定义数据类型（UDT）

在博途左侧项目树中，双击“添加新数据类型”，命名为 `UDT_SensorElement`：

```scl
// 结构体声明：UDT_SensorElement
TYPE "UDT_SensorElement"
VERSION : 0.1
   STRUCT
      rValue : Real;         // 实时采集温度值 (°C)
      rMaxLimit : Real;      // 温度安全上限 (°C)
      rMinLimit : Real;      // 温度安全下限 (°C)
      bBypass : Bool;        // 旁路屏蔽标志 (为 TRUE 时忽略该传感器报警)
      bAlarmActive : Bool;   // 传感器报警输出线圈
   END_STRUCT
END_TYPE
```

---

### 4.3 步骤二：块接口声明（FB_SensorBatchMonitor）

因为我们需要将处理结果进行全局存储，并且可能需要输出，所以我们采用 **FB** 块。我们通过 `IN_OUT` 传入 100 个传感器的数组，以此实现我们第二章学过的“引用传递”性能优化。

```
VAR_INPUT
    rCriticalLimit_High : Real := 90.0; // 危险绝对超限上限 (超过该值触发整柜总警报)
END_VAR

VAR_OUTPUT
    iActiveAlarmsCount : Int;   // 当前处于激活状态的报警传感器总数
    rMaxTempFound : Real;       // 筛选出的最高温度值
    iMaxTemp_SensorID : Int;    // 最高温度传感器编号
    rMinTempFound : Real;       // 筛选出的最低温度值
    iMinTemp_SensorID : Int;    // 最低温度传感器编号
    bCriticalAlarm : Bool;      // 绝对超限危险总警报
END_VAR

VAR_IN_OUT
    arrSensors : Array[*] of "UDT_SensorElement"; // 使用 [*] 开放型数组，自适应各种长度
END_VAR

VAR_TEMP
    diLowBound : DInt;          // 动态获取数组下限
    diHighBound : DInt;         // 动态获取数组上限
    i : DInt;                   // 循环计数变量
    rCurrentVal : Real;         // 临时存储当前遍历到的传感器值
END_VAR
```

---

### 4.4 步骤三：SCL 代码实现

```scl
FUNCTION_BLOCK "FB_SensorBatchMonitor"
{ S7_Optimized_Access := 'TRUE' }
VERSION : 0.1
BEGIN
	// ==========================================
	// 1. 动态获取外部传入的传感器数组边界
	// ==========================================
	#diLowBound := LOWER_BOUND(ARR := #arrSensors, DIM := 1);
	#diHighBound := UPPER_BOUND(ARR := #arrSensors, DIM := 1);
	
	// ==========================================
	// 2. 局部变量初始化（防止残留上个周期的脏数据）
	// ==========================================
	#iActiveAlarmsCount := 0;
	#bCriticalAlarm := FALSE;
	
	// 初始化最值：将最高温度初始化为极小值，最低温度初始化为极大值
	#rMaxTempFound := -9999.0;
	#rMinTempFound := 9999.0;
	#iMaxTemp_SensorID := -1;
	#iMinTemp_SensorID := -1;
	
	// ==========================================
	// 3. 核心 FOR 循环批量数据检索
	// ==========================================
	FOR #i := #diLowBound TO #diHighBound DO
	    
	    // 抓取当前遍历到的传感器实时值，减少后续数组寻址带来的算力消耗
	    #rCurrentVal := #arrSensors[#i].rValue;
	    
	    // ------------------------------------------
	    // A. 报警检测逻辑 (仅在未旁路屏蔽时生效)
	    // ------------------------------------------
	    IF NOT #arrSensors[#i].bBypass THEN
	        
	        // 检测是否超出其设定边界
	        IF (#rCurrentVal > #arrSensors[#i].rMaxLimit) OR 
	           (#rCurrentVal < #arrSensors[#i].rMinLimit) THEN
	            
	            #arrSensors[#i].bAlarmActive := TRUE; // 触发单通道报警自锁线圈
	            #iActiveAlarmsCount := #iActiveAlarmsCount + 1; // 激活报警计数器自增
	        ELSE
	            #arrSensors[#i].bAlarmActive := FALSE; // 恢复正常
	        END_IF;
	        
	        // 危险绝对超限判定
	        IF #rCurrentVal >= #rCriticalLimit_High THEN
	            #bCriticalAlarm := TRUE;
	        END_IF;
	        
	        // ------------------------------------------
	        // B. 最值（最大值/最小值）筛选算法
	        // ------------------------------------------
	        // 筛选最高温度
	        IF #rCurrentVal > #rMaxTempFound THEN
	            #rMaxTempFound := #rCurrentVal;
	            #iMaxTemp_SensorID := DINT_TO_INT(#i); // 记录最热传感器的索引 ID
	        END_IF;
	        
	        // 筛选最低温度
	        IF #rCurrentVal < #rMinTempFound THEN
	            #rMinTempFound := #rCurrentVal;
	            #iMinTemp_SensorID := DINT_TO_INT(#i); // 记录最冷传感器的索引 ID
	        END_IF;
	        
	    ELSE
	        // 如果通道被旁路屏蔽，强制清除该通道的报警输出线圈，且不参与最值筛选
	        #arrSensors[#i].bAlarmActive := FALSE;
	    END_IF;
	    
	END_FOR;
	
	// ==========================================
	// 4. 防御性安全兜底：如果所有通道都被屏蔽，重置最值
	// ==========================================
	IF #iMaxTemp_SensorID = -1 THEN
	    #rMaxTempFound := 0.0;
	    #rMinTempFound := 0.0;
	END_IF;
	
END_FUNCTION_BLOCK
```

## 5. 代码逐行深度解剖与大厂性能优化思维

徒弟，别小看这短短几十行代码，里面包含了几项极其核心的工业级性能优化。

### 5.1 优化一：降低数组间接寻址的算力消耗（第 28 行）
在循环开始处，我们写了：
`#rCurrentVal := #arrSensors[#i].rValue;`
在接下来的代码中，我们多次用到了这个当前的温度值进行“上限比较”、“下限比较”、“绝对超限比较”、“最大值筛选”、“最小值筛选”。

*为什么不直接用 `#arrSensors[#i].rValue` 去写后面的五个比较公式？*
因为在 PLC 的硬件底层，每次执行类似 `#arrSensors[#i]` 这样的符号访问，CPU 都要在后台计算一次 **“基地址 + 偏移量 * 索引”** 的数学乘加公式。如果在循环内部重复写 5 次，CPU 就要在微秒内重复做 500 次无谓的地址计算。
而我们通过临时变量 `#rCurrentVal` 相当于在 **L-Stack 高频缓存中建立了一个副本**。后方的比较直接读取缓存，**运算效率整整提升了 5 倍！**

---

### 5.2 优化二：开放型数组 `Array[*]` 的妙用
在 `IN_OUT` 声明中，我们使用了 `Array[*] of UDT_SensorElement`：
这是 TIA 博途 V14 以后版本引入的高级特性——**变长动态数组**。
有了这个 `[*]`，这个 FB 块就不再死死绑定在 “100 个传感器”上。如果隔壁车间有 50 个传感器，或者另一个大项目有 500 个传感器，**你不需要对该 FB 代码做任何修改，直接调用并把数组扔进去，程序就可以完美自适应运行。**

---

### 5.3 优化三：最值算法的“极端初始化”（第 17~18 行）
在筛选最大值时，我们把初始对比基准 `#rMaxTempFound` 设为了极小的 `-9999.0`。
*为什么不设为 0.0？*
因为如果现场是极度严寒的西伯利亚环境，所有温度传感器的实际读数都是 `-20.0`°C。如果你把最大值初始值设为 `0.0`，那么在执行 `#rCurrentVal > #rMaxTempFound` 时，`-20.0 > 0.0` 永远不成立，最终筛选出的最大温度就会变成荒谬的 `0.0`。
**同理，最小值初始基准必须设为极大值 `9999.0`。**

## 6. 常见错误与避坑指南 (Senior Tips)

### 6.1 错误一：用 `FOR` 循环来做“延时等待”动作
```scl
// ❌ 导致 CPU 直接看门狗超时停机的致命错误写法
FOR #i := 0 TO 10000 DO
    // 企图通过空跑循环让 PLC 延时，这绝对会逼疯看门狗！
    ;
END_FOR;
```
*后果*：PLC 没有任何“空跑等待”的宽容度。如果在单周期内 CPU 运行时间超出限制（比如 150ms），硬件会立刻判定系统失控并锁死停机。
*正确做法*：延时动作必须使用系统自带的 `TON`、`TOF` 定时器，跨越多个 PLC 扫描周期来优雅地计算时间。

---

### 6.2 错误二：循环变量与数组边界“错位 1”（Off-By-One Bug）
很多刚从 C 语言（数组索引从 0 开始）转过来的工程师，在写西门子数组（默认从 1 开始，但也支持从 0 开始）时，经常写错起点。
```scl
VAR
    arrMyData : Array[1..10] of Real; // 边界为 1 到 10
END_VAR

// ❌ 错位运行，导致越界死机
FOR #i := 0 TO 9 DO
    #arrMyData[#i] := 1.0; // 当 i=0 时，访问 arrMyData[0] 发生越界崩溃！
END_FOR;
```
**铁律防守**：**在任何时候，遍历数组均采用 `LOWER_BOUND` 和 `UPPER_BOUND` 保护，彻底断绝因边界计算失误导致的死机。**

### 练习 1：配方系统数据一键比例缩放与安全上下限校验 (FOR 遍历)
某制药厂的配方系统存储在全局数据块中，是一个含有 50 个双精度浮点数的配方原料比例数组：
`arrRecipe : Array[1..50] of LReal;` (比例范围 0.0 ~ 100.0)
为了调整产量，工艺员需要一键对所有 50 种原料进行**放大或缩小系数调整**。
请编写一个 SCL FC：
*   **输入**：
    *   `rScaleFactor : Real`：缩放系数（如输入 1.2 代表放大 1.2 倍；输入 0.8 代表缩小为 80%）。
*   **输入输出 (In_Out)**：
    *   `arrRecipe`：配方数组。
*   **要求**：
    *   使用 `FOR` 循环对 50 个原料值进行缩放计算。
