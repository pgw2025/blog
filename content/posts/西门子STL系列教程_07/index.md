在梯形图（LAD）中，当我们执行一个简单的数学运算（如 `ADD`、`SUB`）时，数据是以直观的“变量到变量”形式传递的。但在 CPU 底层的有源算术逻辑单元（ALU）中，**所有的运算都必须依赖于累加器（Accumulator）这一核心硬件寄存器。**

西门子 S7 CPU 内部配置了超高速的 32 位累加器：**ACCU 1** 和 **ACCU 2**（在高端 S7-400 及 S7-1500 中，这一物理阵列被扩展到了 4 个，即 **ACCU 1~4**）。所有的装载、传送、数字运算、逻辑字运算以及比较操作，都是通过对这些寄存器执行**压栈（Push）**、**出栈（Pop）**、**交换（TAK）**等操作来完成的。

对于只具备 LAD 基础的工程师来说，累加器的滑移和移位机制往往是最难攻克的瓶颈。如果不能在脑海中建立起清晰的“累加器数据管道”模型，编写出的 STL 代码就会经常出现数值覆盖、高低位截断、甚至运算逻辑完全颠倒等严重故障。

本教程将带你深入到 CPU 运算器的微观物理结构中，彻底剖析 ACCU 1 和 ACCU 2 的移位规律、数据滑移机制、以及在多累加器架构（2-ACCU 与 4-ACCU）下的运行机制差异。

---

### 技术兼容性及声明

*   **S7-300**：内部仅配置了 **2 个 32 位累加器（ACCU 1 和 ACCU 2）**。当执行数学运算时，不支持 4 级累加器压栈。
*   **S7-400 / S7-1500**：内部配置了 **4 个 32 位累加器（ACCU 1、ACCU 2、ACCU 3、ACCU 4）**。这 4 个寄存器形成了一个底层的物理硬件堆栈，支持多达 4 级的数据自动滑动与深层压栈保护。
*   **S7-1200**：**由于硬件设计限制，S7-1200 不支持 STL，内部亦无物理累加器架构**。但理解本文中关于算术运算溢出（OV/OS）、比较标志位产生机制以及多级寄存器计算管道的思想，对于在 S7-1200 中编写零 Bug、高效率的有源 SCL 算法同样具有极高的工程价值。

## 1. 累加器物理架构：CPU 运算的核心阵地

累加器是 CPU 内部紧贴有源算术逻辑单元（ALU）的超高速 SRAM 存储单元，其读写周期与 CPU 内核时钟完全同步（通常在纳秒级别）。

### S7 CPU 2-ACCU 与 4-ACCU 物理架构对照图

```
【S7-300 CPU (2-ACCU 架构)】
+--------------------------+  +--------------------------+
| ACCU 2 (32-bit)          |  | ACCU 1 (32-bit)          |
| [0000000000000000000000] |  | [0000000000000000000000] |
+-------------┬------------+  +-------------┬------------+
              │                             │
              └──────────────┬──────────────┘
                             ▼
              +─────────────────────────────+
              | ALU (有源算术逻辑运算单元)  |
              +─────────────────────────────+

【S7-400 / S7-1500 CPU (4-ACCU 堆栈架构)】
+------------------+  +------------------+  +------------------+  +------------------+
| ACCU 4 (32-bit)  |  | ACCU 3 (32-bit)  |  | ACCU 2 (32-bit)  |  | ACCU 1 (32-bit)  |
| [00000000000000] |  | [00000000000000] |  | [00000000000000] |  | [00000000000000] |
+────────┬─────────+  +────────┬─────────+  +────────┬─────────+  +────────┬─────────+
         │                     │                     │                     │
         └────────── (Data shift pipeline during L/T) ◄────────────────────┘
```

#### ACCU 1 (累加器 1)：
这是**主累加器**。所有的装载指令（`L`）其最终目标都是写入 ACCU 1；所有的传送指令（`T`）其物理数据源都来自于 ACCU 1。在算术运算中，计算结果（如加法和、乘积）会无一例外地写回到 ACCU 1 中。

#### ACCU 2 (累加器 2)：
这是**辅助/副累加器**。在双操作数运算（如 `ACCU 2 - ACCU 1`）中，它用于存放第一个操作数。当有新数据装载到 ACCU 1 时，ACCU 1 原本的数据会被自动“挤入”到 ACCU 2 中。

### 1.1 知识点 1 代码：跟踪最基本的数据滑移（Slide）现象

下面我们通过一段基础的数据装载与传送代码，观察累加器 1 与 2 内部数据物理转移的轨迹。

#### 完整 STL 代码

```stl
FUNCTION "Trace_Base_Load" : VOID
TITLE = 跟踪基础装载与滑移机制
VAR_INPUT
   Val_A : INT ;              // 输入值 A (例如 100)
   Val_B : INT ;              // 输入值 B (例如 500)
END_INPUT
VAR_OUTPUT
   Out_Res : INT ;            // 输出
END_VAR
BEGIN
NETWORK
TITLE = 1. 数据依次装载至累加器
      L     #Val_A;           // 1. 装载 Val_A 入 ACCU 1 (触发滑移)
      L     #Val_B;           // 2. 装载 Val_B 入 ACCU 1 (再次触发滑移，Val_A 被挤入 ACCU 2)
      +I    ;                 // 3. 执行整数加法 (ACCU 2 + ACCU 1 -> ACCU 1)
      T     #Out_Res;         // 4. 将结果从 ACCU 1 传出
END_FUNCTION
```

#### 逐行寄存器状态变化与滑移过程跟踪表

假设输入：`Val_A = 100`，`Val_B = 500`。进入块前，假定累加器残存数据为 `999`。

| 步骤 | 指令 | ACCU 1 (32-bit decimal) | ACCU 2 (32-bit decimal) | 状态字 RLO | 物理累加器内幕动作详解 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **0** | (开始前) | `999` | `888` | `0` | 进入本块，寄存器内残存着上个程序块留下的垃圾数据。 |
| **1** | `L #Val_A` | `100` | `999` | `0` | **装载滑移第一步**：CPU 的寄存器滑移电路被触发。首先，ACCU 1 中原本的数据 `999` 被**复制并推入** ACCU 2 中（覆盖了 `888`）。接着，将新数据 `100` 写入 ACCU 1 中。 |
| **2** | `L #Val_B` | `500` | `100` | `0` | **装载滑移第二步**：滑移再次动作。ACCU 1 中的 `100` 被**复制并挤入** ACCU 2 中（覆盖了 `999`）。接着，新数据 `500` 被写入 ACCU 1。此时，两个运算操作数成功“各就各位”。 |
| **3** | `+I` | `600` | `100` | `0` | **有源运算阶段**：ALU 调取 ACCU 2（100）与 ACCU 1（500）执行物理相加。计算结果 `600` 直接写回 ACCU 1。**注意：在 S7-300 (2-ACCU) 中，ACCU 2 中的数据 `100` 依然保持原样，并不会被清零或弹回。** |
| **4** | `T #Out_Res`| `600` | `100` | `0` | **数据传出**：将 ACCU 1 中的 `600` 拷贝至变量物理地址。此时累加器 1 和 2 内部的值依然保留，直到下一次 `L` 指令执行时被覆盖。 |

### 1.2 工程经验：防止因“累加器残存”导致的多重运算污染

从上面的跟踪表（步骤 4）中，我们得出了一个非常关键的工程结论：**传送指令 `T` 之后，累加器内部的数据并不会自动消失或清空。**

*   **隐患场景**：
    有些工程师喜欢编写不规范的 STL 代码，在执行完一次 `T` 之后，直接进行数学逻辑操作：
    ```stl
    L     "Temp_Value"
    T     "Target_1"
    // 很多工程师误以为此时累加器1已经空了。
    // 如果紧跟着写一行只依赖累加器的指令：
    +I                // 此时，ACCU 1 里的 "Temp_Value" 仍在，ACCU 2 里的旧历史值也在！
                      // 这会导致 CPU 执行了一次完全非预期的“历史值 + 当前值”的计算。
    ```
*   **规避法则**：
    在 STL 中，**每一次进行数学运算（如 `+I`、`*R`、`==D`）之前，其前面的代码必须具有明确的、连续的装载（`L`）逻辑。** 绝不依赖上一段已经结束的逻辑残留在累加器中的数据进行计算。

## 2. 压栈机制：PUSH 运算与累加器滑移

当我们在 CPU 中连续执行装载指令（`L`）时，累加器内部会像子弹夹一样进行压栈移动。特别是在 4 级累加器架构（S7-400 / S7-1500）下，这一动作呈现出多级管道滑移的特点。

### 2.1 4 级累加器（ACCU 1~4）的装载压栈（PUSH）流动

当我们在 S7-1500 或 S7-400 中执行一次 `L`（装载）指令时，内部寄存器会发生**全员向下复制移位**，这一硬件动作在西门子底层微码中等效于一次 **`PUSH`**：

```
+---------------------------------------------------------------------------------+
| Accumulator PUSH (Load Operation in 4-ACCU CPU)                                 |
|                                                                                 |
| 1. Prior to Load:                                                               |
|   ACCU 4 [D] ◄─── ACCU 3 [C] ◄─── ACCU 2 [B] ◄─── ACCU 1 [A]                    |
|                                                                                 |
| 2. Executing "L New_Val":                                                       |
|   ACCU 4 [C] ◄─── ACCU 3 [B] ◄─── ACCU 2 [A] ◄─── ACCU 1 [New_Val]              |
|                                                                                 |
|  * [A] slides to ACCU 2, [B] slides to ACCU 3, [C] slides to ACCU 4.             |
|  * Original data [D] in ACCU 4 is pushed out and discarded (Overwritten).       |
+---------------------------------------------------------------------------------+
```

### 2.2 知识点 2 代码：无局部变量的高效多级算术混算

多级累加器堆栈的最大工程优势在于：**在处理复杂的数学公式时，无需定义任何临时变量（Temp）来存放中间计算结果。** 我们完全可以使用 4 级寄存器堆栈，将中间结果压入深层寄存器中。

下面我们来实现这个公式：
$$Result = (Val\_A + Val\_B) \times (Val\_C - Val\_D)$$

#### 完整 STL 代码（基于 S7-1500/400 4级累加器堆栈）

```stl
FUNCTION "Calc_Formula_Fast" : VOID
TITLE = 利用4级累加器栈执行复杂混合计算
VAR_INPUT
   Val_A : DINT ;             // 输入 A (32位有符号DINT)
   Val_B : DINT ;             // 输入 B
   Val_C : DINT ;             // 输入 C
   Val_D : DINT ;             // 输入 D
END_INPUT
VAR_OUTPUT
   Result : DINT ;            // 计算结果
END_VAR
BEGIN
NETWORK
TITLE = 4级累加器无中间变量高速计算
      // 1. 计算前半部分 (Val_A + Val_B)
      L     #Val_A;           // ACCU 1 = Val_A
      L     #Val_B;           // ACCU 1 = Val_B, ACCU 2 = Val_A
      +D    ;                 // ACCU 1 = (Val_A + Val_B)
      
      // 2. 依次加载 Val_C 和 Val_D。此时，前一步的计算和（A+B）将被自动推入深层寄存器中保护
      L     #Val_C;           // PUSH! ACCU 1 = Val_C, ACCU 2 = (Val_A + Val_B)
      L     #Val_D;           // PUSH! ACCU 1 = Val_D, ACCU 2 = Val_C, ACCU 3 = (Val_A + Val_B)
      
      // 3. 计算后半部分 (Val_C - Val_D)
      // 注意：减法在底层执行的是 ACCU 2 - ACCU 1 -> ACCU 1 (即 Val_C - Val_D)
      -D    ;                 // 计算完毕后，ACCU 1 = (Val_C - Val_D)
                              // 此时触发 POP！ACCU 2 自动从 ACCU 3 弹回，
                              // 使得此时的 ACCU 2 = (Val_A + Val_B) 的结果！
      
      // 4. 执行最终乘法
      *D    ;                 // 计算 ACCU 2 * ACCU 1 -> ACCU 1
                              // 即 (Val_A + Val_B) * (Val_C - Val_D)
      T     #Result;          // 传出结果
END_FUNCTION
```

#### 逐行物理寄存器跟踪表（4-ACCU 专有视角）

假设输入：`Val_A = 10`，`Val_B = 20`，`Val_C = 150`，`Val_D = 100`。
数学结果应当为：$(10 + 20) \times (150 - 100) = 30 \times 50 = 1500$。

| 步骤 | 指令 | ACCU 1 | ACCU 2 | ACCU 3 | ACCU 4 | 寄存器级压栈与弹栈微观机制剖析 |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| **0** | (开始) | `0` | `0` | `0` | `0` | 初始状态。 |
| **1** | `L #Val_A` | `10` | `0` | `0` | `0` | `10` 进入 ACCU 1。 |
| **2** | `L #Val_B` | `20` | `10` | `0` | `0` | 触发 PUSH。`10` 被推入 ACCU 2，`20` 写入 ACCU 1。 |
| **3** | `+D` | `30` | `0` | `0` | `0` | 执行加法：`10+20=30`，结果写回 ACCU 1。触发 4-ACCU 的 POP，ACCU2 获得原本在 ACCU3 的 `0`。 |
| **4** | `L #Val_C` | `150` | `30` | `0` | `0` | **关键 PUSH 1**：前半部分和 `30` 被滑入并保护在 ACCU 2 中，`150` 写入 ACCU 1。 |
| **5** | `L #Val_D` | `100` | `150` | `30` | `0` | **关键 PUSH 2**：极具艺术感。`30` 被进一步向下推，安全隐藏到 **ACCU 3** 中！`150` 滑移入 ACCU 2，新值 `100` 写入 ACCU 1。 |
| **6** | `-D` | `50` | `30` | `0` | `0` | **关键 POP**：减法计算 `150 - 100 = 50` 写入 ACCU 1。**由于算术操作消耗了两个寄存器，系统自动触发向上弹栈（POP）动作**：原本在 ACCU 3 里的 `30` 自动复位并弹回到了 ACCU 2 中！ |
| **7** | `*D` | `1500`| `0` | `0` | `0` | 乘法运算：ACCU 2（30）* ACCU 1（50）= `1500` 写入 ACCU 1。 |
| **8** | `T #Result`| `1500`| `0` | `0` | `0` | 结果输出。整个计算过程没有调用任何 Temp 内存，全部由物理寄存器高速打通。 |

### 2.3 工程经验：在 S7-300 (2-ACCU) 上运行上述代码的灾难性后果

**必须极其严肃地提醒大家**：上面的这段代码，如果下载到只有 **2 个累加器（ACCU 1 和 ACCU 2）** 的 S7-300 CPU 中，**计算结果将是完全错误的！**

*   **崩溃原因**：
    在 2 级累加器架构中，不具备 ACCU 3 和 ACCU 4。
    当程序执行到步骤 5 `L #Val_D` 时，由于没有 ACCU 3，前半部分算出来的和 `30` **会被直接挤出累加器边界，发生物理性丢失！**
    在随后的 `-D` 运算中，ACCU 2 此时是 `150`，ACCU 1 是 `100`，相减得到 `50` 写入 ACCU 1。接着进行 `*D` 时，由于原本的 `30` 早已丢失，此时 ACCU 2 里面残留的是 `-D` 运算前的旧操作数，最终乘法将产生一个完全紊乱的错误值。
*   **黄金准则**：
    在进行混合多级计算时，**如果代码需要兼容 S7-300，绝对不允许连续执行 3 次及以上的 `L` 指令。** 必须采用中间变量（Temp）来暂存分步计算结果，或是使用 `SCL` 语言，让博途编译器在编译时根据目标 CPU 的寄存器数量自动安排临时栈。

## 3. 出栈机制：POP 运算与数学消退

当 CPU 执行完算术运算指令后，累加器会自动发生向上弹回（Pop）复位。理解这一消退机制，能够帮助我们精准掌控累加器的生存期。

### 3.1 2-ACCU 与 4-ACCU 在数学运算后的不同弹回硬件设计

有源算术逻辑单元（ALU）在执行双操作数运算（如 `+D`、`/R`）时，需要同时消耗 ACCU 1 和 ACCU 2 中的数据。计算完成后，两个寄存器内部的值如何复位？

```
【S7-400 / S7-1500 (4-ACCU) 在 +D 后的自动向上 POP】
 1. 运算前:
    ACCU 4 [D] ───> ACCU 3 [C] ───> ACCU 2 [B] ───> ACCU 1 [A] (ALU 消耗 A 和 B)

 2. 运算后:
    ACCU 4 [D] ───> ACCU 3 [D] ───> ACCU 2 [C] ───> ACCU 1 [A + B] (堆栈自动向上弹回)
    * 结果写入 ACCU 1。
    * 原本在 ACCU 3 里的 C 弹回 ACCU 2。
    * 原本在 ACCU 4 里的 D 弹回 ACCU 3。
    * ACCU 4 保持不变。

【S7-300 (2-ACCU) 在 +D 后的状态】
 1. 运算前:
    ACCU 2 [B] ───> ACCU 1 [A] (ALU 消耗 A 和 B)

 2. 运算后:
    ACCU 2 [B] ───> ACCU 1 [A + B] (无堆栈可弹)
    * 结果写入 ACCU 1。
    * ACCU 2 保持原值 B 不变，没有任何弹回和复位。
```

### 3.2 知识点 3 代码：利用 2-ACCU 特有的“运算后残留”实现快速连续累加

虽然 2-ACCU（S7-300）在运算后不发生向下弹回，但其“ACCU 2 保持原操作数不变”的硬件设计，有时可以被聪明地用于进行“以同一个基数进行连续累加”的快捷操作。

#### 完整 STL 代码

```stl
FUNCTION "Accum_Same_Base" : VOID
TITLE = 利用 2-ACCU 残留特征进行同一基数的快速倍增
VAR_INPUT
   Base_Val : DINT ;          // 累加基数 (DINT, 例如 5)
END_INPUT
VAR_OUTPUT
   Res_2X   : DINT ;          // 2倍结果输出 (应为 10)
   Res_3X   : DINT ;          // 3倍结果输出 (应为 15)
END_VAR
BEGIN
NETWORK
TITLE = 连续累加
      L     #Base_Val;        // ACCU 1 = 5
      L     #Base_Val;        // ACCU 1 = 5, ACCU 2 = 5
      
      +D    ;                 // 计算 5 + 5 -> ACCU 1。此时 ACCU 1 = 10。
                              // 关键点：在 S7-300 内部，ACCU 2 依然保持原值 5 保持不变！
      T     #Res_2X;          // 传出 2 倍结果 10。ACCU 1 依然是 10。
      
      +D    ;                 // 再次执行 +D！
                              // 此时直接拿 ACCU 2 里的 5，与当前 ACCU 1 里的 10 相加 -> 15！
      T     #Res_3X;          // 传出 3 倍结果 15。
END_FUNCTION
```

#### 逐行剖析与 2-ACCU 专有跟踪表

假设：`Base_Val = 5`。

| 步骤 | 指令 | ACCU 1 | ACCU 2 | 状态字 RLO | 物理寄存器级微观细节解析 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **1** | `L #Base_Val` | `5` | `0` | `0` | `5` 写入 ACCU 1。 |
| **2** | `L #Base_Val` | `5` | `5` | `0` | 触发滑移。`5` 被推入 ACCU 2，新 `5` 写入 ACCU 1。 |
| **3** | `+D` | `10` | `5` | `0` | **核心物理特征**：ALU 将 ACCU 2（5）与 ACCU 1（5）相加，结果 `10` 写入 ACCU 1。**因为是 2-ACCU，ACCU 2 硬件锁死为 5 保持不动！** |
| **4** | `T #Res_2X` | `10` | `5` | `0` | 输出 10。 |
| **5** | `+D` | `15` | `5` | `0` | **直接累加**：无需再次装载！ALU 再次将 ACCU 2 中的 `5` 与当前 ACCU 1 中的 `10` 直接执行加法，得到 `15` 写入 ACCU 1。 |
| **6** | `T #Res_3X` | `15` | `5` | `0` | 输出 15。 |

### 3.3 工程经验：防范多平台移植时，因 2-ACCU 与 4-ACCU 弹回机制差异导致的计算偏差

3.2 节所展示的这种“奇技淫巧”，虽然在 S7-300（2-ACCU）上运行得非常完美，但如果你**直接将该块下载到 S7-400 或 S7-1500（4-ACCU）中运行，计算结果将彻底出错！**

*   **差异崩溃原因**：
    在 S7-1500（4-ACCU）中，当步骤 3 `+D` 执行完后，系统会自动触发向上弹栈（POP）动作。
    这会将原本在 ACCU 3（即 `0`）的内容弹回 ACCU 2，使得 **ACCU 2 的值在运算结束后自动变为了 `0`（而不是 5）**。
    当程序运行到步骤 5 再次执行 `+D` 时，实际执行的是 `0 + 10 = 10`，最终输出的 `Res_3X` 将是 `10`（而不是 15）。
*   **工程最佳移植实践**：
    为了保证代码具有 100% 的跨硬件平台鲁棒性，**严禁利用任何依赖特定硬件累加器数量的“残留、未初始化运算”特征。**
    在进行下一轮算术计算前，必须显式地重新执行 `L`（装载）操作：
    ```stl
    // 标准兼容、无硬件平台绑定的安全写法：
    L     #Base_Val
    L     #Base_Val
    +D    
    T     #Res_2X
    
    L     #Base_Val           // 显式重新装载，消除 2-ACCU 与 4-ACCU 的底层硬件差异影响
    +D    
    T     #Res_3X
    ```

## 4. 动态交换：TAK 指令的妙用

在非交换性（Non-Commutative）算术运算（如减法、除法）中，操作数的顺序有着决定性的物理意义。为了在单扫描周期内灵活调整两个操作数的物理位置，西门子硬件设计了 **`TAK`** 指令。

### 4.1 TAK (Toggle Accumulators) 的底层电路切换机制

`TAK` 指令的物理实现极其精悍。在 S7 CPU 内部，它是通过**两组 32 位寄存器传输门电平的直接交叉对调**实现的。

执行 `TAK` 不需要进行任何内存总线访问，可以在 **1 个时钟周期**内无等待地交换 ACCU 1 和 ACCU 2 的全部 32 位内容：

```
+---------------------------------------------------------------------------------+
| TAK Instruction - Single Cycle Hardware Toggle                                  |
|                                                                                 |
|  ACCU 2 [ 旧数据 B ]  ◄─────────────────────────┐                               |
|        │                                        │  (1-Cycle Cross Transfer)     |
|        └─────────────────────────► ACCU 1 [ 旧数据 A ]                           |
|                                                                                 |
|  * Result: ACCU 1 = [ 旧数据 B ], ACCU 2 = [ 旧数据 A ]                        |
+---------------------------------------------------------------------------------+
```

### 4.2 知识点 4 代码：非交换性工程物理公式计算（带绝对限幅的除法）

本例实现一个典型的工业非交换公式：
$$Result = \frac{Limit\_Value}{Max\_Value - Min\_Value}$$
如果公式中减法（Max - Min）先被算出并存放在 ACCU 1 中，我们需要将其作为除数，而将 `Limit_Value` 作为被除数。此时，必须使用 `TAK` 指令来交换它们在累加器中的物理位置。

#### 完整 STL 代码

```stl
FUNCTION "Calc_Limit_Ratio" : VOID
TITLE = 利用 TAK 实现非交换除法公式计算
VAR_INPUT
   Max_Val : REAL ;           // 上限值 (REAL)
   Min_Val : REAL ;           // 下限值 (REAL)
   Limit_Val : REAL ;         // 限制常数 (REAL)
END_INPUT
VAR_OUTPUT
   Ratio_Out : REAL ;         // 计算比例输出
END_VAR
BEGIN
NETWORK
TITLE = 1. 计算分母：Max - Min
      L     #Max_Val;         // ACCU 1 = Max_Val
      L     #Min_Val;         // ACCU 1 = Min_Val, ACCU 2 = Max_Val
      -R    ;                 // 计算 Max_Val - Min_Val -> ACCU 1 (当前存放的是分母结果)
      
      // 此时，分母（Max - Min）存放在 ACCU 1 中。
      // 根据除法指令 /R 的物理定义：它执行的是 ACCU 2 / ACCU 1 -> ACCU 1。
      // 因此，我们需要：ACCU 1 变为分母，ACCU 2 变为被除数 Limit_Val。

NETWORK
TITLE = 2. 巧妙利用 TAK 调整操作数，执行除法
      L     #Limit_Val;       // 1. 装载被除数。
                              // 此时：ACCU 1 = Limit_Val，
                              //       ACCU 2 = 刚才算出的分母（Max - Min）。
                              
      TAK   ;                 // 2. 交换累加器 1 和 2！
                              // 交换后：ACCU 1 = 分母（Max - Min） —— 正好成为除数！
                              //         ACCU 2 = Limit_Val —— 正好成为被除数！
                              
      /R    ;                 // 3. 执行实数除法：ACCU 2 / ACCU 1 -> ACCU 1
                              // 即 Limit_Val / (Max - Min)
      T     #Ratio_Out;       // 输出结果
END_FUNCTION
```

#### 逐行剖析与 TAK 交换跟踪表

假设输入：`Max_Val = 120.0`，`Min_Val = 20.0`，`Limit_Val = 50.0`。
数学结果应当为：$\frac{50.0}{120.0 - 20.0} = \frac{50.0}{100.0} = 0.5$。

| 步骤 | 指令 | ACCU 1 (float) | ACCU 2 (float) | 状态字 RLO | 寄存器交叉对调及除法微观解析 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **0** | (开始) | `0.0` | `0.0` | `0` | 初始化。 |
| **1** | `L #Max_Val` | `120.0` | `0.0` | `0` | 装载 `120.0`。 |
| **2** | `L #Min_Val` | `20.0` | `120.0` | `0` | `120.0` 移入 ACCU 2，`20.0` 写入 ACCU 1。 |
| **3** | `-R` | `100.0` | `120.0` | `0` | 计算 `120.0 - 20.0 = 100.0`（分母）存入 ACCU 1。 |
| **4** | `L #Limit_Val`| `50.0` | `100.0` | `0` | **滑移关键步**：被除数 `50.0` 写入 ACCU 1，分母 `100.0` 被滑动备份到了 ACCU 2。 |
| **5** | `TAK` | `100.0` | `50.0` | `0` | **TAK 瞬时交换**：两累加器的内容执行物理对调。现在，ACCU 1 变成了除数 `100.0`，ACCU 2 变成了被除数 `50.0`。 |
| **6** | `/R` | `0.5` | `50.0` | `0` | **除法运算**：ALU 执行 `ACCU 2 / ACCU 1`，即 `50.0 / 100.0`。结果 `0.5` 写入 ACCU 1。 |
| **7** | `T #Ratio_Out`| `0.5` | `50.0` | `0` | 传出结果 `0.5`。 |

以下流程图形象地展示了，当需要执行非交换性除法时，`TAK` 是如何瞬间调整寄存器位置，以满足除法器硬件要求的：

### 4.4 工程经验：利用除法运算后 ACCU 2 的“余数残留”获取精密模除（MOD）结果

在进行整数除法（`/I`、`/D`）时，西门子 CPU 的硬件除法器在执行完计算后，会在两个累加器中同时返回两部分数据。这一隐藏的物理特征对于编写高效的数据包解析、时间换算程序极为重要。

*   **硬件返回特征**：
    当执行 16 位整数除法 `/I` 时：
    *   **ACCU 1 的低 16 位**：存放除法计算的**商（Quotient）**。
    *   **ACCU 1 的高 16 位**：存放除法计算的**余数（Remainder）**。
*   **工程快捷取余算法**：
    当你需要执行 `Value MOD 10`（取模/求余数）时，无需再次调用复杂的 MOD 函数。只需在除法后直接对 ACCU 1 进行高字和低字对换（`CAW` 指令），即可直接提取余数：
    ```stl
    L     #Input_Value        // 假设为 13
    L     10                  // 除数
    /I                        // 执行除法。此时商为 1 (存放在 ACCU 1 低16位)
                              //             余数为 3 (存放在 ACCU 1 高16位)
    CAW                       // 交换 ACCU 1 的高低字节/高低字
                              // 交换后，高16位的余数 3 被移动到了低 16 位位置
    T     #Mod_Result         // 传出结果 3！
    ```

## 5. 状态标志位与累加器的深层交织

在算术逻辑单元（ALU）对累加器进行运算时，每次计算的物理状态（如结果是否为负、是否为零、是否溢出）都会实时改写状态字（Status Word）中的状态标志位。

### 5.1 算术结果对状态字核心标志位的驱动规则

在上一篇教程中，我们学习了状态字。现在我们来研究算术指令执行后，有源逻辑单元是如何驱动这些控制位的：

| 运算状态 | 状态位 CC 1 | 状态位 CC 0 | 状态位 OV | 状态位 OS | 物理本质解析 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **计算结果为 0** | `0` | `0` | `0` | 保持不变 | 结果等于 0。 |
| **计算结果为负数** | `0` | `1` | `0` | 保持不变 | 最高符号位（Bit 31）变为 `1`。 |
| **计算结果为正数** | `1` | `0` | `0` | 保持不变 | 结果大于 0。 |
| **算术正向溢出** | `1` | `0` | `1` | `1` | 计算结果超出了该数据类型的最大上限。 |
| **算术负向溢出** | `0` | `1` | `1` | `1` | 计算结果超出了该数据类型的最小下限。 |

### 5.2 知识点 5 代码：具有硬件状态自诊断的高可靠性实数加法器

本例设计一个高可靠性的模拟量加法计算 FC。该函数在每次执行加法时，都会在寄存器级对运算状态进行全方位的安全自检。一旦发生溢出（OV = 1）或出现无效数字（NaN），将自动切断输出并激活报警。

#### 完整 STL 代码

```stl
FUNCTION "Safe_Real_Add" : VOID
TITLE = 具备寄存器级自诊断的安全实数加法器
VAR_INPUT
   Addend_1 : REAL ;          // 加数 1
   Addend_2 : REAL ;          // 加数 2
END_INPUT
VAR_OUTPUT
   Sum_Out  : REAL ;          // 累加和输出
   Error_OS : BOOL ;          // 异常溢出报警锁存器
END_VAR
BEGIN
NETWORK
TITLE = 1. 执行实数相加并立即捕获溢出
      L     #Addend_1;
      L     #Addend_2;
      +R    ;                 // 实数相加 -> ACCU 1
      
      // 物理判定：如果加法计算导致结果越界，FPU 会将状态字中的 OV 和 OS 硬件置 1
      A     OV;               // 检查当前计算是否触发了溢出
      JC    ERR;              // 若 OV = 1，跳转至错误处理

      T     #Sum_Out;         // 正常状态，写回输出
      CLR   ;                 // 清零 RLO
      =     #Error_OS;        // 清除报警
      BEU   ;                 // 安全退出

ERR:  L     0.0;              // 异常状态，强制将输出置为安全默认值 0.0
      T     #Sum_Out;
      SET   ;                 // 强行置 RLO 为 1
      =     #Error_OS;        // 激活异常报警输出
END_FUNCTION
```

#### 逐行剖析与溢出捕获机制

假设输入：`Addend_1 = 3.402823e+38`（接近有源 REAL 单精度浮点数的物理最大上限），`Addend_2 = 1.000000e+38`（相加后结果必定产生严重的算术正向溢出）。

*   **`+R` 执行瞬间**：
    由于计算结果超出了 REAL 的表示极限 $3.402823 \times 10^{38}$。协处理器（FPU）硬件产生状态中断，将状态字中的 **OV（溢出）** 和 **OS（存储溢出）** 位同时置 `1`。
*   **`A OV`**：
    首次检查。由于 `OV = 1`，读取后状态字中的 **RLO 变为了 1**。
*   **`JC ERR`**：
    条件跳转指令。由于 `RLO = 1`，跳转条件成立，程序计数器（PC）强行跳转至 `ERR` 标号处。
*   **`ERR` 标号分支**：
    绕过了非法的异常计算结果，保护性地向外部物理通道写入安全值 `0.0`，并输出错误锁存信号 `"Error_OS" = 1`。

### 5.3 工程经验：在安全连锁系统（F-System）中善用 OS（存储溢出）位进行异常追溯

在设计高炉温度连锁、化工压力紧急切断（ESD）等安全系统时，我们经常需要捕捉一些瞬时发生的计算异常。这些异常可能只存在于某一个毫秒级的扫描周期中，随后就消失了。

*   **状态标志位的物理差异**：
    *   **OV (Overflow)**：它是**易逝的（Volatile）**。每次执行新的算术指令时，OV 都会根据当前计算的最新状态被无情地刷新和覆盖。
    *   **OS (Stored Overflow)**：它是**具有锁存特性的（Sticky）**。只要在当前程序块运行期间，**任何一行计算触发过 OV**，OS 就会被锁定为 `1`。随后的其他正常计算不会将 OS 复位。
*   **工程现场追踪实践**：
    在编写安全联锁 FB 的出口网络时，应当专门加入一行对 OS 的测试指令。这能让你准确抓住在长达数月的运行中，由于外界高频干扰导致的某一次瞬时计算溢出：
    ```stl
    // 在 FB 的最后一行安全网络中
    A     OS;                 // 捕获本块运行期间发生的任何一次瞬时溢出
    S     "Safety_Calc_Flag"; // 一旦发生过溢出，永久锁存异常标志，直到人工复位
    ```

在本教程中，我们完成了对西门子 S7 CPU 内部核心运算部件——累加器（ACCU 1/2）与滑移堆栈的深度考察。

要真正像一位拥有 20 年经验的高级工程师那样自如地掌控累加器，我们需要在心智中沉淀出以下三条最根本的控制原则：

1.  **明确硬件平台的累加器数量**：在编写涉及 3 级以上滑移的复杂公式时，时刻保持警惕：*目标硬件到底是 S7-300 的 2-ACCU 结构，还是 S7-1500 的 4-ACCU 堆栈？* 确保移植时计算顺序的完全一致。
2.  **利用 TAK 驾驭非对称公式**：熟练使用 `TAK` 指令，在单周期内完成除数与被除数、减数与被减数的寄存器物理对调，消除临时变量（Temp）的内存访问延迟。
3.  **用状态标志位构筑主动防御**：在每一次进行大温差计算、高频累加、或敏感的比例除法时，紧跟对 `OV` 或 `OS` 状态位的测试，在寄存器级为 PLC 构筑起铜墙铁壁般的主动安全自检屏障。
