在梯形图（LAD）编程中，比较指令表现为直观的“天平”图形块（如 `CMP ==`、`CMP >`）。我们习惯于将两个变量挂载在它的上下端，当条件满足时，“天平”闭合通电。

然而，在 CPU 底层有源算术逻辑单元（ALU）的物理运行中，**比较指令在本质上执行的是一次“不写入结果的减法”（Subtraction without Writeback）**。

当我们在 STL 中写下 `==I` 或 `>R` 时，CPU 内部的位处理器（BLP）与 ALU 协同工作，执行 `ACCU 2 - ACCU 1` 的减法运算。CPU 硬件强行拦截了这次减法写回到累加器的通道（不破坏累加器中原有的数值），而是将计算结果的物理特征（是正、是负、还是零）直接驱动到状态字中的 **CC1（条件码 1）** 和 **CC0（条件码 0）**。

更加硬核的是，在 STL 架构中，**比较指令被归类为“一等布尔逻辑指令”**。它不仅能产生比较结果，还能直接与状态字中的 **/FC（首次检查位）** 联动。这意味着，西门子 CPU 允许我们在不使用任何括号堆栈的前提下，将比较运算与前序布尔链执行无缝的机器码级直接合并。

本教程将带你深入到西门子 CPU 的微码控制层面，解密比较指令底层的物理运行机制、`NaN`（非数字）在实数比较中的失效陷阱，以及如何编写符合现代工业高安全性规范的比较诊断程序。

## 1. 比较指令的物理本质：不改变数值的减法

在 S7 处理器内部，无论是 16 位整数比较（如 `==I`），还是 32 位双整数比较（如 `>D`），底层的执行机理都是完全一致的：**计算 `ACCU 2 - ACCU 1` 并更新状态字。**

### 1.1 硬件设计内幕：/FC 联动与免括号逻辑合并

在第二章和第八章中，我们详细学习过 `/FC`（首次检查）寄存器。西门子 CPU 最具极客特色的硬件设计之一是：**比较指令本身就是一条布尔指令，它会受到旧 `/FC` 状态的支配，并在执行完后将 `/FC` 置 `1`。**

这导致了以下两种截然不同的底层物理行为：

*   **当 `/FC = 0` 时 (首次检查)**：
    CPU 执行比较（`ACCU 2 - ACCU 1`），根据产生的 CC1/CC0 状态，直接将比较结果（`0` 或 `1`）写入 **RLO 寄存器**。随后，自动将 `/FC` 硬件置 `1`。
*   **当 `/FC = 1` 时 (逻辑链合并)**：
    CPU 执行比较。计算出比较结果后，**BLP（位逻辑处理器）直接将当前的比较结果，与原有的 RLO 进行逻辑“与（AND）”合并**（在没有显式写 `O` 指令的前提下），并更新 RLO。
    **物理红利**：这一设计意味着，我们在进行“布尔条件 AND 比较条件”的复合计算时，**在底层不需要消耗任何 `A(` 和 `)` 嵌套堆栈空间，也不需要执行压栈（PUSH）微码，直接在单条指令内完成了合并！**

#### 比较指令执行后，CC1 与 CC0 的物理映射规则（整型）：

```
+---------------------------------------------------------------------------------+
| Condition Code (CC1, CC0) Generation during Integer Comparison                  |
|                                                                                 |
|                        ALU Calculation: ACCU 2 - ACCU 1                         |
|                                        │                                        |
|         ┌──────────────────────────────┼──────────────────────────────┐         |
|         ▼ (ACCU 2 = ACCU 1)            ▼ (ACCU 2 < ACCU 1)            ▼ (ACCU 2 > ACCU 1)
|       CC1 = 0, CC0 = 0               CC1 = 0, CC0 = 1               CC1 = 1, CC0 = 0|
+---------------------------------------------------------------------------------+
```

### 1.2 知识点 1 代码：免括号级联的加热器超温连锁安全保护

我们来编写一段经典的工业控制代码：当“加热器使能信号通电”且“当前温度超过安全上限”时，强行切断加热器物理输出。我们将展示如何在不使用任何括号堆栈的前提下，实现极速、解耦的 RLO 直接合并。

#### 完整 STL 源代码

```stl
FUNCTION "Heater_Safety_Interlock" : VOID
TITLE = 免括号级联的高速加热器安全保护
VAR_INPUT
   i_Heater_Enable    : BOOL ;    // 输入：加热器系统使能信号 (I0.0)
   i_Current_Temp     : INT ;     // 输入：当前实际检测温度 (MW20)
   i_Temp_Hi_Limit    : INT ;     // 输入：设定的超温安全上限 (例如 150)
END_INPUT
VAR_OUTPUT
   q_Heater_Contactor : BOOL ;    // 输出：加热器接触器控制线圈 (Q0.0)
END_VAR
BEGIN
NETWORK
TITLE = 串联比较计算
      A     #i_Heater_Enable;     // 1. 首检：读取使能状态，直接载入 RLO (当前 /FC=0 -> 1)
      
      // 2. 载入比较操作数
      L     #i_Current_Temp;      // 3. 装载当前温度至 ACCU 1。由于是非逻辑指令，/FC 保持为 1
      L     #i_Temp_Hi_Limit;     // 4. 装载上限至 ACCU 1，当前温度滑动暂存至 ACCU 2。/FC 保持为 1
      
      // 5. 执行比较：检测是否当前温度 <= 上限 (ACCU 2 <= ACCU 1)
      <=I   ;                     // 6. 重点：由于此时 /FC = 1，
                                  //    CPU 自动将本次比较结果（<=I）与前序的 RLO（使能状态）
                                  //    执行直接硬件级“与（AND）”合并，并更新 RLO！
                                  
      =     #q_Heater_Contactor;  // 7. 赋值并复位 /FC 为 0
END_FUNCTION
```

#### 逐行微观寄存器状态变化跟踪表

假设当前状态：`i_Heater_Enable = 1`。
`i_Current_Temp = 160`（已超温越界），`i_Temp_Hi_Limit = 150`。
最终计算：`1` (使能) AND `(160 <= 150)` (比较结果为0) = `1 AND 0 = 0`。`q_Heater_Contactor` 应当输出 `0`（安全切断）。

| 步骤 | 执行指令 | ACCU 1 | ACCU 2 | 当前 `/FC` | 下一步 `/FC` | 当前 `RLO` | 下一步 `RLO` | `CC1 / CC0` | 寄存器与位逻辑处理器微观细节解析 |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| **0** | (开始前) | `0` | `0` | `0` | `0` | `0` | `0` | `0/0` | 状态字干净。 |
| **1** | `A i_Heater_...` | `0` | `0` | `0` | `1` | `0` | `1` | `0/0` | 首检：读取使能 `1` 直接作为新 RLO 写入。`/FC` 置 `1`。 |
| **2** | `L i_Current_...`| `160` | `0` | `1` | `1` | `1` | `1` | `0/0` | 装载当前温 `160` 进入 ACCU 1。非逻辑指令，不改变 RLO 与 `/FC`。 |
| **3** | `L i_Temp_Hi_...`| `150` | `160` | `1` | `1` | `1` | `1` | `0/0` | 装载 `150`。`160` 被滑动移入 ACCU 2。 |
| **4** | `<=I` | `150` | `160` | `1` | `1` | `1` | **`0`** | **`1/0`** | **免括号硬件级合并**：
1. ALU 计算 `ACCU 2 - ACCU 1` 即 `160 - 150 = 10` (正数)。根据映射规则，CC1/CC0 硬件置 `1/0`。
2. `<=I` 算符检查 `CC1=1, CC0=0`（代表大于），得出比较结果为 `0`（FALSE）。
3. **由于 `/FC=1`，BLP 强行执行：新 RLO = 旧 RLO (1) AND 比较结果 (0) = 0**。更新 RLO 为 `0`。 |
| **5** | `= q_Heater_...` | `150` | `160` | `1` | `0` | `0` | `0` | `1/0` | 写入输出，`/FC` 强行清零。 |

### 1.3 工程经验：在 STL 混合逻辑中，绝不让比较指令作为网络的最后一行指令

如 1.2 节所示，比较指令会强制将状态字中的 `/FC` 置为 `1`。

*   **潜在停机灾难**：
    如果你在一个网络的末尾写了比较指令（如 `==I`），但**因为手误，没有写赋值指令（如 `=`）或者任何跳转，直接结束了该网络。**
    这会导致 `/FC = 1` 状态跨越网络边界，发生严重的 **RLO 泄露**。后一个网络的起始逻辑将与本网络的残留比较状态发生不可预知的串联，轻则造成设备误动作，重则因为指针错位引发 `OB121` 造成 PLC 崩溃停机。
*   **黄金开发规范**：
    任何包含比较指令的网络，其尾部**必须通过赋值指令（`=`、`S`、`R`）或者跳转指令（`JC`、`JCN`）进行 RLO 截断**，强行将 `/FC` 硬件复位为 `0`。

## 2. 实数比较的“黑洞”：`NaN`（非数字）失效陷阱

当我们的控制数据类型升级为 **REAL（浮点数）**，使用 `==R`、`>R` 等实数比较指令时，CPU 底层的浮点计算协处理器（FPU）会引入一个整型寻址中绝不存在的魔鬼——**`NaN`（非数字，Not a Number）**。

### 2.1 什么是 `NaN`？它是如何让所有的比较指令瞬间失效的？

在第三篇和第五篇中，我们学习过，浮点数是按照 IEEE 754 存储的。
如果在前序计算中，程序发生了严重的物理错误（如：$0.0 / 0.0$、或者是对负数执行了开平方 `SQRT`），FPU 无法用正常的实数来表示计算结果，就会在累加器 1 中生成一个特定的二进制编码：`16#7FFF_FFFF`。在 IEEE 754 标准中，这代表 **`NaN`（非数字）**。

#### `NaN` 在 FPU 硬件比较器上的灾难特征：
当一个累加器中的值为 `NaN` 时，FPU 执行实数比较，其状态字的 CC1/CC0 会被强制置为特定的 **`1/1`（无序状态，Unordered）**。
这一电平状态会导致以下颠覆常理的后果：
*   **`==R`** 判定结果为 **`0` (FALSE)**。
*   **`<>R`** 判定结果为 **`1` (TRUE)**。（NaN 确实不等于任何数，包括它自己！）
*   **`>R`**、**`<R`**、**`>=R`**、**`<=R`** 其判定结果**通通为 `0` (FALSE)！**
    这意味着，在 FPU 视角中：`NaN` 既不大于 `100.0`，也不小于 `100.0`，更不等于 `100.0`！
*   **失控后果**：如果你利用 `>R` 判定压力是否超限，一旦压力传感器断线导致计算出 `NaN`，`>R` 将永久报出 `0`（未超限）。**安全联锁被彻底废卷，设备将面临超压爆炸的毁灭性灾难！**

### 2.2 知识点 2 代码：具备 `NaN` 硬件安全防护的主动防御型实数比较器

本例设计一个用于冶金高炉冷风壁温度监控的安全比较器 FC。该函数能够在执行实数比较前，首先在硬件级捕获并剔除 `NaN` 脏数据，确保安全联锁的绝对生效。

#### 完整 STL 源代码

```stl
FUNCTION "Safe_Real_Compare" : VOID
TITLE = 具备 NaN 脏数据主动防御的安全实数比较器
VAR_INPUT
   i_Actual_Temp : REAL ;     // 输入：当前实际采集温度 (REAL)
   i_Alarm_Limit : REAL ;     // 输入：超温报警限值 (例如 850.0)
END_INPUT
VAR_OUTPUT
   q_Over_Temp_Alm : BOOL ;   // 输出：超温紧急切断信号 (Q0.1)
   q_Sensor_Fault  : BOOL ;   // 输出：传感器数据异常/NaN 报警
END_VAR
BEGIN
NETWORK
TITLE = 1. 硬件级 NaN 故障探测机制
      // 在 IEEE 754 中，若产生 NaN，状态字的 CC1/CC0 会被锁死为 1/1。
      // 西门子指令集专门设计了无操作数状态测试指令：UO (Unordered, 无序)
      // UO 指令会在检测到 CC1/CC0 为 1/1 时，将 RLO 置 1。
      
      L     #i_Actual_In_Temp; // 装载实际值
      L     #i_Actual_In_Temp; // 自己装载两次
      ==R   ;                  // 自己跟自己比！如果是 NaN，==R 结果为 0！
      NOT   ;                  // 取反：若不相等，说明就是 NaN 脏数据
      
      // 或者直接用更底层的 UO 指令进行瞬时拦截：
      L     #i_Actual_In_Temp;
      L     0.0;
      +R    ;                  // 触发一次加法以刷新 CC1/CC0 状态
      UO    ;                  // 检查是否产生了无序（NaN/无解）状态
      =     #q_Sensor_Fault;   // 报出传感器严重故障
      JC    ERR_HEAL;          // 一旦为 NaN，立刻强行拦截跳转至安全自愈区

NETWORK
TITLE = 2. 安全比较计算 (仅在数据合法时执行)
      L     #i_Actual_In_Temp; 
      L     #i_Alarm_Limit;    
      >R    ;                  // 比较：实际值 > 报警上限 (ACCU 2 > ACCU 1)
      =     #q_Over_Temp_Alm;  // 写入安全切断线圈
      BEU   ;                  // 正常无故障，退块

ERR_HEAL: NOP 0;               // 安全自愈分支
      SET   ;                  
      =     #q_Over_Temp_Alm;  // 一旦传感器发生 NaN 故障，秉承“失能安全”原则，
                               // 强行把超温切断信号置 1，强行关停高炉！
END_FUNCTION
```

#### 逐行微观寄存器状态变化跟踪表

假设当前传感器由于高频电磁干扰，变送器芯片输出异常，导致 PLC 读入数据为了 `NaN`（十六进制 `16#7FFF_FFFF`）。安全上限设为 `850.0`。

| 步骤 | 执行指令 | ACCU 1 | 当前 `/FC` | 下一步 `/FC` | 当前 `RLO` | 下一步 `RLO` | `CC1 / CC0` | 状态字及物理寄存器微观状态跃迁详解 |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| **0** | (开始) | `0` | `0` | `0` | `0` | `0` | `0/0` | 待命。 |
| **1** | `L i_Act_Temp` | `16#7FFF_FFFF`| `0` | `0` | `0` | `0` | `0/0` | 装载 `NaN` 入 ACCU 1。 |
| **2** | `L 0.0` | `16#0000_0000`| `0` | `0` | `0` | `0` | `0/0` | 装载 `0.0` 入 ACCU 1，`NaN` 滑入 ACCU 2。 |
| **3** | `+R` | `16#7FFF_FFFF`| `0` | `0` | `0` | `0` | **`1/1` (无序)**| **FPU 报错**：`NaN + 0.0` 结果依然为 `NaN`。浮点运算协处理器将状态字中的 CC1/CC0 强行锁死在 `1/1`（无序/Unordered 状态）。 |
| **4** | `UO` | - | `0` | `1` | `0` | **`1`** | `1/1` | **无序状态测试**：`UO` 指令属于首检。位处理器检测到此时 `CC1=1, CC0=1`。**判定发生了 NaN。直接将 RLO 置 1！** `/FC` 置 `1`。 |
| **5** | `= q_Sens_Fault`| - | `1` | `0` | `1` | `1` | `1/1` | 写入故障报警。`/FC` 被自动复位。 |
| **6** | `JC ERR_HEAL` | - | `0` | `0` | `1` | `1` | `1/1` | 首次检查跳转。由于 `RLO = 1`，条件成立，**CPU 强行更改 PC 指针，直接跳转到 `ERR_HEAL` 标号处。** |
| **7** | `SET` | - | `0` | `1` | `1` | `1` | `1/1` | 强行置 RLO 为 `1`。 |
| **8** | `= q_O_T_Alm` | - | `1` | `0` | `1` | `1` | `1/1` | **失能保护**：强行将报警切断输出置 `1`，关停设备，成功化解了一次因 NaN 导致的联锁失效灾难。 |

### 2.3 工程经验：为什么在比例积分（PI）调节器中绝不允许直接比较 `==R`？

在闭环控制（如流量调节阀开度控制）中，很多新手程序员喜欢写：
`IF Actual_Flow == Target_Flow THEN ...`（当反馈流速等于设定流速时，停止阀门动作）。

*   **物理失控原因**：
    由于浮点数（REAL）具有 7 位有效数字的物理精度限制。在连续的微积分计算中，反馈流速可能在 `25.00001` 和 `24.99999` 之间高频高精度跳变。
    在芯片级别，这两个浮点数所代表的二进制编码是不一样的。
    因此，**`==R` 成立的概率接近于 0！**
    这会导致阀门执行机构因为永远无法达到完美的“绝对相等”而产生一刻不停的往复振荡，最终将阀门定位器的机械齿轮彻底磨损烧毁。
*   **黄金准则**：
    在任何实数闭环控制中，**禁止使用 `==R`（绝对相等）比较**。必须采用“死区带（Deadband）”判定法：
    $$|Actual - Target| \le \epsilon$$
    在 S7 STL 中，正确的写法是：
    ```stl
    L     #Actual_Flow
    L     #Target_Flow
    -R    ;                   // 计算误差
    ABS   ;                   // 求绝对值（博途 S7-1500 浮点数 ABS 指令）
    L     0.5;                // 设定死区容差为 0.5 L/s
    <=R   ;                   // 判定是否进入死区范围
    =     "Valve_Hold_Cmd";   // 满足则阀门保持不动作
    ```

## 3. 多区间分级排序与范围校验：跳转指令（Jump）的高效分流

在物流分拣（如：根据包裹重量将其分类为“超轻、标准、超重、危险”）、或是多级报警标定中，我们经常需要处理连续的多区间数值判定。

如果在 STL 中一味使用布尔并联，会产生大量冗余的扫描。最优雅、最符合西门子底层架构的设计是：**使用比较指令更新条件码，直接配合跳转指令（`JC` / `JZ` / `JN`）进行物理代码的分流。**

### 3.1 知识点 3 代码：大容量物流行李秤四段重力高速分类器

本例设计一个机场行李称重高速分类算法。程序接收传感器读入的 DINT 重量数据（克，g），并将其高效分流归类。

```
                    【行李称重 DINT 原始值 (g)】
                                 │
                 ┌───────────────┼───────────────┐
                 ▼               ▼               ▼
            [<= 5000g]     [5001 ~ 20000g]   [> 20000g]
                 │               │               │
                 ▼               ▼               ▼
              【超轻】         【标准】         【超重】
```

#### 完整 STL 代码

```stl
FUNCTION "Weight_Classifier" : VOID
TITLE = 高速行李重量四段分流分类器
VAR_INPUT
   i_Weight_Grams : DINT ;    // 输入：行李重量 (DINT, 单位：克)
END_INPUT
VAR_OUTPUT
   q_Light_Lane   : BOOL ;    // 1号分流道：超轻件 (<5kg)
   q_Std_Lane     : BOOL ;    // 2号分流道：标准件 (5kg ~ 20kg)
   q_Heavy_Lane   : BOOL ;    // 3号分流道：超重件 (>20kg)
END_VAR
BEGIN
NETWORK
TITLE = 1. 第一级过滤：判定是否超轻 (< 5000g)
      L     #i_Weight_Grams;  
      L     L#5000;           // 装载双整数 5000
      <D    ;                 // 比较：当前重量是否 < 5000g
      JC    L_LIG;            // 若成立，跳转至 L_LIG 标号（直接入轻件分流道）

NETWORK
TITLE = 2. 第二级过滤：判定是否超重 (> 20000g)
      L     #i_Weight_Grams;  
      L     L#20000;          
      >D    ;                 // 比较：当前重量是否 > 20000g
      JC    L_HEV;            // 若成立，跳转至 L_HEV 标号

NETWORK
TITLE = 3. 正常件逻辑 (5000g ~ 20000g)
      // 如果没有发生上述两个跳转，说明重量正好处于中间的标准范围内。
      // 无需再做任何比较指令，直接无条件分流！
      SET   ;                 
      =     #q_Std_Lane;      // 写入标准道
      CLR   ;                 // 清空其他车道，防止数据交叉污染
      =     #q_Light_Lane;    
      =     #q_Heavy_Lane;    
      BEU   ;                 // 正常结束，直接退块！无需执行后续无关网络代码！

NETWORK
TITLE = 4. 超轻分流处理分支
L_LIG: SET  ;                 
      =     #q_Light_Lane;    
      CLR   ;                 
      =     #q_Std_Lane;      
      =     #q_Heavy_Lane;    
      BEU   ;                 // 结束退出

NETWORK
TITLE = 5. 超重分流处理分支
L_HEV: SET  ;                 
      =     #q_Heavy_Lane;    
      CLR   ;                 
      =     #q_Std_Lane;      
      =     #q_Light_Lane;    
END_FUNCTION
```

### 5.2 知识点 3 的 SCL 规范重构（多段行李重量分类器）

在 SCL 中，实现多段比较分流，**严禁写一堆独立的 `IF`**。
最优雅、执行效率最高、最符合高级语言编译规范的是使用 **`IF-ELSIF-ELSE` 阶梯分流逻辑**。编译器会自动将该结构翻译成类似于 STL 跳转表的极速 CPU 分支预测指令。

```scl
// S7-1200 / 1500 优化的 SCL 阶梯分流算法
IF #i_Weight_Grams < 5000 THEN
    #q_Light_Lane := TRUE;
    #q_Std_Lane   := FALSE;
    #q_Heavy_Lane := FALSE;

ELSIF #i_Weight_Grams > 20000 THEN
    #q_Light_Lane := FALSE;
    #q_Std_Lane   := FALSE;
    #q_Heavy_Lane := TRUE;

ELSE // 处于 5000 ~ 20000g 的标准件
    #q_Light_Lane := FALSE;
    #q_Std_Lane   := TRUE;
    #q_Heavy_Lane := FALSE;
END_IF;
```

## 6. 工业级规范与调试经验总结

在专栏的这一章中，我们为具有 LAD 基础的工程师总结出 2 条在实际调试和优化数值比较逻辑时的金牌经验：

1.  **善用 TIA 在线监控中的“状态值（STA）”与“逻辑结果（RLO）”对照法调试多级比较**：
    在调试复杂的比较链时（特别是包含 `<>`、`<=` 这种带反向、带临界值的比较）。
    *   **调试大招**：紧紧盯着在线 STL 监视框中的 **STA** 列和 **RLO** 列。
        *   **STA**：显示的是**当前行算符比较的绝对判定结果**（例如，当前行写的是 `<D`，若实际数据确实小于，则该行的 STA 列会高亮显示为 `1`）。
        *   **RLO**：显示的是**该比较结果与前序逻辑 AND/OR 级联后的合并结果**。
        只要发现某一行 STA 是 `1`，但 RLO 变成了 `0`，说明正是那一行前面的布尔联锁条件（如使能未通电）强行拉低了判定输出，一秒内即可精准锁定外部硬件联锁故障。
2.  **在涉及高频临界值比对的场合（如：压力、位移到达判定），坚持使用带“迟滞带（Hysteresis）”的区间比较**：
    在工业控制中，物理信号（如液位、高炉温度）是带高频噪声的物理波形。如果你的比较条件写成：
    `A( ; L "Actual_Pressure" ; L "Limit_Value" ; >=R ; )`（当压力大于等于限值时，启动排气阀）。
    *   **物理震荡隐患**：由于物理压力在临界值附近存在微小的物理抖动，这会导致比较器的输出在一秒内高频触发、复位几十次。这会瞬间**让物理排气阀发生毁灭性的“高频空转抖动”**，几个小时内就会让电磁阀线圈过热烧毁。
    *   **黄金标定规范：引入迟滞带**。
        *   开启条件：`Actual_Pressure >= Limit_Value`。
        *   关闭（复位）条件：`Actual_Pressure < (Limit_Value - Hysteresis_Band)`（其中迟滞带 `Hysteresis_Band` 根据物理特性通常设为量程的 1%~2%）。
        通过设计双比较器的空间差，能够彻底滤除临界值附近的物理噪声，保障工业执行机构的长寿命安全运行。
