在软件开发领域有一句名言：“代码首先是写给人看的，其次才是让机器执行的。”

在 PLC 编程中，这一原则同样适用，而在 **STL（指令表）** 这种汇编级语言中更是如此。由于 STL 缺乏图形化语言（如 LAD）那种天然的视觉网络边界，且高度依赖 CPU 内部隐性的物理状态（如 RLO 的连续性、ACCU 的滑移、AR 寄存器的指针漂移），它极易演变为令人望而生畏的“只写（Write-Only）代码”——即编写时逻辑清晰，两周后连开发者自己也无法轻易看懂。

不规范的 STL 代码不仅是设备维护工程师的噩梦，更是导致 PLC 发生偶发性逻辑泄露、物理指针越界甚至 CPU 崩溃停机的最大隐患。

本教程作为本专栏的收官之作，将由浅入深地带你构建一套**工业级、工程化的 STL 编程规范**。我们将从变量命名体系、寄存器状态追踪注释、网络（Network）的物理截断划分、以及消除“魔术常数”的高可维护性架构四个维度，探讨如何写出优雅、安全、极速且易于维护的 STL 经典程序。

## 1. 变量命名与前缀规范：给微观数据穿上制服

在 STL 中，所有的局部变量、接口变量在代码中都以纯文本形式（如 `#My_Var`）呈现。如果缺乏规范，你很难在数百行指令中一眼看出这个变量到底存放在哪里、生命周期有多长。

### 1.1 作用域前缀化命名体系

我们推荐一套在西门子全球大中型项目（如汽车总装、大型钢厂）中被公认的**作用域前缀命名法**。通过变量名首部的小写前缀，直接向阅读者明示其物理内存宿主和生命周期：

| 变量作用域 | 推荐前缀 | 物理存储器位置 | 命名示例 | 物理含义解读 |
| :--- | :--- | :--- | :--- | :--- |
| **Input (输入)** | `i_` 或 `IN_` | 堆栈/背景 DB | `#i_Start_Cmd` | 输入参数：启动指令 |
| **Output (输出)** | `q_` 或 `OUT_` | 堆栈/背景 DB | `#q_Running_State` | 输出参数：运行状态 |
| **In_Out (输入输出)** | `iq_` 或 `IN_OUT_`| 6字节/10字节物理指针 | `#iq_Process_Record`| 双向传参：过程记录结构体 |
| **Static (静态变量)** | `stat_` 或 `s_` | **背景数据块 (Instance DB)**| `#stat_Step_Register`| 保持性静态变量：步骤寄存器 |
| **Temp (临时变量)** | `t_` 或 `temp_` | **局域数据堆栈 (L-Stack)** | `#t_Interlock_OK` | 瞬时临时变量：安全联锁OK |
| **Global DB (全局DB)**| `gdb_` | 系统工作存储区 | `"gdb_Recipe_Data"` | 全局数据块：配方数据 |

#### 100% 符合工业规范的优雅代码示例：

```stl
FUNCTION "Safe_Unpack_Word" : VOID
TITLE = 变量命名规范化示范
VAR_INPUT
   i_DB_No : INT ;            // 输入参数：明确前缀 i_
   i_Byte_Offset : DINT ;     // 输入参数：明确前缀 i_
END_INPUT
VAR_OUTPUT
   q_Extracted_Val : WORD ;   // 输出参数：明确前缀 q_
END_VAR
VAR_TEMP
   t_AR1_Backup : DWORD ;     // 临时变量：明确前缀 t_，警示该数据断电/退块即失
   t_Area_Pointer : DWORD ;   // 临时变量：明确前缀 t_
END_VAR
BEGIN
NETWORK
TITLE = 寄存器备份与指针安全计算
      TAR1  #t_AR1_Backup;    // 备份 AR1，使用 t_ 前缀表明这是栈中备份

      // 计算指针
      L     #i_Byte_Offset;   // 使用 i_ 前缀，一眼看出这是外部传入的控制偏置
      SLD   3;                
      T     #t_Area_Pointer;  // 写入临时变量暂存
      LAR1  ;                 // 载入地址寄存器 AR1

      // 动态安全打开 DB
      OPN   DB [#i_DB_No];     // 使用 i_ 前缀，明确这是外部指定的 DB 块
      L     DBW [AR1, P#0.0]; // 间接装载
      T     #q_Extracted_Val; // 写入输出。使用 q_ 前缀，保证退块时值传递写入外部引脚

      LAR1  #t_AR1_Backup;    // 还原现场
END_FUNCTION
```

### 1.3 工程经验：大型多团队协同项目中的 PLC Tags（变量表）命名规范

在由多家集成商、多名工程师协同开发的联合整车装配线或大型化工中控项目中，全局 PLC 变量表（PLC Tags）的冲突是阻碍项目联调的头号大敌。

#### 黄金命名规范：“模块前缀 + 物理区划 + 功能描述”
在新建全局变量时，严禁使用类似 `"Start_Btn"`、`"Run_Signal"` 这种毫无区划、极易发生命名冲突的变量。建议采用**三段式结构**：

```
+---------------------------------------------------------------------------------+
| Standard Global Tag Naming Structure                                            |
|                                                                                 |
|       [Module Prefix]    _    [Area / Device]    _    [Functional Description]  |
|             │                        │                        │                 |
|             ▼                        ▼                        ▼                 |
|           "PUMP1"       _         "SYS02"        _         "Start_Btn"          |
|                                                                                 |
|  * Result: "PUMP1_SYS02_Start_Btn" (100% Unique, zero collision risk)           |
+---------------------------------------------------------------------------------+
```

通过这一命名规范，即使多名工程师同时往 CPU 中下装各自的控制块，也能实现变量冲突率降低，并在全局交叉引用（Cross-Reference）中做到一目了然。

## 2. 寄存器驱动逻辑的注释艺术：还原 CPU 思考轨迹

由于 STL 是直接面向寄存器运行的汇编语言，其注释规范有着极特殊的硬性要求。

*   **反面教材（无价值的注释）**：
    `L     MW10;         // 装载 MW10` —— 这种注释毫无价值。阅读者能通过指令本身看出是装载 `MW10`，他们需要知道的是**“为什么要装载 MW10，此时累加器里发生了什么变化”**。
*   **正面教材（寄存器示踪法注释）**：
    `L     "Analog_Raw";  // ACCU 1 = 原始模拟量，原 ACCU 1 移入 ACCU 2` —— 这种注释清晰展现了 CPU 运算器的内部状态，能让后续的维护人员瞬间跟上 CPU 的微观思考轨迹。

### 2.1 知识点 2 代码：高精度实数一阶滤波算法中的“寄存器示踪”注释示范

下面我们编写一段高频运行的一阶滞后低通滤波器（Low-Pass Filter）STL 代码，并演示完美的“寄存器示踪”注释排版。

#### 完整 STL 代码

```stl
FUNCTION "Trace_Comment_LPF" : VOID
TITLE = 寄存器示踪注释规范示范
VAR_INPUT
   i_Raw_In   : INT ;         // 原始模拟量通道输入 (0~27648)
   i_Alpha    : REAL ;        // 滤波系数 Alpha (0.0 ~ 1.0)
END_INPUT
VAR_IN_OUT
   iq_Last_Out : REAL ;       // 历史滤波输出 (IN_OUT)
END_VAR
VAR_TEMP
   t_AR1_Backup : DWORD ;     // 现场备份
END_VAR
BEGIN
NETWORK
TITLE = 滤波公式计算: Out = (Alpha * Raw_In) + ((1.0 - Alpha) * Last_Out)
      TAR1  #t_AR1_Backup;    // [Backup AR1] 备份原 AR1
      
      // 1. 转换原始值格式
      L     #i_Raw_In;        // ACCU 1 = Raw_In (INT)
      ITD   ;                 // ACCU 1 = Raw_In (DINT) -> 符号扩展
      DTR   ;                 // ACCU 1 = Raw_In (REAL) -> 转为浮点数
      
      // 2. 计算前半部分: Alpha * Raw_In
      L     #i_Alpha;         // ACCU 1 = Alpha, ACCU 2 = Raw_In (REAL)
      *R    ;                 // ACCU 1 = Alpha * Raw_In (前半部分结果)
      
      // 3. 计算后半部分的前置因子: (1.0 - Alpha)
      L     1.000000e+000;    // ACCU 1 = 1.0, ACCU 2 = (Alpha * Raw_In)
      L     #i_Alpha;         // ACCU 1 = Alpha, ACCU 2 = 1.0, (Alpha * Raw_In) 被滑动暂存
      -R    ;                 // ACCU 1 = (1.0 - Alpha)
      
      // 4. 读取历史值并相乘: (1.0 - Alpha) * Last_Out
      L     P##iq_Last_Out;   // ACCU 1 = iq_Last_Out 物理指针地址
      LAR1  ;                 // AR1 指向外部 iq_Last_Out 实参
      L     D [AR1, P#0.0];   // ACCU 1 = Last_Out (REAL), ACCU 2 = (1.0 - Alpha)
      *R    ;                 // ACCU 1 = (1.0 - Alpha) * Last_Out (后半部分结果)
      
      // 5. 将前半部分与后半部分相加
      TAK   ;                 // ACCU 1 = (Alpha * Raw_In), ACCU 2 = 后半部分结果 -> 交换
      +R    ;                 // ACCU 1 = 最终滤波结果
      T     D [AR1, P#0.0];   // 穿透写入外部 iq_Last_Out 实参保存
      
      LAR1  #t_AR1_Backup;    // [Restore AR1] 还原原 AR1
END_FUNCTION
```

### 2.2 寄存器级变化微观对照分析表

通过我们在代码中精美的示踪注释，我们可以将其与 CPU 底层的物理寄存器变化进行 100% 的完美比对：

| 步骤 | 指令 | ACCU 1 | ACCU 2 | AR 1 指向 | 注释给出的物理状态 | 编译器/ALU 硬件微码动作 |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| **1** | `L #i_Raw_In` | `13824` | `0` | `0.0` | `ACCU 1 = Raw_In` | 从 L 栈中读取整型 Raw_In 值写入 ACCU 1 的低 16 位。 |
| **2** | `ITD` | `13824` (DINT) | `0` | `0.0` | `ACCU 1 = Raw_In` | 符号位扩展。高 16 位填充 0。 |
| **3** | `DTR` | `13824.0` (REAL) | `0` | `0.0` | `ACCU 1 = Raw_In` | 格式转换，转为 32 位浮点数。 |
| **4** | `L #i_Alpha` | `0.2` | `13824.0` | `0.0` | `ACCU 1 = Alpha` | 装载 Alpha 常数，`13824.0` 被滑入 ACCU 2 中。 |
| **5** | `*R` | `2764.8` | `13824.0` | `0.0` | `Alpha * Raw_In` | 浮点数相乘，前半部分和结果 `2764.8` 写入 ACCU 1。 |
| **6** | `L 1.0` | `1.0` | `2764.8` | `0.0` | `ACCU 1 = 1.0` | 装载常数 `1.0`。前半部分结果 `2764.8` 被安全滑动备份到 ACCU 2。 |
| **7** | `L #i_Alpha` | `0.2` | `1.0` | `0.0` | `ACCU 1 = Alpha` | 装载 Alpha。`1.0` 滑入 ACCU 2。 |
| **8** | `-R` | `0.8` | `1.0` | `0.0` | `1.0 - Alpha` | 浮点数相减，得到因子 `0.8` 写入 ACCU 1。 |
| **9** | `L P##iq_L_O`| `16#8400_00B0` | `0.8` | `0.0` | `Last_Out 物理指针`| 装载复杂参数的 6 字节指针地址，因子 `0.8` 滑入 ACCU 2。 |
| **10**| `LAR1` | `16#8400_00B0` | `0.8` | `P#DBX22.0` | `AR1 指向实参` | 将指针载入 AR1。此时 AR1 指向外部 `DB12.DBD22`。 |
| **11**| `L D [AR1..]` | `8000.0` | `0.8` | `P#DBX22.0` | `ACCU 1 = Last_Out`| 间接装载历史值 `8000.0`。因子 `0.8` 滑入 ACCU 2。 |
| **12**| `*R` | `6400.0` | `0.8` | `P#DBX22.0` | `后半部分结果` | 执行乘法：`8000.0 * 0.8 = 6400.0` 写入 ACCU 1。 |
| **13**| `TAK` | `2764.8` | `6400.0` | `P#DBX22.0` | `交换` | **交换累加器 1 和 2**。ACCU 1 变回前半部分结果 `2764.8`。 |
| **14**| `+R` | `9164.8` | `6400.0` | `P#DBX22.0` | `最终滤波结果` | 相加计算，得到最终滤波输出 `9164.8` 写入 ACCU 1。 |
| **15**| `T D [AR1..]` | `9164.8` | `6400.0` | `P#DBX22.0` | `穿透写入` | 穿透块边界，直接将 `9164.8` 写入外部 `DB12.DBD22` 中。 |

### 2.3 工程经验：禁止使用魔术跳转标号（如 M001、M002），统一采用自解释标号

在 STL 中，条件跳转需要定义跳转目标标号（Label）。很多初学者和自动生成工具喜欢使用无意义的标号名（如 `M001`、`M002`、`Label_1`）。

*   **痛点**：
    当一个块内部存在多处复杂的跳转分支时，满屏幕的 `M001`、`M002` 会让代码极其晦涩难读，在线调试时极易跟踪错行。
*   **规范化方案**：
    所有的跳转标号名，**必须采用具备完整自解释功能的英文大写+下划线蛇形命名法**。通过标号名直接明示跳转的目的地状态：
    ```stl
    // 推荐的自解释标号命名示范：
    A     "i_Emergency_Stop"
    JC    LABEL_FAULT_SHUTDOWN  // 一眼看出：条件成立跳转至故障停机保护
    
    A     "i_Auto_Mode"
    JC    LABEL_AUTO_SEQUENCE   // 一眼看出：跳转至自动步进控制
    
    // ... 
    LABEL_FAULT_SHUTDOWN: SET
                          R     "q_Motor_Contactor"
                          BEU   ;
                          
    LABEL_AUTO_SEQUENCE:  NOP   0
                          // 自动逻辑...
    ```

## 3. 网络划分与 RLO 隔离：给连续扫描装上防火墙

在西门子 PLC 编程中，**“网络（Network）”** 是程序块内部最基础的逻辑组织单元。在 LAD 中，网络具有硬性的图形线分界。但在 STL 中，所有的网络代码在物理上是连续紧跟排列的。

因此，在 STL 中，**如何做好网络之间的 RLO 状态完全物理隔离，是检验一个 STL 程序员是否合格的分水岭。**

### 3.1 RLO 隔离的物理机制

在第8篇中，我们学习过 `/FC`（首次检查位）的硬件运行回路。

*   **安全的状态（Network 隔离）**：
    在网络 1 结束时，必须执行一条赋值类指令（如 `=`、`S`、`R`），使 **/FC 自动清零（`0`）**。这样，当 CPU 扫描到下一个网络的起始指令时，能够将其完美判定为**新一轮首检**，从而彻底隔离网络 1 残余 RLO 的状态污染。
*   **极度危险的状态（网络逻辑穿透）**：
    如果网络 1 尾部只进行了一次比较（如 `<>I`）但没有任何输出赋值，且没有强制清空 `/FC`。那么，**网络 1 的 RLO 残留会直接穿透网络边界，与网络 2 的首行指令强行进行逻辑合并。**

---

### 3.2 知识点 3 代码：因网络划分不当引起的“RLO 逻辑穿透”故障复现与隔离重构

本例编写一段气缸到位超时的检测逻辑，复现网络间 RLO 泄漏导致报警误触发的故障，并给出规范的重构方案。

#### 3.2.1 产生逻辑泄露故障的恶劣 STL 代码

```stl
// ==================== 错误示范 ====================
NETWORK 1
TITLE = 1号气缸到位检测
      A     "Cylinder_1_Extend_Sen"; // (I1.0)
      A     "Cylinder_1_Auto_Mode";  // (I1.1)
      // 隐患点：此网络到此为止。没有任何终结类输出赋值指令，
      // 导致 /FC 寄存器此时依然锁死为 1，RLO 寄存器存留着当前气缸状态。

NETWORK 2
TITLE = 2号气缸超时报警逻辑
      A     "Cylinder_2_Extend_Sen"; // (I1.2)
      // 灾难：由于上个网络 /FC = 1，此处的 A 指令不进行首检！
      // 导致 2号气缸 必须在 1号气缸 导通的前提下才会进行逻辑评估！
      A     "Cylinder_2_Timer_Done"; // (I1.3)
      =     "Cylinder_2_Timeout_Alm";// (Q1.0) 输出错误触发
```

---

#### 3.2.2 100% 物理隔离、规范重构后的安全 STL 代码

为了防范上述越界穿透，我们在每个独立网络的尾部，必须保证逻辑链的物理截断。

```stl
// ==================== 规范重构 ====================
NETWORK 1
TITLE = 1号气缸到位检测 (单一网络解决单一问题)
      A     "Cylinder_1_Extend_Sen";
      A     "Cylinder_1_Auto_Mode";
      =     "Cylinder_1_Ready_Temp"; // 黄金规范 1：使用局部临时变量，强制终结逻辑链，将 /FC 归零！

NETWORK 2
TITLE = 2号气缸超时报警逻辑 (彻底安全隔离)
      A     "Cylinder_2_Extend_Sen"; // 此时 /FC 确定为 0。本指令成功执行干净的首检！
      A     "Cylinder_2_Timer_Done";
      =     "Cylinder_2_Timeout_Alm";
```

#### 两者在状态字寄存器级变化的物理对照

### 3.4 工程经验：保障 STL 与 LAD / FBD 混合编程下的可读性转换准则

在 STEP 7 Classic 中，为了方便后续维护工程师阅读，我们编写的 STL 最好能够**无缝切换回 LAD（梯形图）视图**。如果 STL 的网络划分不满足其底层的图形化对齐规则，整个块将永远卡在 STL 视图下，无法转换。

#### 满足 LAD/FBD 切换兼容性的黄金三准则：
1.  **一个 Network 内部，绝对禁止出现两个完全独立的并列赋值逻辑。**（即一个网络内只写一个核心控制线圈的计算）。
2.  **网络内部禁止出现无法折合为并联分支的跳转标号。**所有的跳转目标标号（Label），**必须定义在其后续网络（Network）的首行开始处。**
3.  **在网络的首行，绝不允许使用 `O` 指令作为首检。** 所有的或运算并联分支，必须在后面的指令中通过括号 `O(` 或编译器识别的并联格式来拼装。

## 4. 高可维护性架构：消除魔术常数与指针解耦

在大型、需要跨设备移植的自动化项目中，**“可维护性（Maintainability）”** 与 **“解耦性（Decoupling）”** 具有决定性的地位。

### 4.1 什么是魔术常数？为什么在 STL 中必须彻底消灭它们？

*   **魔术常数（Magic Number）**：指在代码中硬编码的、具体的、无符号描述的物理地址和数值，例如：`L DB10.DBW20`，`T MD100`。
*   **致命隐患**：
    假设你在 50 个网络中都写了 `OPN DB 10` 且硬编码了 `DBD 20` 寻址。
    由于项目升级，外部这个数据块号从 `DB10` 变为了 `DB25`，且偏移地址因为增加了新变量从 `20` 移动到了 `32`。
    由于没有交叉引用，你不得不人工在数千行代码中，去逐行寻找、修改这些硬编码数值。**只要有一处漏改，现场运行的一瞬间，PLC 就会因为写入错误的绝对地址而直接引发 CPU 崩溃停机或执行错误的物理动作！**

### 4.2 知识点 4 代码：寄存器级全备份与物理地址解耦的数据复制块

下面我们编写一个具备工业级安全性的数据搬运块。该块完美规避了所有硬编码地址，全部使用符号接口和动态计算进行偏置寻址，并且**在入口和出口处对 CPU 的全局物理寄存器进行了 100% 的备份与恢复。**

#### 完整 STL 代码

```stl
FUNCTION_BLOCK "Safe_Array_Move"
TITLE = 高安全性、零硬编码解耦数据搬运块
VERSION : 0.1
VAR_INPUT
   i_Src_DB       : DB_ANY ;  // 符号化输入：源背景数据块
   i_Src_Offset   : DINT ;    // 符号化输入：源字节偏置 (DINT)
   i_Dest_DB      : DB_ANY ;  // 符号化输入：目标数据块
   i_Dest_Offset  : DINT ;    // 符号化输入：目标字节偏置 (DINT)
   i_Word_Count   : INT ;     // 搬运字个数
END_INPUT
VAR
   stat_Move_Counter : DINT ; // 静态累积搬运字总数记录 (STAT)
END_VAR
VAR_TEMP
   t_AR1_Backup   : DWORD ;   // 局部栈备份区
   t_AR2_Backup   : DWORD ;
   t_DB_Backup    : WORD ;
   t_DI_Backup    : WORD ;
   t_Loop_Cnt     : INT ;
END_VAR
BEGIN
NETWORK
TITLE = 1. 寄存器物理现场深度备份（工程防护最高规格）
      // 在动用 AR1, AR2, DB, DI 等全局硬寄存器前，必须将其原始数据完整备份到局部 L 栈中
      TAR1  #t_AR1_Backup;    // 备份地址寄存器 AR1
      TAR2  #t_AR2_Backup;    // 备份地址寄存器 AR2
      L     DBNO;             
      T     #t_DB_Backup;     // 备份当前打开的共享 DB 号
      L     DINO;             
      T     #t_DI_Backup;     // 备份当前打开的背景 DI 号

NETWORK
TITLE = 2. 动态解耦指针计算与数据搬运
      // 打开数据源
      OPN   DB [#i_Src_DB];    // 动态载入源 DB (锁入 DB 寄存器)
      OPN   DI [#i_Dest_DB];   // 动态载入目标 DB (锁入 DI 寄存器)

      // 计算源地址指针并载入 AR1
      L     #i_Src_Offset;
      SLD   3;                // 左移 3 位转为位寻址格式
      LAR1  ;                 // AR1 指向数据源起始位置

      // 计算目标地址指针并载入 AR2
      L     #i_Dest_Offset;
      SLD   3;
      LAR2  ;                 // AR2 指向目标起始位置

      L     #i_Word_Count;
      T     #t_Loop_Cnt;      // 初始化循环计数

Loop: T     #t_Loop_Cnt;

      // 双通道免打开切换、寄存器对拷
      L     DBW [AR1, P#0.0]; // 从源 DB 中载入一个字
      T     DIW [AR2, P#0.0]; // 写入目标背景 DB 中

      // 双指针物理漂移
      +AR1  P#2.0;            // 源指针向后移动 2 字节
      +AR2  P#2.0;            // 目标指针向后移动 2 字节

      L     #t_Loop_Cnt;
      LOOP  Loop;             // 循环控制

      // 更新静态计数统计变量 (STAT)
      L     #stat_Move_Counter;
      L     #i_Word_Count;
      +D    ;
      T     #stat_Move_Counter;

NETWORK
TITLE = 3. 物理现场还原（恢复现场，保证调用者安全）
      // 在退出块前，必须按压栈相反顺序，将原本的寄存器环境 100% 还原回去！
      LAR1  #t_AR1_Backup;    // 还原原 AR1 
      LAR2  #t_AR2_Backup;    // 还原原 AR2
      
      OPN   DB [#t_DB_Backup]; // 重新打开原本在调用前就已经打开的共享 DB
      OPN   DI [#t_DI_Backup]; // 重新打开原本在调用前就已经打开的背景 DI
END_FUNCTION_BLOCK
```

### 4.3 现场还原与寄存器状态备份矩阵对比分析

在第9篇关于程序结构的教程中，我们强调过，AR2 和 DI 是父 FB 寻址局部背景变量（`STAT`）的“生命线”。

让我们来剖析，上述备份和还原代码（网络 1 与网络 3）是如何在物理层面上完美避免系统发生“DI 污染崩溃”的：

*   **没有网络 1 与 3 备份还原的毁灭过程**：
    1.  你调用了这个数据搬运块。
    2.  块内部执行了 `OPN DI [#i_Dest_DB]`。**原先父 FB 锁在 DI 寄存器里的背景数据块（如 DB20）瞬间被覆盖篡改为了新 DB 块。**
    3.  块内部执行了静态变量累加 `L #stat_Move_Counter`。由于新 FB 内部要访问 STAT 变量，CPU 硬件无脑调用 `DIX [AR2, P#offset]` 指令。
    4.  **灾难发生**：由于此时 DI 寄存器已经不再是指向原先的 DB20，而是指向了你传入的目标 `i_Dest_DB`，CPU **定位到了完全错误的物理内存区域中！** 造成数据严重错位，引发 PLC 闪红灯崩溃停机。
*   **有网络 1 与 3 保护的安全过程**：
    由于我们在入口处使用 `t_DI_Backup` 完整记下了旧 DI 块号。
    在计算结束后、访问 `#stat_Move_Counter` 前，**由于我们在最后一步无条件执行了 `OPN DI [#t_DI_Backup]` 还原现场**，DI 寄存器重新被拉回了正确的轨道。
    CPU 随后的 STAT 变量寻址（`stat_Move_Counter`）安全、完美地落在了正确的物理内存地址中，保障了系统整体架构的安全屏障。

得益于我们在前几章中严格执行了**“前缀化符号命名”、“网络隔离”与“零绝对地址”**规范，老项目中的 STL 算法在迁移到 SCL 时，能够做到极速重构。

### S7-1200 / S7-1500 高性能、100% 优化访问的 SCL 重构代码

```scl
FUNCTION_BLOCK "Optimized_Array_Move"
{ S7_Optimized_Access := 'true' } // 启用博途优化块访问，物理寻址完全交由编译器托管
VERSION : 0.1
VAR_INPUT
   i_Src_Array    : Array[*] of Word;  // 使用 Variant/动态数组接口，消除硬编码 DB 约束
   i_Dest_Array   : Array[*] of Word;
   i_Word_Count   : Int;               // 搬运个数
END_VAR
VAR_OUTPUT
   q_Status       : Int;               // 搬运成败状态
END_VAR
VAR
   stat_Move_Counter : DINT;           // 保持性静态计数器 (STAT)
END_VAR
VAR_TEMP
   t_Index        : Int;               // 符号化循环下标
   t_Min_Bound    : Dint;              // 动态边界
END_VAR
BEGIN
    // 在博途优化块 SCL 模式下：
    // 1. 我们不需要进行任何 TAR1/TAR2 等复杂的物理寄存器现场备份，
    //    博途编译器在后台会自动使用 CPU 的 General Purpose Registers (GPRs) 极速打通数据流动。
    // 2. 彻底消除了 SLD 3 等位寻址的物理开销。
    
    t_Min_Bound := LOWER_BOUND(ARR := #i_Src_Array, DIM := 1);
    
    // 3. 安全的、带越界防护的符号循环
    FOR #t_Index := 0 TO (#i_Word_Count - 1) DO
        #i_Dest_Array[#t_Index] := #i_Src_Array[#t_Min_Bound + #t_Index];
    END_FOR;
    
    // 4. 更新静态计数器
    #stat_Move_Counter := #stat_Move_Counter + #i_Word_Count;
    #q_Status := 16#0000;              // 报出大获成功状态
    
END_FUNCTION_BLOCK
```
