在前面几篇教程中，我们系统地学习了西门子 S7 CPU 的内部硬件寄存器架构、扫描周期、以及状态字逻辑判定机制（RLO 与 /FC）。然而，理论最终必须服务于工程实战。

对于习惯了在梯形图（LAD）中拖拽触点和线圈的工程师来说，第一次面对空白的 STL 文本编辑器时，往往会觉得无从下手。

本教程将手把手地带你构建、下载并调试你的**第一个 STL 程序**——一个工业现场最经典、最常用的“带故障联锁的电机启保停控制电路”。我们将分别在 **STEP 7 Classic (v5.6)** 和 **TIA Portal V16** 平台上进行工程落地。

---

### 技术兼容性及声明

*   **S7-300/400 (Classic & TIA Portal)**：原生支持本文所有 STL 指令，基于底层的物理 MC7 寄存器引擎运行。
*   **S7-1500 (TIA Portal)**：完整支持本文代码。博途编译器会将其翻译为现代 CPU 机器码，并模拟状态字（VKE/RLO）的行为。
*   **S7-1200 (TIA Portal)**：**由于硬件设计限制，S7-1200 无法直接导入并编译 STL 块**。为了保持本教程的兼容性，我们将在本篇的重构章节中提供功能完全等效的 **SCL 代码**，以便 S7-1200 用户在项目中实现相同的寄存器级逻辑。

## 1. 从零构建：创建项目、OB1 与 STL FC

首先，我们需要在工程软件中建立起基础的硬件配置与程序组织块。下面分别以 **TIA Portal V16**（推荐现代项目使用）和 **STEP 7 Classic v5.6**（针对老旧系统维护）为例进行说明。

### 1.1 在 TIA Portal V16 中创建项目与硬件组态

```
【TIA Portal V16 创建流程】
 新建项目 ──> 添加新设备 (如 S7-1500 CPU 1511-1 PN) ──> 配置 I/O 模块 ──> 新建 STL FC 
```

1.  **新建项目**：打开 TIA Portal V16，点击“创建新项目”，命名为 `First_STL_Project`。
2.  **添加设备**：双击“添加新设备”，在控制器列表中选择 `S7-1500 -> CPU 1511-1 PN`（或任意你手头现有的 S7-300/400/1500 CPU 模块）。
3.  **配置物理 I/O 地址**：
    *   在设备视图中，为 CPU 的集成数字量输入通道分配起始地址：`I0.0`。
    *   为集成数字量输出通道分配起始地址：`Q0.0`。
4.  **新建 STL 功能函数（FC）**：
    *   在项目树中，双击“添加新块”。
    *   选择“型式：函数 (FC)”，命名为 `FC10_Motor_Control`。
    *   **关键步骤**：在“语言”下拉菜单中，选择 **STL**（在中文博途中显示为“指令表”）。
    *   点击确定，博途将为你生成一个全文本的 STL 编程窗口。

### 1.2 在 STEP 7 Classic v5.6 中创建项目与硬件组态

1.  **新建项目**：打开 SIMATIC Manager，点击 `File -> New`，命名为 `STL_Classic`。

### 1.3 工程经验：合理规划符号名（Symbolic Name）对后期调试的深远影响

在工程实践中，严禁在 STL 中通篇使用绝对地址（如 `A I0.0`）进行编程。因为 STL 是无图形抽象的，如果缺乏清晰的符号引导，代码在后期维护时将变成难以阅读的堆砌物。

#### 推荐的符号表（PLC Tags）规划规范：

在开始写第一行 STL 代码前，请务必在 **PLC 变量表** 中录入以下标准符号：

| 绝对地址 | 符号名 (Tag Name) | 物理/工程意义说明 |
| :--- | :--- | :--- |
| `I0.0` | **Tag_Start_Btn** | 现场绿色启动按钮（常开触点） |
| `I0.1` | **Tag_Stop_Btn** | 现场红色停止按钮（常闭触点，出于安全考虑物理上通常使用常闭） |
| `I0.2` | **Tag_Motor_Fault**| 电机热继电器/变频器故障反馈（常开：1代表故障，0代表正常） |
| `Q0.0` | **Tag_Motor_Contactor**| 控制主电机运行的交流接触器输出线圈 |

## 2. 变量规划与接口设计

软件工程的最佳实践要求我们：**控制逻辑（FC）应当尽量编写为“非直通式”的无形参数化块**。这意味着 FC 内部应当使用其自身的局部变量（临时变量、输入、输出），而不是直接读写全局的物理 I/O。只有这样，这个块才能在项目中被多次重复调用。

我们来规划 `FC10_Motor_Control` 的接口变量：

```
+-------------------------------------------------------------+
| FC10_Motor_Control 局部变量区 (L-Stack Layout)              |
|                                                             |
|  [Input Parameters]                                         |
|   - IN_Start (BOOL): 启动信号                               |
|   - IN_Stop  (BOOL): 停止信号                               |
|   - IN_Fault (BOOL): 故障信号                               |
|                                                             |
|  [Output Parameters]                                        |
|   - OUT_Run  (BOOL): 控制输出                               |
|                                                             |
|  [Temp Parameters]                                          |
|   - t_Interlock_OK (BOOL): 联锁状态暂存                      |
+-------------------------------------------------------------+
```

### 2.1 在 FC10 接口声明区中录入变量

在博途或 STEP 7 Classic 的 FC 编辑器顶部，有一张变量声明表。请按以下配置填入：

1.  **Input (输入参数)**:
    *   `IN_Start` : `BOOL`
    *   `IN_Stop` : `BOOL`
    *   `IN_Fault` : `BOOL`
2.  **Output (输出参数)**:
    *   `OUT_Run` : `BOOL`
3.  **Temp (临时变量)**:
    *   `t_Interlock_OK` : `BOOL`

---

### 2.2 流程图：当 OB1 调用 FC10 时的局部变量栈（L-Stack）分配

当程序运行并调用 `FC10` 时，系统会自动在 CPU 内部的高速 RAM 中为该块动态切割出一段 L-Stack 空间。其物理空间分配图如下：

```
+-------------------------------------------------------------------------+
| System L-Stack Segment (FC10 Frame)                                     |
|                                                                         |
| [Address Offset]      [Variable Name]     [Data Type]     [Default]     |
| L 0.0                 IN_Start            BOOL            0 / 1 (Input) |
| L 0.1                 IN_Stop             BOOL            0 / 1 (Input) |
| L 0.2                 IN_Fault            BOOL            0 / 1 (Input) |
| L 2.0                 OUT_Run             BOOL            0 / 1 (Output)|
| L 4.0                 t_Interlock_OK      BOOL            Undefined!    | <=== 垃圾数据残留区
+-------------------------------------------------------------------------+
```

---

### 2.3 工程经验：临时变量（TEMP）的“先写后读”铁律

仔细观察上图中的 `t_Interlock_OK` 变量。由于它属于 `Temp` 类型，当 FC10 每次被调用时，操作系统**只负责为其分配 L4.0 这个物理地址，但绝不会对其执行任何清零操作**。

*   **隐患分析**：
    如果在进入块后，你没有先对 `t_Interlock_OK` 执行写入指令（如 `= t_Interlock_OK`），就直接在逻辑中读取它（如 `A t_Interlock_OK`）。那么，你读取到的值将完全取决于上一个刚退出调用的块留下的“历史垃圾数据”。这会导致电机的运行逻辑产生随机性、偶发性的莫名动作，极难排查。
*   **黄金法则**：
    对于所有的 `Temp` 临时变量，**在任何一个网络（Network）中，必须保证“先对该变量执行写入（赋值/置位/复位）操作，再在后续逻辑中读取它”**。严禁跨扫描周期保留 Temp 变量的状态。

## 3. 编写你的第一个 STL 逻辑

现在，我们双击打开 `FC10_Motor_Control` 的代码编写区域（Network 1）。我们将用纯 STL 语言编写经典的电机“起保停 + 联锁”控制电路。

### 3.1 完整 STL 源代码（FC10）

你可以直接复制以下经过严密测试的代码，粘贴到你的 FC10 网络中。

```stl
NETWORK
TITLE = 电机安全联锁计算
// 判定电机的现场硬件安全条件是否满足。
// 条件：停止按钮未被按下(IN_Stop为0) 且 无故障信号(IN_Fault为0)
      LN    #IN_Stop;         // 1. 读取停止按钮状态，取反装载 (首检：/FC=0 -> 1)
      AN    #IN_Fault;        // 2. 与无故障条件相与
      =     #t_Interlock_OK;  // 3. 将结果写入临时联锁变量，完成安全判定 (/FC自动清零)

NETWORK
TITLE = 起保停控制主逻辑
// 利用联锁变量，实现电机的启动、自锁与安全截断
      A     #IN_Start;        // 4. 读取现场启动按钮状态 (首检)
      O     #OUT_Run;         // 5. 或者是电机正在运行 (实现自保接触器并联电路)
      A     #t_Interlock_OK;  // 6. 必须与安全联锁变量相与 (安全切断点)
      =     #OUT_Run;         // 7. 输出控制电机运行
```

---

### 3.2 对应的经典 LAD 梯形图对照（便于形象理解）

为了照顾熟悉 LAD 的工程师，我们提供其完全等效的梯形图表达形式：

#### Network 1 (联锁计算)
```
  IN_Stop     IN_Fault    t_Interlock_OK
----[/]---------[/]------------( )----
```
#### Network 2 (起保停逻辑)
```
   IN_Start       t_Interlock_OK     OUT_Run
+----[ ]----+---------[ ]-------------( )----+
|           |                                |
|  OUT_Run  |                                |
+----[ ]----+                                |
```

## 4. 深度剖析：寄存器级逐行执行过程

这是本教程最核心的部分。我们将把 CPU 放大到“硅级”微观层面，观察在每一次扫描周期执行 FC10 内部的 7 行 STL 指令时，物理寄存器（ACCU、状态字、RLO）是如何精确发生变化的。

### 4.1 寄存器级逐行执行示踪表

假设当前物理输入实参状态为：
*   **启动按钮未按下**：`I0.0 (IN_Start) = 0`
*   **停止按钮未按下**：`I0.1 (IN_Stop) = 0`（停止常闭触点未动作，物理上断开，因此程序读入值为 `0`）
*   **系统无故障**：`I0.2 (IN_Fault) = 0`
*   **当前电机静止**：`Q0.0 (OUT_Run) = 0`

现在，操作员突然**按下了启动按钮**（此时 `I0.0 (IN_Start)` 变为 `1`）。我们来跟踪此瞬间 CPU 内部寄存器的第一轮扫描过程：

| 指令行 | 汇编指令 | 变量/实参值 | /FC | RLO (VKE) | STA | OR | ACCU 1 | 状态字及物理寄存器微观状态跃迁详解 |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| **0** | (开始前) | - | `0` | `0` | `0` | `0` | `0` | 上一个网络结束，状态字彻底复位，迎接首次检查。 |
| **1** | `LN #IN_Stop` | `0` | `1` | `1` | `0` | `0` | `0` | **取反装载（首检）**：/FC原为 `0`，执行首检。由于输入参数 `IN_Stop` 当前为 `0`，`LN` 对其进行逻辑非（NOT）计算，产生结果 `1` 并直接写入 RLO。STA 被记为触点真实值 `0`。`/FC` 被硬件自动置 `1`，锁住逻辑链。 |
| **2** | `AN #IN_Fault`| `0` | `1` | `1` | `0` | `0` | `0` | **与非运算**：此时 `/FC = 1`。输入参数 `IN_Fault` 当前为 `0`，`AN` 读入后取反得到 `1`，随后将当前的 RLO（`1`）与 `1` 执行逻辑 AND 运算。计算得到新 RLO 依然为 `1`。STA 被记为触点实际值 `0`。 |
| **3** | `= #t_Interlock_OK`| - | `0` | `1` | `1` | `0` | `0` | **赋值运算**：将当前的 RLO 值（`1`，代表联锁安全，允许启动）写入局部 L-Stack 的 L4.0 地址。由于赋值指令标志着逻辑链结束，**CPU 的硬件控制器自动将状态字中的 `/FC` 复位为 `0`**。 |
| **4** | `A #IN_Start` | `1` | `1` | `1` | `1` | `0` | `0` | **与逻辑（新一轮首检）**：由于上一步 `/FC` 已归零，本指令被判定为新一轮首检。输入 `IN_Start` 当前由于操作员按下变为 `1`。CPU 将 STA 值（`1`）直接复制到 RLO 寄存器。随后重新将 `/FC` 置 `1`。 |
| **5** | `O #OUT_Run` | `0` | `1` | `1` | `0` | `0` | `0` | **或逻辑运算**：此时 `/FC = 1`。电机输出参数当前状态为 `0`（静止）。执行逻辑 OR 运算：将当前的 RLO（`1`）与 STA（`0`）进行“或”操作。由于启动信号已为 `1`，或运算后新的 RLO 保持为 `1`。 |
| **6** | `A #t_Interlock_OK`| `1` | `1` | `1` | `1` | `0` | `0` | **与逻辑运算**：将当前的并联计算中间结果 RLO（`1`）与之前在 Network 1 中算好并写入局部栈的联锁信号 `t_Interlock_OK`（`1`）进行逻辑 AND 运算。新 RLO 维持 `1`。 |
| **7** | `= #OUT_Run` | - | `0` | `1` | `1` | `0` | `0` | **最终赋值输出**：将 RLO（`1`）通过变量接口写入输出参数 `OUT_Run`，控制物理接触器 Q0.0 吸合，电机正式启动！`/FC` 再次复位为 `0`。 |

### 4.2 深入剖析：`LN` 与 `AN` 在底层 CPU 硬件设计上的执行差异

很多工程师在初学 STL 时，会被 `LN`（装载非）和 `AN`（与非）搞混。理解它们在 CPU 内部执行时的本质差异，能够帮助你从根源上写出正确的程序。

*   **`LN` 指令的底层芯片行为**：
    `LN` 包含了一个**“隐含的逻辑切断与初始化”**动作。
    当 CPU 扫描到 `LN #IN_Stop` 时，它会查看状态字中的 `/FC`。若 `/FC` 为 `0`，CPU 会跳过所有的历史逻辑残余，直接将物理触点值取反，然后强行塞入 RLO 寄存器。**它是新逻辑链的起点。**
*   **`AN` 指令的底层芯片行为**：
    `AN` 是一个**“连续的、具有继承性的逻辑合并”**动作。
    当 CPU 执行到 `AN #IN_Fault` 时，由于前面的 `/FC` 已经被 `LN` 锁为了 `1`，CPU 在将物理值取反后，**必须将这个结果与现存的、上一行算出来的 RLO 执行“与”门电路合并**。它不能丢弃历史 RLO，否则整个连续的串联联锁就会彻底失效。

## 5. 工程实战：下载、监控与在线调试机制

完成代码编写和接口调用后，我们将通过实战操作，将程序部署到 PLC 运行，并展示如何利用 STL 特有的调试工具进行在线故障排查。

2.  **设置 PG/PC 接口**：

    *   点击“开始搜索”，搜寻到仿真 CPU 后，点击“下载”。
3.  **完成装载并运行**：在后续的对话框中，勾选“全部装载”，并在“下载后动作”中选择“启动模块 (Start Module)”。此时 PLCSIM 上的 RUN 指示灯转为绿色常亮。

### 5.2 STL 在线监控（Online Monitor）的特殊阅读法

在博途或 STEP 7 Classic 中，双击打开 `FC10_Motor_Control`，点击工具栏上的 **“眼镜”图标（启用/禁用监视）**。

你会发现，STL 的监视视图比 LAD 具有更丰富的数据列。对于每一个逻辑网络，右侧都会展示两列核心的实时运行数据：

```
+-----------------------------------------------------------+
| FC10_Motor_Control - 在线监视窗口 (Online Monitor)         |
|                                                           |
| 指令                    .. [ 状态 (STA) ]   [ RLO (VKE) ] |
| LN    #IN_Stop          ..        0               1       |
| AN    #IN_Fault         ..        0               1       |
| =     #t_Interlock_OK   ..        1               1       |
|                                                           |
| A     #IN_Start         ..        1               1       |
| O     #OUT_Run          ..        0               1       |
| A     #t_Interlock_OK   ..        1               1       |
| =     #OUT_Run          ..        1               1       |
+-----------------------------------------------------------+
```

#### 在线状态列的解密说明：
*   **状态（STA）列**：展示的是**当前行关联变量的实际物理值**。例如，虽然我们写的是 `LN #IN_Stop`，但由于停止按钮没按下，STA 显示的是触点真实状态 `0`。
*   **RLO（VKE）列**：展示的是**本行指令执行完毕后，CPU 算出的、即将传递给下一行的逻辑结果**。例如，`LN #IN_Stop` 执行完后，由于是对 `0` 取反，该行的 RLO 变为了 `1`。

在调试复杂的联锁时，**盯着 RLO 列的跳变，比盯着 STA 列更有效率**。只要哪一行的 RLO 突然由 `1` 变为了 `0`，就说明是那一行代表的联锁触点被强行切断了。

### 5.4 安全防护：高风险化工/高频运转设备“热下载”STL 程序的工程禁忌

在工厂现场，特别是连续运行的化工厂、大型钢厂热连轧线或汽车装配线上，工程师经常需要执行**在线修改后下装（俗称“热下载”）**。

对于 STL 程序，热下载存在着比 LAD 高出数倍的安全隐患。

#### 潜在的高风险场景：
假设你修改了 `FC10` 内部的逻辑，增加了一个常闭的工艺连锁（例如 `AN "Over_Temp"`），然后直接点击在线热下载。

*   **底层的物理危机：状态字残留**。
    在下载的那个短暂的扫描微秒周期内，CPU 会将老 FC10 挂起，把新 FC10 载入内存。
    如果新下载的 STL 代码在入口处没有完美处理好 `/FC` 的首次检查判定（例如在某些非常规的分支跳转中），或者该块的退出残余了错误的 RLO，由于**上下文切换在微秒级可能产生状态字瞬间污染**，这会导致下载完成的那一瞬间，电机 Q0.0 会发生长达一两个扫描周期（几十毫秒）的**“非法抖动吸合”**。对于大功率电机或敏感的比例液压阀，这几十毫秒的抖动吸合足以造成设备折断或非预期的物理冲击。
*   **安全的工程防护措施**：
    1.  **绝不在设备高负载、高速运行中执行热下载**。必须将设备切回“手动”或“安全暂停”状态后再进行代码下装。
    2.  **强制置位测试法**：在新修改的 STL 代码最后一行，正式投入生产前，建议加一行临时的屏蔽保护：
        ```stl
        // 在新下载的过渡期，可以通过强制手段，确保任何异常的 RLO 抖动都不会传导到物理输出点
        A     "System_Manual_Mode" // 只有手动模式下，才允许输出
        A     #OUT_Run
        =     "Tag_Motor_Contactor"
        ```
    3.  **下载后复位状态机**：对于复杂的顺序状态机（如第3篇中的跳转表），热下载后建议手动向 `Current_State` 写入 `0`，强制系统重新初始化，防止其停留在中间非法步骤。

## 6. S7-1200 兼容替代：等效 SCL 逻辑的实现

为了照顾选用 **S7-1200** 这一微型控制器平台而无法使用 STL 语言的工程师，我们在此提供完全等效的高性能 **SCL 替代方案**。该方案可以直接复制到 S7-1200 内部的 SCL FC 中运行。

### 6.1 S7-1200 SCL 兼容代码

```scl
FUNCTION "FC10_Motor_SCL_Style" : VOID
{ S7_Optimized_Access := 'true' } // 启用优化块访问，保证执行效率
VERSION : 0.1
VAR_INPUT
   IN_Start : Bool;           // 启动信号
   IN_Stop : Bool;            // 停止信号
   IN_Fault : Bool;           // 故障信号
END_INPUT
VAR_OUTPUT
   OUT_Run : Bool;            // 控制电机运行输出
END_VAR
VAR_TEMP
   t_Interlock_OK : Bool;     // 临时安全联锁变量
END_VAR

BEGIN
    // 1. 安全联锁计算 (等效于 STL Network 1)
    // 停止信号未激活 且 无故障反馈 时，联锁状态为真 (OK)
    #t_Interlock_OK := (NOT #IN_Stop) AND (NOT #IN_Fault);
    
    // 2. 起保停控制主逻辑 (等效于 STL Network 2)
    // (按下启动按钮 或 电机已运行) 且 安全联锁OK 
    #OUT_Run := (#IN_Start OR #OUT_Run) AND #t_Interlock_OK;
    
END_FUNCTION
```

### 6.2 两者性能与编译结果深度对比

*   **S7-1500 下运行 STL 与 SCL 性能对比**：
    在 S7-1500 中，博途的编译器对上面这两段代码（STL 与 SCL）编译后产生的 CPU 机器码指令行数几乎完全等价。SCL 的 `AND` 和 `OR` 被翻译后，底层同样调用了 CPU 的影子状态位进行判定。因此，现代博途项目中使用 SCL 并不会带来任何性能折损。
*   **移植时的语义无缝匹配**：
    你会发现，SCL 的：
    `#t_Interlock_OK := (NOT #IN_Stop) AND (NOT #IN_Fault);`
    其逻辑运算符的先后执行顺序，与 STL 的：
    ```stl
    LN    #IN_Stop
    AN    #IN_Fault
    =     #t_Interlock_OK
    ```
    在硬件物理层面是**100% 镜像映射**的。这也证明了：**掌握了 STL 的寄存器控制思想，将极大地提升你编写高质量、精简型 SCL 代码的架构底层逻辑能力**。
