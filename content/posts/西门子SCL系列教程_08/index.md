在前一章中，我们彻底降伏了计数型循环 `FOR`，并利用变长数组和副本缓存优化，手写了一个高性能的百通道温度分选系统。你应该已经感受到了在 SCL 中进行数据批量处理的高效。

但是，工业现场的工艺需求是千变万化的。有些时候，**我们无法在循环开始前预测到底要循环多少次**。例如：
*   “在配方数据库中检索，直到找到第一个配方名称为空的空闲插槽为止。”
*   “在闭环控制算法中进行数值迭代逼近计算（如牛顿迭代法求物理补偿系数），直到相邻两次计算结果的偏差小于 $0.001$ 或者是计算次数见顶。”

这种“条件驱动”的循环需求，正是 **`WHILE`** 和 **`REPEAT`** 语句的用武之地。

然而，这两门“武器”威力巨大，却也极其危险。如果说 `FOR` 循环是个带着安全护栏的卡丁车，那么 `WHILE` 和 `REPEAT` 就是在没有护栏的悬崖边飞驰的超跑。**一旦逻辑设计不严密，它们就会瞬间在你的 PLC 内部制造出“死循环”，导致 CPU 看门狗超时停机，整条生产线瞬间抱闸锁死。**

今天，师父不仅会带你掌握这两种高级循环控制语句、控制跳转指令（`EXIT`、`CONTINUE`），更会从 **PLC 底层微处理器运行机制和看门狗硬件原理** 的角度，剖析死循环的致命危害，并传授你一套大厂通用的**“双层硬计数防御设计模式”**。

## 1. 条件型循环的双子星：WHILE 与 REPEAT

与已知次数的 `FOR` 循环不同，`WHILE` 和 `REPEAT` 都是由 **布尔条件（Boolean Condition）** 决定生死存亡的循环控制结构。

### 1.1 WHILE 语句：先判定，后执行（Pre-Test Loop）

```scl
WHILE <循环条件表达式> DO
    <循环体代码>;
END_WHILE;
```

*   **运行逻辑**：
    每次循环开始前，CPU 先评估 `<循环条件表达式>`。
    *   如果为 `TRUE`：执行循环体代码。执行完毕后，回到头部**重新评估**条件。
    *   如果为 `FALSE`：连一次都不执行，直接跳过整个 `WHILE`，执行 `END_WHILE` 之后的下一行代码。

```
                   WHILE 循环流向：
                   
                      │
                      ▼
             ┌─────────────────┐
    ┌───────>│ 评估循环条件？  │
    │        └────────┬────────┘
    │                 │
    │                 ├───────── (FALSE) ─────────┐
    │                 │ (TRUE)                    │
    │                 ▼                           ▼
    │        ┌─────────────────┐         ┌─────────────────┐
    │        │   执行循环体    │         │ 跳出 WHILE 循环 │
    └────────│   更新条件变量  │         └─────────────────┘
             └─────────────────┘
```

---

### 1.2 REPEAT 语句：先执行，后判定（Post-Test Loop）

```scl
REPEAT
    <循环体代码>;
UNTIL <终止条件表达式>
END_REPEAT;
```

*   **运行逻辑**：
    CPU 无条件先进入循环体，执行一次 `<循环体代码>`，然后才评估 `UNTIL` 后面的 `<终止条件表达式>`。
    *   如果为 `FALSE`：说明尚未达到终止要求，**继续往回跳转**执行循环体。
    *   如果为 `TRUE`：说明达到了终止要求，**退出循环**。

```
                  REPEAT 循环流向：
                  
                      │
                      ▼
             ┌─────────────────┐
    ┌───────>│   执行循环体    │
    │        │   更新条件变量  │
    │        └────────┬────────┘
    │                 │
    │                 ▼
    │        ┌─────────────────┐
    │        │ 达到终止条件？  │
    │        └────────┬────────┘
    │                 │
    │                 ├───────── (TRUE) ──────────┐
    │                 │ (FALSE)                   │
    │                 ▼                           ▼
    │        ┌─────────────────┐         ┌─────────────────┐
    │        │    往回跳转     │         │ 跳出 REPEAT 循环│
    └────────┴─────────────────┘         └─────────────────┘
```

---

### 1.3 💡 核心对比：双子星的物理差异

徒弟，在现场选择这两种循环时，你脑子里必须有一幅清晰的决策图：

1.  **首轮执行保证**：
    *   `WHILE` 的首轮执行概率是 **$0 \sim N$ 次**。如果一上来条件就不满足，它一次都不会跑。
    *   `REPEAT` 的首轮执行概率是 **$1 \sim N$ 次**。不管终止条件当前如何，它**至少会被雷打不动地执行一次**。
2.  **条件表达式的“逻辑极性”**：
    *   `WHILE` 后面跟的是 **“循环维持条件”**（为 TRUE 时继续跑，为 FALSE 时滚出去）。
    *   `REPEAT` 后面跟的是 **“终止脱离条件”**（为 FALSE 时继续跑，为 TRUE 时滚出去）。

## 2. 循环内部的中道崩殂：EXIT 与 CONTINUE

在复杂的工业遍历中，我们有时需要在循环体执行到一半时，进行紧急干预。SCL 提供了两大法宝：`EXIT` 和 `CONTINUE`。

```
                    [ 循环体开始 ]
                         │
                         ▼
                  ┌──────────────┐
                  │ 遇到 CONTINUE?├───────┐ (跳过本轮后续代码)
                  └──────┬───────┘       │
                         │ (否)          ▼
                         ▼         [ 强制进入下一轮 ]
                  ┌──────────────┐
                  │ 遇到 EXIT?   ├───────┐ (彻底毁灭循环)
                  └──────┬───────┘       │
                         │ (否)          │
                         ▼               ▼
                    [ 循环体结束 ]   [ 彻底跳出循环 ]
```

### 2.1 EXIT（彻底跳出循环）
*   **物理动作**：一旦执行到 `EXIT`，不管循环变量或条件如何，CPU 立即强制终止当前循环，毁灭循环现场，直接跳到该循环的 `END_FOR` / `END_WHILE` / `END_REPEAT` 之后的下一行代码。
*   **适用场景**：搜索算法。一旦找到了目标数据（如找到了第一个空闲插槽），就不需要再往后执行了，用 `EXIT` 节省 CPU 运行时间。

### 2.2 CONTINUE（跳过本轮，进入下轮）
*   **物理动作**：一旦执行到 `CONTINUE`，CPU 立即中断执行本轮循环中剩下的代码，**直接跳转到循环控制头部**（对于 `FOR` 会执行自增并判断，对于 `WHILE`/`REPEAT` 会进行条件评估），重新开始下一轮。
*   **适用场景**：异常数据过滤。在遍历数据时，如果发现当前通道由于传感器损坏被屏蔽了（Bypass），直接执行 `CONTINUE`，跳过该通道复杂的标定计算逻辑，直接看下一个通道。

## 3. 核心痛点：循环风险与 PLC 运行机制的深度博弈

为什么很多老资格的 PLC 工程师一听到你在程序里写了 `WHILE` 就会直摇头？
我们需要从 **西门子 CPU 的运行机制和看门狗硬件原理** 来讲透这里的危险性。

### 3.1 PLC 的实时操作系统（RTOS）与看门狗（WDT）

PLC 与我们的个人电脑（PC）不同。PC 上的操作系统（如 Windows、Linux）采用的是**分时调度系统**。如果一个后台线程陷入了死循环，顶多这个线程占满一个 CPU 核心（CPU 使用率 100%），鼠标依然能动，系统不会死机。

**但 PLC 是极其严苛的实时操作系统。**
PLC 必须保证在毫秒级内对工业现场的危险情况做出响应。
其主循环执行的微观步骤如下：

```
 ┌────────────────────────────────────────────────────────────────────────┐
 │                                 开始                                   │
 └──────────────────────────────────┬─────────────────────────────────────┘
                                    │
                                    ▼
 ┌────────────────────────────────────────────────────────────────────────┐
 │ 1. 启动硬件看门狗定时器 (WDT) -> 写入预设值 (默认 150ms)                 │
 └──────────────────────────────────┬─────────────────────────────────────┘
                                    │
                                    ▼
 ┌────────────────────────────────────────────────────────────────────────┐
 │ 2. 执行 OB1 用户程序 (你的 SCL WHILE 循环在这里运行)                    │
 └──────────────────────────────────┬─────────────────────────────────────┘
                                    │ (如果在这一步由于死循环卡住，时间超过了 150ms)
                                    ├─────────────────────────────────────┐
                                    │                                     │ [触发硬件看门狗复位]
                                    ▼                                     ▼
 ┌────────────────────────────────────────────────────────────────────────┐  ┌────────────────────────┐
 │ 3. 程序正常结束 -> 硬件复位看门狗 (清除 150ms 计时)                      │  │ CPU 瞬间被强行拉入 STOP│
 └────────────────────────────────────────────────────────────────────────┘  │ 所有 Q 点立刻断电归零  │
                                                                             └────────────────────────┘
```

为了防止由于程序逻辑陷入死循环导致整台 PLC “失去知觉”（失去对安全光幕、急停按钮的监控），PLC 硬件芯片中集成了一个 **看门狗定时器（Watchdog Timer, WDT）**。
*   每次循环开始，OS 重置 WDT 为 150ms（可在 CPU 属性里配置）。
*   如果用户程序（你的 `WHILE` 循环）由于某些原因被死死卡住，导致 150ms 内程序没有执行完毕。
*   **硬件看门狗芯片会无视任何软件逻辑，强行把 CPU 的运行状态拉入 STOP 状态！**
*   **后果**：所有输出模块（DO）失电归零，所有运行中的电机抱闸，现场所有执行机构处于无控制失电状态，极其危险。

### 3.2 制造死循环的“经典作案现场”

很多徒弟觉得“我怎么会写死循环呢？”。但在现场高强度脑力劳动下，你极易写出如下“作案现场”：

```scl
// ❌ 现场调试时极易写出的致命死循环代码
#iIndex := 0;
WHILE NOT #arrPartReady[#iIndex] DO
    // 企图寻找第一个准备好的工位
    // 致命漏洞：在循环内部，完全忘记了对 #iIndex 进行自增修改！
    // 导致 #iIndex 永远等于 0，而 arrPartReady[0] 为 FALSE。
    // CPU 就会无限循环执行这三行代码，150毫秒后，CPU 瞬间挂掉红灯暴闪停机！
    ; 
END_WHILE;
```

### 3.3 大厂安全防线：双层硬计数防御设计模式（Loop Sentry Pattern）

作为高级工程师，你的代码必须拥有**绝对不准死机**的强悍韧性。
师父今天传授你一套大厂标准的“双层硬计数防御模式”。**核心思想是：任何条件型循环，都必须人为叠加一个无法被改写的、常数限幅的“硬退出计数器”。**

```scl
//  大厂标准级防御模板
#diSentryCounter := 0; // 物理防御哨兵
#iIndex := 0;
#bFound := FALSE;

WHILE (NOT #arrPartReady[#iIndex]) AND (#diSentryCounter < 1000) DO
    // 第一层：业务条件判定 NOT arrPartReady
    // 第二层：物理哨兵保护 diSentryCounter < 1000（绝不给死循环任何机会）
    
    // 1. 业务逻辑自增
    #iIndex := #iIndex + 1;
    
    // 2. 物理防线自增
    #diSentryCounter := #diSentryCounter + 1;
    
    // 业务边界判定
    IF #iIndex > 49 THEN
        EXIT;
    END_IF;
END_WHILE;

// 哨兵诊断
IF #diSentryCounter >= 1000 THEN
    #bAlarm_ProgramFault := TRUE; // 报告程序自身逻辑 Bug，但 PLC 绝不停机！
END_IF;
```
*师父的忠告*：那个 `1000`（或者任何根据你业务估算的合理上限）就是一张**保命符**。哪怕你的业务自增逻辑写错了、数组下标越界了、或者外部通信变量永远没有返回，哨兵计数器达到 `1000` 时，也会强行把 CPU 拽出这轮循环。PLC 不会停机，你的安全逻辑依然在线，设备依然处于控制之中。

现在，我们把 `WHILE`、`REPEAT`、`EXIT`、`CONTINUE` 和双层硬计数防护融入到两个最真实、最硬核的工业场景中。

### 案例 1：辊筒输送线智能动态优先级分拣调度系统 (WHILE + CONTINUE + EXIT)

**场景描述**：
某电商物流车间有一段辊筒输送线，上面依次排开 20 个托盘位。
*   每个托盘位在 PLC 内部有一个结构体记录托盘数据（UDT_Pallet）。
*   **工艺规则**：
    1.  当新进一个打包指令时，我们必须从 **1 号到 20 号工位** 依次扫描。
    2.  如果某个工位是空置的（`bOccupied` = FALSE），或者由于被锁定（`bLocked` = TRUE），我们直接**跳过（CONTINUE）**不理它。
    3.  我们需要寻找托盘上**优先级最高（iPriority 范围 1~5，5为最高）**的托盘进行发车派送。
    4.  **安全要求**：由于托盘状态是动态由总线交互修改的，必须引入双层硬计数防护。

#### 块接口声明（FB_DynamicDispatcher）：
```
VAR_OUTPUT
    iTargetStationID : Int;     // 筛选出的最优出料工位编号
    bDispatchReady : Bool;      // 出料就绪
    bProgramBugAlarm : Bool;    // 哨兵触发安全报警
END_VAR

VAR_IN_OUT
    arrStations : Array[1..20] of "UDT_Pallet"; // 托盘工位状态结构体数组
END_VAR

VAR_TEMP
    i : Int;                    // 遍历计数
    iMaxPriorityFound : Int;    // 当前找到的最大优先级
    diSentry : DInt;            // 物理防爆哨兵
END_VAR
```

#### SCL 代码实现：

```scl
FUNCTION_BLOCK "FB_DynamicDispatcher"
{ S7_Optimized_Access := 'TRUE' }
VERSION : 0.1
BEGIN
	// ==========================================
	// 1. 变量初始化
	// ==========================================
	#iTargetStationID := -1;
	#bDispatchReady := FALSE;
	#iMaxPriorityFound := -1; // 寻找最大值，初始化为极小值
	
	#i := 1;
	#diSentry := 0;
	
	// ==========================================
	// 2. 双层硬计数安全 WHILE 循环
	// ==========================================
	WHILE (#i <= 20) AND (#diSentry < 500) DO
	    
	    // 物理防御哨兵自增
	    #diSentry := #diSentry + 1;
	    
	    // ------------------------------------------
	    // A. CONTINUE 过滤旁路逻辑
	    // ------------------------------------------
	    // 如果工位空闲，或者被锁定了（检修中），直接跳过本工位，继续看下一个
	    IF (NOT #arrStations[#i].bOccupied) OR #arrStations[#i].bLocked THEN
	        #i := #i + 1; // 必须在 CONTINUE 前手动完成业务指针自增！
	        CONTINUE;     // 立即跳回 WHILE 头部，不再执行后续的最值比较
	    END_IF;
	    
	    // ------------------------------------------
	    // B. 最值检索业务
	    // ------------------------------------------
	    IF #arrStations[#i].iPriority > #iMaxPriorityFound THEN
	        #iMaxPriorityFound := #arrStations[#i].iPriority;
	        #iTargetStationID := #i; // 记录当前最优先的工位
	    END_IF;
	    
	    // ------------------------------------------
	    // C. 快速 EXIT 触发
	    // ------------------------------------------
	    // 工艺规定：5 为绝对紧急最高优先级，一旦扫描到优先级为 5 的托盘，
	    // 没有任何商量余地，立刻提前中断后续的扫描，直接发车！
	    IF #iMaxPriorityFound = 5 THEN
	        #bDispatchReady := TRUE;
	        EXIT; // 彻底跳出整个 WHILE 循环，后续的 11..20 号工位不再遍历
	    END_IF;
	    
	    // 业务自增
	    #i := #i + 1;
	    
	END_WHILE;
	
	// ==========================================
	// 3. 哨兵防御判定与输出
	// ==========================================
	IF #diSentry >= 500 THEN
	    #bProgramBugAlarm := TRUE; // 报告死锁程序隐患
	    #iTargetStationID := -1;
	    #bDispatchReady := FALSE;
	    RETURN; // 退出执行
	END_IF;
	
	// 如果找到了合格的托盘，输出就绪
	IF #iTargetStationID <> -1 THEN
	    #bDispatchReady := TRUE;
	END_IF;
	
END_FUNCTION_BLOCK
```

### 案例 2：高精度智能流体流量计牛顿法二分逼近补偿算法 (REPEAT + 容差退出)

**场景描述**：
化工厂的电磁流量计在采集高温流体体积时，流体膨胀系数是非线性的。为了补偿精度，变送器发回了多项式系数。我们需要根据以下连续渐近的逼近算法求解高精度的补偿系数。
*   **计算逻辑**：
    我们需要不断通过迭代公式 $X_{new} = f(X_{old})$ 来收敛结果。
    当相邻两次迭代计算出来的绝对偏差 $|X_{new} - X_{old}| < 0.0001$（容差范围）时，代表曲线收敛完成，计算精度达标，此时退出。
*   **安全防爆要求**：由于部分工况下（比如传感器发生气蚀、管道空管等极端异常时），多项式公式可能发散，**导致无法收敛**。如果不做限制，`REPEAT` 会陷入死循环直至 PLC 停机。我们需要强行设定：**最大迭代次数绝不能超过 50 次**。

#### 块接口声明（FC_FlowCompensation）：
这是一个高密度的数学计算块，属于无记忆算法，我们选择 **FC**。

```
VAR_INPUT
    rRawFlow : Real;            // 未补偿的原始流量 (m3/h)
    rFluidTemp : Real;          // 实时流体温度 (°C)
END_VAR

VAR_OUTPUT
    rCompensatedFlow : Real;    // 补偿后的高精度流量 (m3/h)
    bConvergenceFail : Bool;    // 迭代发散（收敛失败）故障警报
END_VAR

VAR_TEMP
    rX_Old : Real;              // 上一次迭代值
    rX_New : Real;              // 最新一次迭代值
    iIterationsCount : Int;     // 迭代次数累加器（物理保险）
    rDelta : Real;              // 相邻两次计算的绝对差值
END_VAR
```

#### SCL 代码实现：

```scl
FUNCTION "FC_FlowCompensation" : Void
{ S7_Optimized_Access := 'TRUE' }
VERSION : 0.1
BEGIN
	// ==========================================
	// 1. 初始化物理迭代起点
	// 假设从原始流量开始逼近
	// ==========================================
	#rX_New := #rRawFlow;
	#iIterationsCount := 0;
	#bConvergenceFail := FALSE;
	
	// ==========================================
	// 2. REPEAT 迭代收敛循环
	// ==========================================
	REPEAT
	    
	    // 保存上一次的值
	    #rX_Old := #rX_New;
	    
	    // 物理保险自增
	    #iIterationsCount := #iIterationsCount + 1;
	    
	    // 模拟非线性二分逼近公式：X_new = 0.5 * (X_old + (RawFlow / (1.0 + Temp * 0.0001)))
	    // 现场实际的多项式公式会比这个复杂，但数学结构完全相同
	    #rX_New := 0.5 * (#rX_Old + (#rRawFlow / (1.0 + #rFluidTemp * 0.0001)));
	    
	    // 计算本次计算与上一次计算的绝对偏差
	    #rDelta := ABS(#rX_New - #rX_Old);
	    
	// 终止判定（出口）：
	// 1. 当两次计算差值小于 0.0001 bar 代表精度收敛达标（正常出口）
	// 2. 或者当迭代次数累计达到 50 次时强行停止（安全防御出口）
	UNTIL (#rDelta < 0.0001) OR (#iIterationsCount >= 50)
	END_REPEAT;
	
	// ==========================================
	// 3. 计算结果安全过滤与评估
	// ==========================================
	IF #iIterationsCount >= 50 AND #rDelta >= 0.0001 THEN
	    // 迭代次数见顶，但精度依然没能达标：收敛失败故障！
	    #bConvergenceFail := TRUE;
	    #rCompensatedFlow := #rRawFlow; // 降级处理：直接输出未补偿值，保证控制链不断裂
	ELSE
	    #bConvergenceFail := FALSE;
	    #rCompensatedFlow := #rX_New;   // 输出高精度计算值
	END_IF;
	
END_FUNCTION
```

## 5. 常见错误与避坑指南 (Senior Tips)

### 5.1 错误一：在 `WHILE` 中忘记更新条件变量（最经典的惨案）
这是新手写 `WHILE` 极其容易遗漏的部分：
```scl
// ❌ 导致 CPU 直接暴毙的错误示范
#i := 1;
WHILE #i <= 10 DO
    #arrResult[#i] := 0.0;
    // 致命：忘记在循环体结束前写 #i := #i + 1; 导致循环条件永远为 TRUE
END_WHILE;
```
*师父的避坑法则*：**每当你敲完 `WHILE` 的那一秒，第一件事是立刻在 `END_WHILE` 的上一行，写下条件变量的递增自增代码。** 保证条件有明确的物理流动性，再去写具体的中间业务逻辑。

### 5.2 错误二：在 `CONTINUE` 之前忘记完成指针累加
看这个在现场排故时能让人崩溃的 Bug：
```scl
// ❌ 导致现场随机陷入死循环的 Bug 代码
#i := 1;
WHILE #i <= 20 DO
    IF #arrData[#i].bBypass THEN
        CONTINUE; // 灾难：当遇到 Bypass 的通道时，i 的值没有自增，直接进入下一轮判定。
                  // 下一轮检测的依然是这个通道，导致程序从这一刻起，无限死循环！
    END_IF;
    
    #i := #i + 1; // 正常的递增被越过了
END_WHILE;
```
**避坑秘籍：只要写 `CONTINUE`，必须在它上一行，显式地执行一遍你的指针自增动作！**
```scl
    IF #arrData[#i].bBypass THEN
        #i := #i + 1; // 必须手动提前自增！
        CONTINUE; 
    END_IF;
```

### 练习 1：手写经典冒泡排序算法 (WHILE + EXIT 的性能优化)
现场配方库中有一个双整型数组：`arrAges : Array[1..10] of DInt;`
我们需要对这 10 个数据进行**从小到大**的“冒泡排序（Bubble Sort）”。
*   **工艺算法**：
    我们需要嵌套两个循环。
    *   外循环：控制扫描轮数。
    *   内循环：依次比较相邻的两个数，如果前一个数大于后一个数，进行交换。
*   **高阶优化要求**：
    在排序时，如果某一轮扫描完毕后，发现**没有发生任何一次数据交换**，代表数组其实已经提前排好序了。
    请在内循环中使用一个布尔量 `bSwapped` 进行记录。如果某一轮结束发现其为 `FALSE`，**立刻使用 `EXIT` 提前终止外循环**，为 CPU 节省宝贵的扫描周期开销。
*   **别忘了保命符**：为你的循环套上哨兵硬计数器。

### 练习 2：非标 ASCII 自由口协议特定帧尾字符位置检索器
PLC 通过串口和一台非标地磅仪表通信。PLC 的接收缓冲区为：`arrRxBuffer : Array[0..99] of Byte;`
地磅发送的字符流中，有效数据的结束标志是 **换行符 `16#0A` (LF)** 后面紧跟着一个 **回车符 `16#0D` (CR)**。
请编写一个 SCL FC，使用 `WHILE` 循环，从 `0` 号字节开始检索：
*   找出第一次出现双帧尾 `[16#0A, 16#0D]` 的起始位置索引（即 `16#0A` 所在的数组下标）。
*   **要求**：如果找到了，记录索引并用 `EXIT` 中止循环。如果检索完整条缓冲区（99个字节）都没有找到，输出“帧尾未就绪”布尔标志。
*   **要求**：如果遇到缓冲区中带有 `16#00`（空字符，可能是底层的无用噪点），直接使用 `CONTINUE` 跳过去。
